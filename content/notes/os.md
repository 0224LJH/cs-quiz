# 운영체제

이 문서는 CS 면접 준비를 위해 운영체제의 핵심 개념을 모바일에서 읽기 좋게 정리한 학습 자료입니다. 프로세스, 스케줄링, 동기화, 메모리, 가상 메모리, 캐시를 중심으로 개념과 선택 기준을 함께 다룹니다.

## 운영체제 기초
<!-- category: os-basics -->

### 운영체제의 역할
<!-- article: os-role -->

#### 한눈에 보기
- 운영체제는 하드웨어와 응용 프로그램 사이에서 자원을 관리한다.
- 핵심 기능은 프로세스 관리, 메모리 관리, 파일 시스템, 입출력 관리다.
- 커널은 운영체제의 핵심이며, 민감한 자원 접근을 통제한다.

#### 핵심 정리

운영체제는 CPU, 메모리, 디스크, 네트워크 장치처럼 한정된 자원을 여러 프로그램이 안전하게 나누어 쓰도록 관리한다. 사용자는 프로그램을 실행한다고 느끼지만, 실제로는 운영체제가 CPU 시간을 나누고 메모리를 격리하며 입출력 장치를 대신 제어한다.

운영체제가 없다면 응용 프로그램이 하드웨어를 직접 다뤄야 한다. 그러면 프로그램마다 디스크 읽기, 네트워크 송수신, 메모리 보호를 직접 구현해야 하고, 잘못된 프로그램 하나가 전체 시스템을 망가뜨릴 수 있다.

#### 면접 답변

> 운영체제는 하드웨어 자원을 효율적이고 안전하게 관리하고, 응용 프로그램에 실행 환경과 공통 서비스를 제공하는 시스템 소프트웨어입니다. 대표적으로 프로세스, 메모리, 파일 시스템, 입출력을 관리합니다.

#### 헷갈리는 포인트
- 운영체제 전체와 커널은 같은 말이 아니다. 커널은 운영체제의 핵심 부분이다.
- GUI나 CUI는 사용자 인터페이스이고, 자원 관리는 커널 중심으로 수행된다.

#### 관련 퀴즈
- 아직 직접 연결할 퀴즈 없음

### 커널, 사용자 모드, 커널 모드
<!-- article: os-kernel-user-mode -->

#### 한눈에 보기
- 사용자 모드는 일반 프로그램이 실행되는 제한된 권한의 모드다.
- 커널 모드는 운영체제 커널이 실행되는 높은 권한의 모드다.
- 두 모드를 나누는 이유는 보안과 안정성 때문이다.

#### 핵심 정리

사용자 프로그램이 모든 메모리와 장치를 직접 제어할 수 있다면, 버그나 악성 코드가 시스템 전체를 손상시킬 수 있다. 그래서 CPU는 권한 수준을 나누고, 일반 프로그램은 사용자 모드에서 제한된 명령만 실행하게 한다.

디스크 읽기, 프로세스 생성, 네트워크 송수신처럼 커널 권한이 필요한 작업은 시스템 콜을 통해 요청한다. 이때 CPU는 커널 모드로 전환하고, 커널은 요청이 안전한지 확인한 뒤 서비스를 수행한다.

커널 스택과 사용자 스택도 분리된다. 사용자 스택은 일반 함수 호출에 쓰이고, 커널 스택은 시스템 콜이나 인터럽트 처리 중 커널의 실행 상태를 보관한다.

#### 면접 답변

> 사용자 모드는 응용 프로그램이 제한된 권한으로 실행되는 모드이고, 커널 모드는 운영체제가 모든 자원에 접근할 수 있는 모드입니다. 이 분리는 프로그램 오류가 전체 시스템으로 번지는 것을 막고, 하드웨어 접근을 커널이 통제하게 합니다.

#### 헷갈리는 포인트
- 시스템 콜은 커널 기능을 요청하는 인터페이스다.
- 모드 전환은 비용이 있으므로 잦은 시스템 콜은 성능 병목이 될 수 있다.

#### 관련 퀴즈
- `os-c039`
- `os-o005`
- `os-o006`

### 시스템 콜과 인터럽트
<!-- article: os-system-call-interrupt -->

#### 한눈에 보기
- 시스템 콜은 유저 프로그램이 커널 서비스를 요청하는 공식 통로다.
- 트랩 또는 소프트웨어 인터럽트는 커널로 진입하는 메커니즘이다.
- 하드웨어 인터럽트는 장치나 타이머처럼 CPU 외부에서 발생한다.

#### 핵심 정리

시스템 콜은 `read`, `write`, `fork`처럼 운영체제 서비스를 요청하는 인터페이스다. 사용자가 파일을 읽으려면 라이브러리 함수를 호출하고, 내부적으로 시스템 콜 번호와 인자를 정한 뒤 특수 명령으로 커널에 진입한다.

인터럽트는 CPU가 하던 일을 잠시 멈추고 중요한 이벤트를 처리하게 하는 장치다. 키보드 입력이나 디스크 I/O 완료는 하드웨어 인터럽트이고, 시스템 콜이나 0으로 나누기 같은 예외는 소프트웨어적으로 발생하는 동기 이벤트다.

시스템 콜은 의도적인 요청이고, 예외는 프로그램 실행 중 문제가 생겨 발생할 수 있다는 점이 다르다. 둘 다 커널 코드가 개입하지만 목적과 발생 원인이 다르다.

#### 동작 과정

```text
사용자 코드 -> 시스템 콜 요청 -> 트랩 발생 -> 커널 모드 전환
-> 커널 핸들러 실행 -> 결과 저장 -> 사용자 모드 복귀
```

#### 면접 답변

> 시스템 콜은 사용자 프로그램이 커널 서비스를 요청하는 API이고, 트랩 또는 소프트웨어 인터럽트는 그 요청을 처리하기 위해 커널 모드로 진입하는 방식입니다. 하드웨어 인터럽트는 외부 장치가 CPU에 처리를 요청하는 비동기 이벤트입니다.

#### 관련 퀴즈
- `os-c010`
- `os-d010`
- `os-o005`
- `os-o006`
- `os-s010`

### 입출력, DMA, 폴링
<!-- article: os-io-dma-polling -->

#### 한눈에 보기
- 인터럽트는 I/O 완료를 CPU에 알려 폴링 낭비를 줄인다.
- DMA는 CPU 대신 장치와 메모리 사이의 대용량 전송을 맡는다.
- 매우 빠른 장치에서는 폴링이 인터럽트보다 나을 때도 있다.

#### 핵심 정리

I/O 장치는 CPU보다 느린 경우가 많다. CPU가 장치 상태를 계속 확인하면 계산에 쓸 시간을 낭비한다. 인터럽트를 사용하면 CPU는 다른 작업을 하다가 장치가 완료 신호를 보낼 때만 처리하면 된다.

DMA(Direct Memory Access)는 디바이스 컨트롤러가 메모리에 직접 데이터를 읽고 쓰도록 하는 방식이다. CPU가 매 바이트를 복사하지 않아도 되므로 대용량 디스크나 네트워크 I/O에서 CPU 부담을 줄인다.

다만 고성능 네트워크나 NVMe처럼 응답이 매우 빠른 장치에서는 인터럽트 처리와 컨텍스트 전환 비용이 더 클 수 있다. 이때는 짧게 반복 확인하는 폴링이 지연 시간을 줄이는 선택이 될 수 있다.

#### 면접 답변

> DMA는 입출력 장치가 CPU 개입 없이 메모리와 직접 데이터를 주고받게 해 CPU 부담을 줄이는 방식입니다. 인터럽트는 완료 시점에만 CPU를 깨우지만, 매우 빠른 장치에서는 폴링이 더 효율적인 경우도 있습니다.

#### 관련 퀴즈
- `os-c040`
- `os-t010`

### 커널 구조와 파일 시스템 저널링
<!-- article: os-kernel-filesystem-journaling -->

#### 한눈에 보기
- 모놀리식 커널은 성능이 좋지만 장애 격리에는 약하다.
- 마이크로커널은 안정성과 격리에 유리하지만 IPC 비용이 커질 수 있다.
- 저널링 파일 시스템은 장애 이후 복구 가능성을 높인다.

#### 핵심 정리

모놀리식 커널은 파일 시스템, 네트워크, 장치 드라이버 같은 많은 기능을 커널 공간에서 실행한다. 함수 호출로 빠르게 협력할 수 있어 성능에 유리하지만, 커널 내부 버그 하나가 시스템 전체에 영향을 줄 수 있다.

마이크로커널은 최소 기능만 커널에 두고 나머지 서비스를 사용자 공간 프로세스로 분리한다. 서비스 장애가 전체 커널을 무너뜨릴 가능성은 낮아지지만, 서비스 간 통신이 IPC를 거치므로 오버헤드가 생긴다.

파일 시스템 저널링은 변경 내용을 먼저 기록해 두고 실제 반영을 진행한다. ordered 모드는 데이터 기록 순서를 더 보장해 안전성이 높고, writeback 모드는 순서 보장이 약해 빠르지만 장애 시 파일 내용 손상 위험이 커진다.

#### 면접 답변

> 모놀리식 커널은 커널 공간에서 많은 기능이 직접 통신해 빠르지만 장애 격리가 약하고, 마이크로커널은 서비스를 분리해 안정성은 높지만 IPC 오버헤드가 있습니다. 파일 시스템 저널링은 장애 복구를 위해 변경 이력을 남기는 기법입니다.

#### 관련 퀴즈
- `os-t013`
- `os-t015`

## 프로세스와 스레드
<!-- category: os-process-thread -->

### 프로세스, PCB, 상태 전이
<!-- article: os-process-pcb-state -->

#### 한눈에 보기
- 프로세스는 실행 중인 프로그램이다.
- PCB는 운영체제가 프로세스를 관리하기 위해 보관하는 제어 정보다.
- 프로세스는 생성, 준비, 실행, 대기, 종료 상태를 오간다.

#### 핵심 정리

프로세스는 운영체제로부터 독립된 주소 공간과 자원을 할당받아 실행되는 프로그램이다. 각 프로세스는 코드, 데이터, 힙, 스택 영역을 가지며, 운영체제는 프로세스마다 관리 정보를 따로 유지한다.

PCB(Process Control Block)에는 PID, 프로세스 상태, 프로그램 카운터, CPU 레지스터, 스케줄링 정보, 메모리 관리 정보, 열린 파일 정보 등이 들어간다. 하드디스크의 실제 블록 배치처럼 파일 시스템이 관리하는 정보는 PCB의 핵심 정보가 아니다.

프로세스가 I/O를 요청하면 실행 상태에서 대기 상태로 이동한다. 타임 퀀텀이 끝나거나 더 높은 우선순위 프로세스가 오면 실행 상태에서 준비 상태로 밀릴 수 있다.

#### 면접 답변

> 프로세스는 실행 중인 프로그램이고, PCB는 운영체제가 해당 프로세스의 상태를 저장하고 복원하기 위한 자료구조입니다. PCB에는 PID, 상태, PC, 레지스터, 스케줄링 정보, 메모리 관리 정보 등이 저장됩니다.

#### 관련 퀴즈
- `os-c002`
- `os-c004`
- `os-o001`

### 컨텍스트 스위칭
<!-- article: os-context-switching -->

#### 한눈에 보기
- 컨텍스트 스위칭은 실행 주체를 바꾸기 위해 상태를 저장하고 복원하는 과정이다.
- 프로세스 전환은 PCB 저장과 로드가 핵심이다.
- 너무 자주 발생하면 CPU가 실제 작업보다 전환에 시간을 쓴다.

#### 핵심 정리

컨텍스트는 프로세스나 스레드가 다시 실행되기 위해 필요한 상태 정보다. CPU 레지스터, 프로그램 카운터, 스택 포인터 같은 값이 포함된다. 운영체제는 현재 실행 중인 작업의 컨텍스트를 저장하고 다음 작업의 컨텍스트를 복원한다.

이 과정은 필요한 기능이지만 순수한 오버헤드다. 전환 중 CPU는 사용자 요청을 직접 처리하지 못하고, 캐시 적중률도 떨어질 수 있다. 따라서 타임 퀀텀이 너무 짧거나 락 경쟁이 심하면 컨텍스트 스위칭 비용이 성능을 크게 깎는다.

스레드 간 전환은 같은 주소 공간을 공유하므로 프로세스 전환보다 보통 가볍다. 하지만 스레드도 PC, 레지스터, 스택은 독립적으로 복원해야 하므로 비용이 0은 아니다.

#### 면접 답변

> 컨텍스트 스위칭은 현재 실행 중인 프로세스나 스레드의 CPU 상태를 저장하고, 다음 실행 대상의 상태를 복원하는 과정입니다. 필수 기능이지만 유용한 계산을 하지 않는 오버헤드이므로 지나치게 잦으면 성능이 저하됩니다.

#### 관련 퀴즈
- `os-c003`
- `os-o002`

### 프로세스와 스레드의 차이
<!-- article: os-process-thread-difference -->

#### 한눈에 보기
- 프로세스는 독립된 주소 공간을 가진다.
- 같은 프로세스 안의 스레드는 코드, 데이터, 힙을 공유한다.
- 스레드는 가볍지만 동기화 문제가 더 쉽게 발생한다.

#### 핵심 정리

프로세스는 서로 독립된 메모리 공간을 가진다. 한 프로세스가 죽어도 다른 프로세스가 바로 같은 주소 공간을 잃지는 않으므로 장애 격리에 유리하다. 대신 프로세스 간 데이터 교환에는 IPC가 필요하고 생성과 전환 비용이 크다.

스레드는 한 프로세스 안의 실행 흐름이다. 코드, 데이터, 힙 영역을 공유하므로 데이터 교환이 빠르고 메모리 사용량이 적다. 하지만 공유 자원을 동시에 수정하면 경쟁 조건이 발생할 수 있어 락, 세마포어, 모니터 같은 동기화가 필요하다.

선택 기준은 격리성과 효율성이다. 작업 간 독립성과 안정성이 중요하면 멀티프로세스가 낫고, 데이터 공유가 빈번하고 성능이 중요하면 멀티스레드가 적합하다.

#### 면접 답변

> 프로세스는 독립된 주소 공간을 가진 실행 단위이고, 스레드는 프로세스 내부의 실행 흐름입니다. 같은 프로세스의 스레드들은 코드, 데이터, 힙을 공유하지만 스택과 레지스터 상태는 독립적으로 가집니다.

#### 관련 퀴즈
- `os-c001`
- `os-t001`
- `os-d005`

### 스레드의 스택과 PC 레지스터
<!-- article: os-thread-stack-pc -->

#### 한눈에 보기
- 각 스레드는 독립적인 스택을 가져야 독립적으로 함수 호출을 할 수 있다.
- PC 레지스터는 다음에 실행할 명령어 위치를 기억한다.
- 커널 스택은 시스템 콜과 인터럽트 처리 중 커널 실행 상태를 저장한다.

#### 핵심 정리

스택은 함수 호출의 지역 변수, 매개변수, 복귀 주소를 담는다. 스레드마다 스택이 독립적이지 않다면 서로 다른 함수 호출 흐름이 충돌한다. 그래서 같은 힙을 공유하더라도 스택은 스레드별로 분리한다.

PC 레지스터는 스레드가 어디까지 실행했는지 나타낸다. 스케줄러가 스레드를 중단했다가 다시 실행하려면, 이전 명령어 위치를 정확히 복원해야 한다. 따라서 PC와 레지스터 집합도 스레드별 상태로 관리된다.

시스템 콜이나 인터럽트로 커널에 들어가면 커널 스택이 사용된다. 사용자 스택을 그대로 신뢰하지 않고 별도 스택을 쓰는 것은 보안과 안정성 때문이다.

#### 면접 답변

> 스레드마다 스택이 필요한 이유는 각 스레드가 독립적인 함수 호출 흐름을 가져야 하기 때문입니다. PC 레지스터도 스케줄링으로 중단된 뒤 어디서부터 이어 실행할지 기억하기 위해 스레드마다 독립적으로 관리됩니다.

#### 관련 퀴즈
- `os-c021`
- `os-c022`
- `os-c039`

### 사용자 수준 스레드와 커널 수준 스레드
<!-- article: os-user-kernel-thread -->

#### 한눈에 보기
- 사용자 수준 스레드는 커널 개입 없이 전환할 수 있어 가볍다.
- 커널 수준 스레드는 커널이 스케줄링하므로 멀티코어 활용과 블로킹 처리에 유리하다.
- 사용자 수준 스레드는 하나가 블로킹되면 전체 프로세스가 막힐 수 있다.

#### 핵심 정리

사용자 수준 스레드는 런타임 라이브러리가 스레드 생성과 전환을 관리한다. 커널 모드 전환이 적어 전환 비용이 낮고, 애플리케이션 특성에 맞춘 스케줄링도 가능하다.

하지만 커널이 내부 스레드들을 모르면 하나의 스레드가 블로킹 시스템 콜을 호출했을 때 프로세스 전체가 멈출 수 있다. 또한 커널이 여러 스레드를 여러 코어에 직접 배치하기 어렵다.

커널 수준 스레드는 운영체제가 각 스레드를 인식하고 스케줄링한다. 오버헤드는 상대적으로 크지만, 한 스레드가 블로킹되어도 다른 스레드를 실행할 수 있고 멀티코어 병렬성에도 유리하다.

#### 면접 답변

> 사용자 수준 스레드는 커널 개입 없이 빠르게 전환할 수 있지만 블로킹과 멀티코어 활용에 약합니다. 커널 수준 스레드는 비용은 더 들지만 커널이 직접 스케줄링하므로 병렬 실행과 블로킹 처리에 유리합니다.

#### 관련 퀴즈
- `os-t008`

## 스케줄링
<!-- category: os-scheduling -->

### 장기, 중기, 단기 스케줄러
<!-- article: os-scheduler-types -->

#### 한눈에 보기
- 장기 스케줄러는 어떤 작업을 시스템에 들일지 결정한다.
- 중기 스케줄러는 메모리 압박 시 프로세스를 swap out, swap in 한다.
- 단기 스케줄러는 다음에 CPU를 사용할 ready 프로세스를 고른다.

#### 핵심 정리

스케줄러는 프로세스를 어느 단계에 둘지 결정한다. 장기 스케줄러는 디스크나 작업 큐에 있는 작업 중 어떤 것을 메모리에 올릴지 선택해 다중 프로그래밍 정도를 조절한다.

중기 스케줄러는 메모리가 부족할 때 일부 프로세스를 디스크로 내보내고, 나중에 다시 메모리로 불러온다. 이 과정이 스와핑이다. 단기 스케줄러는 ready queue에서 지금 CPU를 받을 프로세스를 선택한다.

면접에서는 호출 빈도를 함께 말하면 좋다. 단기 스케줄러는 타이머 인터럽트, I/O 대기, 종료 등으로 매우 자주 동작하고, 장기 스케줄러는 상대적으로 드물게 동작한다.

#### 면접 답변

> 장기 스케줄러는 시스템에 들어올 작업을 고르고, 중기 스케줄러는 메모리 압박에 따라 프로세스를 swap in/out 하며, 단기 스케줄러는 ready queue에서 다음 CPU 실행 대상을 고릅니다.

#### 관련 퀴즈
- `os-c023`

### 선점형과 비선점형 스케줄링
<!-- article: os-preemptive-nonpreemptive -->

#### 한눈에 보기
- 선점형은 운영체제가 실행 중인 프로세스의 CPU를 빼앗을 수 있다.
- 비선점형은 프로세스가 자발적으로 CPU를 반납할 때까지 기다린다.
- 응답 시간이 중요한 시스템은 보통 선점형이 유리하다.

#### 핵심 정리

비선점형 스케줄링은 구현이 단순하고 컨텍스트 스위칭이 상대적으로 적다. 하지만 긴 작업이 CPU를 오래 잡으면 짧고 중요한 작업이 기다려야 한다.

선점형 스케줄링은 타이머 인터럽트나 우선순위 변화에 따라 현재 실행 중인 프로세스를 중단하고 다른 프로세스를 실행할 수 있다. 대화형 시스템처럼 응답 시간이 중요한 환경에 적합하지만, 전환 오버헤드와 동기화 복잡성이 증가한다.

실시간 운영체제에서는 예측 가능성이 특히 중요하다. RMS 같은 고정 우선순위 스케줄링은 분석이 쉬워 하드 실시간 임베디드 시스템에서 자주 사용된다.

#### 면접 답변

> 선점형 스케줄링은 운영체제가 실행 중인 프로세스를 강제로 중단하고 다른 프로세스에 CPU를 줄 수 있는 방식입니다. 비선점형은 실행 중인 프로세스가 종료하거나 대기할 때까지 기다리므로 단순하지만 응답성이 떨어질 수 있습니다.

#### 관련 퀴즈
- `os-c011`
- `os-t014`

### FCFS, SJF, 우선순위 스케줄링
<!-- article: os-fcfs-sjf-priority -->

#### 한눈에 보기
- FCFS는 먼저 온 작업을 먼저 실행하는 비선점형 방식이다.
- SJF는 실행 시간이 짧은 작업을 우선해 평균 대기 시간을 줄인다.
- 우선순위 스케줄링은 중요도 차이가 있는 작업에 적합하다.

#### 핵심 정리

FCFS는 단순하지만 긴 작업이 앞에 있으면 뒤의 짧은 작업들이 오래 기다리는 호위 효과가 발생한다. 처리 순서가 중요하거나 작업 시간이 비슷하면 괜찮지만, 대화형 시스템에는 불리할 수 있다.

SJF는 CPU burst가 짧은 작업을 먼저 실행해 평균 대기 시간을 줄인다. 이론적으로 매력적이지만 실제 burst 시간을 정확히 알기 어렵고, 짧은 작업이 계속 들어오면 긴 작업이 굶는 기아 현상이 생길 수 있다.

우선순위 스케줄링은 대화형 요청과 배치 작업처럼 중요도가 다른 작업이 섞인 환경에 적합하다. 낮은 우선순위 작업의 기아는 대기 시간이 길수록 우선순위를 올리는 aging으로 완화한다.

#### 면접 답변

> FCFS는 단순하지만 호위 효과가 있고, SJF는 평균 대기 시간을 줄이지만 실행 시간 예측과 기아 문제가 있습니다. 우선순위 스케줄링은 중요도 차이가 있는 작업에 적합하며 기아 문제는 aging으로 완화합니다.

#### 관련 퀴즈
- `os-c005`
- `os-c024`
- `os-t002`
- `os-t004`
- `os-t017`

### Round Robin과 다단계 피드백 큐
<!-- article: os-round-robin-mlfq -->

#### 한눈에 보기
- Round Robin은 모든 프로세스에 일정한 타임 퀀텀을 부여한다.
- 타임 퀀텀이 너무 작으면 컨텍스트 스위칭 비용이 커진다.
- 다단계 피드백 큐는 실행 특성에 따라 큐 사이 이동이 가능하다.

#### 핵심 정리

Round Robin은 ready queue의 프로세스를 순서대로 실행하고, 정해진 시간이 지나면 다음 프로세스로 넘긴다. 공정하고 응답성이 좋아 대화형 작업에 적합하다.

핵심은 타임 퀀텀 크기다. 너무 작으면 전환이 너무 잦아지고, 너무 크면 FCFS에 가까워져 응답성이 나빠진다. 작업 대부분이 한 퀀텀 안에 끝날 정도가 실용적인 기준으로 자주 언급된다.

다단계 큐는 작업 종류별로 큐를 나누고 프로세스를 고정한다. 다단계 피드백 큐는 CPU를 오래 쓰는 작업을 낮은 우선순위로 내리거나 오래 기다린 작업을 올리는 식으로 동적으로 조정한다.

#### 면접 답변

> Round Robin은 각 프로세스에 같은 타임 퀀텀을 주는 선점형 방식입니다. 퀀텀이 너무 작으면 전환 오버헤드가 커지고, 너무 크면 FCFS처럼 동작합니다. 다단계 피드백 큐는 프로세스 특성에 따라 큐 이동이 가능해 더 유연합니다.

#### 관련 퀴즈
- `os-c019`
- `os-c025`
- `os-d002`

### 동기와 비동기 처리
<!-- article: os-sync-async -->

#### 한눈에 보기
- 동기는 요청한 작업이 끝날 때까지 흐름이 이어지지 않는 방식이다.
- 비동기는 요청 후 다른 작업을 계속하다가 결과를 나중에 받는다.
- 응답성, 복잡도, 자원 사용량을 함께 보고 선택한다.

#### 핵심 정리

동기 처리는 흐름이 단순하다. 호출한 작업이 끝난 뒤 다음 코드를 실행하므로 이해와 디버깅이 쉽다. 대신 느린 I/O를 기다리는 동안 스레드가 막히면 처리량이 떨어질 수 있다.

비동기 처리는 I/O 요청을 보낸 뒤 다른 작업을 계속할 수 있어 높은 동시성과 응답성을 만들기 좋다. 그러나 콜백, 이벤트 루프, 완료 큐, 상태 관리가 들어가면 코드 복잡도가 증가한다.

운영체제 관점에서는 블로킹 I/O, 논블로킹 I/O, 이벤트 기반 I/O, 비동기 I/O가 모두 이 선택과 연결된다. 중요한 것은 작업의 지연 시간, 동시 접속 수, 구현 복잡도를 같이 보는 것이다.

#### 면접 답변

> 동기는 요청 결과가 나올 때까지 흐름이 기다리는 방식이고, 비동기는 요청 후 다른 일을 하다가 완료 시점에 결과를 처리하는 방식입니다. 단순성이 중요하면 동기, 높은 동시성과 응답성이 중요하면 비동기를 선택합니다.

#### 관련 퀴즈
- `os-t016`
- `os-s010`

## 동기화와 교착상태
<!-- category: os-synchronization-deadlock -->

### 경쟁 조건과 임계 영역
<!-- article: os-race-critical-section -->

#### 한눈에 보기
- 경쟁 조건은 실행 순서에 따라 결과가 달라지는 상황이다.
- 임계 영역은 공유 자원을 읽고 쓰는 민감한 코드 구간이다.
- 해결 조건은 상호 배제, 진행, 한정 대기다.

#### 핵심 정리

여러 스레드가 같은 변수를 동시에 수정하면 결과가 실행 순서에 따라 달라질 수 있다. 예를 들어 잔액 확인과 출금이 원자적으로 묶이지 않으면 두 스레드가 같은 잔액을 보고 모두 출금해 음수가 될 수 있다.

임계 영역 문제는 공유 자원 접근을 안전하게 만드는 문제다. 한 번에 하나만 들어가야 하는 상호 배제, 들어갈 수 있는 스레드가 있을 때 결정이 무한히 미뤄지지 않는 진행, 특정 스레드가 계속 밀리지 않는 한정 대기가 필요하다.

단순히 읽기만 하는 공유 데이터는 문제가 덜하지만, 쓰기가 포함되면 동기화가 필요하다. 락 범위는 작게 잡되, 보호해야 할 불변식은 빠뜨리지 않아야 한다.

#### 면접 답변

> 경쟁 조건은 여러 스레드가 동기화 없이 공유 자원을 읽고 쓸 때 실행 순서에 따라 결과가 달라지는 문제입니다. 임계 영역 해결에는 상호 배제, 진행, 한정 대기 조건이 필요합니다.

#### 관련 퀴즈
- `os-c016`
- `os-c026`
- `os-s002`

### 뮤텍스, 세마포어, 모니터
<!-- article: os-mutex-semaphore-monitor -->

#### 한눈에 보기
- 뮤텍스는 소유권이 있는 잠금 도구다.
- 세마포어는 카운터 기반으로 자원 개수를 제어할 수 있다.
- 모니터는 공유 자원과 동기화 코드를 캡슐화한 고수준 도구다.

#### 핵심 정리

뮤텍스는 한 스레드가 락을 획득하면 같은 스레드가 해제해야 한다. 소유권이 명확하므로 단일 임계 영역 보호에 자주 쓰인다.

세마포어는 `wait`와 `signal` 연산으로 카운터를 조작한다. 바이너리 세마포어는 0과 1만 사용해 단일 자원 보호에 쓸 수 있고, 카운팅 세마포어는 연결 풀이나 버퍼 슬롯처럼 여러 개의 동일 자원을 제어하는 데 적합하다.

모니터는 공유 자원과 접근 함수를 하나의 구조로 묶고 내부에 한 번에 하나의 스레드만 들어오게 한다. 조건 변수와 함께 쓰면 생산자-소비자 같은 문제를 더 구조적으로 표현할 수 있다.

#### 면접 답변

> 뮤텍스는 소유권이 있는 락이라 획득한 스레드가 해제해야 하고, 세마포어는 카운터 기반이라 여러 자원 개수를 제어할 수 있습니다. 모니터는 공유 자원과 동기화 로직을 캡슐화한 고수준 동기화 도구입니다.

### Busy Waiting과 스핀락
<!-- article: os-busy-waiting-spinlock -->

#### 한눈에 보기
- Busy Waiting은 조건이 만족될 때까지 CPU를 쓰며 반복 확인하는 방식이다.
- 스핀락은 락을 얻을 때까지 바쁘게 기다리는 락이다.
- 대기 시간이 매우 짧고 멀티코어인 경우에만 신중히 쓴다.

#### 핵심 정리

Busy Waiting은 스레드를 잠재우지 않고 루프를 돌며 상태를 확인한다. 대기 중에도 CPU를 계속 쓰므로 긴 대기에는 비효율적이다. 전력도 더 쓰고, 다른 작업이 CPU를 쓸 기회도 줄어든다.

스핀락은 짧은 임계 영역에서 컨텍스트 스위칭 비용을 피하려는 선택이다. 락이 아주 빨리 풀린다면 스레드를 재우고 깨우는 비용보다 잠깐 도는 비용이 작을 수 있다.

하지만 단일 코어에서는 스핀 중인 스레드가 CPU를 잡고 있으면 락을 해제할 스레드가 실행되기 어렵다. 많은 스레드가 하나의 락을 경쟁하면 직렬화 병목이 생기므로 락 분할, lock-free 구조, 스레드별 집계 후 병합 같은 접근을 고려한다.

#### 면접 답변

> Busy Waiting은 조건이 만족될 때까지 CPU를 점유하며 반복 확인하는 방식입니다. 스핀락은 짧은 대기에서는 컨텍스트 스위칭 비용을 줄일 수 있지만, 대기가 길거나 단일 코어이면 CPU 낭비가 커집니다.

#### 관련 퀴즈
- `os-c018`
- `os-c028`
- `os-s008`

### 교착상태의 발생 조건
<!-- article: os-deadlock-conditions -->

#### 한눈에 보기
- 교착상태는 서로가 가진 자원을 기다리며 모두 멈춘 상태다.
- 필요 조건은 상호 배제, 점유 대기, 비선점, 환형 대기다.
- 네 조건 중 하나라도 깨면 교착상태를 예방할 수 있다.

#### 핵심 정리

교착상태는 여러 프로세스나 스레드가 서로의 자원을 기다려 누구도 진행하지 못하는 상태다. 웹 서버가 CPU를 거의 쓰지 않고 응답도 없는데 프로세스가 살아 있다면 I/O 대기, 락 대기, 교착상태 같은 blocked 상태를 의심할 수 있다.

상호 배제는 자원을 한 번에 하나만 쓸 수 있다는 조건이다. 점유 대기는 자원을 가진 채 다른 자원을 기다리는 상황이다. 비선점은 운영체제가 자원을 강제로 빼앗을 수 없다는 뜻이고, 환형 대기는 기다림 관계가 원형으로 이어지는 상황이다.

이 조건들은 필요 조건이다. 네 가지가 모두 성립해야 교착상태가 발생할 수 있으므로 예방 기법은 보통 이 중 하나를 깨는 방향으로 설계된다.

#### 면접 답변

> 교착상태는 여러 프로세스가 서로 가진 자원을 기다리며 진행하지 못하는 상태입니다. 발생 필요 조건은 상호 배제, 점유 대기, 비선점, 환형 대기입니다.

#### 관련 퀴즈
- `os-c007`
- `os-s001`
- `os-s004`

### 교착상태 처리 전략
<!-- article: os-deadlock-handling -->

#### 한눈에 보기
- 예방은 교착상태 조건 중 하나를 깨는 방식이다.
- 회피는 안전 상태를 유지하도록 자원 할당을 조절한다.
- 탐지는 발생 여부를 찾고, 복구는 프로세스 종료나 자원 회수로 해결한다.

#### 핵심 정리

교착상태 예방은 설계 단계에서 점유 대기나 환형 대기 같은 조건이 성립하지 않게 만든다. 예를 들어 자원에 전순서를 부여해 항상 같은 순서로 요청하게 하면 환형 대기를 줄일 수 있다.

회피는 자원 요청을 받을 때 승인해도 안전 상태가 유지되는지 확인한다. 은행원 알고리즘의 안전 상태는 모든 프로세스가 어떤 순서로든 필요한 자원을 받아 끝날 수 있는 안전 순서가 존재하는 상태다.

탐지는 자원 할당 그래프의 사이클 등을 확인해 교착상태를 찾는다. 일반 목적 운영체제는 교착상태가 드물다면 예방과 회피 비용이 더 크다고 보고 무시하다가 문제가 생기면 종료나 재시작으로 처리하기도 한다.

#### 면접 답변

> 교착상태 처리는 예방, 회피, 탐지와 복구, 무시 전략으로 나뉩니다. 은행원 알고리즘은 회피 기법이고, 자원 할당 그래프의 사이클 탐지는 탐지 기법입니다.

#### 관련 퀴즈

### 생산자-소비자 문제와 조건 알림
<!-- article: os-producer-consumer-signal -->

#### 한눈에 보기
- 생산자-소비자 문제는 버퍼 상태에 따라 스레드가 기다리고 깨는 문제다.
- 빈 버퍼에서 소비자는 기다려야 하고, 생산자는 아이템 추가 후 소비자를 깨워야 한다.
- signal 누락은 정상 코드처럼 보여도 영원한 대기를 만들 수 있다.

#### 핵심 정리

생산자-소비자 문제는 동기화의 대표 예시다. 생산자는 버퍼가 가득 차면 기다리고, 소비자는 버퍼가 비면 기다린다. 버퍼 상태 변경과 대기 큐 알림이 올바르게 묶여야 한다.

생산자가 아이템을 넣었는데 소비자를 깨우는 `signal`을 빠뜨리면, 버퍼에 처리할 일이 생겼는데도 소비자가 계속 잠들 수 있다. 반대로 조건 확인 없이 깨우기만 믿으면 잘못된 실행 순서에서 빈 버퍼를 읽을 수 있다.

실무에서는 조건 변수나 세마포어를 쓸 때 "상태 변경", "조건 확인", "대기와 알림"의 순서를 주의해야 한다. 특히 조건은 보통 `if`보다 반복 확인이 안전하다.

#### 면접 답변

> 생산자-소비자 문제에서는 버퍼 상태와 대기 알림을 함께 관리해야 합니다. 생산자가 데이터를 넣은 뒤 소비자를 깨우는 signal을 누락하면 소비자는 조건이 만족됐는데도 계속 대기할 수 있습니다.

#### 관련 퀴즈
- `os-s006`

## IPC와 시스템 콜
<!-- category: os-ipc-system-call -->

### IPC가 필요한 이유
<!-- article: os-ipc-overview -->

#### 한눈에 보기
- 프로세스는 주소 공간이 분리되어 직접 데이터를 공유하기 어렵다.
- IPC는 프로세스 간 데이터 교환과 협력을 위한 방법이다.
- 공유 메모리, 파이프, 메시지 큐, 소켓, 파일 등이 대표적이다.

#### 핵심 정리

프로세스는 서로의 메모리에 직접 접근하지 못한다. 이 격리는 안정성과 보안에는 좋지만, 여러 프로세스가 협력해야 할 때는 별도의 통신 수단이 필요하다. 이를 IPC(Inter-Process Communication)라고 한다.

IPC를 고를 때는 데이터 크기, 통신 빈도, 동기화 복잡도, 같은 머신인지 네트워크 너머인지가 중요하다. 공유 메모리는 빠르지만 동기화를 직접 해야 하고, 메시지 기반 방식은 구조가 명확하지만 복사와 커널 개입 비용이 생길 수 있다.

#### 면접 답변

> IPC는 독립된 주소 공간을 가진 프로세스들이 데이터를 주고받기 위한 통신 방법입니다. 공유 메모리, 파이프, 메시지 큐, 소켓 등이 있으며, 성능과 동기화 복잡도에 따라 선택합니다.

#### 관련 퀴즈
- `os-d004`

### 공유 메모리와 메시지 패싱
<!-- article: os-shared-memory-message-passing -->

#### 한눈에 보기
- 공유 메모리는 여러 프로세스가 같은 메모리 영역을 매핑해 사용한다.
- 메시지 패싱은 send/receive로 데이터를 주고받는다.
- 대용량 고빈도 통신은 공유 메모리, 명확한 경계와 분산 통신은 메시지 패싱이 유리하다.

#### 핵심 정리

공유 메모리는 한 번 메모리 영역을 공유하도록 설정하면 이후 데이터 교환이 빠르다. 커널을 매번 거쳐 복사하지 않아도 되기 때문이다. 대신 여러 프로세스가 동시에 읽고 쓰므로 뮤텍스나 세마포어 같은 동기화가 필요하다.

메시지 패싱은 데이터를 메시지 단위로 주고받는다. 메시지 경계가 명확하고 send/receive 흐름이 자연스럽게 동기화 역할을 하므로 설계가 단순해질 수 있다. 또한 네트워크 통신으로 확장하기 좋다.

#### 면접 답변

> 공유 메모리는 같은 메모리 영역을 매핑해 빠르게 통신하지만 동기화를 직접 해야 합니다. 메시지 패싱은 명시적인 송수신으로 구조가 분명하고 네트워크 통신에도 적합하지만 복사와 커널 개입 비용이 생길 수 있습니다.

#### 관련 퀴즈
- `os-t009`
- `os-d004`

### 파이프, 메시지 큐, 소켓
<!-- article: os-pipe-message-queue-socket -->

#### 한눈에 보기
- 파이프는 주로 바이트 스트림 기반의 단방향 통신이다.
- 메시지 큐는 메시지 단위와 우선순위를 다루기 쉽다.
- 소켓은 같은 머신뿐 아니라 네트워크 너머 프로세스와도 통신할 수 있다.

#### 핵심 정리

파이프는 데이터를 흐름처럼 전달한다. 익명 파이프는 보통 부모-자식 프로세스처럼 관계가 있는 프로세스 사이에서 많이 쓰고, 명명 파이프는 이름을 통해 독립 프로세스도 접근할 수 있다.

메시지 큐는 커널이 관리하는 큐에 메시지를 넣고 꺼내는 방식이다. 스트림이 아니라 메시지 단위가 유지되므로 경계가 필요한 통신에 유리하다. 구현에 따라 우선순위 메시지도 다룰 수 있다.

소켓은 네트워크 통신까지 포함하는 범용 IPC다. 로컬 소켓은 같은 머신 안의 프로세스 통신에, TCP/UDP 소켓은 다른 머신과의 통신에 사용된다.

#### 면접 답변

> 파이프는 스트림 형태로 데이터를 흘려보내는 IPC이고, 메시지 큐는 메시지 단위로 커널 큐에 저장해 전달합니다. 소켓은 로컬 또는 네트워크 너머 프로세스 간 통신에 사용할 수 있습니다.

#### 관련 퀴즈
- `os-c030`
- `os-d004`

### fork, exec, Copy-on-Write
<!-- article: os-fork-exec-cow -->

#### 한눈에 보기
- `fork()`는 부모와 거의 같은 자식 프로세스를 만든다.
- `exec()`는 현재 프로세스의 주소 공간을 새 프로그램으로 교체한다.
- Copy-on-Write는 실제 쓰기 전까지 페이지 복사를 미룬다.

#### 핵심 정리

유닉스 계열에서 새 프로그램 실행은 흔히 `fork()` 후 자식 프로세스가 `exec()`를 호출하는 흐름으로 설명된다. `fork()`는 부모의 주소 공간을 가진 자식을 만들고, `exec()`는 자식의 메모리 이미지를 새 프로그램으로 바꾼다.

이때 모든 페이지를 즉시 복사하면 낭비가 크다. 특히 자식이 바로 `exec()`를 호출하면 기존 페이지 대부분은 곧 버려진다. Copy-on-Write(CoW)는 부모와 자식이 같은 물리 페이지를 공유하다가, 어느 한쪽이 쓰기를 시도할 때 그 페이지를 복사한다.

CoW는 복사를 없애는 기술이 아니라 지연하는 기술이다. 자식이 `exec()` 없이 데이터를 수정하면 수정된 페이지는 결국 별도로 복사된다.

#### 면접 답변

> `fork()`는 부모와 같은 주소 공간을 가진 자식을 만들고, `exec()`는 그 주소 공간을 새 프로그램으로 교체합니다. CoW는 `fork()` 직후 페이지를 바로 복사하지 않고 쓰기 발생 시점까지 복사를 지연해 성능을 높입니다.

#### 관련 퀴즈
- `os-t012`
- `os-d007`

## 메모리 관리
<!-- category: os-memory-management -->

### 프로세스 메모리 구조
<!-- article: os-process-memory-layout -->

#### 한눈에 보기
- 코드 영역은 실행 명령어를 담고 보통 읽기 전용이다.
- 데이터 영역은 전역 변수와 정적 변수를 담는다.
- 힙은 동적 할당, 스택은 함수 호출에 사용된다.

#### 핵심 정리

프로세스 주소 공간은 보통 코드, 데이터, 힙, 스택으로 설명한다. 코드 영역은 실행할 명령어를 담고, 같은 프로그램을 여러 번 실행할 때 읽기 전용 코드 페이지를 공유할 수 있다.

데이터 영역은 전역 변수와 정적 변수를 담는다. 힙은 런타임에 크기가 변하는 동적 할당 영역이다. 메모리 누수는 보통 힙에 할당한 객체나 자료구조 참조가 계속 남아 회수되지 않을 때 관찰된다.

스택은 함수 호출마다 생기는 지역 변수, 매개변수, 복귀 주소를 담는다. 함수가 끝나면 자동으로 정리되지만, 재귀가 너무 깊으면 스택 오버플로우가 발생할 수 있다.

#### 면접 답변

> 프로세스 메모리는 코드, 데이터, 힙, 스택 영역으로 나눌 수 있습니다. 힙은 동적 할당 영역이고 스택은 함수 호출 흐름을 저장하는 영역이며, 스레드마다 독립적인 스택을 가집니다.

#### 관련 퀴즈
- `os-s003`
- `os-c021`

### 내부 단편화와 외부 단편화
<!-- article: os-fragmentation -->

#### 한눈에 보기
- 내부 단편화는 할당된 블록 내부에 남는 공간이다.
- 외부 단편화는 빈 공간이 흩어져 연속 공간을 못 만드는 상태다.
- 페이징은 외부 단편화를 줄이지만 내부 단편화가 생길 수 있다.

#### 핵심 정리

내부 단편화는 요청보다 큰 고정 크기 블록을 할당할 때 블록 안에 남는 공간이다. 예를 들어 페이지 크기가 4KB인데 마지막 페이지에 1KB만 필요하면 3KB가 내부 단편화가 된다.

외부 단편화는 빈 공간의 총합은 충분하지만 작은 조각으로 흩어져 큰 연속 공간을 할당하지 못하는 상태다. 세그멘테이션처럼 가변 크기 연속 할당을 사용할 때 나타나기 쉽다.

압축(compaction)은 흩어진 빈 공간을 모아 외부 단편화를 줄일 수 있지만 비용이 크다. 페이징은 고정 크기 프레임에 비연속 배치하므로 외부 단편화 문제를 크게 줄인다.

#### 면접 답변

> 내부 단편화는 할당된 블록 안에서 남는 공간이고, 외부 단편화는 빈 공간이 여러 조각으로 흩어져 큰 연속 공간을 만들지 못하는 문제입니다. 페이징은 외부 단편화를 줄이지만 내부 단편화가 생길 수 있습니다.

#### 관련 퀴즈
- `os-c014`

### 페이징과 세그멘테이션
<!-- article: os-paging-segmentation -->

#### 한눈에 보기
- 페이징은 주소 공간을 고정 크기 페이지로 나눈다.
- 세그멘테이션은 코드, 데이터, 스택 같은 논리 단위로 나눈다.
- 페이징은 메모리 관리 효율, 세그멘테이션은 보호와 공유 표현에 장점이 있다.

#### 핵심 정리

페이징은 가상 주소 공간을 고정 크기 페이지로 나누고, 물리 메모리는 같은 크기의 프레임으로 나눈다. 페이지는 물리 메모리에 연속으로 놓일 필요가 없으므로 외부 단편화를 피하기 쉽다.

세그멘테이션은 프로그램의 논리 구조를 반영한다. 코드, 데이터, 스택 같은 의미 있는 단위가 서로 다른 크기의 세그먼트가 된다. 보호 권한을 논리 단위로 주거나 특정 세그먼트를 공유하기에 직관적이다.

단점도 다르다. 페이징은 마지막 페이지에서 내부 단편화가 생길 수 있고, 세그멘테이션은 가변 크기라 외부 단편화가 생길 수 있다.

#### 면접 답변

> 페이징은 고정 크기 페이지로 주소 공간을 나누는 방식이고, 세그멘테이션은 코드나 스택 같은 논리 단위의 가변 크기 세그먼트로 나누는 방식입니다. 페이징은 외부 단편화를 줄이고, 세그멘테이션은 논리적 보호와 공유에 유리합니다.

#### 관련 퀴즈
- `os-c008`
- `os-c031`

### 페이지 크기 선택 기준
<!-- article: os-page-size-tradeoff -->

#### 한눈에 보기
- 페이지가 크면 페이지 테이블 크기와 I/O 횟수가 줄 수 있다.
- 페이지가 크면 내부 단편화가 커지고 불필요한 데이터가 함께 올라올 수 있다.
- 페이지가 작으면 세밀하지만 관리 오버헤드가 커진다.

#### 핵심 정리

페이지 크기는 메모리 관리의 중요한 트레이드오프다. 큰 페이지는 같은 주소 공간을 더 적은 개수의 페이지로 표현하므로 페이지 테이블이 작아지고, 디스크에서 연속 데이터를 가져올 때 I/O 효율이 좋아질 수 있다.

반면 큰 페이지는 실제로 필요한 데이터보다 많은 데이터를 함께 가져올 수 있다. 마지막 페이지에서 남는 공간도 커져 내부 단편화가 증가한다. 작은 페이지는 낭비가 줄고 세밀한 관리가 가능하지만 페이지 테이블과 TLB 부담이 커질 수 있다.

실무에서는 하드웨어 지원, 워크로드의 지역성, TLB 크기, I/O 패턴을 함께 고려한다.

#### 면접 답변

> 페이지 크기를 키우면 페이지 테이블 크기와 I/O 오버헤드는 줄 수 있지만 내부 단편화와 불필요한 적재가 늘 수 있습니다. 작게 만들면 세밀하지만 관리 오버헤드가 커집니다.

#### 관련 퀴즈
- `os-t005`

## 가상 메모리
<!-- category: os-virtual-memory -->

### 가상 메모리가 필요한 이유
<!-- article: os-virtual-memory-purpose -->

#### 한눈에 보기
- 가상 메모리는 프로세스마다 독립된 주소 공간을 제공한다.
- 실제 물리 메모리보다 큰 프로그램도 실행할 수 있게 한다.
- 보호, 격리, 공유를 운영체제가 관리할 수 있게 한다.

#### 핵심 정리

가상 메모리는 프로세스가 보는 주소와 실제 물리 메모리 주소를 분리한다. 각 프로세스는 자신만의 큰 연속 주소 공간을 가진 것처럼 동작하지만, 실제로는 필요한 페이지가 물리 메모리의 프레임에 매핑된다.

이 분리는 보호와 격리에 중요하다. 한 프로세스가 다른 프로세스의 메모리를 직접 침범하지 못하게 하고, 페이지마다 읽기, 쓰기, 실행 권한을 줄 수 있다. 또한 공유 라이브러리처럼 여러 프로세스가 같은 물리 페이지를 읽기 전용으로 공유하는 것도 가능하다.

가상 메모리는 물리 메모리보다 큰 프로그램 실행도 돕는다. 당장 필요한 부분만 메모리에 올리고 나머지는 디스크에 둘 수 있기 때문이다.

#### 면접 답변

> 가상 메모리는 프로세스마다 독립된 주소 공간을 제공하고, 필요한 부분만 물리 메모리에 올려 메모리를 효율적으로 쓰게 합니다. 또한 프로세스 간 격리와 페이지 단위 보호를 가능하게 합니다.

#### 관련 퀴즈
- `os-c009`
- `os-d003`

### 페이지 테이블과 TLB
<!-- article: os-page-table-tlb -->

#### 한눈에 보기
- 페이지 테이블은 가상 페이지와 물리 프레임의 매핑을 저장한다.
- TLB는 최근 주소 변환 결과를 캐싱하는 하드웨어 캐시다.
- TLB miss가 나면 페이지 테이블을 다시 참조해야 한다.

#### 핵심 정리

CPU가 가상 주소로 메모리에 접근하면, 하드웨어와 운영체제는 이 주소가 어느 물리 프레임을 가리키는지 찾아야 한다. 페이지 테이블은 이 변환 정보를 담고, valid bit, 권한 비트, dirty bit 같은 메타데이터도 함께 가진다.

페이지 테이블을 매번 메모리에서 찾으면 느리다. 그래서 CPU는 TLB(Translation Lookaside Buffer)에 최근 주소 변환 결과를 저장한다. TLB hit이면 빠르게 물리 주소를 얻고, miss이면 페이지 테이블을 참조하는 비용이 발생한다.

dirty bit는 페이지가 수정되었는지 알려준다. 수정된 페이지는 교체 전에 디스크에 기록해야 하지만, 수정되지 않은 페이지는 그냥 버릴 수 있어 교체 비용을 줄인다.

#### 면접 답변

> 페이지 테이블은 가상 주소를 물리 주소로 변환하기 위한 매핑 정보를 저장하고, TLB는 최근 변환 결과를 캐싱해 주소 변환을 빠르게 합니다. dirty bit는 수정된 페이지를 교체 전에 디스크에 써야 하는지 판단하는 데 쓰입니다.

#### 관련 퀴즈
- `os-c013`
- `os-c041`

### 요구 페이징과 페이지 폴트
<!-- article: os-demand-paging-page-fault -->

#### 한눈에 보기
- 요구 페이징은 실제 접근이 발생한 페이지를 필요할 때 적재한다.
- 페이지 폴트는 접근한 페이지가 메모리에 없을 때 발생한다.
- 페이지 폴트 처리는 디스크 I/O가 포함될 수 있어 비싸다.

#### 핵심 정리

요구 페이징(Demand Paging)은 프로세스 시작 시 모든 페이지를 메모리에 올리지 않는다. 실제로 접근한 페이지가 없을 때만 디스크에서 메모리로 가져온다. 초기 실행 시간을 줄이고 더 많은 프로세스를 동시에 실행하는 데 도움이 된다.

페이지 폴트가 발생하면 CPU는 예외를 발생시키고 커널이 개입한다. 커널은 접근이 유효한지 확인하고, 빈 프레임을 찾거나 교체할 페이지를 고른 뒤, 필요한 페이지를 디스크에서 읽어 페이지 테이블을 갱신한다.

물리 메모리 전체에 여유가 있어도 특정 프로세스에 할당된 프레임이 워킹셋보다 적으면 해당 프로세스에서는 페이지 폴트가 자주 발생할 수 있다.

#### 동작 과정

```text
페이지 접근 -> page fault -> 유효성 검사 -> 프레임 확보
-> 디스크에서 페이지 적재 -> 페이지 테이블 갱신 -> 명령 재시작
```

#### 면접 답변

> 요구 페이징은 필요한 페이지가 실제 참조될 때 메모리에 적재하는 방식입니다. 페이지 폴트가 발생하면 커널이 유효성을 확인하고 페이지를 디스크에서 읽어온 뒤 페이지 테이블을 갱신합니다.

#### 관련 퀴즈
- `os-c032`
- `os-o003`
- `os-s005`

### 페이지 교체 알고리즘
<!-- article: os-page-replacement-algorithms -->

#### 한눈에 보기
- FIFO는 가장 먼저 들어온 페이지를 교체한다.
- OPT는 미래에 가장 늦게 쓰일 페이지를 교체하지만 실제 구현은 불가능하다.
- LRU는 오래 사용되지 않은 페이지, LFU는 사용 빈도가 낮은 페이지를 교체한다.

#### 핵심 정리

페이지 폴트가 났는데 빈 프레임이 없으면 어떤 페이지를 내보낼지 결정해야 한다. FIFO는 가장 오래 전에 들어온 페이지를 제거한다. 단순하지만 실제 사용 가능성을 보지 않기 때문에 프레임 수를 늘렸는데도 페이지 폴트가 늘어나는 Belady's Anomaly가 발생할 수 있다.

OPT는 앞으로 가장 오래 쓰이지 않을 페이지를 제거해 이론적으로 최적이다. 하지만 실제 시스템은 미래 참조를 알 수 없으므로 구현할 수 없고, 다른 알고리즘의 비교 기준으로 사용된다.

LRU는 시간 지역성을 활용해 가장 오래 참조되지 않은 페이지를 제거한다. LFU는 참조 횟수가 낮은 페이지를 제거하지만, 과거에 많이 쓰였고 지금은 안 쓰는 페이지가 남는 문제가 있어 보정이 필요할 수 있다.

#### 면접 답변

> FIFO는 가장 먼저 들어온 페이지를 교체하고, LRU는 가장 오래 사용되지 않은 페이지를 교체합니다. OPT는 미래 참조를 알아야 하므로 실제 구현은 불가능하며, FIFO는 Belady's Anomaly가 발생할 수 있습니다.

#### 관련 퀴즈
- `os-c012`
- `os-c033`
- `os-c034`
- `os-c035`
- `os-d008`

### 스레싱, 워킹셋, PFF
<!-- article: os-thrashing-working-set-pff -->

#### 한눈에 보기
- 스레싱은 실제 작업보다 페이지 교체에 더 많은 시간을 쓰는 상태다.
- 워킹셋은 일정 시간 동안 자주 참조하는 페이지 집합이다.
- PFF는 페이지 폴트율을 보고 프레임 할당량을 조절한다.

#### 핵심 정리

스레싱은 메모리가 부족해 페이지 폴트가 폭증하고, CPU가 계산보다 디스크 I/O를 기다리는 데 시간을 쓰는 현상이다. 운영체제가 CPU 이용률이 낮다고 보고 프로세스를 더 올리면 메모리 압박이 심해져 악순환이 된다.

워킹셋 모델은 프로세스가 최근에 자주 참조한 페이지 집합을 메모리에 유지하려는 접근이다. 워킹셋보다 적은 프레임을 받으면 페이지 폴트가 계속 발생하므로, 충분한 프레임이 없으면 프로세스를 잠시 내보낼 수 있다.

PFF(Page Fault Frequency)는 페이지 폴트율의 상한선과 하한선을 둔다. 폴트율이 너무 높으면 프레임을 늘리고, 너무 낮으면 과하게 할당된 프레임을 줄여 전체 메모리 사용을 조절한다.

#### 면접 답변

> 스레싱은 메모리 부족으로 페이지 폴트와 디스크 I/O가 폭증해 실제 작업을 거의 못 하는 상태입니다. 워킹셋은 자주 쓰는 페이지 집합을 유지하고, PFF는 페이지 폴트율을 기준으로 프레임 수를 조절해 스레싱을 줄입니다.

#### 관련 퀴즈
- `os-t007`
- `os-c036`
- `os-s007`

### 공유 라이브러리와 메모리 공유
<!-- article: os-shared-library-memory -->

#### 한눈에 보기
- 공유 라이브러리의 읽기 전용 코드 페이지는 여러 프로세스가 공유할 수 있다.
- 각 프로세스는 자신만의 가상 주소에서 라이브러리를 보는 것처럼 동작한다.
- 실제 물리 프레임을 공유하면 중복 적재를 줄일 수 있다.

#### 핵심 정리

가상 메모리는 같은 물리 페이지를 여러 프로세스의 가상 주소 공간에 매핑할 수 있다. 공유 라이브러리의 코드 영역처럼 수정되지 않는 페이지는 여러 프로세스가 안전하게 함께 사용할 수 있다.

각 프로세스 입장에서는 자기 주소 공간 안에 라이브러리가 있는 것처럼 보인다. 하지만 물리 메모리 관점에서는 동일한 프레임을 가리키므로 메모리 사용량이 줄어든다.

쓰기 가능한 데이터 영역은 조심해야 한다. 공유하면 동기화 문제가 생기고, 공유하지 않으면 프로세스별로 별도 페이지를 가져야 한다. CoW도 이런 매핑과 보호 비트를 활용해 쓰기 시점에 복사를 수행한다.

#### 면접 답변

> 공유 라이브러리는 여러 프로세스의 가상 주소 공간이 같은 물리 페이지의 읽기 전용 코드 영역을 매핑하는 방식으로 공유됩니다. 각 프로세스는 독립적으로 보는 것 같지만 실제 메모리 중복은 줄어듭니다.

#### 관련 퀴즈
- `os-c042`
- `os-t012`

## 캐시와 지역성
<!-- category: os-cache-locality -->

### 메모리 계층 구조
<!-- article: os-memory-hierarchy -->

#### 한눈에 보기
- 메모리는 빠를수록 작고 비싸며, 느릴수록 크고 저렴하다.
- 일반적인 속도 순서는 레지스터, 캐시, 메인 메모리, SSD, HDD다.
- 운영체제와 하드웨어는 지역성을 이용해 계층 간 속도 차이를 줄인다.

#### 핵심 정리

CPU 레지스터는 가장 빠르지만 용량이 매우 작다. 캐시는 CPU 가까이에 위치해 자주 쓰는 데이터를 저장한다. 메인 메모리는 실행 중인 프로그램과 데이터를 담고, SSD나 HDD는 영구 저장을 담당한다.

이 계층 구조가 효과적인 이유는 프로그램이 모든 데이터를 균등하게 접근하지 않기 때문이다. 최근에 사용한 데이터나 그 주변 데이터를 다시 사용할 가능성이 높다는 지역성을 이용하면, 느린 계층까지 매번 내려가지 않아도 된다.

운영체제의 페이지 캐시, CPU 캐시, TLB는 모두 "최근 또는 가까운 것을 다시 쓸 가능성"을 활용하는 예로 볼 수 있다.

#### 면접 답변

> 메모리 계층은 레지스터, 캐시, 메인 메모리, SSD, HDD처럼 속도와 용량의 트레이드오프로 구성됩니다. 빠른 계층은 작고 비싸며, 지역성을 활용해 자주 쓰는 데이터를 위 계층에 둡니다.

#### 관련 퀴즈
- `os-c017`

### 캐시 지역성
<!-- article: os-cache-locality-types -->

#### 한눈에 보기
- 시간 지역성은 최근 사용한 데이터를 다시 사용할 가능성이다.
- 공간 지역성은 접근한 주소 주변을 곧 사용할 가능성이다.
- 순차 지역성은 명령어나 데이터가 연속 접근되는 경향이다.

#### 핵심 정리

시간 지역성은 반복문에서 같은 변수나 명령어가 반복 사용되는 상황에서 나타난다. LRU가 "오래 쓰지 않은 것은 앞으로도 덜 쓸 가능성이 높다"고 가정하는 것도 시간 지역성에 기반한다.

공간 지역성은 배열 순회처럼 인접한 메모리 주소를 이어서 접근할 때 나타난다. CPU 캐시는 보통 한 바이트만 가져오지 않고 캐시 라인 단위로 주변 데이터를 함께 가져온다.

순차 지역성은 공간 지역성의 특수한 형태로 볼 수 있다. 프로그램 카운터가 다음 명령어로 증가하거나 파일을 순서대로 읽는 패턴이 대표적이다.

#### 면접 답변

> 캐시 지역성에는 최근 사용한 데이터를 다시 쓸 가능성인 시간 지역성, 접근한 주소 주변을 곧 쓸 가능성인 공간 지역성, 연속적으로 접근하는 순차 지역성이 있습니다.

#### 관련 퀴즈
- `os-c037`
- `os-c012`

### 캐시 라인과 매핑 방식
<!-- article: os-cache-line-mapping -->

#### 한눈에 보기
- 캐시는 데이터를 캐시 라인 단위로 저장한다.
- 직접 매핑은 빠르지만 충돌이 많을 수 있다.
- 완전 연관은 유연하지만 검색 비용이 크고, 세트 연관은 절충안이다.

#### 핵심 정리

캐시 라인은 캐시가 메모리에서 데이터를 가져와 저장하는 기본 단위다. CPU가 특정 주소를 읽으면 그 주변 바이트까지 함께 가져와 공간 지역성을 활용한다. 각 라인에는 데이터뿐 아니라 태그, valid bit, dirty bit 같은 메타데이터가 붙는다.

직접 매핑은 메모리 블록이 들어갈 캐시 위치가 하나로 정해져 빠르지만, 같은 위치로 매핑되는 블록들이 서로 밀어내 충돌 miss가 많을 수 있다.

완전 연관은 어느 위치에나 넣을 수 있어 충돌이 적지만 모든 라인을 비교해야 하므로 비싸다. 세트 연관은 캐시를 세트로 나누고 세트 안에서만 비교해 둘 사이의 균형을 잡는다.

#### 면접 답변

> 캐시 라인은 캐시가 데이터를 저장하는 기본 단위입니다. 직접 매핑은 빠르지만 충돌이 많고, 완전 연관은 유연하지만 비싸며, 세트 연관은 성능과 비용 사이의 절충안입니다.

#### 관련 퀴즈
- `os-c017`
- `os-c037`

### 캐시 일관성과 NUMA
<!-- article: os-cache-coherence-numa -->

#### 한눈에 보기
- 캐시 일관성 문제는 여러 코어의 캐시가 같은 데이터의 다른 값을 볼 수 있는 문제다.
- MESI 같은 프로토콜은 캐시 사본 상태를 관리해 일관성을 유지한다.
- NUMA에서는 스레드와 메모리를 같은 노드에 배치하는 것이 중요하다.

#### 핵심 정리

멀티코어 CPU에서는 각 코어가 자기 캐시를 가진다. 같은 메모리 위치의 사본이 여러 캐시에 있을 때 한 코어가 값을 바꾸면, 다른 코어가 오래된 값을 볼 수 있다. 이것이 캐시 일관성 문제다.

하드웨어는 MESI 같은 프로토콜로 캐시 라인의 상태를 관리한다. 하지만 개발자 입장에서도 공유 데이터를 과도하게 여러 코어가 수정하면 캐시 라인이 계속 이동해 성능이 떨어질 수 있다.

NUMA 환경에서는 CPU 소켓 또는 노드마다 가까운 메모리가 다르다. 스레드가 원격 노드 메모리를 자주 접근하면 지연이 커진다. 따라서 스레드와 데이터가 같은 NUMA 노드에 놓이도록 배치하는 것이 성능에 중요하다.

#### 면접 답변

> 캐시 일관성 문제는 여러 코어의 캐시에 같은 데이터 사본이 있을 때 수정 내용이 즉시 맞지 않아 값이 달라질 수 있는 문제입니다. NUMA에서는 로컬 메모리 접근이 빠르므로 스레드와 메모리를 같은 노드에 배치하는 것이 중요합니다.

#### 관련 퀴즈
- `os-c038`
- `os-t011`

## 면접 질문 정리
<!-- category: os-interview -->

### 핵심 개념 답변 흐름
<!-- article: os-interview-answer-flow -->

#### 한눈에 보기
- 정의를 먼저 말하고, 왜 필요한지 설명한다.
- 장단점이나 비용을 붙이면 면접 답변이 깊어진다.
- 마지막에 적용 상황이나 주의점을 말하면 좋다.

#### 핵심 정리

운영체제 면접 답변은 "정의 -> 목적 -> 동작 방식 -> 트레이드오프" 순서가 안정적이다. 예를 들어 컨텍스트 스위칭을 묻는다면 "상태 저장과 복원 과정"이라고 정의한 뒤, 왜 필요한지와 어떤 비용이 있는지를 이어 말한다.

동기화 문제는 항상 공유 자원과 실행 순서를 함께 설명한다. "여러 스레드가 동시에 접근한다"에서 끝내지 말고, 읽기와 쓰기가 섞여 결과가 순서 의존적이 된다는 점을 말해야 한다.

메모리 문제는 주소 변환, 페이지 테이블, TLB, 페이지 폴트 비용을 연결하면 좋다. 가상 메모리를 단순히 "메모리를 늘리는 기술"로만 말하지 말고 보호와 격리까지 포함한다.

#### 면접 답변

> 운영체제 질문은 개념 정의만 말하기보다, 그 기능이 필요한 이유와 비용까지 함께 설명하겠습니다. 예를 들어 스레드는 가볍지만 공유 자원 동기화가 필요하고, 가상 메모리는 효율뿐 아니라 보호와 격리를 제공합니다.

#### 관련 퀴즈

- `os-c003`

### 장애 상황별 진단 포인트
<!-- article: os-interview-troubleshooting -->

#### 한눈에 보기
- CPU 0%인데 응답이 없으면 blocked 상태를 의심한다.
- 메모리가 계속 증가하면 메모리 누수나 과도한 보관을 의심한다.
- OOM Killer는 시스템 생존을 위해 프로세스를 종료한다.

#### 핵심 정리

운영체제 시나리오 문제는 지표를 먼저 해석한다. CPU가 높으면 무한 루프나 계산 폭주를 의심하고, CPU가 0%인데 응답이 없으면 I/O, 네트워크, DB, 락 대기를 의심한다.

프로세스 메모리가 시간이 갈수록 증가하고 재시작하면 정상으로 돌아온다면 메모리 누수 패턴이다. 네이티브 프로그램은 해제 누락을, GC 언어는 참조가 계속 남아 수집되지 않는 구조를 확인한다.

물리 메모리와 스왑이 모두 소진되면 리눅스는 OOM Killer로 프로세스를 종료해 시스템을 살리려 한다. 중요한 프로세스는 `oom_score_adj`를 조정할 수 있지만, 근본 해결은 메모리 프로파일링과 할당량 관리다.

#### 면접 답변

> 장애 상황에서는 CPU, 메모리, I/O 대기, 락 대기를 나눠 봅니다. CPU가 0%인데 응답이 없으면 blocked 상태를, 메모리가 계속 증가하면 누수를, 메모리와 스왑이 모두 소진되면 OOM Killer 동작을 의심합니다.

#### 관련 퀴즈
- `os-s001`
- `os-s003`
- `os-s004`
- `os-s007`

### 컨테이너와 자원 제한
<!-- article: os-container-resource-limits -->

#### 한눈에 보기
- 컨테이너는 프로세스 격리와 자원 제한을 함께 사용한다.
- cgroups는 CPU, 메모리, 프로세스 수 같은 자원 사용량을 그룹별로 제한한다.
- pids 제한은 fork bomb이 시스템 전체를 마비시키는 것을 막는다.

#### 핵심 정리

컨테이너는 가상 머신처럼 완전히 별도 커널을 띄우는 방식이 아니라, 같은 커널 위에서 프로세스를 격리하고 제한하는 방식으로 이해하면 좋다. 네임스페이스는 보이는 자원을 분리하고, cgroups는 사용할 수 있는 자원량을 제한한다.

fork bomb은 프로세스를 무한 생성해 PID나 메모리 같은 시스템 자원을 고갈시킨다. 컨테이너별 pids 제한을 두면 한 컨테이너가 생성할 수 있는 프로세스 수를 제한해 다른 컨테이너와 호스트를 보호할 수 있다.

운영체제 관점에서는 컨테이너도 결국 프로세스다. 따라서 스케줄링, 메모리 제한, OOM, 파일 시스템 격리 같은 기본 OS 개념과 직접 연결된다.

#### 면접 답변

> 컨테이너는 네임스페이스로 보이는 자원을 격리하고 cgroups로 CPU, 메모리, 프로세스 수 같은 사용량을 제한합니다. pids 제한은 fork bomb이 전체 시스템을 마비시키는 것을 막는 대표적인 보호 장치입니다.

#### 관련 퀴즈
- `os-s009`
