# 데이터베이스

이 문서는 CS 면접 준비를 위해 데이터베이스 핵심 개념을 모바일에서 읽기 좋게 정리한 학습 자료입니다. 관계형 데이터베이스의 기본기, 인덱스, 정규화, 트랜잭션, 조인, 스토리지 엔진, NoSQL, 쿼리 최적화를 하나의 흐름으로 연결합니다.

## 데이터베이스 기본 개념
<!-- category: db-basics -->

### 데이터베이스를 사용하는 이유
<!-- article: db-why-database -->

#### 한눈에 보기
- 파일 시스템은 데이터 중복, 애플리케이션 종속성, 무결성 관리가 약하다.
- 데이터베이스는 데이터를 구조화해 일관성, 보안성, 무결성을 높인다.
- 뷰(View)는 복잡한 조회를 감추고 필요한 데이터만 노출하는 가상 테이블이다.

#### 핵심 정리

파일로 데이터를 관리하면 각 애플리케이션이 자기 방식대로 파일 구조를 알고 있어야 한다. 파일 구조가 바뀌면 프로그램도 함께 바뀌기 쉽고, 같은 데이터가 여러 파일에 흩어지면 어느 값이 최신인지 판단하기 어렵다.

데이터베이스는 데이터를 공통 저장소에 두고 스키마, 제약 조건, 권한, 트랜잭션으로 관리한다. 덕분에 중복을 줄이고, 잘못된 값이 들어오는 것을 막고, 여러 사용자가 동시에 접근해도 일관된 상태를 유지할 수 있다.

뷰는 실제 데이터를 복사하지 않고 저장된 SELECT 결과처럼 보이게 하는 객체다. 자주 쓰는 복잡한 조회를 단순화하거나, 민감한 컬럼을 숨긴 채 일부 데이터만 보여줄 때 유용하다.

#### 면접 답변

> 데이터베이스는 파일 시스템의 중복, 종속성, 무결성 문제를 줄이기 위해 사용합니다. 데이터를 중앙에서 구조화해 관리하고, 제약 조건과 트랜잭션, 권한 관리를 통해 일관성과 보안성을 보장합니다.

#### 헷갈리는 포인트
- 뷰는 일반적으로 데이터를 물리적으로 복사하지 않는다.
- 단순 뷰를 수정하면 원본 테이블에 반영될 수 있으므로 읽기 전용 객체라고 단정하면 안 된다.

#### 관련 퀴즈

- `db-c021`

### 데이터 독립성, 무결성, 성능
<!-- article: db-independence-integrity-performance -->

#### 한눈에 보기
- 물리적 독립성은 저장 구조가 바뀌어도 애플리케이션 영향이 적은 성질이다.
- 논리적 독립성은 논리 스키마 변화와 사용자 관점을 분리하는 성질이다.
- 데이터베이스 성능 이슈는 많은 경우 디스크 I/O를 얼마나 줄이느냐로 이어진다.

#### 핵심 정리

데이터 독립성은 데이터베이스를 쓰는 중요한 이유 중 하나다. 물리적 독립성은 파일 배치, 인덱스, 저장 공간 같은 내부 저장 방식이 바뀌어도 애플리케이션을 크게 수정하지 않아도 되는 성질이다.

논리적 독립성은 테이블이나 뷰 같은 논리 구조가 사용자별 요구를 흡수할 수 있게 해준다. 예를 들어 특정 사용자는 필요한 컬럼만 보는 뷰를 사용하고, 내부 테이블 구조는 더 정규화된 형태로 유지할 수 있다.

성능 관점에서는 디스크 I/O가 핵심 병목이 된다. 순차 I/O는 연속된 데이터를 한 번에 읽기 쉬워 빠르고, 랜덤 I/O는 위치 이동이 잦아 비싸다. 쿼리 튜닝은 결국 불필요한 랜덤 I/O와 읽는 데이터 양을 줄이는 작업이다.

#### 면접 답변

> 데이터베이스는 데이터 독립성을 통해 저장 구조와 애플리케이션의 결합을 낮추고, 무결성 제약으로 유효한 상태를 유지합니다. 성능은 주로 디스크 I/O 비용에 좌우되므로, 인덱스와 실행 계획을 통해 읽는 범위를 줄이는 것이 중요합니다.

#### 헷갈리는 포인트
- 순차 I/O가 항상 모든 상황에서 빠른 것은 아니지만, 대량 연속 읽기에는 랜덤 I/O보다 유리하다.
- 인덱스도 결국 I/O를 줄이기 위한 도구라서, 잘못 만들면 오히려 비용이 늘어난다.

#### 관련 퀴즈
- `db-c022`
- `db-c023`

### 엔터티, 릴레이션, 키
<!-- article: db-entity-relation-key -->

#### 한눈에 보기
- 엔터티는 현실 세계의 객체나 개념이고, 릴레이션은 이를 테이블 구조로 표현한 것이다.
- 기본 키는 행을 유일하게 식별하며 NULL과 중복을 허용하지 않는다.
- 외래 키는 다른 테이블의 키를 참조해 참조 무결성을 지킨다.

#### 핵심 정리

엔터티는 학생, 주문, 상품처럼 저장하고 싶은 대상이다. 관계형 데이터베이스에서는 엔터티가 보통 테이블로 표현되고, 엔터티의 속성은 컬럼이 되며, 실제 데이터 한 건은 레코드 또는 튜플이 된다.

기본 키(Primary Key)는 테이블에서 각 행을 구분하는 기준이다. 기본 키가 안정적으로 정해져야 수정, 삭제, 조인, 참조가 명확해진다. 한 테이블에는 논리적으로 하나의 기본 키가 있고, 여러 컬럼을 묶은 복합 키도 가능하다.

외래 키(Foreign Key)는 두 테이블 사이의 관계를 강제한다. 주문의 `user_id`가 사용자 테이블의 `id`를 참조한다면, 존재하지 않는 사용자의 주문이 들어오지 않도록 막을 수 있다.

#### 면접 답변

> 엔터티는 저장 대상이 되는 현실 세계의 객체이고, 릴레이션은 이를 테이블 형태로 표현한 구조입니다. 기본 키는 각 행을 유일하게 식별하고, 외래 키는 다른 테이블을 참조해 테이블 간 참조 무결성을 보장합니다.

#### 헷갈리는 포인트
- 기본 키는 NULL을 허용하지 않는다.
- 외래 키는 중복 제거 자체가 아니라 참조 관계의 유효성을 보장하는 장치다.

#### 관련 퀴즈
- `db-c003`
- `db-c004`
- `db-c024`

### 타입 선택과 NULL 처리
<!-- article: db-types-null -->

#### 한눈에 보기
- `CHAR`는 고정 길이, `VARCHAR`는 가변 길이 문자열에 적합하다.
- `TEXT`는 긴 본문처럼 크고 검색 조건으로 자주 쓰지 않는 데이터에 맞다.
- `NULL`은 빈 문자열이나 0이 아니라 값이 없거나 알 수 없음을 뜻한다.

#### 핵심 정리

컬럼 타입은 저장 공간과 조회 성능에 영향을 준다. 길이가 항상 일정한 코드 값은 `CHAR`가 단순하고, 사용자 이름이나 제목처럼 길이가 달라지는 값은 `VARCHAR`가 자연스럽다.

`VARCHAR`는 행 내부에 저장되며 인덱스와 조회에 상대적으로 다루기 좋다. `TEXT`는 긴 글을 저장할 때 적합하지만, 조건 검색이나 인덱싱이 자주 필요한 컬럼이라면 설계와 제한을 신경 써야 한다.

`NULL`은 값이 비어 있다는 별도 상태다. `WHERE column = NULL`처럼 비교하면 원하는 결과가 나오지 않고, 보통 `IS NULL`을 사용해야 한다. 집계나 정렬에서도 DBMS별 처리 차이를 확인해야 한다.

#### 면접 답변

> CHAR는 길이가 고정된 값에, VARCHAR는 길이가 변하는 문자열에 적합합니다. TEXT는 긴 본문 저장에 사용하지만 인덱싱과 검색 비용을 고려해야 합니다. NULL은 빈 값이 아니라 미정 상태이므로 비교와 집계에서 별도 처리가 필요합니다.

#### 헷갈리는 포인트
- 빈 문자열 `''`, 숫자 `0`, `NULL`은 서로 다른 값이다.
- 긴 문자열이라고 무조건 `TEXT`를 쓰기보다 조회 패턴을 함께 봐야 한다.

#### 관련 퀴즈
- `db-c026`
- `db-c027`
- `db-d010`

### SQL 실행 흐름과 집계
<!-- article: db-sql-order-grouping -->

#### 한눈에 보기
- SQL의 논리적 실행 순서는 작성 순서와 다르다.
- `WHERE`는 그룹화 전 행을 필터링하고, `HAVING`은 그룹 결과를 필터링한다.
- `GROUP BY` 이후에는 그룹 기준 컬럼이나 집계 결과 중심으로 조회한다.

#### 핵심 정리

SQL은 보통 `SELECT`, `FROM`, `WHERE`, `GROUP BY`, `HAVING`, `ORDER BY` 순서로 쓰지만, 논리적으로는 `FROM`으로 대상을 정하고 `WHERE`로 행을 거른 뒤 그룹화와 집계를 수행한다. 그 다음 `HAVING`으로 그룹을 거르고, 마지막에 조회 컬럼과 정렬을 결정한다.

`WHERE`와 `HAVING`을 구분하는 기준은 집계 전인지 후인지다. 예를 들어 주문 금액이 0보다 큰 행만 대상으로 삼으려면 `WHERE amount > 0`을 쓰고, 사용자별 총 주문액이 100만 원 이상인 그룹을 찾으려면 `HAVING SUM(amount) >= 1000000`처럼 쓴다.

#### 면접 답변

> GROUP BY는 같은 값을 가진 행을 그룹으로 묶고, HAVING은 집계된 그룹 결과에 조건을 거는 절입니다. WHERE는 그룹화 전에 개별 행을 필터링하므로 집계 함수 결과 조건에는 HAVING을 사용해야 합니다.

#### 헷갈리는 포인트
- SQL 작성 순서와 논리적 실행 순서를 혼동하면 별칭이나 집계 조건에서 실수하기 쉽다.
- `GROUP BY` 뒤에 그룹 기준이 아닌 개별 행 컬럼을 그대로 SELECT하면 DBMS 설정에 따라 오류나 비결정적 결과가 생길 수 있다.

#### 관련 퀴즈
- `db-c009`
- `db-o001`

### 인덱스의 목적과 비용
<!-- article: db-index-purpose-cost -->

#### 한눈에 보기
- 인덱스는 특정 컬럼 기준으로 데이터를 빠르게 찾기 위한 보조 자료구조다.
- 조회는 빨라질 수 있지만 삽입, 수정, 삭제 때 인덱스 유지 비용이 든다.
- 작은 테이블이나 쓰기 비중이 매우 큰 테이블에는 인덱스가 오히려 부담이 될 수 있다.

#### 핵심 정리

인덱스는 책의 색인처럼 특정 값이 어디에 있는지 빠르게 찾도록 도와준다. 조건 컬럼에 적절한 인덱스가 있으면 전체 테이블을 처음부터 끝까지 훑지 않고 필요한 범위만 읽을 수 있다.

하지만 인덱스는 공짜가 아니다. 행을 삽입하면 인덱스에도 키가 추가되고, 값을 수정하면 기존 키를 지우고 새 키를 반영해야 한다. 삭제도 인덱스 정리 비용을 만든다. 그래서 모든 컬럼에 인덱스를 만드는 것은 좋은 전략이 아니다.

실무에서는 자주 조회되는 조건, 조인 키, 정렬 기준, 외래 키 등을 우선 검토한다. 반대로 데이터가 적거나, 조회보다 쓰기가 압도적으로 많거나, 값 종류가 너무 적은 컬럼은 신중해야 한다.

#### 면접 답변

> 인덱스는 WHERE, JOIN, ORDER BY 등에 쓰이는 컬럼을 빠르게 찾기 위한 자료구조입니다. 조회 성능을 높이는 대신 쓰기 작업과 저장 공간 비용이 증가하므로, 조회 패턴과 선택도를 기준으로 필요한 인덱스만 설계해야 합니다.

#### 헷갈리는 포인트
- 인덱스가 있어도 옵티마이저가 비용상 불리하다고 판단하면 사용하지 않을 수 있다.
- 인덱스는 중복 방지 자체가 목적이 아니다. 중복 방지는 유니크 제약과 함께 봐야 한다.

#### 관련 퀴즈
- `db-c002`
- `db-c020`
- `db-c030`
- `db-s001`

### B+트리와 해시 인덱스
<!-- article: db-index-btree-hash -->

#### 한눈에 보기
- B+트리는 정렬된 구조라 동등 검색과 범위 검색에 모두 유리하다.
- 해시 인덱스는 동등 비교에는 빠르지만 범위 검색에는 맞지 않는다.
- 데이터베이스 인덱스는 디스크 접근 패턴과 범위 질의를 함께 고려해야 한다.

#### 핵심 정리

데이터베이스에서 가장 널리 쓰이는 인덱스 구조는 B+트리다. 내부 노드는 탐색을 위한 키를 갖고, 실제 데이터나 데이터 포인터는 리프 노드에 모인다. 리프 노드가 연결되어 있어 `BETWEEN`, `>`, `<`, 정렬된 스캔에 강하다.

해시 인덱스는 키를 해시 값으로 바꿔 찾기 때문에 `=` 조건은 매우 빠를 수 있다. 그러나 해시 값은 원래 값의 순서를 보존하지 않으므로 범위 조건이나 정렬에는 적합하지 않다.

B+트리 탐색은 루트에서 내부 노드를 거쳐 리프 노드로 내려간다. 트리가 균형을 유지하기 때문에 데이터가 많아져도 높이가 천천히 증가하고, 큰 테이블에서도 안정적인 탐색 비용을 기대할 수 있다.

#### 면접 답변

> 데이터베이스는 동등 검색뿐 아니라 범위 검색과 정렬도 자주 수행하므로 B+트리 인덱스를 많이 사용합니다. B+트리는 리프 노드가 연결되어 범위 스캔이 효율적이고, 균형 트리라 데이터가 많아져도 탐색 비용이 안정적입니다.

#### 헷갈리는 포인트
- 해시가 평균 O(1)이라고 해서 모든 데이터베이스 질의에 더 좋은 것은 아니다.
- B트리와 B+트리는 데이터 저장 위치와 리프 노드 연결 방식에서 차이가 난다.

#### 관련 퀴즈
- `db-c012`
- `db-c028`
- `db-o003`

### 클러스터드, 세컨더리, 커버링 인덱스
<!-- article: db-index-clustered-covering -->

#### 한눈에 보기
- 클러스터드 인덱스는 데이터 행의 저장 순서와 밀접하게 연결된다.
- 논클러스터드 또는 세컨더리 인덱스는 별도 구조에서 실제 행 위치를 찾아간다.
- 커버링 인덱스는 쿼리에 필요한 컬럼을 인덱스만으로 해결해 테이블 접근을 줄인다.

#### 핵심 정리

클러스터드 인덱스는 리프 노드가 실제 데이터 행에 해당하거나 데이터가 인덱스 순서대로 배치되는 구조다. 테이블당 하나만 가능하며, 범위 검색에 유리하지만 삽입 위치가 중간이면 페이지 분할 같은 비용이 생길 수 있다.

논클러스터드 인덱스는 실제 데이터와 별도로 존재한다. 리프 노드에는 데이터 자체가 아니라 행을 찾기 위한 포인터나 클러스터드 키가 들어간다. 여러 개 만들 수 있지만, 조회 후 실제 행을 다시 읽는 비용이 생길 수 있다.

커버링 인덱스는 WHERE, JOIN, SELECT, ORDER BY에 필요한 컬럼이 모두 인덱스에 포함된 경우다. 이때 DB는 테이블 본문을 추가로 읽지 않고 인덱스만으로 결과를 만들 수 있어 랜덤 I/O를 줄일 수 있다.

#### 면접 답변

> 클러스터드 인덱스는 데이터가 인덱스 순서와 함께 저장되어 테이블당 하나만 가능하고 범위 검색에 유리합니다. 논클러스터드 인덱스는 별도 인덱스에서 행 위치를 찾아가며 여러 개 만들 수 있습니다. 커버링 인덱스는 필요한 컬럼을 모두 포함해 테이블 접근을 줄입니다.

#### 헷갈리는 포인트
- MySQL InnoDB에서는 기본 키가 클러스터드 인덱스 역할을 한다.
- 커버링 인덱스는 인덱스를 많이 만들라는 뜻이 아니라 특정 쿼리를 인덱스만으로 처리하게 설계한다는 뜻이다.

### 복합 인덱스와 선택도
<!-- article: db-index-composite-selectivity -->

#### 한눈에 보기
- 복합 인덱스는 컬럼 순서가 성능을 크게 좌우한다.
- 왼쪽부터 사용하는 규칙을 이해해야 인덱스를 제대로 탈 수 있다.
- 선택도가 높은 컬럼은 후보가 되지만, 실제 순서는 쿼리 조건과 정렬까지 함께 본다.

#### 핵심 정리

복합 인덱스 `(a, b, c)`는 보통 왼쪽부터 이어지는 조건에 잘 동작한다. `a`만 쓰거나 `a`와 `b`를 함께 쓰면 효과가 있지만, `b`만 조건에 쓰면 인덱스 장점을 충분히 얻기 어렵다.

컬럼 순서는 등호 조건, 정렬 조건, 범위 조건을 함께 고려한다. 일반적으로 등호로 좁히는 컬럼을 앞에 두고, 정렬이나 범위 조건은 뒤쪽에 두는 식으로 설계한다. 다만 카디널리티, 쿼리 빈도, DBMS 옵티마이저 특성까지 실행 계획으로 확인해야 한다.

`ORDER BY created_at DESC LIMIT 10`처럼 최신 데이터 일부만 가져오는 쿼리는 정렬 컬럼에 맞는 인덱스가 큰 효과를 낼 수 있다. 반대로 `LIKE '%검색어%'`처럼 앞부분이 와일드카드인 조건은 일반 B+트리 인덱스를 쓰기 어렵다.

#### 면접 답변

> 복합 인덱스는 왼쪽 컬럼부터 조건에 사용될 때 효과가 큽니다. 등호 조건으로 먼저 범위를 좁히고, 정렬이나 범위 조건을 뒤에 배치하는 식으로 쿼리 패턴에 맞춰 설계합니다. 최종 판단은 EXPLAIN으로 실제 사용 여부를 확인해야 합니다.

#### 헷갈리는 포인트
- 카디널리티가 높다고 항상 첫 번째 컬럼이 되는 것은 아니다.
- 인덱스 컬럼에 함수를 적용하거나 앞쪽 와일드카드를 쓰면 인덱스 사용이 어려워질 수 있다.

#### 관련 퀴즈
- `db-c031`
- `db-t005`
- `db-s004`
- `db-d002`

### 실행 계획과 Full Table Scan
<!-- article: db-explain-full-scan -->

#### 한눈에 보기
- Full Table Scan은 테이블 전체를 읽는 접근 방식이다.
- 인덱스가 없거나, 있어도 비용상 불리하면 옵티마이저가 전체 스캔을 선택한다.
- EXPLAIN은 사용 인덱스, 조인 순서, 예상 행 수, 필터링 비율을 확인하는 출발점이다.

#### 핵심 정리

쿼리가 느릴 때는 감으로 인덱스를 추가하기보다 실행 계획을 먼저 확인해야 한다. EXPLAIN을 보면 어떤 테이블을 어떤 순서로 읽는지, 어떤 인덱스를 쓰는지, 몇 행 정도를 읽을 것으로 예상하는지 확인할 수 있다.

Full Table Scan은 항상 나쁜 것은 아니다. 작은 테이블은 전체를 읽는 편이 더 빠를 수 있고, 조건이 대부분의 행을 반환한다면 인덱스를 거치는 비용이 더 클 수 있다. 문제는 큰 테이블에서 의도치 않게 전체를 읽는 경우다.

최적화는 읽는 데이터 양을 줄이는 방향으로 접근한다. 필요한 컬럼만 조회하고, 조건을 인덱스가 사용 가능한 형태로 바꾸고, 조인 조건을 명확히 하며, 통계와 인덱스 설계를 점검한다.

#### 면접 답변

> Full Table Scan은 조건에 맞는 인덱스가 없거나, 인덱스를 써도 비용이 더 크다고 옵티마이저가 판단할 때 발생합니다. EXPLAIN으로 접근 방식과 예상 행 수를 확인하고, 선택도 높은 인덱스와 조건식 개선으로 읽는 범위를 줄여야 합니다.

#### 헷갈리는 포인트
- `SELECT *`는 불필요한 컬럼 I/O를 늘릴 수 있다.
- 인덱스 추가는 처방 중 하나일 뿐이고, 쿼리 구조나 데이터 분포가 더 중요한 원인일 수 있다.

#### 관련 퀴즈
- `db-c018`
- `db-c050`
- `db-c051`
- `db-d008`

### 함수적 종속성
<!-- article: db-functional-dependency -->

#### 한눈에 보기
- 함수적 종속은 한 속성 값이 다른 속성 값을 유일하게 결정하는 관계다.
- 완전 함수 종속은 키 전체가 있어야 종속 값이 결정되는 상태다.
- 부분 종속과 이행 종속은 정규화 과정에서 제거해야 할 대표 대상이다.

#### 핵심 정리

함수적 종속은 `X -> Y`로 표현한다. X 값이 정해지면 Y 값이 하나로 결정될 때, Y는 X에 함수적으로 종속된다고 말한다. 예를 들어 사번이 정해지면 직원 이름이 정해지는 구조라면 `사번 -> 직원명`이 성립한다.

완전 함수 종속은 복합 키 전체가 있어야 비키 속성을 결정할 수 있는 경우다. 반대로 복합 키의 일부만으로도 결정된다면 부분 함수 종속이다. 부분 종속은 같은 정보가 반복 저장되는 원인이 된다.

이행적 함수 종속은 `A -> B`, `B -> C`라서 결과적으로 `A -> C`가 되는 형태다. 직원 테이블에 부서명과 부서 전화번호를 함께 두면, 직원이 아니라 부서가 전화번호를 결정하므로 분리 대상이 될 수 있다.

#### 면접 답변

> 함수적 종속은 어떤 속성 집합이 다른 속성 값을 유일하게 결정하는 관계입니다. 정규화는 함수적 종속을 분석해 부분 종속과 이행 종속을 제거하고, 중복과 갱신 이상을 줄이는 과정입니다.

#### 헷갈리는 포인트
- 부분 함수 종속은 보통 복합 키가 있을 때 문제가 된다.
- 이행적 종속은 기본 키가 아닌 속성이 또 다른 비키 속성을 결정할 때 자주 나타난다.

#### 관련 퀴즈
- `db-c025`
- `db-c032`
- `db-c033`

### 정규형과 이상 현상
<!-- article: db-normal-forms-anomaly -->

#### 한눈에 보기
- 정규화는 중복을 줄이고 삽입, 수정, 삭제 이상을 방지한다.
- 1NF는 컬럼 값의 원자성을 요구한다.
- 2NF는 부분 종속 제거, 3NF는 이행 종속 제거, BCNF는 모든 결정자가 후보 키인 상태를 요구한다.

#### 핵심 정리

정규화가 필요한 이유는 이상 현상 때문이다. 같은 정보가 여러 행에 중복되면 일부만 수정되어 데이터가 불일치할 수 있고, 어떤 데이터를 넣기 위해 원치 않는 다른 정보까지 필요할 수 있으며, 한 행을 지우면서 필요한 정보까지 사라질 수 있다.

제1정규형은 한 칸에 여러 값을 넣지 않는 것에서 시작한다. 전화번호 컬럼에 여러 번호를 쉼표로 저장하면 검색, 수정, 제약 조건 적용이 어려워진다.

제2정규형은 복합 키의 일부에만 의존하는 속성을 분리한다. 제3정규형은 키가 아닌 속성이 다른 키가 아닌 속성을 결정하는 구조를 제거한다. BCNF는 더 엄격하게 모든 결정자가 후보 키여야 한다.

#### 면접 답변

> 정규화는 데이터 중복과 삽입, 수정, 삭제 이상을 줄이기 위해 릴레이션을 분해하는 과정입니다. 1NF는 원자값, 2NF는 부분 종속 제거, 3NF는 이행 종속 제거, BCNF는 모든 결정자가 후보 키라는 조건을 요구합니다.

#### 헷갈리는 포인트
- 정규화는 성능 향상이 아니라 데이터 구조의 일관성과 무결성이 1차 목적이다.
- 정규형이 올라갈수록 조인이 늘 수 있어 조회 성능과 균형을 봐야 한다.

#### 관련 퀴즈
- `db-c006`
- `db-c014`
- `db-d003`

### 반정규화 선택 기준
<!-- article: db-denormalization-tradeoff -->

#### 한눈에 보기
- 반정규화는 조회 성능과 단순성을 위해 중복을 의도적으로 허용하는 설계다.
- 읽기 중심 시스템에서 복잡한 조인이 병목일 때 고려한다.
- 중복 데이터 갱신 비용과 무결성 위험을 반드시 함께 관리해야 한다.

#### 핵심 정리

정규화를 많이 한다고 항상 좋은 것은 아니다. 테이블을 지나치게 잘게 나누면 조회할 때 조인이 많아지고, 자주 호출되는 API에서 응답 시간이 늘어날 수 있다. 이때 읽기 성능을 위해 일부 값을 중복 저장하는 반정규화를 고려한다.

예를 들어 상품 목록 API가 상품, 카테고리, 브랜드 테이블을 매번 조인해 느리다면, 변경이 드문 카테고리명이나 브랜드명을 조회용 테이블에 함께 저장할 수 있다. 다만 원본 값이 바뀔 때 중복된 값도 함께 갱신해야 한다.

반정규화는 성능 문제를 해결하는 도구지만, 무결성 책임을 애플리케이션이나 배치 작업으로 옮기는 선택이기도 하다. 그래서 읽기 빈도, 변경 빈도, 불일치 허용 범위를 함께 따져야 한다.

#### 면접 답변

> 반정규화는 정규화로 인한 조인 비용이 커서 읽기 성능이 심하게 떨어질 때, 중복을 허용해 조회를 단순화하는 설계입니다. 대신 중복 데이터의 동기화와 무결성 문제가 생기므로 변경 빈도가 낮고 읽기가 많은 경우에 신중히 적용합니다.

#### 헷갈리는 포인트
- 반정규화는 정규화를 모르는 설계가 아니라, 비용을 알고 의도적으로 선택하는 설계다.
- 쓰기가 많은 OLTP 핵심 테이블에서는 중복 갱신 비용이 더 큰 문제가 될 수 있다.

#### 관련 퀴즈
- `db-t001`
- `db-t016`
- `db-s002`

## 트랜잭션
<!-- category: db-transactions -->

### 트랜잭션과 ACID
<!-- article: db-transaction-acid -->

#### 한눈에 보기
- 트랜잭션은 여러 DB 작업을 하나의 논리적 작업 단위로 묶는다.
- COMMIT은 변경을 확정하고, ROLLBACK은 트랜잭션 시작 전 상태로 되돌린다.
- ACID는 원자성, 일관성, 격리성, 지속성을 뜻한다.

#### 핵심 정리

트랜잭션은 작업의 완전성을 보장한다. 계좌 이체에서 출금은 성공했는데 입금이 실패하면 데이터는 깨진다. 출금과 입금을 하나의 트랜잭션으로 묶으면 모두 성공할 때만 반영하고, 중간 실패 시 모두 되돌릴 수 있다.

원자성은 모두 수행되거나 전혀 수행되지 않는 성질이다. 일관성은 트랜잭션 전후 데이터가 제약 조건을 만족하는 유효한 상태여야 한다는 뜻이다. 격리성은 동시에 실행되는 트랜잭션이 서로의 중간 상태를 함부로 보지 못하게 한다.

지속성은 커밋된 결과가 장애 이후에도 보존되는 성질이다. DBMS는 로그와 복구 메커니즘을 통해 커밋된 데이터를 유지하려고 한다.

#### 면접 답변

> 트랜잭션은 여러 연산을 하나의 작업 단위로 묶어 모두 성공하거나 모두 실패하게 합니다. COMMIT은 변경을 확정하고 ROLLBACK은 취소합니다. ACID는 원자성, 일관성, 격리성, 지속성으로 트랜잭션의 신뢰성을 설명합니다.

#### 헷갈리는 포인트
- COMMIT 이후에는 일반적으로 해당 트랜잭션을 ROLLBACK할 수 없다.
- 트랜잭션은 정합성의 단위이고, 락은 동시 접근을 제어하는 수단이다.

#### 관련 퀴즈
- `db-c001`
- `db-c007`
- `db-s003`
- `db-o002`

### 격리 수준과 이상 읽기
<!-- article: db-isolation-levels-anomalies -->

#### 한눈에 보기
- 격리 수준은 동시 트랜잭션 사이에서 어떤 읽기 이상을 허용할지 정한다.
- Dirty Read는 커밋되지 않은 변경을 읽는 현상이다.
- REPEATABLE READ는 같은 트랜잭션 안에서 동일 조회 결과의 반복성을 강화한다.

#### 핵심 정리

READ UNCOMMITTED는 가장 낮은 격리 수준으로 커밋되지 않은 데이터까지 읽을 수 있어 Dirty Read가 발생할 수 있다. READ COMMITTED는 커밋된 데이터만 읽게 해 Dirty Read를 막지만, 같은 행을 다시 읽었을 때 값이 달라지는 Non-repeatable Read가 가능하다.

REPEATABLE READ는 같은 트랜잭션 안에서 동일 행을 반복 조회할 때 일관성을 높인다. MySQL InnoDB의 기본 격리 수준이며, MVCC를 통해 트랜잭션 시작 시점 기준의 읽기를 제공한다.

SERIALIZABLE은 가장 강한 격리 수준으로 트랜잭션을 순차 실행한 것과 비슷한 결과를 목표로 한다. 정합성은 강하지만 잠금과 대기 시간이 늘어 처리량이 떨어질 수 있다.

#### 면접 답변

> 격리 수준은 동시 트랜잭션 사이에서 Dirty Read, Non-repeatable Read, Phantom Read 같은 현상을 얼마나 막을지 정합니다. InnoDB는 기본적으로 REPEATABLE READ를 사용해 스냅샷 기반 반복 읽기와 동시성의 균형을 맞춥니다.

#### 헷갈리는 포인트
- Phantom Read는 같은 조건으로 범위 조회했을 때 새 행이 나타나거나 사라지는 현상이다.
- DBMS마다 같은 격리 수준 이름이라도 MVCC와 락 구현 때문에 세부 동작이 다를 수 있다.

### 격리 수준 선택의 비용
<!-- article: db-isolation-tradeoff -->

#### 한눈에 보기
- 격리 수준을 높이면 정합성은 강해지지만 동시성 비용이 커질 수 있다.
- 재고 차감처럼 경쟁 조건이 중요한 작업은 락이나 조건부 갱신을 함께 고려한다.
- SERIALIZABLE은 강력하지만 모든 업무에 기본값으로 쓰기에는 비용이 크다.

#### 핵심 정리

격리 수준 선택은 정합성과 처리량의 트레이드오프다. 낮은 격리 수준은 동시성이 좋지만 읽기 이상을 허용할 수 있고, 높은 격리 수준은 이상 현상을 더 강하게 막지만 락 대기, 데드락, 응답 지연을 늘릴 수 있다.

예를 들어 마지막 재고 1개를 동시에 구매하는 상황에서는 단순 조회 후 업데이트가 위험하다. 조건부 UPDATE, 비관적 락, 낙관적 락, 적절한 트랜잭션 격리 수준을 조합해 재고가 음수가 되지 않게 해야 한다.

READ COMMITTED보다 SERIALIZABLE을 선택해야 하는 경우는 결과의 직렬 가능성이 매우 중요하고 동시성 저하를 감수할 수 있을 때다. 대부분의 서비스는 업무별로 필요한 정합성 수준을 나눠 설계한다.

#### 면접 답변

> 격리 수준을 높이면 읽기 이상을 더 잘 막지만 락 범위와 대기 시간이 늘어 처리량이 낮아지고 데드락 가능성이 커질 수 있습니다. 그래서 업무의 정합성 요구와 동시성 요구를 기준으로 수준을 선택해야 합니다.

#### 헷갈리는 포인트
- 격리 수준만 높이면 모든 동시성 문제가 해결된다고 보면 안 된다.
- 재고 차감 같은 문제는 쿼리 조건, 락 전략, 재시도 정책까지 함께 설계해야 한다.

#### 관련 퀴즈

- `db-t017`

### WAL과 장애 복구
<!-- article: db-wal-recovery -->

#### 한눈에 보기
- WAL은 데이터 페이지를 쓰기 전에 변경 로그를 먼저 기록하는 방식이다.
- 장애가 나면 로그를 기준으로 커밋된 변경은 재적용하고, 미완료 변경은 되돌린다.
- 지속성은 트랜잭션 로그와 복구 절차에 크게 의존한다.

#### 핵심 정리

데이터베이스는 장애 상황에서도 커밋된 변경을 잃지 않기 위해 로그를 사용한다. Write-Ahead Logging은 이름 그대로 실제 데이터 페이지를 디스크에 반영하기 전에 변경 내용을 로그에 먼저 남긴다.

장애가 발생하면 DBMS는 로그를 읽어 어떤 트랜잭션이 커밋되었고 어떤 트랜잭션이 미완료였는지 판단한다. 커밋된 변경이 데이터 파일에 완전히 반영되지 않았다면 다시 적용하고, 미완료 변경은 롤백한다.

이 과정은 ACID의 지속성과 원자성을 현실에서 구현하는 기반이다. 사용자는 COMMIT을 호출하지만, 내부적으로는 로그 기록, 플러시, 체크포인트, 복구 절차가 함께 작동한다.

#### 면접 답변

> WAL은 데이터 페이지보다 변경 로그를 먼저 기록해 장애 복구를 가능하게 하는 방식입니다. 장애 후 로그를 기준으로 커밋된 트랜잭션은 재적용하고 미완료 트랜잭션은 되돌려 원자성과 지속성을 보장합니다.

#### 헷갈리는 포인트
- 로그가 있다고 백업이 필요 없는 것은 아니다.
- 체크포인트는 복구 범위를 줄이는 데 도움을 주지만, 로그 복구 자체를 대체하지 않는다.

#### 관련 퀴즈
- `db-o004`

### 공유 락과 배타 락
<!-- article: db-shared-exclusive-lock -->

#### 한눈에 보기
- 공유 락은 읽기용이며 여러 트랜잭션이 동시에 가질 수 있다.
- 배타 락은 쓰기용이며 다른 락과 충돌한다.
- 락은 격리성을 구현하는 수단이고, 트랜잭션 자체와는 역할이 다르다.

#### 핵심 정리

공유 락(S-Lock)은 데이터를 읽는 동안 다른 트랜잭션도 같은 데이터를 읽을 수 있게 한다. 읽기끼리는 충돌하지 않기 때문이다. 그러나 누군가 쓰기 위해 배타 락을 잡으려면 기존 공유 락이 해제될 때까지 기다려야 한다.

배타 락(X-Lock)은 데이터를 변경할 때 사용한다. 하나의 트랜잭션이 배타 락을 잡으면 다른 트랜잭션은 같은 자원에 공유 락이나 배타 락을 잡기 어렵다. 잘못된 동시 쓰기와 중간 상태 읽기를 막기 위한 장치다.

트랜잭션은 작업 단위와 정합성을 설명하고, 락은 여러 트랜잭션이 같은 자원을 동시에 다룰 때 접근 순서를 제어한다. 둘은 함께 쓰이지만 같은 개념이 아니다.

#### 면접 답변

> 공유 락은 읽기 시 여러 트랜잭션이 동시에 획득할 수 있고, 배타 락은 쓰기 시 하나의 트랜잭션만 획득할 수 있습니다. 트랜잭션은 정합성을 보장하는 작업 단위이고, 락은 동시 접근을 제어해 격리성을 돕는 수단입니다.

#### 헷갈리는 포인트
- 락을 쓴다고 트랜잭션이 필요 없어지는 것은 아니다.
- 인덱스를 잘못 타면 잠금 범위가 예상보다 넓어질 수 있다.

#### 관련 퀴즈

- `db-c034`
- `db-d007`

### 교착상태 탐지
<!-- article: db-deadlock-detection -->

#### 한눈에 보기
- 교착상태는 서로가 가진 자원을 기다리며 더 이상 진행하지 못하는 상태다.
- 상호 배제, 점유와 대기, 비선점, 순환 대기 조건이 함께 만족되면 발생할 수 있다.
- Wait-for Graph는 대기 관계의 사이클로 데드락을 탐지한다.

#### 핵심 정리

교착상태는 두 트랜잭션이 서로의 락을 기다릴 때 대표적으로 발생한다. 트랜잭션 A가 1번 자원을 잡고 2번 자원을 기다리고, 트랜잭션 B가 2번 자원을 잡고 1번 자원을 기다리면 둘 다 진행할 수 없다.

DBMS는 이런 상태를 탐지하면 보통 한 트랜잭션을 희생자로 골라 롤백한다. 애플리케이션은 데드락이 실패가 아니라 재시도 가능한 충돌 상황일 수 있음을 이해하고, 짧은 트랜잭션과 재시도 정책을 준비해야 한다.

Wait-for Graph 방식은 트랜잭션을 노드로, 대기 관계를 간선으로 표현한다. 그래프에 사이클이 생기면 서로를 기다리는 구조이므로 교착상태로 판단할 수 있다.

#### 면접 답변

> 교착상태는 트랜잭션들이 서로 보유한 락을 기다려 진행하지 못하는 상태입니다. Wait-for Graph는 트랜잭션 간 대기 관계를 그래프로 만들고 사이클이 있는지 확인해 데드락을 탐지합니다.

#### 헷갈리는 포인트
- DB가 한 트랜잭션을 자동 롤백해도 재발 방지 설계는 애플리케이션 몫이다.
- 데드락은 락을 오래 잡거나 자원 접근 순서가 뒤섞일 때 더 자주 발생한다.

#### 관련 퀴즈
- `db-c036`
- `db-s015`

### 교착상태 예방과 실무 팁
<!-- article: db-deadlock-prevention -->

#### 한눈에 보기
- 자원 접근 순서를 통일하면 순환 대기 가능성을 줄일 수 있다.
- 트랜잭션은 짧게 유지하고, 사용자 입력이나 외부 API 호출을 트랜잭션 안에 오래 두지 않는다.
- Wait-Die와 Wound-Wait는 타임스탬프 기반 예방 방식이다.

#### 핵심 정리

실무에서 가장 먼저 적용할 수 있는 방법은 접근 순서 통일이다. 모든 코드가 `계좌 A -> 계좌 B`처럼 같은 기준으로 락을 잡으면 서로 반대 순서로 기다리는 상황을 줄일 수 있다.

트랜잭션 안에서 오래 걸리는 작업을 줄이는 것도 중요하다. 락을 잡은 채 네트워크 호출, 파일 처리, 사용자 응답 대기를 하면 다른 트랜잭션의 대기 시간이 길어지고 데드락 가능성도 커진다.

Wait-Die와 Wound-Wait는 트랜잭션의 나이를 기준으로 기다릴지 롤백할지 정하는 방식이다. Wait-Die는 오래된 트랜잭션은 기다리고 젊은 트랜잭션은 물러나게 한다. Wound-Wait는 오래된 트랜잭션이 젊은 트랜잭션을 선점할 수 있게 한다.

#### 면접 답변

> 교착상태를 줄이려면 자원 접근 순서를 통일하고, 트랜잭션을 짧게 유지하며, 실패 시 재시도할 수 있게 설계해야 합니다. Wait-Die와 Wound-Wait는 트랜잭션 타임스탬프를 기준으로 대기 또는 롤백을 결정하는 데드락 예방 방식입니다.

#### 헷갈리는 포인트
- 데드락을 완전히 없애기보다 발생 빈도를 낮추고 안전하게 재시도하는 전략이 현실적이다.
- 긴 트랜잭션은 정합성을 높이는 것이 아니라 동시성 비용을 키우는 원인이 될 수 있다.

#### 관련 퀴즈
- `db-c037`

## 조인
<!-- category: db-joins -->

### 조인의 종류
<!-- article: db-join-types -->

#### 한눈에 보기
- INNER JOIN은 양쪽 테이블에서 조건에 맞는 행만 반환한다.
- LEFT JOIN과 RIGHT JOIN은 기준 테이블의 모든 행을 보존한다.
- CROSS JOIN은 모든 조합을 만들기 때문에 결과 수가 급격히 커질 수 있다.

#### 핵심 정리

조인은 정규화로 나뉜 테이블을 관계 조건에 따라 다시 결합하는 SQL 기능이다. 주문 테이블과 사용자 테이블을 `user_id`로 연결하면 주문 정보와 주문자 정보를 함께 조회할 수 있다.

INNER JOIN은 양쪽에 매칭되는 행만 반환한다. LEFT JOIN은 왼쪽 테이블의 모든 행을 유지하고 오른쪽에 매칭되는 값이 없으면 NULL을 채운다. RIGHT JOIN은 기준이 반대지만, 실무에서는 가독성을 위해 기준 테이블을 왼쪽에 두고 LEFT JOIN을 많이 쓴다.

CROSS JOIN은 조건 없이 두 테이블의 모든 조합을 만든다. 행 수가 `N x M`으로 증가하므로 테스트 데이터 생성처럼 의도가 분명한 경우가 아니라면 주의해야 한다.

#### 면접 답변

> JOIN은 둘 이상의 테이블을 관련 컬럼 기준으로 결합해 하나의 결과 집합으로 조회하는 기능입니다. INNER JOIN은 조건에 맞는 행만, LEFT JOIN은 왼쪽 기준 테이블의 모든 행과 매칭 결과를 반환하며, CROSS JOIN은 모든 조합을 생성합니다.

#### 헷갈리는 포인트
- OUTER JOIN 결과에서 매칭되지 않은 쪽 컬럼은 NULL이 된다.
- NATURAL JOIN은 같은 이름 컬럼을 자동으로 기준 삼기 때문에 명시적 JOIN보다 의도를 파악하기 어렵다.

### 조인 알고리즘
<!-- article: db-join-algorithms -->

#### 한눈에 보기
- 중첩 루프 조인은 한쪽 행마다 다른 쪽을 탐색한다.
- 정렬 병합 조인은 조인 키로 정렬된 입력을 순차 병합한다.
- 해시 조인은 한쪽을 해시 테이블로 만들고 다른 쪽을 probe하며, 동등 조인에 적합하다.

#### 핵심 정리

중첩 루프 조인은 이해하기 쉽고 작은 데이터나 인덱스가 잘 잡힌 조건에서 효과적이다. 바깥 테이블의 각 행마다 안쪽 테이블을 찾기 때문에, 안쪽 접근이 인덱스로 빠르게 해결되지 않으면 비용이 커진다.

정렬 병합 조인은 양쪽 입력을 조인 키 기준으로 정렬한 뒤 순서대로 병합한다. 이미 정렬된 인덱스나 결과를 활용할 수 있거나, 대용량 데이터에서 순차 처리가 유리할 때 후보가 된다.

해시 조인은 보통 더 작은 쪽을 해시 테이블로 만들고, 큰 쪽을 읽으며 같은 해시 버킷을 찾아 매칭한다. 키의 동일성을 해시로 판단하므로 `=` 조건 조인에 적합하고, 범위 조건에는 맞지 않는다.

#### 면접 답변

> 조인 알고리즘에는 중첩 루프, 정렬 병합, 해시 조인이 있습니다. 중첩 루프는 작은 데이터나 인덱스 활용에 유리하고, 정렬 병합은 정렬된 입력에 유리하며, 해시 조인은 한쪽을 해시 테이블로 만들어 동등 조인을 빠르게 처리합니다.

#### 헷갈리는 포인트
- 해시 조인은 크다/작다 같은 범위 관계를 해시 값만으로 판단하기 어렵다.
- 옵티마이저가 데이터 크기, 통계, 인덱스, 메모리를 바탕으로 조인 방식을 선택한다.

## 스토리지 엔진
<!-- category: db-storage-engine -->

### InnoDB와 MyISAM
<!-- article: db-innodb-myisam -->

#### 한눈에 보기
- InnoDB는 트랜잭션, 외래 키, 행 레벨 락, MVCC를 지원한다.
- MyISAM은 트랜잭션을 지원하지 않고 테이블 레벨 락 중심이다.
- 일반적인 서비스에서는 안정성과 동시성 때문에 InnoDB가 기본 선택이다.

#### 핵심 정리

스토리지 엔진은 데이터를 실제로 저장하고 읽고, 락과 트랜잭션, 복구를 처리하는 핵심 구성 요소다. MySQL에서 InnoDB는 기본 엔진으로, ACID 트랜잭션과 외래 키, 행 레벨 락을 제공한다.

행 레벨 락은 변경 대상 행만 잠가 서로 다른 행을 다루는 트랜잭션이 동시에 진행될 수 있게 한다. 테이블 전체를 잠그는 방식보다 OLTP 환경에서 동시성이 높다.

MyISAM은 구조가 단순하고 일부 읽기 중심 워크로드에서 오버헤드가 낮을 수 있지만, 트랜잭션과 외래 키를 지원하지 않는다. 장애 복구와 정합성 요구가 있는 일반 서비스라면 InnoDB를 우선 고려한다.

#### 면접 답변

> InnoDB는 트랜잭션, 외래 키, MVCC, 행 레벨 락을 지원해 정합성과 동시성이 중요한 서비스에 적합합니다. MyISAM은 트랜잭션을 지원하지 않고 테이블 레벨 락 중심이라 읽기 위주의 특수한 상황에서만 제한적으로 고려합니다.

#### 헷갈리는 포인트
- 읽기만 빠르다는 이유로 MyISAM을 기본 선택하면 복구와 정합성에서 문제가 생길 수 있다.
- 행 레벨 락도 인덱스를 제대로 사용하지 못하면 잠금 범위가 커질 수 있다.

#### 관련 퀴즈

### MVCC와 2PL
<!-- article: db-mvcc-2pl -->

#### 한눈에 보기
- MVCC는 여러 버전의 데이터를 두어 읽기와 쓰기의 충돌을 줄인다.
- 2PL은 락 획득과 해제 규칙으로 직렬 가능성을 보장하려는 접근이다.
- MVCC는 읽기 성능과 동시성에 유리하지만 버전 관리 비용이 든다.

#### 핵심 정리

MVCC(Multi-Version Concurrency Control)는 데이터를 갱신할 때 기존 값을 바로 덮어쓰기보다 여러 버전을 관리해, 읽는 트랜잭션이 자기 시점에 맞는 버전을 보게 한다. 덕분에 읽기가 쓰기를 무조건 기다리지 않아도 되는 경우가 많다.

2PL(Two-Phase Locking)은 트랜잭션이 필요한 락을 얻는 단계와 락을 해제하는 단계를 구분해 충돌을 제어한다. 정합성을 강하게 만들 수 있지만 락 대기와 데드락 비용이 커질 수 있다.

MVCC는 동시성을 높이는 대신 오래된 버전 정리, 언두 로그 관리, 스냅샷 판단 비용이 필요하다. 그래서 장시간 열린 트랜잭션은 버전 정리를 방해하고 성능 문제를 만들 수 있다.

#### 면접 답변

> MVCC는 데이터의 여러 버전을 관리해 읽기 트랜잭션이 일관된 스냅샷을 보게 하고, 읽기와 쓰기 충돌을 줄입니다. 2PL은 락 규칙으로 충돌을 막지만 대기와 데드락 비용이 커질 수 있어 MVCC가 읽기 동시성에 유리합니다.

#### 헷갈리는 포인트
- MVCC가 모든 락을 없애는 것은 아니다. 쓰기 충돌이나 무결성 보장을 위해 락이 여전히 필요하다.
- 긴 트랜잭션은 MVCC 환경에서도 비용이 크다.

#### 관련 퀴즈
- `db-t013`

## Statement와 PreparedStatement
<!-- category: db-statement-preparedstatement -->

### PreparedStatement와 SQL Injection
<!-- article: db-preparedstatement-sql-injection -->

#### 한눈에 보기
- Statement는 SQL 문자열을 그대로 만들어 실행한다.
- PreparedStatement는 SQL 구조를 준비하고 값은 파라미터로 바인딩한다.
- 파라미터 바인딩은 입력값을 SQL 문법이 아니라 데이터로 처리해 SQL Injection을 막는 데 유리하다.

#### 핵심 정리

Statement로 사용자 입력을 문자열 결합해 SQL을 만들면 입력값이 쿼리 구조를 바꿀 수 있다. 예를 들어 로그인 아이디 입력에 SQL 조각이 섞이면 인증 조건이 왜곡될 수 있다.

PreparedStatement는 `WHERE email = ?`처럼 SQL 구조를 먼저 정의하고, 실제 값은 별도로 바인딩한다. 이때 입력값은 명령어가 아니라 데이터로 다뤄지므로 SQL Injection 방어에 효과적이다.

반복 실행에서도 PreparedStatement가 유리할 수 있다. DBMS와 드라이버가 SQL 파싱과 실행 계획 준비 비용을 줄일 수 있기 때문이다. 다만 전체 성능은 인덱스, 읽는 행 수, 네트워크 비용의 영향도 함께 받는다.

#### 면접 답변

> Statement는 완성된 SQL 문자열을 실행하고, PreparedStatement는 SQL 템플릿을 미리 준비한 뒤 값을 파라미터로 바인딩합니다. PreparedStatement는 입력값이 SQL 구조로 해석되지 않게 하므로 SQL Injection 방어와 반복 실행 성능에 유리합니다.

#### 헷갈리는 포인트
- PreparedStatement를 써도 테이블명, 컬럼명 같은 SQL 구조는 파라미터로 바인딩할 수 없다.
- 보안은 파라미터 바인딩, 권한 최소화, 에러 메시지 관리까지 함께 봐야 한다.

#### 관련 퀴즈

- `db-c046`

### ORM, 직접 SQL, N+1
<!-- article: db-orm-sql-n-plus-one -->

#### 한눈에 보기
- ORM은 객체와 테이블 매핑을 도와 생산성을 높인다.
- 복잡한 쿼리 튜닝이나 대량 처리에는 직접 SQL이 더 명확할 수 있다.
- N+1 문제는 목록 조회 뒤 연관 데이터를 건별로 추가 조회할 때 발생한다.

#### 핵심 정리

ORM은 반복적인 CRUD와 매핑 코드를 줄여준다. 하지만 생성되는 SQL을 이해하지 못하면 성능 문제를 늦게 발견한다. 복잡한 조인, 집계, DBMS별 튜닝이 필요한 경우에는 직접 SQL이나 쿼리 DSL이 더 적합할 수 있다.

N+1 문제는 게시글 100개를 조회한 뒤 각 게시글의 작성자나 댓글을 하나씩 추가 조회해 총 101개 쿼리가 나가는 식으로 발생한다. 데이터 수가 늘수록 쿼리 횟수가 선형으로 증가한다.

해결 방법은 fetch join, entity graph, batch size, 필요한 데이터만 조회하는 DTO 쿼리 등이다. 중요한 것은 ORM을 쓰더라도 실제 SQL과 실행 계획을 확인하는 습관이다.

#### 면접 답변

> ORM은 개발 생산성을 높이지만 복잡한 튜닝 쿼리나 대량 처리에서는 직접 SQL이 더 적합할 수 있습니다. N+1 문제는 연관 데이터를 건별로 추가 조회하는 문제이며, fetch join, batch size, DTO 조회 등으로 쿼리 수를 줄입니다.

#### 헷갈리는 포인트
- ORM을 쓰면 SQL을 몰라도 된다는 뜻이 아니다.
- fetch join은 유용하지만 페이징, 중복 행, 컬렉션 조인에서 주의가 필요하다.

#### 관련 퀴즈
- `db-t008`
- `db-s005`

## NoSQL과 분산 데이터
<!-- category: db-nosql-distributed -->

### NoSQL을 선택하는 기준
<!-- article: db-nosql-choice -->

#### 한눈에 보기
- NoSQL은 유연한 스키마와 수평 확장에 유리한 데이터 저장소 계열이다.
- 문서 지향 DB는 중첩 구조를 함께 읽는 서비스에 잘 맞을 수 있다.
- 강한 조인, 복잡한 트랜잭션, 엄격한 정규화가 핵심이면 RDBMS가 더 적합할 수 있다.

#### 핵심 정리

NoSQL은 하나의 제품군이라기보다 관계형 모델을 벗어난 여러 저장소를 포괄하는 말이다. 키-값, 문서, 와이드 컬럼, 그래프 모델처럼 데이터 접근 패턴에 맞춰 구조가 다르다.

문서 지향 DB는 JSON과 비슷한 문서 단위로 데이터를 저장해 스키마 변경이 잦고 관련 데이터를 함께 읽는 경우에 편하다. 예를 들어 사용자 설정이나 콘텐츠 메타데이터처럼 구조가 자주 변하는 데이터에 맞을 수 있다.

반대로 복잡한 다단계 트랜잭션, 강한 참조 무결성, 복잡한 JOIN과 정규화가 핵심인 업무라면 RDBMS가 더 자연스럽다. 선택 기준은 유행이 아니라 데이터 모델, 조회 패턴, 정합성 요구다.

#### 면접 답변

> NoSQL은 유연한 스키마와 수평 확장, 특정 접근 패턴에 최적화된 저장 모델이 필요할 때 선택합니다. 하지만 복잡한 트랜잭션과 강한 무결성, JOIN 중심 모델이 중요하면 관계형 데이터베이스가 더 적합합니다.

#### 헷갈리는 포인트
- NoSQL이 SQL을 전혀 쓰지 않는다는 뜻으로만 이해하면 부족하다.
- NoSQL도 제품에 따라 트랜잭션, 인덱스, 집계 지원 수준이 다르다.

#### 관련 퀴즈

- `db-t002`

### CAP, 최종 일관성, 복제
<!-- article: db-cap-eventual-replication -->

#### 한눈에 보기
- CAP은 분산 시스템에서 일관성, 가용성, 분단 내성을 동시에 완벽히 만족하기 어렵다는 관점이다.
- 최종 일관성은 시간이 지나면 복제본들이 같은 상태로 수렴하는 모델이다.
- Master-Slave 복제는 읽기 분산에 유리하지만 복제 지연과 장애 전환 문제가 있다.

#### 핵심 정리

분산 시스템에서는 네트워크 분단을 현실적으로 피하기 어렵다. 분단 상황에서 모든 요청에 응답할지, 최신 일관성을 강하게 지킬지 선택해야 한다. CAP은 이 선택의 긴장을 설명한다.

최종 일관성은 즉시 모든 노드가 같은 값을 보장하지는 않지만, 새로운 변경이 없다면 복제와 동기화를 통해 결국 같은 상태로 수렴한다는 모델이다. 가용성과 지연 시간을 중시하는 서비스에서 자주 등장한다.

Master-Slave 복제는 쓰기를 Master에 모으고 Slave 또는 Replica가 데이터를 복제해 읽기 부하를 나눈다. 그러나 비동기 복제에서는 읽기 복제본이 최신 값을 아직 반영하지 못하는 replication lag가 생길 수 있다.

#### 면접 답변

> CAP 이론은 분산 시스템에서 일관성, 가용성, 분단 내성을 동시에 완벽히 보장하기 어렵다는 관점입니다. 최종 일관성은 일시적으로 복제본 값이 달라도 시간이 지나면 수렴하는 모델이며, 복제 구조는 읽기 분산에 유리하지만 복제 지연을 고려해야 합니다.

#### 헷갈리는 포인트
- 복제본이 있다고 백업과 장애 복구 전략이 자동으로 완성되는 것은 아니다.
- 읽기 복제본을 쓸 때는 최신 데이터가 필요한 요청을 Master로 보내는 전략이 필요할 수 있다.

#### 관련 퀴즈
- `db-c017`
- `db-c047`
- `db-t007`
- `db-t019`
- `db-d009`
- `db-s011`

### MongoDB, Redis, 와이드 컬럼 스토어
<!-- article: db-nosql-models-mongodb-redis-cassandra -->

#### 한눈에 보기
- MongoDB는 문서 기반 저장소로 유연한 스키마와 샤딩을 지원한다.
- Redis는 빠른 키-값 접근과 캐시 용도에 자주 쓰인다.
- Cassandra 같은 와이드 컬럼 스토어는 높은 쓰기 처리량과 수평 확장에 강하다.

#### 핵심 정리

MongoDB는 BSON 문서로 데이터를 저장하고, 문서 단위로 함께 읽는 구조에 적합하다. 다만 관계가 많은 데이터를 무리하게 문서에 중복하면 동기화 비용과 불일치 위험이 커진다.

Redis를 캐시로 사용할 때 Cache-Aside는 필요한 순간 캐시에 올리고, Write-Through는 쓰기 시점에 캐시와 저장소를 함께 갱신한다. Write-Through는 쓰기 비용은 늘지만 캐시 누락이나 초기 읽기 지연을 줄이고 싶은 경우에 맞을 수 있다.

Cassandra 같은 와이드 컬럼 스토어는 대량 쓰기와 분산 확장에 강하다. 사용자 활동 이벤트처럼 초당 쓰기량이 매우 크고, 정해진 파티션 키로 빠르게 조회하는 패턴에 적합하다.

#### 면접 답변

> MongoDB는 유연한 문서 구조와 수평 확장에 유리하지만 관계가 많은 데이터에서는 중복과 일관성 관리가 중요합니다. Redis는 빠른 캐시에, Cassandra 같은 와이드 컬럼 스토어는 대량 쓰기와 확장성이 필요한 이벤트성 데이터에 적합합니다.

#### 헷갈리는 포인트
- MongoDB가 스키마가 없다는 말은 설계를 안 해도 된다는 뜻이 아니다.
- 캐시는 원본 저장소와의 일관성, 만료 정책, 장애 시 동작까지 설계해야 한다.

- `db-t009`
- `db-t014`

### 샤딩과 파티셔닝
<!-- article: db-sharding-partitioning -->

#### 한눈에 보기
- 샤딩은 데이터를 여러 DB 서버에 수평 분산하는 방식이다.
- 파티셔닝은 큰 테이블을 논리적 또는 물리적 구간으로 나눠 관리하는 방식이다.
- 샤드 키와 파티션 키는 조회 패턴과 데이터 분포를 기준으로 골라야 한다.

#### 핵심 정리

샤딩은 데이터를 여러 서버에 나눠 저장해 쓰기와 저장 공간 부하를 분산한다. 사용자 ID를 기준으로 샤딩하면 특정 사용자의 데이터가 어느 샤드에 있는지 계산하거나 메타데이터에서 찾아 해당 DB로 질의한다.

파티셔닝은 하나의 큰 테이블을 날짜나 범위 기준으로 나눠 관리하는 데 유용하다. 최근 7일 로그만 자주 조회하고 오래된 데이터는 보관만 한다면 날짜 파티션으로 조회 범위를 줄이고 오래된 파티션을 빠르게 아카이빙할 수 있다.

글로벌 서비스에서는 데이터 주권 요구 때문에 지역별 DB를 둘 수 있다. 이 경우 지역 내 데이터 저장 원칙을 지키면서, 글로벌 조회는 집계 저장소나 검색 인덱스, 제한된 복제 정책으로 분리하는 식의 설계가 필요하다.

#### 면접 답변

> 샤딩은 데이터를 여러 DB 서버에 수평 분산해 저장하는 방식이고, 파티셔닝은 큰 테이블을 범위나 해시 기준으로 나눠 관리하는 방식입니다. 둘 다 키 선택이 중요하며, 조회 패턴과 데이터 분포가 맞지 않으면 오히려 복잡도만 커질 수 있습니다.

#### 헷갈리는 포인트
- 샤딩은 운영 복잡도, 크로스 샤드 조인, 재분배 비용이 크다.
- 파티셔닝은 오래된 데이터 삭제나 특정 기간 조회에 강하지만 모든 쿼리를 자동으로 빠르게 하지는 않는다.

#### 관련 퀴즈

### 분산 트랜잭션과 이벤트 소싱
<!-- article: db-distributed-transaction-event-sourcing -->

#### 한눈에 보기
- 2PC는 여러 시스템의 커밋을 조율하지만 지연과 가용성 비용이 크다.
- 마이크로서비스에서는 Saga처럼 보상 트랜잭션을 쓰는 패턴도 고려한다.
- 이벤트 소싱은 현재 상태 대신 변경 이벤트를 저장해 상태를 재구성한다.

#### 핵심 정리

여러 서비스가 각자 다른 데이터베이스를 쓰면 하나의 로컬 트랜잭션으로 모든 변경을 원자적으로 묶기 어렵다. 2PC는 준비 단계와 커밋 단계로 참여자들을 조율해 원자성을 높이지만, 코디네이터 장애와 대기 시간, 잠금 유지 비용이 생긴다.

Saga는 긴 분산 업무를 여러 로컬 트랜잭션으로 나누고, 중간 실패 시 보상 작업으로 되돌리는 방식이다. 주문 생성 후 재고 차감이 실패하면 주문을 취소하는 이벤트를 발행하는 식이다.

이벤트 소싱은 마지막 상태만 저장하지 않고 상태 변화 이벤트를 순서대로 저장한다. 감사 추적과 상태 재생에 강하지만, 조회 모델 구성과 이벤트 스키마 진화가 복잡해질 수 있다.

#### 면접 답변

> 분산 DB나 마이크로서비스에서 2PC는 원자성을 높일 수 있지만 지연, 락 유지, 가용성 저하 비용이 큽니다. 그래서 서비스 경계가 분리된 환경에서는 Saga와 보상 트랜잭션, 이벤트 기반 일관성을 함께 고려합니다.

#### 헷갈리는 포인트
- 분산 환경에서 로컬 트랜잭션과 같은 감각으로 모든 것을 묶으려 하면 가용성이 크게 떨어질 수 있다.
- 이벤트 소싱은 감사와 재생에 좋지만 단순 CRUD 시스템에는 과한 선택일 수 있다.

#### 관련 퀴즈
- `db-t011`
- `db-t012`
- `db-s010`

## 쿼리 최적화와 실무 이슈
<!-- category: db-query-optimization -->

### OLTP와 OLAP 쿼리 분리
<!-- article: db-oltp-olap-separation -->

#### 한눈에 보기
- OLTP는 짧고 빈번한 거래 처리에 최적화된다.
- OLAP은 대량 데이터를 읽고 집계하는 분석 작업에 가깝다.
- 운영 DB에서 무거운 분석 쿼리를 직접 돌리면 서비스 트랜잭션에 영향을 줄 수 있다.

#### 핵심 정리

OLTP 시스템은 주문 생성, 결제, 재고 차감처럼 빠르고 정확한 단건 또는 소량 변경이 핵심이다. 인덱스와 트랜잭션, 락 전략도 짧은 응답 시간과 동시성에 맞춰 설계된다.

OLAP 쿼리는 긴 기간의 데이터를 스캔하고 그룹화, 정렬, 집계를 수행하는 경우가 많다. 운영 DB에서 이런 쿼리가 오래 실행되면 CPU, I/O, 버퍼 캐시를 많이 사용해 실제 서비스 요청이 느려질 수 있다.

해결책은 읽기 복제본, 데이터 웨어하우스, 배치 ETL, 집계 테이블 등으로 분석 부하를 분리하는 것이다. 핵심은 운영 트랜잭션 경로와 무거운 분석 경로를 같은 자원 위에서 경쟁시키지 않는 것이다.

#### 면접 답변

> OLTP는 짧고 빈번한 거래 처리, OLAP은 대량 데이터 분석에 초점이 있습니다. 운영 DB에서 무거운 분석 쿼리를 직접 실행하면 서비스 성능에 영향을 주므로 복제본, 집계 테이블, 데이터 웨어하우스 등으로 부하를 분리합니다.

#### 헷갈리는 포인트
- 읽기 복제본도 복제 지연이 있으므로 실시간 정합성이 필요한 분석에는 주의가 필요하다.
- 집계 테이블은 조회를 빠르게 하지만 갱신 전략을 별도로 설계해야 한다.

#### 관련 퀴즈
- `db-t010`

### 검색 쿼리와 LIKE 최적화
<!-- article: db-search-like-optimization -->

#### 한눈에 보기
- `LIKE 'abc%'`는 인덱스를 활용할 가능성이 있지만 `LIKE '%abc%'`는 어렵다.
- 부분 문자열 검색은 전문 검색 인덱스나 검색 엔진을 고려한다.
- 검색 요구가 복잡하면 RDBMS 기본 인덱스만으로 해결하기 어렵다.

#### 핵심 정리

B+트리 인덱스는 정렬된 키를 기준으로 탐색한다. 접두어가 고정된 `LIKE 'kim%'` 조건은 정렬 순서상 범위를 잡을 수 있지만, 앞쪽이 열린 `LIKE '%kim%'`은 어디서부터 찾을지 정하기 어렵다.

사용자 검색 기능에서 부분 문자열, 형태소, 오타 허용, 랭킹이 필요하다면 전문 검색 인덱스나 Elasticsearch 같은 검색 엔진을 고려할 수 있다. 단순 조건 검색과 검색 서비스는 요구가 다르다.

검색 최적화도 먼저 사용 패턴을 봐야 한다. 자동완성인지, 정확히 일치하는 코드 검색인지, 긴 본문 검색인지에 따라 인덱스와 저장 구조가 달라진다.

#### 면접 답변

> B+트리 인덱스는 앞부분이 고정된 조건에는 유리하지만 `LIKE '%검색어%'`처럼 앞쪽 와일드카드가 있으면 인덱스를 사용하기 어렵습니다. 이런 검색은 전문 검색 인덱스나 검색 엔진을 검토해야 합니다.

#### 헷갈리는 포인트
- 모든 검색 문제를 일반 인덱스 추가로 해결할 수는 없다.
- 검색 엔진을 도입하면 색인 동기화와 최종 일관성 문제도 함께 생긴다.

#### 관련 퀴즈
- `db-s008`

### 대량 쓰기와 삭제
<!-- article: db-bulk-write-delete -->

#### 한눈에 보기
- 초당 쓰기량이 매우 크면 단일 RDBMS 서버가 병목이 될 수 있다.
- 대량 DELETE는 락, 언두 로그, 복제 지연을 유발할 수 있다.
- 인덱스가 너무 많으면 INSERT와 UPDATE 성능이 크게 떨어진다.

#### 핵심 정리

사용자 활동 이벤트처럼 초당 수만 건 이상 쓰기가 필요한 데이터는 단일 MySQL 서버만으로 버티기 어렵다. 이 경우 샤딩, 파티셔닝, 큐 기반 버퍼링, 와이드 컬럼 스토어 같은 쓰기 확장 전략을 검토한다.

대량 DELETE는 삭제 대상 행에 락을 걸고 언두 로그를 크게 만들 수 있다. 오래된 데이터를 지우는 목적이라면 날짜 파티션을 사용해 파티션 단위로 제거하거나, 작은 배치로 나누어 처리하는 편이 안정적이다.

인덱스가 20개처럼 과도하게 많으면 한 행을 쓰는 데 모든 관련 인덱스를 갱신해야 한다. 실제 사용되지 않는 인덱스, 중복 인덱스, 낮은 선택도의 인덱스를 실행 계획과 모니터링으로 정리해야 한다.

#### 면접 답변

> 대량 쓰기는 샤딩, 파티셔닝, 큐, 쓰기 특화 저장소로 부하를 분산하고, 대량 삭제는 파티션 드롭이나 배치 삭제로 락과 언두 로그 부담을 줄입니다. 인덱스가 많으면 쓰기마다 갱신 비용이 늘어 사용하지 않는 인덱스를 제거해야 합니다.

#### 헷갈리는 포인트
- DELETE 한 번으로 끝나는 작업처럼 보여도 운영 DB에서는 긴 트랜잭션과 복제 지연을 만들 수 있다.
- 쓰기 성능 문제를 인스턴스 증설만으로 해결하려 하면 곧 구조적 병목을 다시 만난다.

#### 관련 퀴즈

- `db-s013`
- `db-s014`

### 커넥션 풀 크기
<!-- article: db-connection-pool-size -->

#### 한눈에 보기
- 커넥션 풀은 DB 연결 생성 비용을 줄이고 재사용하기 위한 장치다.
- 풀 크기가 너무 작으면 대기가 늘고, 너무 크면 DB가 과부하된다.
- 적정 크기는 애플리케이션 처리 시간, DB 처리량, 서버 리소스를 함께 보고 정한다.

#### 핵심 정리

커넥션 풀은 미리 만들어둔 DB 연결을 애플리케이션 요청들이 빌려 쓰게 한다. 매 요청마다 연결을 새로 만들지 않아도 되므로 지연과 리소스 비용을 줄인다.

하지만 풀 크기를 무조건 크게 잡으면 좋아지지 않는다. DB는 동시에 처리할 수 있는 쿼리 수가 한정되어 있고, 너무 많은 연결은 컨텍스트 스위칭, 락 경합, 메모리 사용량 증가를 만든다. 결과적으로 평균 응답 시간이 더 나빠질 수 있다.

적정 풀 크기는 부하 테스트로 찾는 것이 안전하다. 대기 시간, 활성 커넥션 수, DB CPU와 I/O, 쿼리 지연, 타임아웃을 함께 보며 병목이 애플리케이션인지 DB인지 구분해야 한다.

#### 면접 답변

> 커넥션 풀은 DB 연결을 재사용해 연결 생성 비용을 줄입니다. 하지만 풀을 너무 크게 잡으면 DB가 동시에 처리해야 할 작업이 늘어 경합과 메모리 사용량이 증가하므로, 부하 테스트와 모니터링으로 적정 크기를 정해야 합니다.

#### 헷갈리는 포인트
- 풀 대기가 있다고 항상 풀 크기를 늘리는 것이 답은 아니다. 느린 쿼리가 원인일 수 있다.
- 애플리케이션 서버 수가 늘면 전체 DB 연결 수도 함께 늘어난다.

#### 관련 퀴즈
- `db-t015`

## 면접 질문 정리
<!-- category: db-interview-summary -->

### 핵심 질문에 답하는 순서
<!-- article: db-interview-answer-flow -->

#### 한눈에 보기
- 정의를 먼저 말하고, 필요성이나 동작 원리를 이어서 설명한다.
- 장단점 질문은 어떤 상황에서 선택하는지까지 말해야 한다.
- 성능 질문은 실행 계획, 인덱스, 읽는 데이터 양, 트랜잭션 영향을 함께 본다.

#### 핵심 정리

데이터베이스 면접 답변은 개념 하나만 외우기보다 연결해서 말하는 것이 좋다. 예를 들어 인덱스 질문은 "조회 속도를 높인다"에서 끝내지 말고, B+트리, 범위 검색, 쓰기 비용, 선택도, 실행 계획까지 이어가면 좋다.

트랜잭션 질문은 ACID를 나열한 뒤 실제 예시를 붙이면 선명하다. 계좌 이체, 재고 차감, 주문 생성처럼 실패 시 무엇을 되돌려야 하는지 말하면 원자성과 일관성이 자연스럽게 설명된다.

확장 질문은 정답 하나보다 트레이드오프를 말하는 것이 중요하다. NoSQL, 샤딩, 복제, 2PC, Saga, 캐시는 모두 문제를 해결하지만 새로운 복잡도를 만든다. 어떤 요구 때문에 선택했고 어떤 비용을 감수하는지까지 답해야 한다.

#### 면접 답변

> 데이터베이스 답변은 정의, 필요한 이유, 동작 방식, 장단점, 실무 주의점 순서로 말하면 안정적입니다. 특히 인덱스와 트랜잭션, NoSQL 같은 주제는 성능과 정합성의 트레이드오프를 함께 설명해야 합니다.

#### 헷갈리는 포인트
- "항상", "무조건"이라는 표현은 데이터베이스 답변에서 위험하다.
- 실행 계획과 실제 데이터 분포를 확인하지 않은 성능 주장은 면접에서 쉽게 약해진다.
