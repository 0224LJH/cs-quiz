# Spring

이 문서는 CS 면접 준비를 위해 Spring과 JPA의 핵심 개념을 모바일에서 읽기 좋게 정리한 학습 자료입니다. 스프링 컨테이너, 의존성 주입, 웹 요청 처리, 트랜잭션, JPA 영속성 컨텍스트와 실무 이슈를 질문 단위로 빠르게 복습할 수 있게 구성했습니다.

## Spring 기본
<!-- category: spring-basics -->

### Spring과 Spring Boot의 역할
<!-- article: spring-core-and-boot -->

#### 한눈에 보기
- Spring은 객체 생성, 의존관계, 트랜잭션, 웹 요청 처리 같은 인프라 코드를 프레임워크가 맡아 주는 기반이다.
- Spring Boot는 Spring 애플리케이션을 더 빨리 실행할 수 있게 자동 설정, 내장 서버, starter 의존성을 제공한다.
- `@SpringBootApplication`은 설정 클래스, 자동 설정, 컴포넌트 스캔을 한 번에 켜는 진입점이다.

#### 핵심 정리

Spring의 핵심은 애플리케이션 객체를 직접 만들고 연결하는 부담을 컨테이너로 넘기는 것이다. 개발자는 "이 객체가 필요하다"는 의존관계와 역할을 선언하고, 컨테이너가 객체 생성과 연결을 담당한다.

Spring Boot는 여기에 실행 편의성을 더한다. 일반적인 웹 서버, JSON 변환기, 데이터베이스 연동, MVC 설정을 관례와 조건에 따라 자동으로 구성한다. 그래서 별도 WAS 배포나 긴 XML 설정 없이도 독립 실행 가능한 애플리케이션을 만들 수 있다.

`@SpringBootApplication`은 `@SpringBootConfiguration`, `@EnableAutoConfiguration`, `@ComponentScan`을 합친 복합 어노테이션이다. 여기서 `@SpringBootConfiguration`은 Spring 설정 클래스 역할을 하는 `@Configuration`의 특수화라고 보면 된다.

#### 면접 답변

> Spring은 객체 생성과 의존관계, 트랜잭션 같은 공통 인프라를 컨테이너와 추상화로 관리하는 프레임워크입니다. Spring Boot는 자동 설정, starter 의존성, 내장 서버를 제공해 Spring 애플리케이션을 빠르게 만들고 실행할 수 있게 해 줍니다.

#### 헷갈리는 포인트
- Spring Boot는 Spring을 대체하는 별도 기술이 아니라 Spring을 쉽게 쓰기 위한 확장이다.
- `@SpringBootApplication`의 컴포넌트 스캔은 기본적으로 해당 클래스가 위치한 패키지와 하위 패키지를 대상으로 한다.
- 내장 서버가 있어도 Spring MVC의 핵심 요청 처리는 여전히 `DispatcherServlet`이 담당한다.

### 자동 설정과 조건부 Bean 등록
<!-- article: spring-auto-configuration -->

#### 한눈에 보기
- 자동 설정은 클래스패스, 설정 값, 기존 Bean 존재 여부를 보고 필요한 Bean을 등록한다.
- 조건이 맞지 않으면 설정 클래스가 있어도 Bean이 등록되지 않을 수 있다.
- Spring Boot 3 계열에서는 자동 설정 목록을 `AutoConfiguration.imports` 방식으로 읽는다.

#### 핵심 정리

Spring Boot 자동 설정은 `@EnableAutoConfiguration`에서 시작된다. Boot는 미리 준비된 자동 설정 후보를 읽고, `@ConditionalOnClass`, `@ConditionalOnMissingBean`, `@ConditionalOnProperty` 같은 조건을 평가한 뒤 조건을 만족하는 Bean만 등록한다.

예를 들어 `feature.enabled=true`일 때만 등록되도록 만든 Bean은 해당 설정 값이 없거나 값이 다르면 등록되지 않는다. 이런 구조 덕분에 starter를 추가하는 것만으로도 기본 구성이 잡히지만, 사용자가 직접 등록한 Bean이 있으면 자동 설정이 물러나는 방식도 가능하다.

#### 면접 답변

> Spring Boot 자동 설정은 애플리케이션 시작 시 자동 설정 후보를 읽고 조건부 어노테이션을 평가해 필요한 Bean을 등록하는 기능입니다. 클래스패스와 프로퍼티, 기존 Bean 상태에 따라 설정이 달라지므로, 같은 코드라도 환경 설정에 따라 등록되는 Bean이 달라질 수 있습니다.

#### 헷갈리는 포인트
- 자동 설정은 무조건 모든 Bean을 등록하는 기능이 아니다.
- `@ConditionalOnProperty`는 프로퍼티 이름과 값이 정확히 맞아야 동작한다.
- 자동 설정 문제를 추적할 때는 어떤 조건이 실패했는지 보는 것이 핵심이다.

### PSA와 기술 추상화
<!-- article: spring-psa -->

#### 한눈에 보기
- PSA는 세부 기술이 달라도 같은 방식으로 접근하게 해 주는 Spring의 추상화다.
- 대표 예시는 트랜잭션, 캐시, 메시징, 데이터 접근이다.
- 추상화 덕분에 기술 교체 시 애플리케이션 코드 변경을 줄일 수 있다.

#### 핵심 정리

PSA(Portable Service Abstraction)는 특정 구현 기술에 직접 묶이지 않도록 공통 접근 방식을 제공한다. 예를 들어 `@Transactional`은 내부 구현이 JDBC, JPA, Hibernate 중 무엇인지와 관계없이 같은 방식으로 트랜잭션 경계를 선언하게 해 준다.

이 장점은 기술 선택의 유연성이다. 단, 추상화가 모든 차이를 없애 주지는 않는다. JPA와 MyBatis는 접근 방식이 다르고, RestTemplate, WebClient, Feign Client도 사용 모델이 다르다. Spring의 추상화는 공통 인터페이스를 제공하지만, 성능 특성과 제약은 여전히 이해해야 한다.

#### 면접 답변

> PSA는 트랜잭션이나 캐시처럼 여러 구현 기술이 존재하는 기능을 일관된 방식으로 사용할 수 있게 해 주는 Spring의 추상화입니다. 덕분에 구현 기술을 바꿔도 비즈니스 코드 변경을 줄일 수 있지만, 각 구현체의 성능 특성과 제약까지 사라지는 것은 아닙니다.

#### 헷갈리는 포인트
- PSA는 "아무 기술이나 완전히 똑같이 동작한다"는 뜻이 아니다.
- JPA와 MyBatis 선택은 추상화보다 개발 방식, SQL 제어 필요성, 도메인 모델 복잡도에 더 큰 영향을 받는다.
- 외부 HTTP 클라이언트도 동기, 비동기, 선언형 호출 방식에 따라 선택 기준이 달라진다.

#### 관련 퀴즈

- `spring-t014`
- `spring-t015`

## IoC와 DI
<!-- category: spring-ioc-di -->

### IoC와 IoC 컨테이너
<!-- article: spring-ioc-container -->

#### 한눈에 보기
- IoC는 객체 생성과 생명주기 제어권이 개발자 코드에서 프레임워크로 넘어가는 구조다.
- Spring의 대표 IoC 컨테이너는 `ApplicationContext`다.
- 컨테이너는 Bean 생성, 의존성 주입, 초기화, 소멸까지 관리한다.

#### 핵심 정리

전통적인 코드는 필요한 객체를 직접 `new`로 만들고, 어떤 구현체를 쓸지도 직접 결정한다. IoC에서는 이 제어 흐름이 바뀐다. 개발자는 Bean 정의와 의존관계를 선언하고, Spring 컨테이너가 객체를 만들고 연결한다.

`ApplicationContext`는 BeanFactory 기능에 국제화, 이벤트, 리소스 로딩, AOP 연동 같은 애플리케이션 기능을 더한 컨테이너다. 실무에서는 대부분 `ApplicationContext`를 Spring 컨테이너라고 부른다.

#### 면접 답변

> IoC는 객체 생성과 의존관계 관리의 제어권을 개발자가 아니라 Spring 컨테이너가 갖는 구조입니다. 컨테이너는 설정 정보를 바탕으로 Bean을 만들고, 필요한 의존성을 주입하며, 초기화와 소멸까지 관리합니다.

#### 헷갈리는 포인트
- IoC는 원칙이고, DI는 IoC를 구현하는 대표적인 방법이다.
- 컨테이너가 관리하는 객체만 Spring Bean이다.
- 직접 생성한 객체는 기본적으로 Spring의 생명주기와 AOP 적용 대상이 아니다.

#### 관련 퀴즈
- `spring-c001`
- `spring-c016`

### DI 방식과 생성자 주입
<!-- article: spring-di-constructor -->

#### 한눈에 보기
- DI는 필요한 의존 객체를 외부에서 주입받는 방식이다.
- 생성자 주입, 세터 주입, 필드 주입이 있다.
- 필수 의존성은 생성자 주입이 가장 권장된다.

#### 핵심 정리

생성자 주입은 객체가 만들어질 때 필요한 의존성을 모두 받는다. 그래서 필수 의존성이 빠진 상태로 객체가 존재하기 어렵고, `final` 필드를 사용할 수 있어 불변성을 유지하기 좋다. 테스트에서도 Spring 컨테이너 없이 순수 Java 객체처럼 생성해 검증할 수 있다.

세터 주입은 선택 의존성을 나중에 바꿀 수 있다는 장점이 있지만, 객체가 불완전한 상태로 존재할 수 있다. 필드 주입은 코드가 짧지만 테스트가 어렵고, `final` 사용이 불가능하며, 순환 의존성을 늦게 발견하기 쉽다.

#### 면접 답변

> DI는 객체가 직접 의존 객체를 생성하지 않고 외부에서 주입받는 방식입니다. 실무에서는 필수 의존성을 생성 시점에 보장하고 불변성을 유지할 수 있는 생성자 주입을 권장합니다. 필드 주입은 편하지만 테스트와 순환 의존성 감지 측면에서 불리합니다.

#### 헷갈리는 포인트
- 생성자 주입이라고 해서 항상 DIP를 지키는 것은 아니다.
- 구체 클래스에 직접 의존하면 주입 방식이 생성자여도 결합도가 높다.
- 순환 의존성은 설계 문제의 신호일 수 있으므로 지연 주입으로 덮기보다 역할 분리를 먼저 검토한다.

#### 관련 퀴즈
- `spring-c002`
- `spring-c017`
- `spring-t001`
- `spring-s002`

### DIP와 여러 구현체 주입
<!-- article: spring-dip-and-bean-choice -->

#### 한눈에 보기
- DIP는 상위 모듈과 하위 모듈이 모두 추상화에 의존해야 한다는 원칙이다.
- 인터페이스 타입으로 주입받아도 후보 Bean이 여러 개면 선택 기준이 필요하다.
- `@Primary`, `@Qualifier`, 명시적 Bean 이름으로 주입 대상을 좁힐 수 있다.

#### 핵심 정리

DIP를 지키려면 서비스가 구체 구현체가 아니라 인터페이스나 추상 타입에 의존해야 한다. 예를 들어 주문 서비스가 `CardPayment`를 직접 알기보다 `PaymentService` 인터페이스에 의존하면 결제 구현을 바꾸기 쉽다.

하지만 같은 인터페이스 구현체가 여러 개 등록되어 있으면 Spring은 어떤 Bean을 주입해야 할지 결정하지 못한다. 이때 `NoUniqueBeanDefinitionException`이 발생한다. 해결하려면 기본 후보를 `@Primary`로 지정하거나, 필요한 위치에서 `@Qualifier`로 특정 Bean을 명시한다.

#### 면접 답변

> DI는 DIP를 실현하기 좋은 방법이지만, 구체 클래스에 직접 의존하면 DIP를 지킨다고 보기 어렵습니다. 인터페이스에 의존하게 만들고, 구현체가 여러 개라면 `@Primary`나 `@Qualifier`로 선택 기준을 명확히 해야 합니다.

#### 헷갈리는 포인트
- 인터페이스 타입으로 주입받는 것과 Bean 선택 문제가 사라지는 것은 별개다.
- 계층 간 의존성 방향은 컨트롤러에서 서비스, 서비스에서 리포지토리로 흐르되, 구현 상세에는 덜 의존하도록 설계한다.
- `@Autowired` 자체가 DIP를 보장하지 않는다.

#### 관련 퀴즈
- `spring-c018`
- `spring-d001`

## Bean과 ApplicationContext
<!-- category: spring-bean-application-context -->

### Bean 등록과 컴포넌트 어노테이션
<!-- article: spring-bean-registration -->

#### 한눈에 보기
- Spring Bean은 Spring 컨테이너가 생성하고 관리하는 객체다.
- `@Component`, `@Service`, `@Repository`, `@Controller`는 컴포넌트 스캔 대상이다.
- `@Bean`은 개발자가 직접 생성 로직을 제공할 때 주로 사용한다.

#### 핵심 정리

`@Component`는 일반적인 컴포넌트 등록 어노테이션이다. `@Service`, `@Repository`, `@Controller`는 `@Component`를 기반으로 하되 계층 의미를 더 명확히 표현한다. 이 구분은 코드 가독성과 계층 책임을 드러내는 데 도움이 된다.

외부 라이브러리 객체처럼 클래스에 직접 어노테이션을 붙일 수 없거나, 생성 과정에 명시적 제어가 필요한 객체는 `@Configuration` 클래스 안에서 `@Bean` 메서드로 등록한다.

#### 면접 답변

> Spring Bean은 컨테이너가 관리하는 객체입니다. 일반 애플리케이션 클래스는 `@Component` 계열 어노테이션으로 등록하고, 생성 과정을 직접 제어해야 하는 객체는 `@Bean`으로 등록합니다. `@Service`, `@Repository`, `@Controller`는 모두 `@Component`의 특수화입니다.

#### 헷갈리는 포인트
- `@Service`가 자동으로 트랜잭션을 걸어 주는 것은 아니다.
- `@Repository`는 데이터 접근 계층이라는 의미와 예외 변환 역할을 가진다.
- 컴포넌트 스캔 범위 밖에 있는 클래스는 어노테이션이 있어도 Bean으로 등록되지 않는다.

#### 관련 퀴즈
- `spring-c004`
- `spring-c019`

### Bean 스코프와 싱글톤 주의점
<!-- article: spring-bean-scope -->

#### 한눈에 보기
- 기본 Bean 스코프는 싱글톤이다.
- 싱글톤 Bean은 여러 요청과 스레드가 공유할 수 있으므로 무상태로 설계해야 한다.
- 프로토타입은 조회할 때마다 새 인스턴스를 만들지만, 생성 이후 관리는 제한적이다.

#### 핵심 정리

Spring의 싱글톤은 `ApplicationContext` 안에서 Bean 정의 하나당 인스턴스 하나를 공유한다는 뜻이다. 웹 요청이 많아도 서비스 Bean은 보통 하나만 만들어져 여러 스레드가 함께 사용한다.

따라서 싱글톤 Bean에 요청별 상태를 필드로 저장하면 동시성 문제가 생긴다. 요청 데이터는 지역 변수, 메서드 파라미터, 별도 요청 스코프 객체로 다루는 것이 안전하다. 상태가 꼭 필요하면 스코프를 분리하거나 설계를 다시 검토해야 한다.

#### 면접 답변

> Spring Bean의 기본 스코프는 싱글톤입니다. 컨테이너 안에서 하나의 인스턴스를 공유하므로 서비스 Bean은 무상태로 설계해야 합니다. 요청마다 달라지는 값을 필드에 저장하면 멀티스레드 환경에서 데이터가 섞일 수 있습니다.

#### 헷갈리는 포인트
- Spring 싱글톤은 JVM 전체의 단 하나가 아니라 컨테이너 기준의 단 하나다.
- 프로토타입 Bean을 싱글톤 Bean에 직접 주입하면 매번 새로 만들어지는 효과를 얻지 못할 수 있다.
- 웹 스코프인 request, session은 웹 환경에서만 의미가 있다.

#### 관련 퀴즈
- `spring-c003`
- `spring-t012`

### Bean 생명주기와 초기화 콜백
<!-- article: spring-bean-lifecycle -->

#### 한눈에 보기
- Bean은 인스턴스 생성, 의존성 주입, 초기화 콜백, 사용, 소멸 콜백 순서로 관리된다.
- 초기화에는 `@PostConstruct`, `InitializingBean`, 커스텀 init 메서드가 쓰인다.
- 종료 시점에는 `@PreDestroy`나 destroy 메서드가 호출될 수 있다.

#### 핵심 정리

Spring 컨테이너는 Bean 인스턴스를 만든 뒤 필요한 의존성을 주입하고 초기화 콜백을 호출한다. 이 시점 이후 Bean은 애플리케이션 로직에서 사용된다. 컨테이너가 종료될 때는 소멸 콜백을 호출해 리소스를 정리할 기회를 준다.

초기화 로직은 생성자에 모두 넣기보다 의존성 주입이 끝난 뒤 실행되어야 하는 작업만 콜백으로 분리하는 편이 좋다. 예를 들어 외부 연결 확인, 캐시 준비, 설정 검증은 초기화 단계에서 처리할 수 있다.

#### 면접 답변

> Spring Bean은 생성 후 의존성 주입을 받고, 초기화 콜백을 거친 뒤 사용됩니다. 컨테이너 종료 시에는 소멸 콜백을 통해 리소스를 정리할 수 있습니다. 초기화는 의존성 주입이 끝난 뒤 실행된다는 점이 중요합니다.

#### 헷갈리는 포인트
- 생성자에서는 아직 세터 주입 의존성이 들어오지 않았을 수 있다.
- `@PostConstruct`는 의존성 주입이 끝난 뒤 실행된다.
- 모든 Bean이 소멸 콜백 대상처럼 보이더라도 프로토타입 Bean은 컨테이너가 완전한 종료 관리를 해 주지 않는다.

#### 관련 퀴즈

- `spring-o002`

### AOP와 프록시 동작
<!-- article: spring-aop-proxy -->

#### 한눈에 보기
- AOP는 로깅, 트랜잭션, 보안, 캐싱처럼 여러 계층에 반복되는 횡단 관심사를 분리한다.
- Spring AOP는 주로 프록시 기반으로 동작한다.
- 핵심 용어는 Aspect, Join Point, Advice, Pointcut, Target, Weaving이다.

#### 핵심 정리

AOP는 비즈니스 로직에 섞이기 쉬운 공통 기능을 별도 모듈로 분리한다. 예를 들어 실행 시간 측정 로직을 모든 서비스 메서드에 직접 넣으면 중복이 늘어난다. AOP를 사용하면 특정 Pointcut에 매칭되는 메서드 앞뒤로 Advice를 적용할 수 있다.

Spring AOP는 대상 Bean을 감싸는 프록시를 만들고, 외부 호출이 프록시를 통과할 때 Advice를 실행한다. 인터페이스 기반이면 JDK Dynamic Proxy를 사용할 수 있고, 클래스 기반이면 CGLIB 프록시를 사용할 수 있다.

#### 면접 답변

> AOP는 여러 모듈에 반복되는 횡단 관심사를 비즈니스 로직과 분리하는 방식입니다. Spring에서는 주로 프록시가 대상 Bean 호출을 가로채고, Pointcut에 매칭되면 Advice를 실행하는 방식으로 동작합니다.

#### 헷갈리는 포인트
- Join Point는 Advice를 적용할 수 있는 지점이고, Pointcut은 그중 실제 적용 대상을 고르는 조건이다.
- `@Around`는 대상 메서드 실행 전후와 예외를 모두 제어할 수 있다.
- Pointcut 표현식은 패키지, 클래스명, 메서드명, 파라미터 조건을 조합해 매칭한다.

### `@Transactional`과 AOP의 관계
<!-- article: spring-transactional-aop -->

#### 한눈에 보기
- `@Transactional`은 Spring AOP의 대표적인 활용 사례다.
- 프록시가 메서드 호출을 가로채 트랜잭션 시작, 커밋, 롤백을 처리한다.
- 내부 호출이나 private 메서드에는 프록시가 개입하지 못해 적용되지 않을 수 있다.

#### 핵심 정리

`@Transactional`이 붙은 메서드를 외부에서 호출하면 실제 객체가 아니라 트랜잭션 프록시가 먼저 호출을 받는다. 프록시는 트랜잭션을 시작하고 실제 메서드를 실행한 뒤, 정상 종료면 커밋하고 예외가 발생하면 롤백 여부를 판단한다.

문제는 같은 객체 내부에서 `this.save()`처럼 호출하는 경우다. 이 호출은 프록시를 거치지 않고 자기 자신의 메서드를 직접 호출하므로 트랜잭션 Advice가 실행되지 않는다. private 메서드도 프록시가 오버라이드하거나 인터셉트하기 어렵기 때문에 트랜잭션 경계로 적합하지 않다.

#### 면접 답변

> `@Transactional`은 프록시 기반 AOP로 구현됩니다. 외부 호출이 프록시를 통과할 때 트랜잭션을 시작하고 메서드 종료 후 커밋 또는 롤백합니다. 그래서 자기 내부 호출이나 private 메서드에는 트랜잭션이 기대대로 적용되지 않을 수 있습니다.

#### 헷갈리는 포인트
- 어노테이션이 붙어 있어도 호출 경로가 프록시를 통과해야 동작한다.
- 트랜잭션 경계는 보통 public 서비스 메서드에 둔다.
- AOP 적용 순서는 프록시 인터셉트, 트랜잭션 시작, 실제 로직 실행, 커밋 또는 롤백 순서다.

#### 관련 퀴즈
- `spring-c021`
- `spring-s001`
- `spring-d002`
- `spring-o003`

### 프록시 기반 기능의 공통 함정
<!-- article: spring-proxy-limitations -->

#### 한눈에 보기
- `@Cacheable`, `@Async`, `@Transactional`은 프록시 호출 경로의 영향을 받는다.
- 자기 내부 호출은 프록시를 거치지 않아 부가 기능이 적용되지 않을 수 있다.
- 비동기나 캐시 기능은 활성화 어노테이션과 실행 환경 설정도 함께 필요하다.

#### 핵심 정리

Spring의 여러 선언형 기능은 프록시로 구현된다. 캐시 메서드를 같은 클래스 내부에서 `this.getProduct()`처럼 호출하면 캐시 프록시가 끼어들지 못해 `@Cacheable`이 동작하지 않는다. 외부 Bean으로 분리하거나 프록시를 통해 호출되도록 구조를 바꿔야 한다.

`@Async`는 비동기 실행기를 사용하므로 `@EnableAsync`와 스레드 풀 설정을 확인해야 한다. 반환 타입, 예외 처리, 트랜잭션 경계가 동기 호출과 달라질 수 있다. `@Scheduled`도 스케줄링을 켜는 설정이 빠지면 메서드가 실행되지 않는다.

#### 면접 답변

> Spring의 선언형 기능은 대개 프록시를 통해 적용되므로, 같은 객체 내부 호출에서는 동작하지 않을 수 있습니다. `@Cacheable`, `@Async`, `@Transactional` 문제를 볼 때는 프록시를 통과하는 호출인지, 기능 활성화 설정이 있는지 먼저 확인합니다.

#### 헷갈리는 포인트
- `@Async` 메서드 안의 예외는 호출자에게 일반 동기 예외처럼 전달되지 않을 수 있다.
- 캐시는 키 설계, 무효화 전략, 데이터 최신성 문제가 함께 따라온다.
- `@Scheduled`는 메서드에 어노테이션만 붙인다고 항상 실행되는 것이 아니다.

#### 관련 퀴즈
- `spring-c014`
- `spring-t003`
- `spring-s015`
- `spring-d006`
- `spring-o009`

### DispatcherServlet 요청 처리 흐름
<!-- article: spring-mvc-dispatcher-servlet -->

#### 한눈에 보기
- `DispatcherServlet`은 모든 HTTP 요청을 받는 프론트 컨트롤러다.
- `HandlerMapping`이 처리할 컨트롤러를 찾고, `HandlerAdapter`가 실행한다.
- View 기반 응답은 `ViewResolver`가 뷰를 찾아 렌더링한다.

#### 핵심 정리

Spring MVC는 웹 요청을 한 지점에서 받아 공통 흐름으로 처리한다. 클라이언트 요청이 들어오면 `DispatcherServlet`이 먼저 받고, `HandlerMapping`을 통해 어떤 컨트롤러 메서드가 처리할지 찾는다.

그다음 `HandlerAdapter`가 컨트롤러 메서드를 실행한다. 컨트롤러가 모델과 뷰 이름을 반환하면 `ViewResolver`가 실제 뷰를 찾고 렌더링한다. REST API에서는 뷰 렌더링 대신 메시지 컨버터가 객체를 JSON 같은 응답 본문으로 변환한다.

#### 면접 답변

> Spring MVC에서 `DispatcherServlet`은 프론트 컨트롤러입니다. 요청을 받아 `HandlerMapping`으로 컨트롤러를 찾고, `HandlerAdapter`로 실행한 뒤, 뷰 응답이면 `ViewResolver`를 통해 렌더링하고 REST 응답이면 메시지 컨버터로 본문을 작성합니다.

#### 헷갈리는 포인트
- `DispatcherServlet`은 서블릿 필터보다 뒤에서 동작한다.
- 컨트롤러 메서드 자체를 직접 호출하는 것이 아니라 어댑터가 실행한다.
- REST 응답에서는 ViewResolver가 아니라 HTTP 메시지 컨버터가 핵심이다.

#### 관련 퀴즈
- `spring-c006`
- `spring-o001`
- `spring-o011`

### Controller, 요청 바인딩, 검증
<!-- article: spring-mvc-controller-binding -->

#### 한눈에 보기
- `@Controller`는 주로 뷰 이름을 반환하고, `@RestController`는 응답 본문을 직접 반환한다.
- `@RequestBody`는 요청 본문을 객체로 읽고, `@ResponseBody`는 반환 객체를 응답 본문으로 쓴다.
- `@PathVariable`은 경로 일부, `@RequestParam`은 쿼리 파라미터를 바인딩한다.

#### 핵심 정리

REST API에서는 보통 `@RestController`를 사용한다. 이는 `@Controller`와 `@ResponseBody`를 합친 형태라서 메서드 반환값이 뷰 이름이 아니라 HTTP 응답 본문으로 처리된다. JSON 요청 본문은 `@RequestBody`로 객체에 매핑한다.

경로 자체가 자원을 식별하면 `@PathVariable`이 어울리고, 검색 조건이나 옵션처럼 부가 조건이면 `@RequestParam`이 자연스럽다. 파라미터 이름은 기본적으로 대소문자를 구분하므로 `name`을 기대하는데 `Name`으로 보내면 바인딩이 되지 않을 수 있다.

검증에서는 `@Valid` 뒤에 `BindingResult`를 바로 두면 검증 실패를 직접 처리할 수 있다. 없으면 예외 흐름으로 넘어가 전역 예외 처리에서 다루는 경우가 많다.

#### 면접 답변

> `@RestController`는 `@Controller`에 `@ResponseBody`가 결합된 형태라 반환값을 응답 본문으로 씁니다. 요청 본문은 `@RequestBody`, 경로 변수는 `@PathVariable`, 쿼리 파라미터는 `@RequestParam`으로 받습니다. 검증 실패를 직접 처리하려면 `@Valid`와 `BindingResult`를 함께 사용합니다.

#### 헷갈리는 포인트
- `@RequestParam String name`은 기본적으로 `?name=...`을 찾는다.
- 같은 URL과 HTTP 메서드에 여러 핸들러가 매핑되면 애플리케이션 시작 시 충돌이 난다.
- `@ControllerAdvice`는 여러 컨트롤러의 예외 처리를 한곳으로 모으는 방식이다.

#### 관련 퀴즈

- `spring-c022`
- `spring-t011`
- `spring-s014`

### 예외 처리와 핸들러 매핑 문제
<!-- article: spring-mvc-exception-handler -->

#### 한눈에 보기
- `@ExceptionHandler`는 특정 컨트롤러나 advice에서 예외 응답을 만든다.
- `@ControllerAdvice`는 여러 컨트롤러에 공통 예외 처리를 적용한다.
- 동일한 URL 매핑이 중복되면 ambiguous handler 오류가 발생한다.

#### 핵심 정리

컨트롤러별로 예외 처리가 작고 독립적이면 `@ExceptionHandler`를 컨트롤러 안에 둘 수 있다. 여러 API에서 같은 오류 응답 형식을 사용해야 한다면 `@ControllerAdvice`에 모아 중복을 줄이는 편이 좋다.

Spring MVC는 애플리케이션 시작 시 URL, HTTP 메서드, consumes, produces 조건을 기준으로 핸들러 매핑을 구성한다. 같은 조건을 만족하는 핸들러가 둘 이상이면 어떤 메서드를 실행해야 할지 모호해져 `BeanCreationException`이나 ambiguous handler 오류가 발생할 수 있다.

#### 면접 답변

> 개별 컨트롤러에 국한된 예외는 `@ExceptionHandler`로 처리하고, 여러 컨트롤러에 공통으로 적용할 예외 응답은 `@ControllerAdvice`로 모으는 것이 좋습니다. 핸들러 매핑은 URL과 HTTP 메서드 조건이 겹치면 모호해지므로 중복 매핑을 피해야 합니다.

#### 헷갈리는 포인트
- `@ControllerAdvice`가 많아지면 적용 범위와 우선순위가 복잡해질 수 있다.
- 같은 경로라도 HTTP 메서드가 다르면 다른 매핑으로 볼 수 있다.
- 검증 실패, 비즈니스 예외, 시스템 예외는 응답 코드와 메시지 정책을 분리해서 설계한다.

#### 관련 퀴즈
- `spring-t013`
- `spring-d003`
- `spring-s014`

### Filter, Interceptor, AOP 선택 기준
<!-- article: spring-filter-interceptor-aop -->

#### 한눈에 보기
- Filter는 서블릿 스펙이며 `DispatcherServlet` 이전에 동작한다.
- Interceptor는 Spring MVC 영역에서 컨트롤러 전후를 가로챈다.
- AOP는 웹 요청 여부와 무관하게 Bean 메서드 단위로 적용할 수 있다.

#### 핵심 정리

Filter는 가장 앞단에서 요청과 응답 전체를 다룬다. 인코딩, CORS, 보안 필터, 로깅처럼 Spring MVC로 들어오기 전 처리해야 하는 작업에 적합하다.

Interceptor는 `DispatcherServlet` 이후 컨트롤러 호출 전후에 개입한다. 인증 정보 확인, 컨트롤러 공통 로깅, 요청별 권한 체크처럼 MVC 핸들러 정보를 활용해야 하는 작업에 잘 맞는다.

AOP는 서비스나 리포지토리 메서드에도 적용할 수 있다. 트랜잭션, 캐싱, 실행 시간 측정처럼 웹 계층에 묶이지 않는 공통 로직은 AOP가 더 자연스럽다.

#### 면접 답변

> 요청 전체를 서블릿 앞단에서 처리해야 하면 Filter, 컨트롤러 전후의 MVC 흐름을 다뤄야 하면 Interceptor, 웹과 무관하게 서비스 메서드의 공통 관심사를 처리해야 하면 AOP를 선택합니다.

#### 헷갈리는 포인트
- 실행 순서는 대략 Filter, DispatcherServlet, Interceptor, Controller, AOP 적용 메서드 흐름으로 이해하면 된다.
- 보안은 FilterChain 기반으로 처리되는 경우가 많다.
- AOP는 HTTP 요청 객체를 직접 다루는 용도보다 메서드 관심사 분리에 적합하다.

#### 관련 퀴즈
- `spring-t017`

### Spring MVC와 WebFlux 선택
<!-- article: spring-mvc-webflux -->

#### 한눈에 보기
- Spring MVC는 서블릿 기반의 동기 처리 모델이다.
- WebFlux는 리액티브 스트림 기반의 비동기 논블로킹 처리 모델이다.
- 대부분의 일반 CRUD 서비스는 MVC가 단순하고 운영 난도가 낮다.

#### 핵심 정리

Spring MVC는 요청당 스레드 모델을 기반으로 이해하기 쉽고 생태계가 넓다. 블로킹 JDBC, JPA, 전통적인 MVC 라이브러리와 잘 맞는다. 일반적인 백오피스, CRUD API, 트래픽 규모가 예측 가능한 서비스에서는 MVC가 실용적이다.

WebFlux는 외부 API 호출이 많거나 동시 연결이 매우 많고 논블로킹 I/O의 장점이 필요한 경우에 고려한다. 다만 JPA처럼 블로킹 기반 기술과 섞으면 장점이 줄어든다. 팀이 리액티브 프로그래밍 모델을 이해하고 운영할 준비가 있는지도 중요하다.

#### 면접 답변

> 일반적인 동기 CRUD 서비스와 JPA 중심 애플리케이션은 Spring MVC가 단순하고 적합합니다. 대량의 동시 연결이나 외부 I/O가 많아 논블로킹 처리의 이점이 큰 서비스라면 WebFlux를 고려합니다. 단, 블로킹 기술과 섞으면 WebFlux의 장점이 줄어듭니다.

#### 헷갈리는 포인트
- WebFlux를 쓴다고 자동으로 성능이 좋아지는 것은 아니다.
- 리액티브 체인 중간에 블로킹 호출이 들어가면 병목이 생긴다.
- 운영 복잡도와 디버깅 난도도 선택 기준에 포함해야 한다.

#### 관련 퀴즈

## 트랜잭션
<!-- category: spring-transaction -->

### `@Transactional` 기본 동작
<!-- article: spring-transaction-basic -->

#### 한눈에 보기
- 기본 전파 속성은 `REQUIRED`다.
- 정상 종료하면 커밋하고, 기본적으로 unchecked 예외가 발생하면 롤백한다.
- JPA 변경 작업은 트랜잭션 안에서 수행되어야 한다.

#### 핵심 정리

`@Transactional`은 메서드 진입 시 기존 트랜잭션이 있으면 참여하고, 없으면 새로 시작한다. 메서드가 정상 종료하면 커밋하고, `RuntimeException`이나 `Error`가 발생하면 기본적으로 롤백한다.

JPA에서는 트랜잭션 경계가 특히 중요하다. 영속성 컨텍스트의 변경 감지, flush, 커밋은 트랜잭션 흐름과 함께 움직인다. 읽기 전용 조회에는 `readOnly = true`를 사용해 의도를 표현하고 일부 최적화를 기대할 수 있지만, 데이터 변경이 필요한 로직에는 쓰면 안 된다.

#### 면접 답변

> `@Transactional`은 기본적으로 기존 트랜잭션이 있으면 참여하고 없으면 새로 만듭니다. 메서드가 정상 종료되면 커밋하고, unchecked 예외가 발생하면 롤백합니다. JPA 변경 작업은 트랜잭션 안에서 수행되어야 flush와 변경 감지가 정상적으로 동작합니다.

#### 헷갈리는 포인트
- checked 예외는 기본 롤백 대상이 아니다.
- `readOnly = true`는 읽기 의도를 나타내지만 모든 DB와 드라이버에서 같은 최적화를 보장하지는 않는다.
- 트랜잭션 어노테이션은 프록시 적용 조건을 만족해야 동작한다.

#### 관련 퀴즈
- `spring-c008`
- `spring-t005`

### 전파 속성과 `REQUIRES_NEW`
<!-- article: spring-transaction-propagation -->

#### 한눈에 보기
- 전파 속성은 이미 트랜잭션이 있을 때 새 메서드가 어떻게 참여할지 정한다.
- `REQUIRED`는 기존 트랜잭션에 참여하거나 새로 만든다.
- `REQUIRES_NEW`는 기존 트랜잭션을 잠시 중단하고 새 트랜잭션을 시작한다.

#### 핵심 정리

트랜잭션 전파는 서비스 메서드가 서로 호출될 때 중요하다. 기본값인 `REQUIRED`는 외부 트랜잭션이 있으면 같은 트랜잭션으로 묶인다. 내부 메서드에서 예외가 발생하면 외부 작업까지 함께 롤백될 수 있다.

`REQUIRES_NEW`는 외부 트랜잭션을 suspend하고 독립 트랜잭션을 만든다. 감사 로그처럼 본 작업이 롤백되어도 기록을 남겨야 하는 경우에 쓸 수 있다. 하지만 커넥션을 추가로 사용할 수 있고, 독립 커밋 때문에 데이터 일관성 설계를 더 신중히 해야 한다.

#### 면접 답변

> `REQUIRED`는 기존 트랜잭션에 참여하고 없으면 새로 만들지만, `REQUIRES_NEW`는 기존 트랜잭션을 중단한 뒤 완전히 새로운 트랜잭션을 시작합니다. 새 트랜잭션이 끝나면 외부 트랜잭션이 다시 재개됩니다.

#### 헷갈리는 포인트
- `REQUIRES_NEW`는 단순한 중첩 트랜잭션이 아니라 독립 트랜잭션이다.
- 외부 트랜잭션이 롤백되어도 내부 `REQUIRES_NEW` 커밋은 유지될 수 있다.
- 커넥션 풀 여유가 부족하면 `REQUIRES_NEW` 남용이 문제를 만들 수 있다.

#### 관련 퀴즈
- `spring-c012`
- `spring-o010`

### 예외 처리와 롤백 전략
<!-- article: spring-transaction-rollback -->

#### 한눈에 보기
- 기본 롤백 대상은 unchecked 예외다.
- checked 예외도 롤백하려면 `rollbackFor`를 명시해야 한다.
- 예외를 잡고 삼키면 프록시는 실패를 알지 못해 커밋할 수 있다.

#### 핵심 정리

`@Transactional` 메서드에서 `IOException` 같은 checked 예외가 발생해도 기본 설정만으로는 롤백되지 않는다. 이 경우 `rollbackFor = IOException.class`처럼 정책을 명시해야 한다.

또 하나의 흔한 문제는 catch 블록에서 예외를 로그만 남기고 삼키는 것이다. 메서드가 정상 종료되면 트랜잭션 프록시는 성공으로 판단하고 커밋할 수 있다. 롤백이 필요하면 예외를 다시 던지거나, 명시적으로 rollback-only 상태를 표시해야 한다.

#### 면접 답변

> Spring 트랜잭션은 기본적으로 unchecked 예외에서 롤백합니다. checked 예외도 롤백하려면 `rollbackFor`를 지정해야 합니다. 또한 예외를 catch해서 삼키면 트랜잭션 프록시가 실패를 알 수 없어 커밋될 수 있으므로 롤백 정책을 명확히 해야 합니다.

#### 헷갈리는 포인트
- 모든 예외가 자동 롤백되는 것은 아니다.
- 비즈니스 예외를 checked로 둘지 unchecked로 둘지는 팀의 오류 처리 정책과 연결된다.
- catch 후 보상 로직을 수행하더라도 롤백 여부를 별도로 결정해야 한다.

#### 관련 퀴즈

- `spring-s009`

## JPA와 ORM
<!-- category: spring-jpa-orm -->

### JPA와 ORM의 관계
<!-- article: spring-jpa-orm -->

#### 한눈에 보기
- JPA는 Java에서 ORM을 사용하기 위한 표준 명세다.
- Hibernate는 대표적인 JPA 구현체다.
- Spring Data JPA는 JPA 사용을 편하게 해 주는 Spring Data 모듈이다.

#### 핵심 정리

ORM(Object-Relational Mapping)은 객체와 관계형 데이터베이스 테이블을 매핑해 주는 기술이다. 객체는 참조와 메서드 중심이고, 관계형 데이터베이스는 테이블과 외래키 중심이라 둘 사이에는 패러다임 차이가 있다. ORM은 이 차이를 줄여 객체 중심으로 데이터 접근 코드를 작성하게 한다.

JPA는 ORM 사용 방법을 정의한 표준 인터페이스다. JPA 자체가 SQL을 실행하는 구현체는 아니며, Hibernate 같은 구현체가 실제 동작을 담당한다. Spring Data JPA는 Repository 추상화와 쿼리 메서드 등을 제공해 JPA 사용을 더 간결하게 만든다.

#### 면접 답변

> JPA는 Java ORM 표준 명세이고, Hibernate는 그 명세를 구현한 대표 구현체입니다. ORM은 객체와 테이블을 매핑해 패러다임 불일치를 줄이는 기술입니다. Spring Data JPA는 JPA를 더 편하게 쓰도록 Repository 추상화를 제공하는 모듈입니다.

#### 헷갈리는 포인트
- JPA, Hibernate, Spring Data JPA는 같은 층위의 기술이 아니다.
- ORM을 쓰면 SQL을 전혀 몰라도 된다는 뜻이 아니다.
- 복잡한 SQL 제어가 중요하면 MyBatis가 더 적합한 상황도 있다.

#### 관련 퀴즈
- `spring-c025`
- `spring-t014`

### Spring Data JPA 저장과 쿼리 메서드
<!-- article: spring-data-jpa-save-query-method -->

#### 한눈에 보기
- `repository.save()`는 새 엔티티면 `persist`, 준영속 엔티티면 `merge` 흐름으로 이어질 수 있다.
- Spring Data JPA 쿼리 메서드는 엔티티 필드명을 기준으로 파싱된다.
- DB 컬럼명이 아니라 엔티티 프로퍼티명이 기준이라는 점이 중요하다.

#### 핵심 정리

JPA에서 새 엔티티를 저장할 때는 영속성 컨텍스트에 등록하는 `persist`가 사용된다. 이미 식별자가 있는 준영속 엔티티를 다시 반영할 때는 `merge`가 사용될 수 있다. `merge`는 넘긴 객체 자체를 영속 상태로 바꾸는 것이 아니라, 병합된 새 영속 객체를 반환한다는 점을 주의해야 한다.

Spring Data JPA의 `findByNameAndAge` 같은 메서드명 쿼리는 DB 컬럼명이 아니라 엔티티 필드나 프로퍼티 이름을 기반으로 해석된다. 실제 컬럼명이 `username`이어도 엔티티 필드가 `name`이면 `findByName`이 맞고, 반대로 필드명이 다르면 메서드명이 실패할 수 있다.

#### 면접 답변

> Spring Data JPA의 `save`는 엔티티 상태에 따라 `persist`나 `merge` 흐름으로 처리됩니다. 준영속 엔티티를 병합할 때는 반환된 영속 엔티티를 사용해야 합니다. 쿼리 메서드는 DB 컬럼명이 아니라 엔티티 필드명을 기준으로 만들어야 합니다.

#### 헷갈리는 포인트
- `merge`에 전달한 준영속 객체가 그대로 영속 상태가 되는 것이 아니다.
- 엔티티 필드명과 DB 컬럼명은 `@Column`으로 다를 수 있다.
- 저장 순서와 flush 시점은 트랜잭션 커밋 흐름과 함께 이해해야 한다.

#### 관련 퀴즈
- `spring-o006`
- `spring-d004`
- `spring-d007`

### 운영 환경 설정과 DDL 자동 생성
<!-- article: spring-jpa-operational-settings -->

#### 한눈에 보기
- `ddl-auto=create-drop`은 시작 시 테이블을 만들고 종료 시 삭제할 수 있어 테스트 환경에 가깝다.
- 운영 환경에서 SQL 로그를 과도하게 켜면 성능과 보안 문제가 생길 수 있다.
- JPA 설정은 환경별 프로파일로 분리하는 것이 안전하다.

#### 핵심 정리

인메모리 H2와 `ddl-auto=create-drop` 조합은 테스트나 로컬 실습에서 편리하다. 애플리케이션 시작 시 스키마를 만들고 종료 시 제거하므로 반복 테스트에 적합하다. 하지만 운영 DB에 이런 설정이 들어가면 데이터 손실 위험이 매우 크다.

`show-sql`, `format_sql`, SQL DEBUG 로그는 개발 중 쿼리 확인에는 유용하다. 운영에서는 로그 양이 급증하고 민감 정보가 노출될 수 있으며, I/O 비용으로 성능이 떨어질 수 있다. 운영 관찰에는 필요한 수준의 로깅과 모니터링 도구를 선택해야 한다.

#### 면접 답변

> `ddl-auto=create-drop`이나 상세 SQL 로그는 개발과 테스트에는 편하지만 운영에서는 위험합니다. 스키마 자동 생성은 데이터 손실을 만들 수 있고, SQL DEBUG 로그는 성능 저하와 민감 정보 노출을 유발할 수 있으므로 환경별 설정을 분리해야 합니다.

#### 헷갈리는 포인트
- 테스트 편의 설정과 운영 안전 설정은 다르다.
- SQL 로그는 문제 분석에 좋지만 항상 켜 둘수록 좋은 설정은 아니다.
- 운영 스키마 변경은 마이그레이션 도구와 배포 절차로 관리하는 편이 안전하다.

#### 관련 퀴즈
- `spring-s004`
- `spring-d009`

## 영속성 컨텍스트
<!-- category: spring-persistence-context -->

### 영속성 컨텍스트와 1차 캐시
<!-- article: spring-persistence-context-first-cache -->

#### 한눈에 보기
- 영속성 컨텍스트는 엔티티를 관리하는 1차 캐시이자 작업 공간이다.
- 같은 트랜잭션 안에서 같은 엔티티를 다시 조회하면 같은 인스턴스를 반환할 수 있다.
- 쓰기 지연, 동일성 보장, 변경 감지, 지연 로딩이 여기에서 나온다.

#### 핵심 정리

영속성 컨텍스트는 애플리케이션과 데이터베이스 사이에서 엔티티를 관리하는 공간이다. `EntityManager`가 엔티티를 조회하거나 저장하면 해당 엔티티는 영속성 컨텍스트에 올라간다.

1차 캐시는 영속성 컨텍스트 내부의 캐시다. 같은 식별자의 엔티티를 다시 조회하면 DB를 다시 조회하지 않고 캐시된 객체를 반환할 수 있다. 이 덕분에 같은 트랜잭션 안에서는 동일성 보장이 가능하다.

#### 면접 답변

> 영속성 컨텍스트는 JPA가 엔티티를 관리하는 환경입니다. 내부 1차 캐시를 통해 같은 트랜잭션 안에서 동일한 엔티티의 동일성을 보장하고, 쓰기 지연, 변경 감지, 지연 로딩 같은 JPA 핵심 기능의 기반이 됩니다.

#### 헷갈리는 포인트
- 1차 캐시는 애플리케이션 전체 공유 캐시가 아니라 영속성 컨텍스트 범위의 캐시다.
- 2차 캐시는 여러 트랜잭션에서 공유될 수 있는 별도 캐시 영역이다.
- 영속성 컨텍스트 범위가 끝나면 관리되던 엔티티는 더 이상 자동 변경 감지 대상이 아니다.

#### 관련 퀴즈
- `spring-c026`

### 엔티티 생명주기와 준영속
<!-- article: spring-entity-lifecycle -->

#### 한눈에 보기
- 엔티티 상태는 비영속, 영속, 준영속, 삭제로 나눌 수 있다.
- 영속 상태는 영속성 컨텍스트가 관리하는 상태다.
- 준영속 상태는 한 번 관리되었지만 현재는 분리된 상태다.

#### 핵심 정리

비영속 상태는 단순히 `new`로 만든 객체처럼 영속성 컨텍스트와 관계없는 상태다. `persist`나 조회를 통해 영속성 컨텍스트에 들어가면 영속 상태가 된다. 영속 상태의 엔티티는 변경 감지와 1차 캐시의 대상이다.

`detach`, `clear`, `close` 등으로 컨텍스트에서 분리되면 준영속 상태가 된다. 준영속 엔티티를 수정해도 자동으로 DB에 반영되지 않는다. 다시 반영해야 한다면 `merge`를 사용할 수 있지만, 반환된 영속 엔티티를 사용해야 한다.

#### 면접 답변

> JPA 엔티티는 비영속, 영속, 준영속, 삭제 상태를 가집니다. 영속 상태일 때만 영속성 컨텍스트의 관리와 변경 감지 대상이 됩니다. 준영속 엔티티를 다시 반영하려면 `merge`를 사용할 수 있지만 반환된 영속 객체를 사용해야 합니다.

#### 헷갈리는 포인트
- `find`로 조회한 엔티티는 이미 영속 상태다.
- 준영속 엔티티의 필드 변경은 자동 업데이트되지 않는다.
- 삭제 상태는 트랜잭션 커밋 시점에 DELETE SQL로 반영된다.

#### 관련 퀴즈
- `spring-c027`
- `spring-d004`

### 더티 체킹과 flush
<!-- article: spring-dirty-checking-flush -->

#### 한눈에 보기
- 더티 체킹은 영속 엔티티의 변경을 자동 감지해 UPDATE로 반영하는 기능이다.
- JPA는 엔티티가 영속 상태가 될 때 스냅샷을 저장한다.
- 트랜잭션 커밋 전 flush 시점에 스냅샷과 현재 값을 비교한다.

#### 핵심 정리

JPA에서는 영속 상태의 엔티티를 조회한 뒤 setter로 값을 바꾸고 별도 update 메서드를 호출하지 않아도 된다. 영속성 컨텍스트가 최초 상태 스냅샷과 현재 상태를 비교하고, 변경이 있으면 flush 시점에 UPDATE SQL을 만든다.

기본적으로 JPA는 전체 필드를 업데이트하는 쿼리를 사용할 수 있다. 변경 컬럼만 업데이트하려면 Hibernate의 `@DynamicUpdate` 같은 기능을 고려할 수 있지만, 모든 곳에 쓰는 기본값으로 삼기보다 쿼리 재사용성과 성능 특성을 함께 봐야 한다.

#### 면접 답변

> 더티 체킹은 영속성 컨텍스트가 관리하는 엔티티의 변경을 감지해 트랜잭션 커밋 전 flush 시점에 UPDATE SQL로 반영하는 기능입니다. 영속 상태가 아니면 변경 감지가 동작하지 않습니다.

#### 헷갈리는 포인트
- `save()`를 호출하지 않아도 영속 엔티티 변경은 반영될 수 있다.
- 트랜잭션 밖이거나 준영속 상태라면 더티 체킹이 동작하지 않는다.
- flush는 SQL을 DB에 보내는 것이고, commit은 트랜잭션을 확정하는 것이다.

#### 관련 퀴즈
- `spring-s008`
- `spring-o012`

## 연관관계와 로딩 전략
<!-- category: spring-association-loading -->

### 연관관계 매핑과 주인
<!-- article: spring-association-owner -->

#### 한눈에 보기
- JPA 연관관계는 `@ManyToOne`, `@OneToMany`, `@OneToOne`, `@ManyToMany`로 표현한다.
- 양방향 연관관계에서는 외래키를 관리하는 쪽이 주인이다.
- 주인이 아닌 쪽은 `mappedBy`로 주인을 가리키며 읽기 관점으로 사용한다.

#### 핵심 정리

관계형 데이터베이스에서는 외래키 하나가 관계를 결정한다. JPA에서 양방향 객체 참조를 만들더라도 실제 외래키를 관리하는 쪽은 하나여야 한다. 보통 외래키가 있는 다대일 쪽, 예를 들어 `Member.team`이 연관관계의 주인이 된다.

주인이 아닌 컬렉션 쪽은 `mappedBy`로 "이 관계는 상대 필드가 관리한다"고 표시한다. 실무에서는 다대다는 중간 테이블에 추가 컬럼이 생기기 쉽기 때문에, 중간 테이블을 엔티티로 승격해 다대일과 일대다로 푸는 편이 안전하다.

#### 면접 답변

> JPA 연관관계의 주인은 외래키를 관리하는 쪽입니다. 양방향 관계에서는 한쪽만 외래키 등록과 수정을 담당해야 하며, 반대쪽은 `mappedBy`로 주인을 지정합니다. 보통 외래키가 있는 다대일 쪽을 주인으로 둡니다.

#### 헷갈리는 포인트
- 객체 참조가 두 방향이어도 DB 외래키는 보통 한쪽 테이블에만 있다.
- 주인이 아닌 쪽의 변경만으로는 외래키가 업데이트되지 않는다.
- `@ManyToMany`는 실무에서 중간 엔티티로 풀어내는 경우가 많다.

#### 관련 퀴즈
- `spring-c030`

### 즉시 로딩과 지연 로딩
<!-- article: spring-eager-lazy-loading -->

#### 한눈에 보기
- 즉시 로딩은 엔티티 조회 시 연관 엔티티도 함께 조회한다.
- 지연 로딩은 실제 연관 객체를 사용할 때 조회한다.
- 실무에서는 기본적으로 LAZY로 두고 필요한 조회에서 명시적으로 fetch 전략을 정하는 편이 좋다.

#### 핵심 정리

EAGER는 편해 보이지만 예상하지 못한 조인과 추가 쿼리를 만들 수 있다. 특정 API에서는 필요 없는 연관 데이터까지 매번 로딩되어 성능을 떨어뜨릴 수 있고, 여러 관계가 얽히면 조회 계획을 예측하기 어려워진다.

LAZY는 프록시를 두고 실제 접근 시점에 조회한다. 필요한 데이터만 가져올 수 있어 유연하지만, 영속성 컨텍스트가 닫힌 뒤 지연 로딩 프록시에 접근하면 `LazyInitializationException`이 발생할 수 있다. 따라서 API 응답에 필요한 데이터는 트랜잭션 안에서 fetch join, EntityGraph, DTO 조회 등으로 명시적으로 준비하는 편이 좋다.

#### 면접 답변

> 즉시 로딩은 엔티티 조회 시 연관 엔티티를 함께 가져오고, 지연 로딩은 실제 사용할 때 가져옵니다. 실무에서는 예상치 못한 SQL을 줄이기 위해 기본적으로 LAZY를 사용하고, 필요한 화면이나 API에서 fetch join이나 EntityGraph로 명시적으로 로딩합니다.

#### 헷갈리는 포인트
- LAZY가 항상 빠른 것이 아니라 조회 시점을 늦추는 전략이다.
- 트랜잭션 밖에서 LAZY 프록시에 접근하면 예외가 날 수 있다.
- EAGER는 N+1 문제를 완전히 막아 주지 못한다.

#### 관련 퀴즈
- `spring-t004`
- `spring-s003`

### N+1 문제와 Fetch Join
<!-- article: spring-n-plus-one-fetch-join -->

#### 한눈에 보기
- N+1은 목록 1번 조회 후 각 행의 연관 데이터를 N번 추가 조회하는 문제다.
- fetch join은 한 번의 JPQL로 연관 엔티티까지 함께 조회한다.
- `@EntityGraph`도 연관 로딩을 지정하는 데 사용할 수 있다.

#### 핵심 정리

N+1 문제는 연관관계가 있는 엔티티 목록을 조회한 뒤 반복문에서 연관 데이터를 접근할 때 자주 발생한다. 예를 들어 사용자 목록을 한 번 조회하고 각 사용자의 주문 컬렉션을 접근하면 사용자 수만큼 추가 쿼리가 나갈 수 있다.

가장 흔한 해결책은 fetch join이다. JPQL에 `join fetch`를 사용해 필요한 연관 엔티티를 한 번에 가져온다. 간단한 관계 로딩은 `@EntityGraph`로 선언할 수도 있다. 다만 컬렉션 fetch join은 페이징이 어렵고, 둘 이상의 컬렉션을 동시에 fetch join하면 문제가 생길 수 있다.

#### 면접 답변

> N+1 문제는 목록 조회 1번 뒤에 각 엔티티의 연관 데이터를 조회하는 쿼리가 N번 추가로 발생하는 문제입니다. fetch join이나 `@EntityGraph`, batch size 설정으로 해결할 수 있으며, 컬렉션 fetch join에서는 페이징과 중복 문제를 조심해야 합니다.

#### 헷갈리는 포인트
- fetch join은 조회 성능을 개선하지만 모든 관계를 무조건 가져오라는 뜻은 아니다.
- `@EntityGraph`는 재사용성이 좋고 간단한 로딩에 적합하지만 복잡한 조건은 fetch join이나 QueryDSL이 더 낫다.
- 컬렉션 fetch join 결과는 중복 행이 생길 수 있어 `distinct`가 필요할 수 있다.

#### 관련 퀴즈
- `spring-c015`
- `spring-s005`

## JPA 실무 이슈
<!-- category: spring-jpa-practical -->

### 영속성 전이와 고아 객체 제거
<!-- article: spring-cascade-orphan-removal -->

#### 한눈에 보기
- Cascade는 부모의 영속 상태 변화를 자식에게 전파한다.
- `orphanRemoval`은 부모와 관계가 끊긴 자식을 삭제한다.
- 참조 주체가 하나인 강한 소유 관계에서만 신중하게 사용한다.

#### 핵심 정리

Cascade는 부모 엔티티를 저장할 때 자식 엔티티도 함께 저장하거나, 부모 삭제 시 자식 삭제를 함께 처리하게 해 준다. 부모가 자식의 생명주기를 정말 소유하는 구조라면 코드가 간결해진다.

`orphanRemoval = true`는 컬렉션에서 자식 엔티티를 제거하는 것만으로 DELETE 대상이 되게 한다. 이 기능은 편하지만 다른 곳에서도 참조하는 엔티티에 쓰면 의도치 않은 삭제가 발생할 수 있다. 게시글과 첨부파일처럼 소유 관계가 명확할 때 적합하다.

#### 면접 답변

> Cascade는 부모 엔티티의 저장, 삭제 같은 상태 변화를 자식에게 전파하는 기능입니다. 고아 객체 제거는 부모와의 연관관계가 끊긴 자식을 자동 삭제합니다. 둘 다 자식 생명주기를 부모가 독점적으로 관리할 때만 신중하게 사용해야 합니다.

#### 헷갈리는 포인트
- Cascade는 연관관계 매핑을 대신해 주는 기능이 아니다.
- `orphanRemoval`은 컬렉션에서 제거하는 동작만으로 삭제를 유발할 수 있다.
- 공유 참조가 있는 엔티티에는 고아 객체 제거가 위험하다.

#### 관련 퀴즈
- `spring-t018`

### JPQL과 QueryDSL
<!-- article: spring-jpql-querydsl -->

#### 한눈에 보기
- JPQL은 테이블이 아니라 엔티티 객체를 대상으로 하는 쿼리 언어다.
- JPQL은 문자열 기반이라 런타임 오류 가능성이 있다.
- QueryDSL은 타입 안전성과 동적 쿼리 작성 편의성을 제공한다.

#### 핵심 정리

JPQL은 SQL과 비슷해 보이지만 테이블명과 컬럼명이 아니라 엔티티명과 필드명을 사용한다. JPA 구현체가 JPQL을 실제 DB 방언에 맞는 SQL로 변환한다. 덕분에 DB 독립성을 얻지만, 문자열 오타나 필드명 변경은 컴파일 시점에 잡기 어렵다.

QueryDSL은 Q타입을 기반으로 Java 코드처럼 쿼리를 만든다. IDE 자동완성과 컴파일 검사를 받을 수 있고, 조건을 조합하는 동적 쿼리에 강하다. 다만 설정과 학습 비용이 추가되고, 단순 쿼리까지 모두 QueryDSL로 작성할 필요는 없다.

#### 면접 답변

> JPQL은 엔티티 객체를 대상으로 하는 객체 지향 쿼리 언어이고, SQL은 DB 테이블을 대상으로 합니다. QueryDSL은 JPQL의 문자열 기반 단점을 줄여 타입 안전하게 쿼리를 작성하고 동적 조건을 조합하기 쉽게 해 줍니다.

#### 헷갈리는 포인트
- JPQL의 `Member`는 테이블명이 아니라 엔티티명이다.
- QueryDSL을 써도 결국 실행되는 것은 SQL이다.
- 단순 CRUD는 Repository 메서드나 JPQL로 충분할 수 있다.

#### 관련 퀴즈
- `spring-c028`

### 낙관적 락과 비관적 락
<!-- article: spring-jpa-locking -->

#### 한눈에 보기
- 낙관적 락은 충돌이 드물다고 보고 버전으로 충돌을 감지한다.
- 비관적 락은 충돌 가능성이 높다고 보고 DB 락을 먼저 잡는다.
- 동시성, 대기 시간, 충돌 빈도를 기준으로 선택한다.

#### 핵심 정리

낙관적 락은 `@Version` 필드를 두고 업데이트 시 버전이 맞는지 확인한다. 동시에 수정이 일어나면 나중에 커밋하는 쪽에서 충돌 예외가 발생할 수 있다. 대기 시간이 짧고 락 경합이 적은 서비스에 적합하다.

비관적 락은 DB의 `SELECT FOR UPDATE` 같은 락을 사용해 다른 트랜잭션의 수정을 막는다. 충돌 가능성이 높거나 중복 처리 비용이 큰 경우 고려하지만, 락 대기와 데드락 위험이 커질 수 있다.

#### 면접 답변

> 낙관적 락은 충돌이 적다고 가정하고 버전 필드로 커밋 시 충돌을 감지합니다. 비관적 락은 충돌이 많다고 보고 DB 락을 먼저 획득합니다. 충돌 빈도가 낮으면 낙관적 락, 재시도 비용이 크고 충돌이 잦으면 비관적 락을 고려합니다.

#### 헷갈리는 포인트
- 낙관적 락은 DB 락을 오래 잡는 방식이 아니다.
- 비관적 락은 정확성이 필요한 대신 처리량과 대기 시간에 영향을 줄 수 있다.
- 트랜잭션 격리 수준과 락 전략은 함께 봐야 한다.

#### 관련 퀴즈
- `spring-c029`

### Spring Security, 세션, JWT
<!-- article: spring-security-session-jwt -->

#### 한눈에 보기
- Spring Security는 FilterChain 기반으로 인증과 인가를 처리한다.
- 세션 기반 인증은 서버가 로그인 상태를 저장한다.
- JWT는 토큰에 인증 정보를 담아 stateless 구조를 만들기 쉽다.

#### 핵심 정리

Spring Security에서는 요청이 `SecurityFilterChain`을 지나며 인증 정보가 추출되고, `AuthenticationManager`가 검증한 뒤 `SecurityContextHolder`에 인증 결과가 저장된다. 이후 인가 규칙에 따라 URL 접근이 허용되거나 거부된다.

세션 방식은 서버에서 상태를 관리하므로 토큰 탈취 시 대응과 강제 로그아웃이 비교적 쉽다. JWT는 서버 확장성이 좋고 모바일, API 환경에 편하지만 토큰 폐기와 만료 정책을 신중히 설계해야 한다. CSRF를 무조건 끄고 모든 요청을 허용하는 설정은 운영 보안상 위험하다.

#### 면접 답변

> Spring Security는 FilterChain에서 인증과 인가를 처리합니다. 세션 방식은 서버가 로그인 상태를 관리하고, JWT는 토큰 기반으로 stateless 구성이 쉽습니다. JWT는 확장성 장점이 있지만 토큰 탈취, 만료, 폐기 전략을 반드시 함께 설계해야 합니다.

#### 헷갈리는 포인트
- `permitAll()`은 인증 없이 접근을 허용한다는 뜻이다.
- `hasRole("ADMIN")`은 일반적으로 `ROLE_ADMIN` 권한과 연결된다.
- CSRF 비활성화와 전체 요청 허용을 함께 두면 보호 장치가 크게 약해진다.

### 캐시와 이벤트 처리
<!-- article: spring-cache-event -->

#### 한눈에 보기
- `@Cacheable`은 메서드 결과를 캐시에 저장하고 같은 키 요청에서 재사용한다.
- 캐시는 응답 속도를 높이지만 데이터 최신성과 무효화 전략이 필요하다.
- Spring 이벤트는 애플리케이션 내부 결합을 줄이고 후속 작업을 분리할 때 쓴다.

#### 핵심 정리

캐시는 반복 조회 비용이 큰 데이터를 빠르게 반환하는 데 유용하다. 하지만 데이터가 바뀌었을 때 언제 캐시를 지울지, 어떤 키를 쓸지, TTL을 어떻게 둘지 정하지 않으면 오래된 데이터를 보여줄 수 있다. `@Cacheable`도 프록시 기반이므로 내부 호출 문제를 주의해야 한다.

`ApplicationEventPublisher`는 같은 애플리케이션 안에서 이벤트를 발행하고 리스너가 후속 작업을 처리하게 한다. 트랜잭션 커밋 이후에만 이벤트를 처리해야 한다면 `@TransactionalEventListener`가 중요하다. Kafka나 RabbitMQ 같은 브로커는 프로세스 간 비동기 통신과 내구성이 필요할 때 선택한다.

#### 면접 답변

> 캐시는 반복 조회 성능을 높이지만 키 설계와 무효화 전략이 핵심입니다. Spring 이벤트는 애플리케이션 내부 후속 작업을 느슨하게 분리할 때 좋고, 트랜잭션 완료 이후 처리가 필요하면 `@TransactionalEventListener`를 고려합니다. 서비스 간 내구성 있는 비동기는 메시지 브로커가 더 적합합니다.

#### 헷갈리는 포인트
- 캐시 히트 시 실제 메서드는 실행되지 않는다.
- 이벤트 리스너가 트랜잭션 커밋 전 실행되면 롤백된 데이터에 기반한 후속 작업이 생길 수 있다.
- 내부 호출은 캐시 프록시를 통과하지 않는다.

#### 관련 퀴즈

### 비동기, 스케줄링, 배치
<!-- article: spring-async-scheduling-batch -->

#### 한눈에 보기
- `@Async`는 별도 스레드에서 메서드를 실행하게 한다.
- `@Scheduled`는 주기적 작업 실행에 사용하지만 스케줄링 활성화가 필요하다.
- Spring Batch는 대량 데이터 처리, Quartz는 시간 기반 작업 스케줄링에 강하다.

#### 핵심 정리

`@Async`는 응답 시간을 줄이거나 독립적인 후속 작업을 분리할 때 쓴다. 하지만 스레드 풀 크기, 예외 처리, 트랜잭션 경계가 중요하다. 비동기 메서드에서 새 트랜잭션이 필요한지, 호출자가 결과를 기다려야 하는지도 결정해야 한다.

Spring Batch는 reader, processor, writer를 구성하고 chunk 단위로 읽고 처리하고 쓰는 대량 처리 프레임워크다. 재시작, skip, retry, 작업 이력 관리가 중요할 때 적합하다. Quartz는 특정 시간이나 cron 조건에 맞춘 스케줄 실행이 핵심인 작업에 더 잘 맞는다.

#### 면접 답변

> `@Async`는 별도 스레드에서 작업을 실행하므로 스레드 풀과 예외 처리 전략이 필요합니다. 주기 실행은 `@Scheduled`나 Quartz를 쓰고, 대량 데이터를 안정적으로 읽고 처리하고 쓰는 작업은 Spring Batch가 적합합니다.

#### 헷갈리는 포인트
- 비동기는 트랜잭션 컨텍스트가 호출자와 자연스럽게 공유된다고 가정하면 안 된다.
- `@Scheduled`는 기능 활성화 설정이 빠지면 실행되지 않는다.
- Batch는 단순 예약 실행이 아니라 대량 처리와 재시작 가능성이 핵심이다.

#### 관련 퀴즈

### 외부 호출 클라이언트와 서비스 경계
<!-- article: spring-http-client-and-service-boundary -->

#### 한눈에 보기
- RestTemplate은 전통적인 동기 HTTP 클라이언트다.
- WebClient는 비동기 논블로킹 호출에 적합하다.
- Feign Client는 인터페이스 선언형 클라이언트로 서비스 간 호출 코드를 줄인다.

#### 핵심 정리

외부 API 호출이 단순하고 동기 흐름이면 RestTemplate 계열 방식이 이해하기 쉽다. 다만 새 프로젝트에서는 WebClient나 선언형 클라이언트를 검토하는 경우가 많다. WebClient는 논블로킹 호출과 스트리밍에 유리하지만, 전체 호출 체인이 비동기 모델과 잘 맞아야 장점이 살아난다.

Feign Client는 인터페이스와 어노테이션으로 외부 API 호출을 선언할 수 있어 마이크로서비스 환경에서 코드가 간결해진다. 대신 장애 전파, 타임아웃, 재시도, 서킷 브레이커 같은 운영 정책을 함께 설계해야 한다.

단일 애플리케이션에서 마이크로서비스로 전환할 때는 단순히 프로젝트를 나누는 것보다 데이터 소유권, 배포 독립성, 장애 격리, 팀 구조가 준비되어 있는지가 더 중요하다.

#### 면접 답변

> 동기 호출이 단순하면 RestTemplate 방식이 쉽고, 논블로킹 I/O가 필요하면 WebClient를 고려합니다. 서비스 간 선언형 호출은 Feign Client가 편리합니다. 마이크로서비스 전환은 기술 선택보다 데이터 소유권, 배포 독립성, 장애 격리 같은 경계를 먼저 봐야 합니다.

#### 헷갈리는 포인트
- WebClient를 사용해도 중간에 블로킹 호출을 섞으면 장점이 줄어든다.
- Feign은 호출 코드를 줄여 주지만 네트워크 장애를 없애 주지는 않는다.
- 마이크로서비스는 규모가 커질수록 관측성, 배포, 데이터 정합성 비용도 커진다.

#### 관련 퀴즈
- `spring-t015`
- `spring-t007`

### Spring 핵심 질문 빠른 답변
<!-- article: spring-interview-core-review -->

#### 한눈에 보기
- IoC, DI, Bean, AOP, MVC는 Spring 면접의 기본 축이다.
- 답변은 "정의 → 왜 쓰는지 → 주의점" 순서로 말하면 안정적이다.
- 프록시 기반 기능은 내부 호출 문제를 함께 언급하면 좋다.

#### 핵심 정리

Spring 질문은 대부분 컨테이너와 프록시로 연결된다. IoC는 제어권을 컨테이너로 넘기는 원칙이고, DI는 의존성을 외부에서 주입받는 방식이다. Bean은 컨테이너가 관리하는 객체이며, 기본 스코프는 싱글톤이다.

AOP는 횡단 관심사를 분리하고, Spring에서는 프록시를 통해 적용된다. 그래서 `@Transactional`, `@Cacheable`, `@Async` 같은 기능은 호출이 프록시를 통과하는지 확인해야 한다. MVC는 `DispatcherServlet`을 중심으로 요청을 컨트롤러에 연결한다.

#### 면접 답변

> Spring의 핵심은 컨테이너가 Bean 생명주기와 의존관계를 관리하고, 프록시와 추상화를 통해 트랜잭션, 캐시, 웹 요청 처리 같은 공통 기능을 일관되게 제공한다는 점입니다. 면접에서는 정의와 장점뿐 아니라 싱글톤 상태 공유, 내부 호출, 자동 설정 조건 같은 주의점까지 함께 말하는 것이 좋습니다.

#### 헷갈리는 포인트
- "Spring이 해 준다"에서 멈추지 말고 컨테이너, 프록시, 조건부 설정 중 무엇이 해 주는지 구체화한다.
- 쉬운 질문일수록 단점이나 주의점을 한 문장 붙이면 답변이 탄탄해진다.
- Bean 등록, 요청 매핑, 트랜잭션 적용은 모두 시작 시점 또는 호출 경로 문제로 이어질 수 있다.

### JPA 핵심 질문 빠른 답변
<!-- article: spring-interview-jpa-review -->

#### 한눈에 보기
- JPA 질문은 ORM, 영속성 컨텍스트, 연관관계, 로딩 전략, 트랜잭션으로 이어진다.
- N+1, LazyInitializationException, 더티 체킹은 실무형 단골 질문이다.
- 엔티티 상태와 트랜잭션 경계를 함께 설명해야 한다.

#### 핵심 정리

JPA는 ORM 표준 명세이고 Hibernate 같은 구현체가 실제 동작을 담당한다. 영속성 컨텍스트는 1차 캐시, 동일성 보장, 쓰기 지연, 변경 감지의 기반이다. 엔티티가 영속 상태일 때만 더티 체킹이 동작한다.

연관관계에서는 외래키를 관리하는 쪽이 주인이다. 로딩 전략은 기본적으로 LAZY를 두고, 필요한 조회에서 fetch join, EntityGraph, DTO 조회를 선택하는 방향이 안전하다. N+1 문제는 목록 조회와 연관 데이터 접근이 분리될 때 발생하므로 SQL 발생 시점을 설명할 수 있어야 한다.

#### 면접 답변

> JPA는 객체와 테이블을 매핑하는 ORM 표준이고, 핵심은 영속성 컨텍스트입니다. 영속성 컨텍스트는 1차 캐시와 변경 감지를 제공하며 트랜잭션 안에서 동작합니다. 실무에서는 연관관계를 LAZY로 두고 필요한 조회에서 fetch join이나 EntityGraph로 N+1 문제를 제어합니다.

#### 헷갈리는 포인트
- 영속성 컨텍스트와 DB는 같은 것이 아니다.
- LAZY 전략만으로 N+1 문제가 자동 해결되지는 않는다.
- 엔티티를 API 응답으로 직접 노출하면 지연 로딩과 순환 참조 문제가 생길 수 있다.

#### 관련 퀴즈
- `spring-c025`
- `spring-c026`
- `spring-c030`
- `spring-s003`
- `spring-s005`
