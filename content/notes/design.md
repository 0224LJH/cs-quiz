# 디자인패턴

이 문서는 CS 면접 준비를 위해 디자인패턴, SOLID, DI를 모바일에서 읽기 좋게 정리한 학습 자료입니다. 패턴 이름을 외우는 것보다 "어떤 문제를 해결하려고 쓰는가", "대안보다 나은 점과 불리한 점은 무엇인가"를 말할 수 있게 만드는 데 초점을 둡니다.

## 디자인패턴 개요

### 디자인패턴을 사용하는 이유
<!-- article: design-why-patterns -->

#### 한눈에 보기
- 디자인패턴은 반복해서 등장하는 설계 문제에 대한 검증된 해결 방식이다.
- 팀원이 같은 용어로 설계를 설명할 수 있어 의사소통 비용이 줄어든다.
- 패턴은 정답이 아니라 선택지이며, 상황에 맞지 않게 쓰면 복잡도만 늘어난다.

#### 핵심 정리

디자인패턴은 프로그램을 설계할 때 자주 만나는 문제를 객체 간의 역할과 협력으로 풀어낸 설계 관용구입니다. 예를 들어 객체 생성을 한곳에 모으고 싶을 때 Factory를 떠올리고, 상태 변화 알림을 여러 객체에 전파해야 할 때 Observer를 떠올릴 수 있습니다.

패턴의 가장 큰 장점은 설명력입니다. "결제 방식을 Strategy로 분리하자"라고 말하면, 알고리즘을 객체로 캡슐화하고 런타임에 교체하자는 의도를 짧게 전달할 수 있습니다. 라이브러리나 프레임워크 내부 구조를 이해할 때도 패턴을 알면 흐름을 더 빠르게 잡을 수 있습니다.

다만 패턴은 코드에 클래스와 인터페이스를 추가합니다. 단순한 조건문 하나로 충분한 문제에 무리하게 패턴을 적용하면 가독성이 오히려 떨어집니다. 좋은 설계는 패턴을 많이 쓰는 설계가 아니라, 변경 가능성이 높은 부분을 적절히 분리한 설계입니다.

#### 면접 답변

> 디자인패턴은 자주 반복되는 설계 문제에 대한 재사용 가능한 해결 방식입니다. 객체 생성, 객체 구조, 객체 간 책임 분배를 일관된 용어로 설명할 수 있게 해 협업과 유지보수에 도움이 됩니다. 다만 패턴 자체가 목적은 아니므로 변경 가능성, 복잡도, 테스트성을 보고 필요한 곳에만 적용해야 합니다.

#### 헷갈리는 포인트
- 패턴을 적용하면 항상 좋은 코드가 되는 것은 아니다.
- 같은 패턴도 언어, 프레임워크, 요구사항에 따라 구현 방식이 달라진다.
- 면접에서는 정의보다 "언제 쓰고 언제 피하는지"가 더 중요하게 평가된다.

### 생성, 구조, 행동 패턴
<!-- article: design-pattern-categories -->

#### 한눈에 보기
- 생성 패턴은 객체를 어떻게 만들지 다룬다.
- 구조 패턴은 객체와 클래스를 어떻게 조합할지 다룬다.
- 행동 패턴은 객체들이 책임을 나누고 협력하는 방식을 다룬다.

#### 핵심 정리

GoF 디자인패턴은 목적에 따라 크게 생성, 구조, 행동 패턴으로 나눕니다. 생성 패턴은 객체 생성 책임을 클라이언트 코드에서 분리합니다. Singleton, Factory Method, Abstract Factory, Builder, Prototype이 여기에 들어갑니다.

구조 패턴은 여러 객체를 조합해 더 큰 구조를 만드는 방법입니다. Adapter는 인터페이스 불일치를 맞추고, Decorator는 객체를 감싸 기능을 더하며, Facade는 복잡한 서브시스템 앞에 단순한 입구를 둡니다. Proxy, Composite, Bridge, Flyweight도 구조 패턴입니다.

행동 패턴은 객체 사이의 알고리즘, 상태 변화, 요청 처리 흐름을 다룹니다. Strategy는 알고리즘을 교체 가능하게 만들고, Observer는 상태 변화를 구독자에게 알립니다. Command, Chain of Responsibility, Template Method, State, Iterator, Mediator 등이 대표적입니다.

#### 면접 답변

> 디자인패턴은 생성, 구조, 행동 패턴으로 분류합니다. 생성 패턴은 객체 생성 책임을 분리하고, 구조 패턴은 객체 조합과 관계를 유연하게 만들며, 행동 패턴은 객체 간 책임 분배와 상호작용을 다룹니다.

#### 헷갈리는 포인트
- Singleton은 전역 접근점이 핵심처럼 보이지만 분류상 생성 패턴이다.
- Decorator와 Proxy는 둘 다 감싸는 구조지만, 목적이 다르다.
- Strategy와 State는 둘 다 행동을 위임하지만 선택 주체가 다르다.

## 생성 패턴

### Singleton 패턴
<!-- article: design-singleton -->

#### 한눈에 보기
- 클래스의 인스턴스를 하나만 만들고 전역 접근점을 제공한다.
- 설정, 로깅, 공용 자원처럼 하나의 인스턴스로 충분한 곳에 쓰인다.
- 전역 상태, 테스트 어려움, 스레드 안전성 문제가 단점이다.

#### 핵심 정리

Singleton은 특정 클래스의 인스턴스가 애플리케이션 안에서 하나만 존재하도록 제한하는 패턴입니다. 데이터베이스 연결 관리자, 설정 객체, 로거처럼 여러 곳에서 같은 자원을 사용해야 할 때 떠올리기 쉽습니다.

장점은 생성 비용을 줄이고 접근 지점을 단순하게 만들 수 있다는 점입니다. 하지만 그 편리함 때문에 객체가 사실상 전역 변수처럼 쓰이기 쉽습니다. 전역 상태가 생기면 테스트에서 대체 객체를 주입하기 어렵고, 실행 순서나 공유 상태에 따라 예측하기 어려운 버그가 생길 수 있습니다.

멀티스레드 환경에서는 초기화 방식도 중요합니다. 즉시 초기화는 단순하지만 사용하지 않아도 비용이 발생할 수 있고, 지연 초기화는 실제 사용 시점까지 생성을 미룰 수 있지만 동시 접근을 안전하게 처리해야 합니다. Spring 같은 DI 컨테이너의 singleton scope는 객체 생명주기를 컨테이너가 관리한다는 점에서 직접 구현한 Singleton과 구분해야 합니다.

#### 면접 답변

> Singleton은 인스턴스를 하나만 생성하고 전역 접근점을 제공하는 생성 패턴입니다. 자원 공유에는 편리하지만 전역 상태로 인해 결합도가 높아지고 테스트가 어려워질 수 있습니다. 멀티스레드 환경에서는 지연 초기화와 동기화 방식도 함께 고려해야 합니다.

#### 헷갈리는 포인트
- Singleton이 스레드 안전성을 자동으로 보장하지 않는다.
- DI 컨테이너의 싱글톤 빈은 직접 작성한 Singleton 클래스와 목적과 관리 방식이 다르다.
- 파일 I/O처럼 무거운 초기화가 있으면 즉시 초기화가 불필요한 비용을 만들 수 있다.

### Factory Method와 Abstract Factory
<!-- article: design-factory-method-abstract-factory -->

#### 한눈에 보기
- Factory 계열 패턴은 객체 생성 로직을 클라이언트에서 분리한다.
- Factory Method는 단일 객체 생성을 서브클래스나 메서드에 위임한다.
- Abstract Factory는 관련 객체군을 일관되게 생성한다.

#### 핵심 정리

Factory 패턴의 핵심은 `new`로 구체 클래스를 직접 생성하는 코드를 감추는 것입니다. 클라이언트가 `new KakaoPayClient()`처럼 구체 구현에 직접 의존하면 새 구현체가 추가될 때 클라이언트 코드가 자주 바뀝니다. Factory를 두면 클라이언트는 추상 타입을 받고, 어떤 구현체를 만들지는 생성 책임을 가진 객체가 결정합니다.

Factory Method는 하나의 제품을 생성하는 메서드를 정의하고, 구체적인 생성은 하위 클래스나 구현체에 맡기는 방식입니다. Abstract Factory는 한 단계 더 나아가 관련된 객체 패밀리를 함께 생성합니다. 예를 들어 OS 테마별로 Button, TextField, Checkbox를 함께 만들어야 하면 Abstract Factory가 적합합니다.

문자열 분기만 있는 단순 `ShapeFactory`는 겉보기에는 Factory지만 새 타입이 생길 때마다 Factory 내부의 `if-else`를 수정해야 합니다. 이런 구조는 OCP를 깨기 쉽습니다. 등록 기반 Map, DI 컨테이너, 하위 Creator 분리 같은 방식으로 확장 지점을 열어둘 수 있습니다.

#### 면접 답변

> Factory Method는 객체 생성 인터페이스를 정의하고 어떤 구체 클래스를 만들지는 하위 클래스나 구현체가 결정하게 합니다. Abstract Factory는 관련 객체군을 한 번에 생성하는 인터페이스를 제공합니다. 둘 다 클라이언트가 구체 클래스 생성에 직접 의존하지 않도록 해 확장성을 높입니다.

#### 헷갈리는 포인트
- 단순 정적 메서드 Factory와 GoF Factory Method는 같은 이름으로 불리지만 구조가 다를 수 있다.
- Abstract Factory는 "여러 종류의 관련 제품"을 함께 만들 때 강점이 있다.
- 문자열 분기 Factory는 타입이 자주 늘어나면 OCP 위반 지점이 된다.

### Builder 패턴
<!-- article: design-builder -->

#### 한눈에 보기
- 매개변수가 많거나 선택 값이 많은 객체 생성에 유용하다.
- 생성 과정을 단계별 메서드 체이닝으로 표현할 수 있다.
- `build()` 시점에 유효성 검사와 불변 객체 생성을 함께 처리하기 좋다.

#### 핵심 정리

Builder는 복잡한 객체를 단계별로 조립하는 생성 패턴입니다. 생성자의 파라미터가 많아지면 `new User(a, b, true, null, 3)`처럼 각 값의 의미가 흐려집니다. 선택적 값이 많으면 생성자 오버로딩도 빠르게 지저분해집니다.

Builder를 사용하면 필수 값과 선택 값을 이름 있는 메서드로 설정할 수 있어 읽기 쉬운 생성 코드가 됩니다. 마지막 `build()` 단계에서 필수 값 누락, 값 범위, 조합 규칙을 검증하면 잘못된 객체 생성을 한곳에서 막을 수 있습니다.

단점도 있습니다. 단순 객체에는 Builder 클래스가 과한 구조일 수 있고, 객체 하나를 만들기 위해 더 많은 코드가 필요합니다. 따라서 생성자가 짧고 명확한 경우에는 생성자나 정적 팩토리 메서드가 더 낫습니다. Builder는 "객체 생성 규칙이 복잡할 때" 빛납니다.

#### 면접 답변

> Builder 패턴은 매개변수가 많거나 선택 값이 많은 객체를 단계적으로 생성할 때 사용합니다. 생성자보다 가독성이 좋고 `build()`에서 유효성 검사를 모을 수 있습니다. 다만 단순 객체에는 코드가 과해질 수 있으므로 복잡한 생성 규칙이 있을 때 적합합니다.

#### 헷갈리는 포인트
- Builder는 객체 생성을 빠르게 만들기 위한 패턴이 아니다.
- 필수 값은 Builder 생성자나 별도 메서드로 강제할 수 있다.
- Query Builder처럼 메서드 체이닝 API에도 Builder 아이디어가 자주 쓰인다.

## 구조 패턴

### Adapter 패턴
<!-- article: design-adapter -->

#### 한눈에 보기
- 서로 맞지 않는 인터페이스 사이를 변환한다.
- 레거시 코드나 서드파티 라이브러리를 새 시스템에 맞출 때 유용하다.
- 기존 코드를 수정하지 않고 클라이언트가 기대하는 인터페이스를 제공한다.

#### 핵심 정리

Adapter는 호환되지 않는 인터페이스를 가진 객체를 함께 동작하게 만드는 구조 패턴입니다. 클라이언트는 `JsonParser` 인터페이스를 기대하지만 실제로 가진 라이브러리가 `XmlParser`라면, Adapter가 중간에서 호출 형식과 반환 형식을 변환합니다.

이 패턴은 레거시 시스템 통합에서 자주 등장합니다. 이미 검증된 코드를 직접 고치면 위험이 크고, 새 시스템 전체를 레거시 인터페이스에 맞추면 설계가 오염될 수 있습니다. Adapter를 두면 레거시 변경을 격리하면서 새 코드의 인터페이스를 유지할 수 있습니다.

주의할 점은 Adapter가 너무 많은 변환 책임을 떠안는 경우입니다. 단순 호출 변환을 넘어 데이터 해석, 검증, 비즈니스 규칙까지 들어가면 Adapter가 비대한 서비스 객체가 됩니다. Adapter는 경계의 모양을 맞추는 데 집중시키는 편이 좋습니다.

#### 면접 답변

> Adapter는 클라이언트가 기대하는 인터페이스와 기존 객체의 인터페이스가 맞지 않을 때 중간에서 변환해 주는 패턴입니다. 레거시 코드나 외부 라이브러리를 수정하지 않고 새 시스템의 인터페이스에 맞춰 사용할 수 있게 합니다.

#### 헷갈리는 포인트
- Adapter는 기능을 추가하기보다 인터페이스 호환을 맞추는 것이 목적이다.
- Decorator처럼 감쌀 수 있지만, Decorator는 책임 추가가 목적이다.
- 레거시 코드를 직접 수정하기 어렵거나 위험할 때 특히 유용하다.

#### 관련 퀴즈
- `design-c006`
- `design-s008`

### Decorator 패턴
<!-- article: design-decorator -->

#### 한눈에 보기
- 객체를 래퍼로 감싸 런타임에 책임을 추가한다.
- 상속보다 기능 조합이 유연하다.
- 래퍼가 많이 중첩되면 흐름 추적이 어려워질 수 있다.

#### 핵심 정리

Decorator는 원본 객체와 같은 인터페이스를 가진 래퍼 객체로 기능을 덧붙이는 구조 패턴입니다. 커피 예시로 보면 `Espresso`를 `MilkDecorator`, `SyrupDecorator`로 감싸면서 가격 계산과 설명을 누적할 수 있습니다.

상속으로 기능 조합을 만들면 "우유 커피", "시럽 커피", "우유 시럽 커피"처럼 조합마다 클래스를 만들어야 할 수 있습니다. Decorator는 필요한 기능을 런타임에 감싸 조합하므로 클래스 폭발을 줄입니다. 그래서 "상속보다 합성"이라는 설계 원칙을 보여주는 대표적인 패턴입니다.

단점은 작은 래퍼 객체가 많아지고 호출 경로가 길어진다는 점입니다. 같은 Decorator를 실수로 여러 번 적용하는 문제도 생길 수 있습니다. 실무에서는 중복 적용을 막는 검증, 설명 메서드, Builder와의 조합 등으로 사용 실수를 줄입니다.

#### 면접 답변

> Decorator는 원본 객체를 같은 인터페이스의 래퍼로 감싸 런타임에 기능을 동적으로 추가하는 구조 패턴입니다. 상속보다 조합이 유연하지만 래퍼가 많아지면 디버깅과 중복 적용 관리가 어려워질 수 있습니다.

#### 헷갈리는 포인트
- Decorator는 기존 객체의 인터페이스를 유지하면서 책임을 추가한다.
- 상속은 정적 확장, Decorator는 런타임 조합에 가깝다.
- Proxy와 모양은 비슷하지만 Proxy는 접근 제어나 지연 로딩이 주목적이다.

#### 관련 퀴즈

### Facade 패턴
<!-- article: design-facade -->

#### 한눈에 보기
- 복잡한 서브시스템 앞에 단순한 인터페이스를 제공한다.
- 클라이언트가 내부 클래스들의 호출 순서를 몰라도 되게 한다.
- 책임이 너무 커지면 God Object가 될 수 있다.

#### 핵심 정리

Facade는 여러 클래스와 단계로 이루어진 복잡한 기능을 단순한 API로 감싸는 구조 패턴입니다. 예를 들어 주문 생성 과정에 검증, 재고 확인, 결제, 알림, 배송 요청이 있다면 클라이언트가 각 객체를 직접 호출하는 대신 `orderFacade.placeOrder()`처럼 하나의 입구를 사용할 수 있습니다.

장점은 결합도 감소와 사용성입니다. 클라이언트는 서브시스템 내부 구조를 몰라도 되고, 내부 구현이 바뀌어도 Facade의 공개 인터페이스가 유지되면 영향이 작습니다. 계층 간 경계나 외부 API 제공 계층에서도 자주 쓰입니다.

하지만 Facade가 모든 요청을 다 받기 시작하면 책임이 집중됩니다. 단순 입구 역할을 넘어 정책 판단, 데이터 변환, 상태 관리까지 모두 들고 있으면 거대한 객체가 됩니다. Facade는 내부 복잡도를 숨기되, 도메인 책임을 독점하지 않게 경계를 잡아야 합니다.

#### 면접 답변

> Facade는 복잡한 서브시스템을 단순한 인터페이스로 감싸 클라이언트가 쉽게 사용하게 하는 패턴입니다. 결합도를 낮추고 API를 단순화하지만, 너무 많은 책임이 모이면 God Object가 될 수 있어 주의해야 합니다.

#### 헷갈리는 포인트
- Facade는 서브시스템 접근을 금지하는 패턴이 아니라 기본 사용 경로를 단순화하는 패턴이다.
- 고급 기능이 필요한 클라이언트는 Facade를 우회해야 할 수도 있다.
- 단순화 계층과 비즈니스 책임 계층을 구분해야 한다.

#### 관련 퀴즈
- `design-c008`
- `design-t006`

### Composite 패턴
<!-- article: design-composite -->

#### 한눈에 보기
- 단일 객체와 객체 그룹을 같은 인터페이스로 다룬다.
- 트리 구조를 표현할 때 자연스럽다.
- 파일과 폴더, UI 컴포넌트 계층이 대표 예시다.

#### 핵심 정리

Composite는 객체를 트리 구조로 구성하고, 개별 객체와 복합 객체를 동일하게 처리하게 만드는 구조 패턴입니다. 파일 시스템에서 파일은 단일 객체이고 폴더는 파일과 다른 폴더를 포함하는 복합 객체입니다. 클라이언트가 둘 다 `size()`나 `print()` 같은 공통 인터페이스로 다루면 코드가 단순해집니다.

이 패턴의 장점은 재귀적 구조를 자연스럽게 표현한다는 점입니다. UI 컴포넌트도 버튼 하나와 패널 전체를 같은 `render()` 인터페이스로 처리할 수 있습니다. 클라이언트는 현재 대상이 Leaf인지 Composite인지 계속 분기하지 않아도 됩니다.

주의할 점은 공통 인터페이스 설계입니다. Leaf에는 의미 없는 메서드가 Composite 인터페이스에 들어가면 ISP를 해칠 수 있습니다. 예를 들어 파일 객체에 `addChild()`가 노출되면 호출해도 안 되는 메서드가 생깁니다. 투명성과 안전성 사이에서 선택이 필요합니다.

#### 면접 답변

> Composite는 단일 객체와 복합 객체를 같은 인터페이스로 처리하게 해 트리 구조를 표현하는 패턴입니다. 파일과 폴더, UI 컴포넌트 계층처럼 부분과 전체를 동일하게 다뤄야 할 때 적합합니다.

#### 헷갈리는 포인트
- Composite는 컬렉션을 단순히 감싸는 패턴이 아니라 부분-전체 계층을 다룬다.
- Leaf와 Composite에 억지로 같은 메서드를 넣으면 인터페이스가 어색해질 수 있다.
- 재귀 호출 구조가 자연스럽지만 깊은 트리에서는 성능과 순환 참조를 주의한다.

#### 관련 퀴즈
- `design-c010`
- `design-s005`

### Proxy 패턴
<!-- article: design-proxy -->

#### 한눈에 보기
- 실제 객체 앞에 대리 객체를 둔다.
- 접근 제어, 지연 로딩, 캐싱, 로깅, 원격 호출에 사용된다.
- 클라이언트는 실제 객체와 같은 인터페이스로 Proxy를 사용한다.

#### 핵심 정리

Proxy는 실제 대상 객체와 같은 인터페이스를 제공하는 대리 객체입니다. 클라이언트 호출을 바로 원본에 전달하지 않고 중간에서 부가 처리를 수행합니다. 권한 검사 후 호출을 허용하는 보호 프록시, 무거운 객체를 실제 필요할 때 생성하는 가상 프록시, 원격 객체를 로컬 객체처럼 보이게 하는 원격 프록시가 대표적입니다.

Spring AOP의 프록시는 메서드 호출 앞뒤에 트랜잭션, 로깅, 보안 같은 공통 관심사를 끼워 넣는 데 쓰입니다. ORM의 지연 로딩도 실제 데이터 조회를 나중으로 미루기 위해 프록시 아이디어를 활용합니다.

Proxy와 Decorator는 둘 다 원본을 감싸고 같은 인터페이스를 제공할 수 있습니다. 차이는 의도입니다. Decorator는 기능이나 책임을 동적으로 추가하는 데 초점이 있고, Proxy는 실제 객체에 대한 접근을 제어하거나 대리하는 데 초점이 있습니다.

#### 면접 답변

> Proxy는 실제 객체에 대한 대리 객체를 두고, 호출을 중간에서 제어하는 구조 패턴입니다. 접근 제어, 지연 로딩, 캐싱, 로깅, 원격 호출 등에 쓰이며 클라이언트는 실제 객체와 같은 인터페이스로 사용할 수 있습니다.

#### 헷갈리는 포인트
- 프록시가 항상 성능을 높이는 것은 아니다. 중간 계층 비용이 있다.
- Decorator와 구조는 비슷하지만 목적은 접근 제어와 대리이다.
- 지연 로딩 프록시는 세션 생명주기와 함께 이해해야 실무 버그를 줄일 수 있다.

#### 관련 퀴즈
- `design-c013`
- `design-c019`

## 행동 패턴

### Strategy 패턴
<!-- article: design-strategy -->

#### 한눈에 보기
- 알고리즘군을 각각 캡슐화하고 교체 가능하게 만든다.
- 결제 방식, 정렬 방식, 인증 방식처럼 선택지가 많은 동작에 적합하다.
- 단순한 분기에는 오히려 과할 수 있다.

#### 핵심 정리

Strategy는 같은 목적을 가진 여러 알고리즘을 객체로 분리하고, Context가 전략 인터페이스에 의존하게 만드는 행동 패턴입니다. 결제 시스템에서 신용카드, 계좌이체, 간편결제를 각각 `PaymentStrategy` 구현체로 만들면 런타임에 결제 방식을 바꿀 수 있습니다.

이 패턴은 `if-else`나 `switch`가 늘어나는 문제를 줄입니다. 새 전략이 추가될 때 기존 분기문을 계속 수정하지 않고 새 클래스를 추가하면 됩니다. 이런 점에서 OCP와 잘 맞습니다. 테스트할 때도 전략별로 독립적인 단위 테스트를 만들기 쉽습니다.

다만 선택지가 2-3개이고 변경 가능성이 낮다면 조건문이 더 읽기 쉽습니다. Strategy를 적용하면 인터페이스와 구현 클래스가 늘어나므로 구조가 복잡해집니다. 또한 Context 내부에서 기본 전략을 직접 `new`로 생성하면 다시 구체 클래스에 의존하므로 DI로 주입하는 편이 좋습니다.

#### 면접 답변

> Strategy는 알고리즘을 각각 캡슐화해 런타임에 교체 가능하게 만드는 패턴입니다. 분기문이 많거나 알고리즘 추가가 잦은 경우 OCP를 지키기 좋습니다. 단순한 분기에는 과할 수 있고, 전략은 외부에서 주입해 결합도를 낮추는 편이 좋습니다.

#### 헷갈리는 포인트
- Strategy는 클라이언트나 Context 외부에서 전략을 선택하는 경우가 많다.
- State와 비슷하지만 State는 객체 내부 상태 전이에 따라 행동이 바뀐다.
- 내부에서 구체 전략을 직접 생성하면 Strategy의 장점이 약해진다.

#### 관련 퀴즈
- `design-c004`
- `design-t005`
- `design-s003`
- `design-d005`

### Observer 패턴
<!-- article: design-observer -->

#### 한눈에 보기
- Subject의 상태 변화가 발생하면 등록된 Observer들에게 알린다.
- 발행자와 구독자를 느슨하게 연결한다.
- 구독 해제 누락, 알림 순서, 과도한 이벤트를 주의해야 한다.

#### 핵심 정리

Observer는 한 객체의 상태 변화가 여러 객체에 전파되어야 할 때 쓰는 행동 패턴입니다. Subject는 Observer 목록을 가지고 있다가 상태가 바뀌면 `notify()`를 호출합니다. MVC에서 모델 변경이 뷰에 반영되는 구조, 이벤트 리스너, 알림 시스템, 리액티브 프로그래밍에서 자주 보입니다.

장점은 느슨한 결합입니다. Subject는 구독자들이 구체적으로 어떤 일을 하는지 몰라도 됩니다. 런타임에 구독자를 추가하거나 제거할 수 있어 확장성이 좋습니다. 새 알림 채널이 추가되어도 발행자 코드를 크게 바꾸지 않을 수 있습니다.

단점은 흐름 추적입니다. 이벤트가 여러 구독자에게 전파되면 어떤 순서로 어떤 부작용이 발생하는지 보기 어려울 수 있습니다. 특히 Subject가 Observer를 강한 참조로 계속 들고 있고 구독 해제가 없다면 메모리 누수가 발생합니다. 이벤트 기반 구조에서는 실패 재처리와 중복 처리도 함께 설계해야 합니다.

#### 면접 답변

> Observer는 Subject의 상태 변화가 발생했을 때 등록된 Observer들에게 자동으로 알리는 일대다 의존 관계 패턴입니다. 느슨한 결합과 확장성이 장점이지만, 구독 해제 누락으로 인한 메모리 누수, 알림 순서, 디버깅 어려움이 단점입니다.

#### 헷갈리는 포인트
- Observer가 항상 비동기라는 뜻은 아니다.
- 구독자 수가 늘면 알림 비용도 늘 수 있다.
- Mediator는 복잡한 다대다 통신을 중앙 중재자로 정리할 때 더 적합하다.

#### 관련 퀴즈
- `design-c003`
- `design-t003`
- `design-t008`
- `design-d002`
- `design-o002`

### Command 패턴
<!-- article: design-command -->

#### 한눈에 보기
- 요청을 객체로 캡슐화한다.
- 실행 취소, 재실행, 큐잉, 로깅에 유리하다.
- 요청자와 실제 수행 객체를 분리한다.

#### 핵심 정리

Command는 "무엇을 실행할지"라는 요청 자체를 객체로 만드는 행동 패턴입니다. 텍스트 에디터의 글자 입력, 삭제, 서식 변경을 각각 Command 객체로 만들면 실행한 명령을 스택에 저장해 Undo와 Redo를 구현할 수 있습니다.

Command 객체는 실행에 필요한 정보와 실행 메서드를 함께 가집니다. Invoker는 Command 인터페이스만 호출하고, 실제 작업은 Receiver가 수행합니다. 이 구조 덕분에 요청을 큐에 넣거나, 로그로 남기거나, 나중에 재실행하는 식의 처리가 쉬워집니다.

단점은 명령마다 클래스나 객체가 늘어난다는 점입니다. 단순한 버튼 클릭 하나에 모든 것을 Command로 만들면 구조가 무거워질 수 있습니다. 반대로 실행 이력, 트랜잭션, 보상 작업이 중요한 시스템에서는 Command가 흐름을 명확하게 만들어 줍니다.

#### 면접 답변

> Command 패턴은 요청을 객체로 캡슐화해 실행 취소, 재실행, 큐잉, 로깅을 가능하게 합니다. 요청자와 실제 수행자를 분리할 수 있어 확장성이 좋아지지만, 명령 객체가 많아져 구조가 복잡해질 수 있습니다.

#### 헷갈리는 포인트
- Command는 단순 콜백과 비슷해 보이지만 요청을 객체로 저장하고 조작할 수 있다는 점이 중요하다.
- Undo를 하려면 이전 상태 저장이나 반대 명령 실행 전략이 필요하다.
- 모든 UI 이벤트를 Command로 만들 필요는 없다.

#### 관련 퀴즈

- `design-s002`

### Chain of Responsibility 패턴
<!-- article: design-chain-of-responsibility -->

#### 한눈에 보기
- 요청을 처리할 수 있는 핸들러를 찾을 때까지 체인을 따라 전달한다.
- 필터, 검증 단계, 보안 체인에 자주 쓰인다.
- 처리 순서와 체인 구성 방식이 중요하다.

#### 핵심 정리

Chain of Responsibility는 여러 핸들러를 연결하고, 요청을 첫 핸들러부터 차례로 전달하는 행동 패턴입니다. 각 핸들러는 자신이 처리할 수 있으면 처리하고, 아니면 다음 핸들러로 넘깁니다. 주문 처리에서 유효성 검사, 재고 확인, 결제, 배송 요청을 단계별로 배치하는 식으로 이해할 수 있습니다.

이 패턴은 요청자와 처리자를 분리합니다. 클라이언트는 누가 요청을 처리할지 몰라도 첫 핸들러에만 요청을 보내면 됩니다. Java Servlet Filter, Spring Security FilterChain처럼 요청 전후 처리를 연결하는 구조에서 자주 등장합니다.

주의할 점은 요청이 끝까지 처리되지 않을 수 있다는 점입니다. 체인 끝에 도달했을 때의 동작을 명확히 해야 하고, 각 핸들러가 다음으로 넘길지 중단할지 규칙도 분명해야 합니다. 순서가 바뀌면 동작이 달라질 수 있으므로 구성 테스트가 중요합니다.

#### 면접 답변

> Chain of Responsibility는 요청을 처리할 수 있는 핸들러를 찾을 때까지 핸들러 체인을 따라 전달하는 패턴입니다. 필터나 검증 단계처럼 처리 흐름을 유연하게 조합할 수 있지만, 순서와 미처리 요청에 대한 규칙을 명확히 해야 합니다.

#### 헷갈리는 포인트
- 모든 핸들러가 반드시 요청을 처리하는 것은 아니다.
- 파이프라인처럼 모든 단계를 통과시키는 구조와는 의도가 다를 수 있다.
- 체인 순서가 비즈니스 규칙이면 설정 변경도 테스트 대상이다.

#### 관련 퀴즈
- `design-c014`
- `design-s006`
- `design-o003`

### Template Method와 State
<!-- article: design-template-method-state -->

#### 한눈에 보기
- Template Method는 알고리즘 골격을 상위 클래스에 둔다.
- State는 객체 내부 상태에 따라 행동을 바꾼다.
- 둘 다 공통 흐름과 변화 지점을 분리한다.

#### 핵심 정리

Template Method는 상위 클래스가 알고리즘의 전체 순서를 정하고, 일부 단계만 하위 클래스가 오버라이드하게 하는 패턴입니다. 예를 들어 파일 처리의 "열기 -> 파싱 -> 검증 -> 저장" 흐름은 고정하고, 파싱 방식만 CSV와 JSON 하위 클래스에서 다르게 구현할 수 있습니다.

이 패턴은 중복 흐름을 줄이는 데 좋지만 상속에 의존합니다. 하위 클래스가 많아지거나 런타임에 알고리즘을 바꿔야 한다면 Strategy가 더 나을 수 있습니다. Template Method는 "흐름은 고정, 일부 단계만 변경"일 때 적합합니다.

State는 객체의 내부 상태가 바뀌면서 허용 동작이 달라질 때 사용합니다. 주문이 대기, 처리 중, 완료, 취소 상태를 가진다면 각 상태 객체가 가능한 동작을 정의합니다. 조건문으로 상태를 계속 검사하는 대신 상태별 클래스로 분리해 새 상태 추가를 쉽게 만듭니다.

#### 면접 답변

> Template Method는 상위 클래스에서 알고리즘의 골격을 정하고 일부 단계를 하위 클래스가 구현하게 합니다. State는 객체 내부 상태에 따라 행동이 바뀌도록 상태별 객체에 책임을 나눕니다. Template Method는 상속 기반의 흐름 재사용, State는 상태 전이와 조건문 제거에 강점이 있습니다.

#### 헷갈리는 포인트
- Template Method는 Strategy보다 상속 결합이 강하다.
- State와 Strategy는 구조가 비슷하지만 State는 내부 상태 전이가 핵심이다.
- 상태 전이가 복잡하면 전이 규칙을 코드나 표로 명확히 관리해야 한다.

#### 관련 퀴즈
- `design-c009`
- `design-c015`
- `design-t011`
- `design-s010`

### Iterator 패턴
<!-- article: design-iterator -->

#### 한눈에 보기
- 컬렉션 내부 구조를 노출하지 않고 순회 방법을 제공한다.
- 배열, 리스트, 트리 등 서로 다른 자료구조를 같은 방식으로 접근하게 한다.
- Java의 Iterator, JavaScript의 iterator protocol이 익숙한 예시다.

#### 핵심 정리

Iterator는 컬렉션의 내부 표현을 감추고 원소를 순차적으로 접근하는 방법을 제공하는 행동 패턴입니다. 클라이언트는 컬렉션이 배열 기반인지 트리 기반인지 몰라도 `hasNext()`, `next()` 같은 공통 방식으로 원소를 가져올 수 있습니다.

이 패턴은 컬렉션과 순회 로직을 분리합니다. 자료구조가 바뀌어도 외부 순회 코드를 크게 바꾸지 않을 수 있고, 같은 컬렉션에 여러 순회 방식을 제공할 수도 있습니다. 예를 들어 트리에는 전위, 중위, 후위 순회 Iterator를 따로 둘 수 있습니다.

주의할 점은 순회 중 컬렉션이 변경될 때입니다. 언어와 구현에 따라 예외가 발생하거나, 변경을 반영하거나, 스냅샷 기준으로 순회할 수 있습니다. 면접에서는 Iterator의 목적을 "내부 구조 은닉과 순회 인터페이스 통일"로 잡으면 충분합니다.

#### 면접 답변

> Iterator는 컬렉션 내부 구조를 노출하지 않고 원소를 순차적으로 접근하는 인터페이스를 제공하는 패턴입니다. 클라이언트는 자료구조 구현을 몰라도 동일한 방식으로 순회할 수 있고, 컬렉션과 순회 로직을 분리할 수 있습니다.

#### 헷갈리는 포인트
- Iterator는 단순 반복문 문법이 아니라 순회 책임을 분리하는 설계이다.
- 순회 중 변경 정책은 구현마다 다르다.
- Composite와 함께 쓰면 트리 구조 전체 순회도 깔끔하게 만들 수 있다.

#### 관련 퀴즈
- `design-c017`
- `design-s011`

## SOLID와 설계 원칙

### SOLID 개요
<!-- article: design-solid-overview -->

#### 한눈에 보기
- SOLID는 객체 지향 설계를 유지보수하기 쉽게 만드는 다섯 가지 원칙이다.
- SRP, OCP, LSP, ISP, DIP로 구성된다.
- 원칙은 서로 연결되어 있으며 테스트성과 변경 용이성을 높이는 방향으로 작동한다.

#### 핵심 정리

SOLID는 객체 지향 코드가 시간이 지나도 변경하기 쉽게 유지되도록 돕는 설계 원칙입니다. SRP는 하나의 책임, OCP는 확장에는 열리고 수정에는 닫힌 구조, LSP는 하위 타입의 대체 가능성, ISP는 인터페이스 분리, DIP는 추상화에 대한 의존을 말합니다.

이 원칙들은 디자인패턴과 함께 자주 등장합니다. Strategy는 OCP를 지키기 좋은 구조이고, Factory와 DI는 DIP를 적용하는 데 도움을 줍니다. Decorator는 상속보다 합성을 선호하는 설계를 보여주며, Observer는 발행자와 구독자의 결합을 낮춥니다.

SOLID를 적용할 때는 순서가 중요할 수 있습니다. 보통 책임이 뒤섞인 클래스를 먼저 나누고, 상속 관계가 계약을 깨지 않는지 확인한 뒤, 변경이 잦은 지점에 추상화를 둡니다. 처음부터 모든 것을 추상화하면 설계가 과해질 수 있습니다.

#### 면접 답변

> SOLID는 객체 지향 설계를 변경과 테스트에 강하게 만들기 위한 다섯 원칙입니다. 클래스는 하나의 변경 이유를 가져야 하고, 확장은 쉽게 하되 기존 코드는 덜 수정해야 하며, 하위 타입은 상위 타입을 대체할 수 있어야 합니다. 또한 인터페이스는 클라이언트 기준으로 나누고, 고수준 모듈은 구체 구현이 아니라 추상화에 의존해야 합니다.

#### 헷갈리는 포인트
- SOLID는 모든 코드에 인터페이스를 붙이라는 뜻이 아니다.
- 원칙끼리 충돌해 보일 때는 변경 가능성과 복잡도를 함께 본다.
- 리팩토링은 보통 SRP로 책임을 분리하는 것에서 시작한다.

### SRP, LSP, ISP
<!-- article: design-srp-lsp-isp -->

#### 한눈에 보기
- SRP는 클래스의 변경 이유를 하나로 줄이는 원칙이다.
- LSP는 하위 타입이 상위 타입의 기대 동작을 깨지 않아야 한다는 원칙이다.
- ISP는 큰 인터페이스를 클라이언트가 쓰는 역할별로 나누라는 원칙이다.

#### 핵심 정리

SRP는 "클래스는 하나의 책임만 가져야 한다"로 자주 설명되지만, 더 실무적인 표현은 "변경 이유가 하나여야 한다"입니다. 예를 들어 주문 서비스가 주문 검증, 결제, 이메일 발송, 로그 저장을 모두 직접 한다면 변경 이유가 너무 많습니다. 역할별로 분리하면 테스트와 변경 영향 분석이 쉬워집니다.

LSP는 다형성의 안전장치입니다. 상위 타입을 사용하는 코드가 하위 타입을 받아도 기대한 계약이 유지되어야 합니다. 하위 클래스가 부모 메서드의 의미를 바꾸거나, 부모는 허용하던 입력을 갑자기 예외 처리하면 상위 타입 기반 코드가 깨집니다.

ISP는 클라이언트가 쓰지 않는 메서드에 의존하지 않게 하라는 원칙입니다. 거대한 `Machine` 인터페이스에 `print()`, `scan()`, `fax()`가 모두 있으면 단순 프린터 구현체도 필요 없는 `fax()`를 억지로 구현해야 합니다. 역할별 인터페이스로 나누면 변경 영향이 줄어듭니다.

#### 면접 답변

> SRP는 클래스의 변경 이유를 하나로 줄이는 원칙이고, LSP는 하위 타입이 상위 타입을 대체해도 기대 동작을 깨지 않아야 한다는 원칙입니다. ISP는 클라이언트가 사용하지 않는 메서드에 의존하지 않도록 인터페이스를 작고 역할 중심으로 나누라는 원칙입니다.

#### 헷갈리는 포인트
- SRP는 메서드가 하나여야 한다는 뜻이 아니다.
- 상속을 사용한다고 LSP가 자동으로 만족되는 것은 아니다.
- ISP는 구현체 편의가 아니라 클라이언트 사용 관점에서 인터페이스를 나누는 것이다.

#### 관련 퀴즈

### OCP, DIP, DI
<!-- article: design-ocp-dip-di -->

#### 한눈에 보기
- OCP는 새 기능 추가 시 기존 코드 수정을 줄이는 원칙이다.
- DIP는 고수준 모듈이 구체 구현이 아니라 추상화에 의존하라는 원칙이다.
- DI는 의존 객체를 외부에서 주입해 DIP를 실현하기 쉽게 만든다.

#### 핵심 정리

OCP는 "확장에는 열려 있고 변경에는 닫혀 있어야 한다"는 원칙입니다. 결제 수단이 늘어날 때마다 `PaymentService`의 `if-else`를 수정한다면 기존 코드가 계속 흔들립니다. 결제 전략을 인터페이스로 분리하고 새 구현체를 추가하는 구조로 만들면 확장은 쉬워지고 기존 코드 수정은 줄어듭니다.

DIP는 의존 방향에 관한 원칙입니다. 고수준 정책을 담은 `OrderService`가 `KakaoPayClient` 같은 구체 구현에 직접 의존하면 결제 업체가 바뀔 때 서비스 코드가 함께 바뀝니다. `PaymentClient` 같은 추상화에 의존하고 구체 구현은 외부에서 연결하면 변경 영향이 작아집니다.

DI는 의존성 주입입니다. 객체가 필요한 의존성을 직접 만들지 않고 생성자, 메서드, 프레임워크 컨테이너 등을 통해 외부에서 받습니다. 테스트에서는 목 객체를 주입하기 쉬워지고, 운영에서는 실제 구현체를 주입할 수 있습니다. 단, 의존성 그래프가 커지면 설정과 추적이 어려워질 수 있습니다.

#### 면접 답변

> OCP는 새 요구사항이 들어와도 기존 코드를 덜 수정하고 확장으로 대응하라는 원칙입니다. DIP는 고수준 모듈과 저수준 구현체가 모두 추상화에 의존해야 한다는 원칙입니다. DI는 의존 객체를 외부에서 주입해 구체 클래스 결합을 낮추고 테스트하기 쉬운 구조를 만드는 방법입니다.

#### 헷갈리는 포인트
- 생성자 주입을 쓴다고 DIP가 자동으로 지켜지는 것은 아니다. 타입이 구체 클래스면 여전히 결합되어 있다.
- OCP를 위해 모든 곳에 추상화를 만들면 오히려 복잡해질 수 있다.
- DI는 장점이 크지만 순환 의존성과 설정 복잡도를 관리해야 한다.

## 실무 선택 기준

### 상속보다 합성
<!-- article: design-composition-over-inheritance -->

#### 한눈에 보기
- 상속은 `is-a` 관계가 명확할 때 적합하다.
- 합성은 필요한 역할을 객체로 주입하거나 위임해 유연하게 조합한다.
- Decorator, Strategy는 합성의 장점을 보여주는 대표 패턴이다.

#### 핵심 정리

상속은 코드 재사용이 직관적이고 공통 동작을 부모 클래스에 모을 수 있습니다. 하지만 부모 클래스 변경이 모든 자식에게 영향을 주고, 런타임에 조합을 바꾸기 어렵습니다. 상속 계층이 깊어지면 실제 동작이 어디서 결정되는지 추적하기도 힘듭니다.

합성은 객체가 필요한 기능을 다른 객체에게 위임하는 방식입니다. `PaymentService`가 결제 알고리즘을 상속받기보다 `PaymentStrategy`를 들고 있게 만들면 런타임 교체와 테스트가 쉬워집니다. Decorator는 기능을 감싸 조합하고, Strategy는 알고리즘을 주입해 조합합니다.

그렇다고 합성이 항상 정답은 아닙니다. 하위 타입이 상위 타입의 계약을 자연스럽게 만족하고 변경 가능성이 낮다면 상속이 간결할 수 있습니다. 판단 기준은 관계의 의미, 변경 가능성, 테스트 필요성, 조합 수입니다.

#### 면접 답변

> 상속은 `is-a` 관계가 명확할 때 간결하지만 부모 변경의 영향이 크고 런타임 조합이 어렵습니다. 합성은 객체를 위임해 유연하게 조합할 수 있고 테스트도 쉬우나 구조가 조금 더 복잡해질 수 있습니다. 그래서 변경 가능성이 큰 동작은 합성으로 분리하는 편이 좋습니다.

#### 헷갈리는 포인트
- "합성이 항상 상속보다 낫다"가 아니라 변경이 잦은 동작에는 합성이 유리하다는 뜻이다.
- LSP를 깨는 상속이라면 합성으로 바꾸는 것이 안전하다.
- Decorator와 Strategy는 합성을 사용하는 방식이 다르다.

### 이벤트 방식과 직접 호출
<!-- article: design-event-driven-vs-direct-call -->

#### 한눈에 보기
- 직접 호출은 흐름이 명확하고 즉각적인 결과를 확인하기 쉽다.
- 이벤트 방식은 발행자와 구독자를 느슨하게 연결해 확장성이 좋다.
- 이벤트 방식은 순서, 재처리, 최종 일관성 관리가 필요하다.

#### 핵심 정리

직접 메서드 호출은 이해하기 쉽습니다. `orderService.pay()` 다음에 `deliveryService.ship()`을 호출하면 실행 흐름이 코드에 그대로 드러납니다. 즉각적인 일관성이 중요하고 처리 순서가 단순하다면 직접 호출이 더 적합할 수 있습니다.

이벤트 방식은 어떤 일이 발생했다는 사실을 발행하고, 관심 있는 구독자들이 각자 처리하게 합니다. 주문 완료 이벤트를 발행하면 포인트 적립, 알림 발송, 추천 데이터 업데이트가 독립적으로 붙을 수 있습니다. 발행자는 구독자 목록과 세부 처리를 몰라도 되므로 확장성이 좋습니다.

대신 운영 복잡도가 올라갑니다. 이벤트 순서가 보장되는지, 실패한 구독자 처리를 어떻게 재시도할지, 같은 이벤트가 두 번 도착해도 안전한지, 여러 서비스의 데이터가 잠시 불일치해도 되는지 결정해야 합니다. 이벤트 방식은 결합도를 낮추지만 흐름 추적 비용을 만든다는 점을 함께 말해야 합니다.

#### 면접 답변

> 직접 호출은 흐름이 명확하고 즉각적인 일관성을 다루기 쉽지만 호출자와 수신자가 강하게 결합됩니다. 이벤트 방식은 발행자와 구독자를 분리해 확장성이 좋지만, 이벤트 순서, 실패 재처리, 중복 처리, 최종 일관성 같은 운영 복잡도를 관리해야 합니다.

#### 헷갈리는 포인트
- 이벤트 방식이 항상 더 빠르거나 더 단순한 것은 아니다.
- Observer는 애플리케이션 내부의 이벤트 구조로, 분산 이벤트 아키텍처와는 고려할 운영 문제가 다를 수 있다.
- 이벤트 핸들러는 멱등성을 고려하는 것이 좋다.

### 아키텍처 패턴 선택
<!-- article: design-architecture-pattern-choice -->

#### 한눈에 보기
- MVC, MVVM, Repository, MSA도 설계 패턴 관점에서 장단점이 있다.
- 패턴 선택은 변경 지점, 팀 규모, 테스트 전략, 운영 복잡도에 따라 달라진다.
- 작은 시스템에 큰 구조를 먼저 도입하면 생산성이 떨어질 수 있다.

#### 핵심 정리

MVC는 Model, View, Controller로 책임을 나눠 UI와 비즈니스 흐름을 분리합니다. 구조가 익숙하고 서버 애플리케이션에서 설명하기 좋지만, 화면과 상태 흐름이 복잡해지면 Controller나 Model-View 관계가 비대해질 수 있습니다.

MVVM은 ViewModel을 통해 View 상태와 동작을 추상화합니다. 데이터 바인딩과 상태 관리가 중요한 UI에서 장점이 있지만, ViewModel이 커지면 추적이 어려워집니다. Repository 패턴은 데이터 접근을 도메인 로직에서 분리해 테스트와 저장소 교체를 돕지만, 단순 CRUD에서는 얇은 중간 계층만 늘어날 수 있습니다.

MSA는 서비스를 작게 나눠 독립 배포와 확장을 가능하게 합니다. 하지만 네트워크 호출, 분산 트랜잭션, 관측성, 장애 전파, 데이터 일관성 문제가 생깁니다. 면접에서는 "어떤 패턴이 더 좋은가"보다 "현재 문제에 비해 복잡도를 감당할 수 있는가"를 기준으로 답하는 것이 좋습니다.

#### 면접 답변

> 아키텍처 패턴은 책임 분리와 변경 관리 방식의 선택입니다. MVC는 역할 분리가 단순하고, MVVM은 UI 상태 추상화에 강하며, Repository는 데이터 접근 분리에 유리합니다. MSA는 독립 배포와 확장성이 장점이지만 분산 시스템 복잡도를 감당해야 하므로 규모와 운영 역량을 보고 선택해야 합니다.

#### 헷갈리는 포인트
- Repository는 ORM을 쓰면 자동으로 생기는 계층이 아니라 도메인과 저장소 경계를 표현하는 패턴이다.
- MSA는 코드 구조 문제가 아니라 운영 구조 문제까지 포함한다.
- MVC와 MVVM은 프레임워크 이름이 아니라 책임 분리 방식이다.

## 면접 질문 정리

### 패턴 답변 구조
<!-- article: design-interview-answer-structure -->

#### 한눈에 보기
- 정의, 해결 문제, 구조, 장단점, 예시 순서로 답하면 안정적이다.
- 패턴끼리 비교할 때는 목적과 선택 주체를 기준으로 구분한다.
- 마지막에는 남용 시 단점을 덧붙이면 깊이가 생긴다.

#### 핵심 정리

디자인패턴 면접 답변은 이름 암기보다 사고 흐름이 중요합니다. 먼저 "무엇을 해결하는 패턴인지"를 말하고, 다음으로 "어떤 객체들이 어떤 역할을 나누는지"를 설명합니다. 그 뒤 장점과 단점, 실무 예시를 붙이면 답변이 단단해집니다.

예를 들어 Strategy는 "알고리즘 교체"가 해결 문제이고, Context와 Strategy 인터페이스, ConcreteStrategy가 구조입니다. 장점은 분기 제거와 OCP, 단점은 클래스 증가입니다. 예시는 결제 방식, 정렬 방식, 인증 방식입니다.

비교 질문에서는 키워드를 잡으면 쉽습니다. Decorator와 Proxy는 둘 다 감싸지만 Decorator는 기능 추가, Proxy는 접근 제어입니다. Strategy와 State는 둘 다 위임하지만 Strategy는 외부 선택, State는 내부 상태 전이입니다. Template Method와 Strategy는 알고리즘 변화 지점을 분리하지만 전자는 상속, 후자는 합성 중심입니다.

#### 면접 답변

> 디자인패턴 질문에는 먼저 패턴의 목적을 말하고, 참여 객체와 동작 흐름을 설명한 뒤 장단점과 사용 예시를 덧붙이면 좋습니다. 비교 질문은 구조보다 의도를 기준으로 구분합니다. 같은 래퍼 구조라도 Decorator는 책임 추가, Proxy는 접근 제어가 핵심입니다.

#### 헷갈리는 포인트
- "어디에 쓰나요?" 질문에는 프레임워크 예시와 직접 만든 예시를 하나씩 준비하면 좋다.
- 단점 없이 장점만 말하면 실제 설계 경험이 부족해 보일 수 있다.
- 패턴 비교는 클래스 다이어그램보다 목적 차이로 설명하는 것이 빠르다.

### 패턴 선택 빠른 기준
<!-- article: design-pattern-selection-checklist -->

#### 한눈에 보기
- 객체 생성이 복잡하면 Factory나 Builder를 검토한다.
- 인터페이스가 맞지 않으면 Adapter, 접근을 제어하려면 Proxy를 검토한다.
- 분기가 알고리즘 선택이면 Strategy, 상태 전이면 State를 검토한다.

#### 핵심 정리

패턴을 고를 때는 먼저 변경 지점을 찾습니다. 새 객체 종류가 자주 추가된다면 Factory 계열을, 생성 파라미터가 많고 조립 순서가 중요하다면 Builder를 검토합니다. 단 하나의 공유 인스턴스가 필요해 보여도 테스트와 전역 상태 문제 때문에 Singleton은 신중히 선택합니다.

객체 구조 문제가 중심이면 구조 패턴을 봅니다. 기존 인터페이스를 새 인터페이스에 맞춰야 하면 Adapter, 기능을 조합해 덧붙여야 하면 Decorator, 복잡한 내부 기능을 단순한 입구로 제공하려면 Facade, 실제 객체 접근 전 제어가 필요하면 Proxy, 트리 구조를 같은 방식으로 다루려면 Composite가 어울립니다.

동작 흐름이 중심이면 행동 패턴을 봅니다. 알고리즘 선택은 Strategy, 상태별 동작은 State, 일대다 알림은 Observer, 요청 이력과 Undo는 Command, 여러 처리 단계 중 담당자를 찾는 흐름은 Chain of Responsibility, 컬렉션 순회는 Iterator가 자연스럽습니다.

#### 면접 답변

> 패턴은 문제 유형으로 고릅니다. 생성 책임이 문제면 Factory나 Builder, 객체 조합이 문제면 Adapter, Decorator, Facade, Proxy, Composite를 봅니다. 동작 변경이나 협력이 문제면 Strategy, State, Observer, Command, Chain of Responsibility, Iterator를 검토합니다. 단순한 문제에는 패턴보다 간단한 코드가 더 낫습니다.

#### 헷갈리는 포인트
- 패턴 이름보다 변경 이유를 먼저 찾아야 한다.
- 같은 상황에도 팀의 숙련도와 프레임워크 기능에 따라 선택이 달라진다.
- 패턴 적용 후 테스트가 쉬워졌는지, 변경 범위가 줄었는지 확인해야 한다.
