# 네트워크

이 문서는 CS 면접 준비를 위해 네트워크 핵심 개념을 모바일에서 읽기 좋게 정리한 학습 자료입니다. 계층 구조부터 HTTP, TCP, DNS, 인증 상태 관리, CORS, 로드밸런싱과 장애 대응까지 질문에 답할 수 있는 단위로 나누었습니다.

## 네트워크 계층
<!-- category: network-layers -->

### OSI 7계층과 TCP/IP 4계층
<!-- article: network-osi-tcpip -->

#### 한눈에 보기
- OSI 7계층은 통신 과정을 역할별로 나눈 개념 모델이다.
- TCP/IP 4계층은 실제 인터넷 프로토콜 스택에 더 가깝다.
- 면접에서는 각 계층의 주소, 장비, PDU를 함께 묶어 말하면 좋다.

#### 핵심 정리

OSI 7계층은 아래에서부터 물리, 데이터링크, 네트워크, 전송, 세션, 표현, 응용 계층으로 올라간다. 각 계층은 자기 역할만 수행하고, 바로 위아래 계층과 데이터를 주고받는다.

TCP/IP 4계층은 네트워크 접근, 인터넷, 전송, 애플리케이션 계층으로 본다. OSI의 세션, 표현, 응용 계층은 TCP/IP에서 애플리케이션 계층으로 묶어 이해하는 경우가 많다.

실무에서는 "브라우저가 HTTP 요청을 만들고, TCP가 포트를 붙이고, IP가 목적지 주소를 붙이고, 이더넷이 MAC 주소로 같은 네트워크 안에서 전달한다"처럼 흐름으로 설명하는 편이 기억하기 쉽다.

#### 면접 답변

> OSI 7계층은 네트워크 통신을 물리부터 응용까지 역할별로 나눈 모델입니다. 실제 인터넷은 TCP/IP 4계층 구조로 많이 설명하며, HTTP 같은 애플리케이션 데이터가 TCP, IP, 이더넷 계층을 거쳐 전달됩니다.

#### 헷갈리는 포인트
- 스위치는 주로 데이터링크 계층에서 MAC 주소를 보고 전달한다.
- 라우터는 네트워크 계층에서 IP 주소를 보고 다른 네트워크로 전달한다.
- OSI는 개념 모델이고, TCP/IP는 실제 프로토콜 묶음을 설명할 때 자주 쓴다.

### 캡슐화와 PDU
<!-- article: network-encapsulation-pdu -->

#### 한눈에 보기
- 캡슐화는 하위 계층으로 내려가며 헤더를 붙이는 과정이다.
- 역캡슐화는 수신 측에서 헤더를 제거하며 상위 계층으로 올리는 과정이다.
- 계층마다 부르는 데이터 단위인 PDU 이름이 다르다.

#### 핵심 정리

송신 측 애플리케이션이 만든 데이터는 전송 계층에서 TCP 세그먼트나 UDP 데이터그램이 된다. 여기에 인터넷 계층의 IP 헤더가 붙으면 패킷이 되고, 데이터링크 계층의 이더넷 헤더와 트레일러가 붙으면 프레임이 된다.

수신 측은 반대로 동작한다. 네트워크 인터페이스가 프레임을 받고, 데이터링크 계층이 자기 헤더를 확인한 뒤 IP 계층으로 넘긴다. IP 계층은 목적지 IP를 확인하고, TCP나 UDP 계층은 포트 번호를 기준으로 적절한 프로세스에 데이터를 전달한다.

이 구조 덕분에 각 계층은 자기 관심사에 집중할 수 있다. 예를 들어 IP는 목적지 네트워크까지 가는 일을 담당하고, TCP는 순서와 신뢰성을 담당한다.

#### 면접 답변

> 캡슐화는 송신 측에서 상위 계층 데이터에 TCP, IP, 이더넷 헤더처럼 각 계층의 정보를 붙이는 과정입니다. 수신 측은 반대로 헤더를 제거하며 원래 데이터를 애플리케이션까지 전달하는데, 이를 역캡슐화라고 합니다.

#### 헷갈리는 포인트
- 모든 계층의 PDU를 무조건 패킷이라고 부르지는 않는다.
- TCP는 세그먼트, UDP는 데이터그램, IP는 패킷, 데이터링크는 프레임이라고 구분한다.
- 암호화와 캡슐화는 같은 말이 아니다.

#### 관련 퀴즈

- `network-c022`

### 포트, 소켓, 라우터와 스위치
<!-- article: network-port-socket-router-switch -->

#### 한눈에 보기
- IP 주소는 호스트를 찾고, 포트 번호는 호스트 안의 프로세스를 찾는다.
- 소켓은 보통 IP, 포트, 프로토콜을 포함한 통신 끝점으로 이해한다.
- 라우터와 스위치는 보는 주소와 동작 계층이 다르다.

#### 핵심 정리

하나의 서버 안에는 웹 서버, 데이터베이스, SSH 서버처럼 여러 프로세스가 동시에 실행될 수 있다. 포트 번호는 이 중 어느 프로세스와 통신할지 구분한다. HTTP는 80, HTTPS는 443, SSH는 22처럼 잘 알려진 포트가 있다.

소켓은 네트워크 통신의 끝점이다. 예를 들어 클라이언트의 임시 포트와 서버의 443 포트가 TCP 연결을 만들면, 운영체제는 이 연결을 소켓 단위로 관리한다.

라우터는 IP 주소를 보고 서로 다른 네트워크 사이에서 패킷의 다음 경로를 정한다. 스위치는 같은 네트워크 안에서 MAC 주소 테이블을 보고 프레임을 어느 포트로 내보낼지 결정한다.

#### 면접 답변

> IP 주소가 네트워크상의 장치를 찾는 논리적 주소라면, 포트 번호는 그 장치 안의 특정 애플리케이션을 구분하는 번호입니다. 라우터는 IP 기반으로 네트워크 간 라우팅을 하고, 스위치는 MAC 기반으로 같은 네트워크 안에서 프레임을 전달합니다.

#### 헷갈리는 포인트
- 포트 번호는 데이터링크 계층이 아니라 전송 계층에서 사용한다.
- MAC 주소만으로 인터넷 전체 라우팅을 하지는 않는다.
- 라우터가 항상 물리 장비만 의미하는 것은 아니며, 소프트웨어 라우팅도 가능하다.

#### 관련 퀴즈

- `network-c008`

### HTTP 요청과 메서드
<!-- article: network-http-methods -->

#### 한눈에 보기
- HTTP는 클라이언트와 서버가 요청, 응답 메시지를 주고받는 프로토콜이다.
- GET은 조회, POST는 생성이나 처리를 표현할 때 주로 사용한다.
- PUT, PATCH, DELETE는 리소스 변경 의도를 더 명확히 드러낸다.

#### 핵심 정리

GET은 리소스를 조회할 때 사용한다. 요청 데이터가 URL 쿼리 문자열에 드러날 수 있고, 캐시나 북마크에 유리하다. 조회 목적이므로 같은 요청을 여러 번 보내도 서버 상태가 바뀌지 않는 방향으로 설계하는 것이 좋다.

POST는 요청 본문에 데이터를 담아 서버에 전달한다. 회원 가입, 주문 생성, 검색 조건이 복잡한 제출처럼 서버가 새 리소스를 만들거나 처리를 수행할 때 자주 쓴다. 민감 정보가 URL에 남지 않는다는 장점이 있지만, HTTPS 없이 보내면 본문도 안전하지 않다.

PUT은 리소스 전체 교체, PATCH는 일부 수정, DELETE는 삭제를 표현한다. RESTful API에서는 URI가 리소스를 나타내고 HTTP 메서드가 행위를 나타내도록 설계한다.

#### 면접 답변

> GET은 주로 리소스 조회에 사용하고 요청 정보가 URL에 드러날 수 있어 캐시와 북마크에 유리합니다. POST는 요청 본문에 데이터를 담아 생성이나 처리를 수행할 때 사용하며, REST API에서는 URI는 리소스, 메서드는 행위를 표현하도록 설계합니다.

#### 헷갈리는 포인트
- POST라고 해서 자동으로 암호화되는 것은 아니다.
- GET은 "본문을 절대 못 가진다"보다 "조회 의미와 캐시 특성 때문에 쿼리로 표현하는 경우가 많다"로 이해한다.
- 삭제를 `GET /deleteUser`처럼 표현하면 HTTP 메서드의 의미를 살리지 못한다.

### HTTP 상태 코드와 캐시
<!-- article: network-http-status-cache -->

#### 한눈에 보기
- 2xx는 성공, 3xx는 리다이렉션, 4xx는 클라이언트 오류, 5xx는 서버 오류다.
- HTTP 캐시는 네트워크 비용과 서버 부하를 줄인다.
- `no-cache`와 `no-store`는 이름이 비슷하지만 의미가 다르다.

#### 핵심 정리

상태 코드는 응답 결과를 빠르게 알려준다. 200은 성공, 201은 생성 성공, 204는 본문 없는 성공이다. 301과 302는 리다이렉션, 304는 캐시된 응답을 다시 써도 된다는 의미다. 400번대는 요청 문제, 500번대는 서버나 게이트웨이 문제로 본다.

캐시는 같은 리소스를 반복해서 받을 때 대역폭과 응답 시간을 줄인다. 서버는 `Cache-Control`, `ETag`, `Last-Modified` 같은 헤더로 캐시 정책을 전달한다. 클라이언트는 조건부 요청을 보내 변경 여부를 확인하고, 변경이 없으면 304 응답을 받아 기존 캐시를 사용한다.

`no-cache`는 캐시 저장은 가능하지만 사용하기 전에 서버 검증을 요구한다. `no-store`는 민감 정보처럼 저장 자체를 막아야 할 때 사용한다.

#### 면접 답변

> HTTP 상태 코드는 응답 결과를 범주화합니다. 2xx는 성공, 3xx는 리다이렉션, 4xx는 클라이언트 오류, 5xx는 서버 오류입니다. 캐시는 응답을 재사용해 성능을 높이고, no-cache는 재검증 후 사용, no-store는 저장 금지를 의미합니다.

#### 헷갈리는 포인트
- 304는 실패가 아니라 캐시 재사용을 의미하는 응답이다.
- 민감한 사용자별 응답은 프록시나 CDN에 저장되지 않도록 명확한 캐시 헤더가 필요하다.
- 504 Gateway Timeout은 보통 중간 프록시나 게이트웨이가 뒤 서버의 응답을 제때 받지 못할 때 발생한다.

### HTTPS와 TLS 인증서
<!-- article: network-https-tls-certificate -->

#### 한눈에 보기
- HTTPS는 HTTP에 TLS를 더해 암호화, 인증, 무결성을 제공한다.
- 인증서는 서버 공개키와 도메인 정보를 CA 서명으로 검증하게 해준다.
- 실제 데이터 전송은 대칭키로 처리하고, 키 합의에 공개키 기반 절차를 사용한다.

#### 핵심 정리

HTTP는 평문 통신이므로 중간에서 내용을 볼 수 있고 변조할 수도 있다. HTTPS는 TLS를 통해 통신 내용을 암호화하고, 서버가 진짜 해당 도메인의 서버인지 인증서로 확인하며, 전송 중 변경 여부도 검증한다.

TLS 핸드셰이크에서 클라이언트와 서버는 사용할 암호 스위트와 키 재료를 협상한다. 인증서는 서버의 공개키와 도메인 정보를 담고 있으며, 신뢰할 수 있는 인증 기관의 서명으로 검증된다. 검증이 끝나면 양쪽은 같은 세션 키를 만들어 이후 데이터를 빠른 대칭 암호화로 주고받는다.

현대 TLS에서는 키 교환에 ECDHE를 사용해 순방향 비밀성을 제공하는 방식이 일반적이다. 즉 장기 개인키가 나중에 노출되더라도 과거 세션의 내용을 바로 복호화하기 어렵게 만든다.

#### 면접 답변

> HTTPS는 HTTP 위에 TLS를 적용해 기밀성, 무결성, 서버 인증을 제공합니다. TLS 인증서는 서버 공개키와 도메인 정보를 CA가 서명한 문서이고, 클라이언트는 이를 검증한 뒤 세션 키를 합의해 이후 데이터를 대칭키로 암호화합니다.

#### 헷갈리는 포인트
- 인증서는 대칭키 자체를 클라이언트에게 전달하는 문서가 아니다.
- HTTPS도 기본적으로 HTTP 메시지를 사용하며, 전송 전에 TLS로 보호한다.
- 비대칭 암호화는 주로 인증과 키 합의에 쓰고, 대량 데이터는 대칭키로 암호화한다.

### HTTP 버전의 흐름
<!-- article: network-http-versions -->

#### 한눈에 보기
- HTTP/1.0은 요청마다 연결을 새로 여는 비용이 컸다.
- HTTP/1.1은 지속 연결과 Host 헤더 등으로 효율을 개선했다.
- HTTP/2는 멀티플렉싱을 도입했고, HTTP/3는 QUIC 기반으로 전환했다.

#### 핵심 정리

HTTP/1.0은 요청마다 TCP 연결을 새로 만드는 방식이 일반적이었다. 작은 리소스가 많은 웹 페이지에서는 3-way handshake가 반복되어 지연이 커졌다.

HTTP/1.1은 지속 연결을 기본으로 사용해 하나의 TCP 연결에서 여러 요청을 처리할 수 있게 했다. Host 헤더가 중요해지면서 하나의 IP에서 여러 도메인을 서비스하는 가상 호스팅도 쉬워졌다. 다만 파이프라이닝과 순서 문제 때문에 Head-of-Line Blocking이 남아 있었다.

HTTP/2는 하나의 TCP 연결에서 여러 스트림을 동시에 처리하는 멀티플렉싱, HPACK 헤더 압축, 서버 푸시를 도입했다. 그러나 TCP 위에서 동작하므로 TCP 레벨의 패킷 손실이 모든 스트림에 영향을 줄 수 있다. HTTP/3는 UDP 위의 QUIC을 사용해 연결 설정 지연을 줄이고 스트림 단위 손실 격리를 제공한다.

#### 면접 답변

> HTTP/1.1은 HTTP/1.0보다 지속 연결과 Host 헤더, 캐시 제어가 개선되었습니다. HTTP/2는 TCP 연결 하나에서 여러 요청을 멀티플렉싱하고 헤더를 압축하지만 TCP 수준의 HOL Blocking은 남습니다. HTTP/3는 QUIC 기반으로 이 한계를 완화합니다.

#### 헷갈리는 포인트
- HTTP/2는 UDP 기반이 아니라 TCP 기반이다.
- HTTP/3가 암호화를 제거한 것이 아니라 QUIC 안에 TLS 1.3을 통합한다.
- Keep-Alive는 연결 비용을 줄이지만, 유휴 연결이 너무 많으면 서버 자원을 잡아먹을 수 있다.

## TCP와 UDP
<!-- category: network-tcp-udp -->

### TCP와 UDP 선택 기준
<!-- article: network-tcp-udp-choice -->

#### 한눈에 보기
- TCP는 연결 지향, 신뢰성, 순서 보장을 제공한다.
- UDP는 비연결, 낮은 오버헤드, 빠른 전송에 유리하다.
- 정확성이 중요한지, 지연이 중요한지가 선택 기준이다.

#### 핵심 정리

TCP는 연결을 수립한 뒤 데이터를 순서대로, 손실 없이 전달하려고 노력한다. ACK, 재전송, 흐름 제어, 혼잡 제어가 있기 때문에 파일 전송, 웹 요청, 이메일, SSH처럼 정확성이 중요한 통신에 적합하다.

UDP는 연결 수립 과정이 없고 헤더가 단순하다. 손실이나 순서 뒤바뀜을 전송 계층에서 보장하지 않지만 지연이 작고 구현이 단순하다. DNS, 실시간 음성, 게임, 스트리밍처럼 일부 손실보다 빠른 응답이 중요한 경우에 어울린다.

UDP를 쓴다고 신뢰성을 전혀 만들 수 없는 것은 아니다. 애플리케이션이 필요한 재전송, 순서 제어, 오류 복구를 직접 구현할 수 있다. QUIC이 대표적으로 UDP 위에서 더 높은 수준의 전송 기능을 제공한다.

#### 면접 답변

> TCP는 연결 지향 프로토콜로 신뢰성과 순서 보장을 제공해 정확한 데이터 전송에 적합합니다. UDP는 비연결 방식이라 오버헤드가 작고 빠르지만 손실을 보장하지 않으므로 실시간 스트리밍이나 게임처럼 지연이 더 중요한 상황에 적합합니다.

#### 헷갈리는 포인트
- UDP가 항상 TCP보다 좋은 성능을 낸다는 뜻은 아니다.
- 파일 전송처럼 무결성이 중요한 경우는 보통 TCP가 더 자연스럽다.
- HTTP/3는 UDP를 직접 쓰는 것이 아니라 UDP 위의 QUIC을 사용한다.

#### 관련 퀴즈
- `network-c002`
- `network-t001`
- `network-d001`

### 흐름 제어와 혼잡 제어
<!-- article: network-flow-congestion-control -->

#### 한눈에 보기
- 흐름 제어는 수신자가 감당할 수 있는 속도로 보내게 한다.
- 혼잡 제어는 네트워크 전체가 막히지 않도록 송신량을 조절한다.
- Sliding Window, Slow Start, AIMD는 TCP 제어를 이해하는 핵심 키워드다.

#### 핵심 정리

흐름 제어는 송신자와 수신자 사이의 속도 차이를 다룬다. 수신자가 처리하지 못할 만큼 빠르게 보내면 버퍼가 넘치므로, 수신 윈도우를 통해 "이만큼까지 받을 수 있다"는 정보를 알려준다.

Stop-and-Wait은 하나를 보내고 ACK를 받은 뒤 다음 데이터를 보내는 단순한 방식이다. 구현은 쉽지만 RTT가 큰 환경에서는 기다리는 시간이 길다. Sliding Window는 ACK를 기다리는 동안에도 허용된 범위 안에서 여러 세그먼트를 연속으로 보내 대역폭 활용률을 높인다.

혼잡 제어는 수신자 한 명이 아니라 네트워크 전체의 혼잡을 다룬다. TCP는 Slow Start로 조심스럽게 시작해 ACK를 받으며 전송량을 빠르게 늘리고, 혼잡 신호가 보이면 AIMD처럼 윈도우를 크게 줄인다. 중복 ACK가 반복되면 Fast Retransmit으로 타임아웃 전에 재전송하고, Fast Recovery로 전송을 회복한다.

#### 면접 답변

> 흐름 제어는 수신자 버퍼가 넘치지 않도록 수신자의 처리 가능량에 맞춰 송신 속도를 조절하는 것입니다. 혼잡 제어는 네트워크 전체의 혼잡을 감지해 혼잡 윈도우를 조절하는 것이며, Slow Start, AIMD, Fast Retransmit 같은 기법이 사용됩니다.

#### 헷갈리는 포인트
- 흐름 제어는 주로 수신자 보호, 혼잡 제어는 네트워크 보호에 가깝다.
- Stop-and-Wait은 단순하지만 링크 활용률이 낮다.
- Slow Start는 이름과 달리 초기에 윈도우를 지수적으로 빠르게 늘린다.

#### 관련 퀴즈
- `network-c011`
- `network-c034`
- `network-c035`
- `network-c036`
- `network-c037`
- `network-c038`

### 실시간 통신: WebSocket, SSE, 폴링
<!-- article: network-realtime-communication -->

#### 한눈에 보기
- REST 요청은 기본적으로 클라이언트가 필요할 때 요청하는 방식이다.
- WebSocket은 연결을 유지하며 양방향 실시간 통신을 제공한다.
- SSE는 서버가 클라이언트로 이벤트를 지속 전송할 때 단순한 선택지가 될 수 있다.

#### 핵심 정리

일반적인 REST API는 요청과 응답이 한 번씩 오가는 구조다. 상품 조회, 주문 생성, 설정 변경처럼 명확한 CRUD 작업에는 단순하고 캐시하기 쉬운 REST가 적합하다.

채팅, 실시간 알림, 협업 편집, 주식 시세처럼 서버가 먼저 데이터를 밀어줘야 한다면 WebSocket이 더 자연스럽다. WebSocket은 처음에 HTTP 업그레이드로 연결을 맺고, 이후 하나의 연결에서 양방향 메시지를 주고받는다.

SSE(Server-Sent Events)는 서버에서 클라이언트로 단방향 이벤트를 보내는 데 적합하다. 클라이언트가 계속 질문하는 폴링보다 불필요한 요청을 줄일 수 있고, WebSocket보다 단순하게 구현할 수 있다. 다만 양방향 상호작용이 핵심이면 WebSocket이 더 맞다.

#### 면접 답변

> REST는 클라이언트가 필요할 때 요청하고 응답받는 구조라 CRUD에 적합합니다. WebSocket은 연결을 유지하며 서버와 클라이언트가 양방향으로 메시지를 보낼 수 있어 채팅이나 실시간 알림에 적합하고, SSE는 서버 단방향 이벤트 전송에 적합합니다.

#### 헷갈리는 포인트
- WebSocket은 UDP 기반이 아니라 보통 TCP 연결 위에서 동작한다.
- `wss://`는 TLS로 보호되는 WebSocket이다.
- 실시간 요구가 약하면 주기적 폴링이나 캐시 전략이 더 단순할 수 있다.

## 연결 설정과 종료
<!-- category: network-connections -->

### TCP 3-way handshake
<!-- article: network-tcp-three-way-handshake -->

#### 한눈에 보기
- TCP는 데이터를 보내기 전에 연결을 수립한다.
- 3-way handshake는 SYN, SYN+ACK, ACK 순서로 진행된다.
- 양쪽의 송수신 가능성과 초기 시퀀스 번호를 확인한다.

#### 핵심 정리

클라이언트가 서버에 연결하고 싶으면 먼저 SYN을 보낸다. 서버는 요청을 받고 자신의 SYN과 클라이언트 SYN에 대한 ACK를 함께 보낸다. 클라이언트가 마지막 ACK를 보내면 양쪽 모두 연결이 수립된 상태가 된다.

이 과정은 단순히 "접속 가능 여부"만 확인하지 않는다. TCP는 바이트 스트림을 순서대로 전달해야 하므로 양쪽의 초기 시퀀스 번호를 교환하고 동기화한다. 이후 데이터는 이 시퀀스 번호와 ACK를 바탕으로 순서와 손실을 관리한다.

HTTPS라면 TCP 연결 뒤에 TLS 핸드셰이크가 추가된다. 따라서 첫 HTTP 응답 바이트를 받기까지 URL 해석, DNS 조회, TCP 연결, TLS 연결, HTTP 요청의 순서가 이어진다.

#### 면접 답변

> TCP 3-way handshake는 클라이언트의 SYN, 서버의 SYN+ACK, 클라이언트의 ACK로 연결을 수립하는 과정입니다. 이때 양쪽의 송수신 가능성을 확인하고 초기 시퀀스 번호를 동기화해 신뢰성 있는 바이트 스트림 전송을 준비합니다.

#### 헷갈리는 포인트
- 3-way handshake는 TLS 키 교환 절차가 아니다.
- DNS 조회와 TCP 연결은 서로 다른 단계다.
- 서버가 SYN 대기 큐를 과도하게 점유당하면 SYN flood 공격에 취약해질 수 있다.

#### 관련 퀴즈

- `network-o001`
- `network-s007`

### TCP 4-way handshake와 TIME_WAIT
<!-- article: network-tcp-four-way-time-wait -->

#### 한눈에 보기
- TCP는 전이중 통신이므로 양방향 종료를 각각 처리한다.
- 종료는 보통 FIN, ACK, FIN, ACK 흐름으로 진행된다.
- TIME_WAIT은 지연 패킷과 마지막 ACK 유실에 대비한다.

#### 핵심 정리

TCP 연결은 양쪽이 동시에 데이터를 주고받을 수 있다. 한쪽이 더 이상 보낼 데이터가 없다고 FIN을 보내도, 반대 방향의 데이터 전송은 아직 남아 있을 수 있다. 그래서 종료 과정은 각 방향의 송신 종료를 따로 확인한다.

일반적으로 먼저 끊는 쪽이 FIN을 보내고, 상대가 ACK를 보낸다. 상대도 보낼 데이터를 모두 마치면 FIN을 보내고, 먼저 끊은 쪽이 마지막 ACK를 보낸다. 이 마지막 ACK를 보낸 쪽은 바로 사라지지 않고 TIME_WAIT 상태에서 일정 시간 기다린다.

TIME_WAIT은 이전 연결의 지연 세그먼트가 새 연결에 섞이는 것을 막고, 마지막 ACK가 유실되었을 때 상대의 FIN에 다시 응답할 수 있게 한다.

#### 면접 답변

> TCP 4-way handshake는 양방향 연결을 각각 종료하기 위해 FIN과 ACK를 주고받는 과정입니다. TIME_WAIT은 마지막 ACK 유실에 대비하고, 이전 연결의 지연 패킷이 새 연결에 섞이지 않도록 일정 시간 기다리는 상태입니다.

#### 헷갈리는 포인트
- UDP에는 TCP 같은 연결 종료 핸드셰이크가 없다.
- TIME_WAIT은 DNS 캐시를 비우는 상태가 아니다.
- 항상 클라이언트만 TIME_WAIT에 들어가는 것은 아니고, 먼저 능동 종료한 쪽이 주로 들어간다.

#### 관련 퀴즈
- `network-c024`
- `network-o002`

### Keep-Alive와 연결 비용
<!-- article: network-keep-alive-cost -->

#### 한눈에 보기
- 연결을 매번 새로 만들면 TCP 핸드셰이크 비용이 반복된다.
- Keep-Alive는 연결을 재사용해 지연을 줄인다.
- 유휴 연결이 많으면 서버 자원 비용이 커질 수 있다.

#### 핵심 정리

웹 페이지 하나를 그릴 때 HTML, CSS, JavaScript, 이미지처럼 여러 리소스를 요청한다. 요청마다 TCP 연결을 새로 만들면 handshake 비용과 지연이 반복된다. HTTP/1.1의 지속 연결은 하나의 연결을 재사용해 이 비용을 줄였다.

그러나 연결을 오래 유지하면 서버는 소켓, 메모리, 파일 디스크립터 같은 자원을 계속 잡아둔다. 요청이 드문 클라이언트가 매우 많다면 유휴 연결 유지가 오히려 부담이 될 수 있다.

면접에서는 Keep-Alive의 장점과 비용을 함께 말하는 것이 좋다. 트래픽 패턴, 서버 자원, 타임아웃 설정을 보고 적절한 연결 유지 시간을 정해야 한다.

#### 면접 답변

> Keep-Alive는 TCP 연결을 재사용해 반복적인 3-way handshake 비용을 줄이고 응답 지연을 낮춥니다. 다만 유휴 연결이 많으면 서버의 소켓과 메모리를 점유하므로, 트래픽 패턴에 맞는 타임아웃과 연결 수 제한이 필요합니다.

#### 헷갈리는 포인트
- 연결 재사용은 항상 무조건 이득이 아니다.
- HTTP/2도 TCP 연결 하나에 여러 스트림을 싣지만, TCP 레벨 손실 영향은 남을 수 있다.
- 모바일 환경처럼 네트워크가 자주 바뀌면 연결 재수립 비용이 체감될 수 있다.

#### 관련 퀴즈
- `network-c025`
- `network-t009`
- `network-s012`

## IP, MAC, ARP
<!-- category: network-addressing -->

### IP 주소와 MAC 주소
<!-- article: network-ip-mac-address -->

#### 한눈에 보기
- IP 주소는 네트워크 간 라우팅에 쓰는 논리적 주소다.
- MAC 주소는 같은 네트워크 안에서 프레임 전달에 쓰는 물리적 주소다.
- IP는 목적지까지의 큰 경로, MAC은 다음 홉까지의 전달에 가깝다.

#### 핵심 정리

IP 주소는 네트워크 계층에서 사용한다. 라우터는 목적지 IP와 라우팅 테이블을 보고 다음 홉을 결정한다. IPv4는 32비트 주소를 사용하고, 사설 IP 대역은 내부 네트워크에서 자유롭게 사용할 수 있다.

MAC 주소는 데이터링크 계층에서 사용한다. 같은 LAN 안에서 이더넷 프레임을 어느 장치로 전달할지 구분하는 주소다. 스위치는 MAC 주소 테이블을 학습해 프레임을 적절한 포트로 보낸다.

인터넷 너머 목적지로 패킷을 보낼 때도 매 홉의 실제 프레임 목적지 MAC은 계속 바뀐다. 목적지 IP는 최종 목적지를 가리키지만, MAC 주소는 현재 링크에서 다음 장치를 가리키기 때문이다.

#### 면접 답변

> IP 주소는 네트워크 계층의 논리적 주소로 라우터가 목적지 네트워크까지 경로를 선택할 때 사용합니다. MAC 주소는 데이터링크 계층의 물리적 주소로 같은 네트워크 안에서 프레임을 전달할 때 사용합니다.

#### 헷갈리는 포인트
- IP 주소는 네트워크 환경에 따라 바뀔 수 있다.
- MAC 주소는 같은 네트워크 안의 프레임 전달에 필요하다.
- 라우터를 지날 때 이더넷 헤더의 MAC 주소는 다음 링크에 맞게 바뀐다.

#### 관련 퀴즈
- `network-c005`

### DNS와 재귀 조회
<!-- article: network-dns-resolution -->

#### 한눈에 보기
- DNS는 도메인 이름을 IP 주소로 바꿔주는 분산 시스템이다.
- 재귀 리졸버는 루트, TLD, 권한 있는 DNS를 따라가며 답을 찾는다.
- 캐시는 빠르지만 TTL과 장애 전파를 함께 고려해야 한다.

#### 핵심 정리

사용자가 `www.example.com`을 입력하면 브라우저와 운영체제는 먼저 로컬 캐시를 확인한다. 없으면 설정된 재귀 DNS 리졸버에 질의한다. 리졸버는 루트 DNS, TLD DNS, 권한 있는 DNS를 거쳐 최종 IP 주소를 얻고 이를 클라이언트에게 돌려준다.

DNS 응답은 TTL 동안 캐시될 수 있다. TTL이 길면 조회 비용은 줄지만 IP 변경이나 장애 전환이 늦게 반영될 수 있다. TTL이 짧으면 변경 반영은 빠르지만 DNS 질의가 늘어난다.

도메인으로 접속이 안 되는데 IP 직접 접속은 된다면 DNS 설정, 레코드 누락, 잘못된 CNAME, 로컬 DNS 캐시 문제를 먼저 의심할 수 있다.

#### 면접 답변

> DNS는 사람이 읽는 도메인 이름을 IP 주소로 변환하는 시스템입니다. 재귀 리졸버는 루트 DNS, TLD DNS, 권한 있는 DNS를 차례로 조회해 IP를 얻고, TTL 동안 결과를 캐시해 다음 조회를 빠르게 처리합니다.

#### 헷갈리는 포인트
- DNS는 IP를 MAC 주소로 바꾸는 프로토콜이 아니다. 그 역할은 ARP다.
- TTL을 줄이면 장애 전환에는 유리하지만 질의량이 늘 수 있다.
- 애플리케이션에서 DNS 지연이 문제면 캐시, 타임아웃, 재시도 전략을 함께 본다.

#### 관련 퀴즈

### ARP, ICMP, ping, NAT
<!-- article: network-arp-icmp-nat -->

#### 한눈에 보기
- ARP는 같은 네트워크에서 IP에 해당하는 MAC 주소를 찾는다.
- ICMP는 IP 계층의 오류 보고와 진단에 사용된다.
- NAT는 사설 IP를 공인 IP로 변환해 주소 부족을 완화한다.

#### 핵심 정리

IP 패킷을 실제 이더넷 프레임으로 보내려면 다음 홉의 MAC 주소가 필요하다. ARP는 "이 IP를 가진 장치는 MAC 주소를 알려달라"고 브로드캐스트하고, 해당 장치가 유니캐스트로 응답하게 한다. 결과는 ARP 캐시에 저장된다.

ICMP는 네트워크 오류와 진단을 위한 프로토콜이다. ping은 ICMP Echo Request와 Echo Reply를 사용해 대상까지 도달 가능한지와 왕복 시간을 확인한다. 방화벽이 ICMP를 차단하면 실제 서비스는 살아 있어도 ping이 실패할 수 있다.

NAT는 내부 사설 IP와 외부 공인 IP를 변환한다. 여러 내부 장치가 하나의 공인 IP를 공유할 수 있어 IPv4 주소 부족을 완화하고 내부 구조를 감출 수 있다. 다만 포트 매핑과 연결 추적이 필요하다.

#### 면접 답변

> ARP는 같은 네트워크 안에서 목적지 IP에 대응하는 MAC 주소를 찾는 프로토콜입니다. ICMP는 오류 보고와 진단에 사용되며 ping은 ICMP Echo로 도달성과 RTT를 확인합니다. NAT는 사설 IP를 공인 IP로 변환해 여러 장치가 외부와 통신하게 합니다.

#### 헷갈리는 포인트
- ARP는 도메인 이름을 해석하지 않는다.
- ping 실패가 항상 서버 장애를 의미하지는 않는다.
- NAT는 속도를 높이기 위한 기술이라기보다 주소 변환과 내부망 은닉에 가깝다.

#### 관련 퀴즈
- `network-c020`
- `network-c027`
- `network-c028`

## 쿠키, 세션, JWT
<!-- category: network-auth-state -->

### HTTP 무상태와 쿠키
<!-- article: network-cookie-stateless -->

#### 한눈에 보기
- HTTP는 기본적으로 이전 요청 상태를 기억하지 않는다.
- 쿠키는 클라이언트에 저장되고 요청마다 서버로 전송될 수 있다.
- 보안 속성은 XSS와 CSRF 위험을 줄이는 데 중요하다.

#### 핵심 정리

HTTP는 무상태 프로토콜이므로 서버는 각 요청을 독립적으로 처리한다. 로그인 상태, 장바구니, 사용자 설정처럼 요청 사이에 유지해야 하는 정보가 있으면 별도의 상태 관리 방식이 필요하다.

쿠키는 브라우저에 저장되는 작은 데이터다. 서버는 `Set-Cookie` 응답 헤더로 쿠키를 설정하고, 브라우저는 조건이 맞는 요청에 `Cookie` 헤더를 자동으로 붙인다. Domain, Path, Expires, Max-Age 같은 속성으로 전송 범위와 만료를 조절한다.

보안 관점에서는 `HttpOnly`, `Secure`, `SameSite`가 중요하다. `HttpOnly`는 JavaScript 접근을 막아 XSS 피해를 줄이고, `Secure`는 HTTPS에서만 전송하게 하며, `SameSite`는 CSRF 위험을 낮추는 데 도움을 준다.

#### 면접 답변

> HTTP는 무상태라서 요청 사이의 사용자 상태를 기본적으로 기억하지 않습니다. 쿠키는 클라이언트에 저장되어 요청마다 서버로 전송될 수 있는 데이터이며, 로그인 유지나 사용자 설정 저장에 쓰입니다. 보안을 위해 HttpOnly, Secure, SameSite 같은 속성을 설정합니다.

#### 헷갈리는 포인트
- 쿠키는 클라이언트에 저장되므로 민감 정보를 그대로 넣으면 위험하다.
- HTTPS가 아니면 Secure 쿠키가 전송되지 않는다.
- 쿠키 자체와 세션은 같은 개념이 아니다.

#### 관련 퀴즈
- `network-c009`

### 세션과 Sticky Session
<!-- article: network-session-sticky-session -->

#### 한눈에 보기
- 세션은 서버가 사용자 상태를 저장하고 클라이언트는 세션 ID만 들고 있는 방식이다.
- 서버 로컬 세션은 수평 확장 시 요청이 다른 서버로 가면 문제가 될 수 있다.
- Sticky Session은 같은 사용자를 같은 서버로 보내는 해결책 중 하나다.

#### 핵심 정리

세션 방식에서는 서버가 로그인 사용자 정보를 메모리, 데이터베이스, Redis 같은 저장소에 보관한다. 클라이언트는 세션 ID를 쿠키로 들고 있다가 요청마다 서버에 전달한다. 서버는 이 ID로 사용자를 식별한다.

서버가 여러 대이고 세션을 각 서버의 로컬 메모리에만 저장하면 문제가 생긴다. 첫 요청은 A 서버에서 로그인했는데 다음 요청이 B 서버로 가면 B 서버는 세션을 모른다. Sticky Session은 로드밸런서가 같은 클라이언트의 요청을 같은 서버로 보내도록 고정한다.

다만 Sticky Session은 특정 서버에 부하가 몰리거나 장애 시 세션 유지가 어려울 수 있다. 실무에서는 Redis 같은 외부 세션 저장소로 세션을 공유하는 방식도 많이 사용한다.

#### 면접 답변

> 세션은 서버에 사용자 상태를 저장하고 클라이언트는 세션 ID만 쿠키로 보관하는 방식입니다. 서버 로컬 세션을 쓰면 로드밸런싱 환경에서 같은 사용자의 요청이 같은 서버로 가야 하므로 Sticky Session이 필요할 수 있지만, 외부 세션 저장소가 더 확장성 있는 대안입니다.

#### 헷갈리는 포인트
- Sticky Session은 로드밸런싱 성능을 무조건 높이는 기능이 아니다.
- 세션 저장소가 서버 로컬인지 외부 공유 저장소인지에 따라 설계가 달라진다.
- 쿠키는 세션 ID를 담는 수단으로 사용될 수 있다.

#### 관련 퀴즈
- `network-c014`
- `network-t007`

### JWT와 토큰 인증
<!-- article: network-jwt-token-auth -->

#### 한눈에 보기
- JWT는 Header, Payload, Signature 세 부분으로 구성된다.
- 서버가 세션 상태를 저장하지 않아도 검증 가능한 토큰을 만들 수 있다.
- Payload는 기본적으로 암호화가 아니라 인코딩이므로 민감 정보를 넣으면 안 된다.

#### 핵심 정리

JWT(JSON Web Token)는 `header.payload.signature` 형태의 토큰이다. Header에는 알고리즘과 타입, Payload에는 클레임, Signature에는 위변조 검증을 위한 서명이 들어간다.

JWT의 장점은 서버가 중앙 세션 저장소를 조회하지 않아도 토큰의 서명을 검증해 인증 정보를 확인할 수 있다는 점이다. 여러 서버나 도메인에서 인증 정보를 공유해야 하는 환경에서는 확장성 측면에서 유리할 수 있다.

단점도 분명하다. 토큰이 탈취되면 만료 전까지 무효화하기 어렵고, Payload는 누구나 디코딩할 수 있으므로 민감 정보를 넣으면 안 된다. 만료 시간을 짧게 두고, Refresh Token, 블랙리스트, 키 회전 같은 보완 전략을 함께 고려한다.

#### 면접 답변

> JWT는 Header, Payload, Signature를 점으로 구분한 토큰입니다. 서버는 서명을 검증해 토큰 위변조 여부를 확인할 수 있어 상태 저장 부담을 줄일 수 있지만, Payload는 암호화가 아니므로 민감 정보를 넣으면 안 되고 탈취 시 무효화 전략이 필요합니다.

#### 헷갈리는 포인트
- JWT를 쓴다고 완전히 안전한 인증이 자동으로 완성되지는 않는다.
- 토큰 기반 인증도 쿠키에 담아 전송할 수 있다.
- Stateless 인증은 확장성에 유리하지만 즉시 폐기와 권한 변경 반영이 어려울 수 있다.

#### 관련 퀴즈
- `network-c029`
- `network-t007`

## REST API와 CORS
<!-- category: network-rest-cors -->

### REST API 설계
<!-- article: network-rest-api-design -->

#### 한눈에 보기
- REST는 자원을 URI로 표현하고 행위를 HTTP 메서드로 표현하는 아키텍처 스타일이다.
- 각 요청은 필요한 정보를 담아 독립적으로 처리되는 무상태성을 따른다.
- 캐시 가능성, 균일한 인터페이스, 계층화도 중요한 제약이다.

#### 핵심 정리

REST API에서 URI는 명사를 중심으로 리소스를 표현한다. 예를 들어 `/users/123`은 사용자 123이라는 자원을 의미하고, 조회는 GET, 생성은 POST, 전체 수정은 PUT, 일부 수정은 PATCH, 삭제는 DELETE로 표현한다.

무상태성은 서버가 클라이언트의 대화 상태를 기억하지 않는다는 뜻이다. 각 요청은 인증 정보와 필요한 컨텍스트를 포함해야 한다. 이 덕분에 서버를 수평 확장하기 쉬워지지만, 요청마다 필요한 정보가 누락되면 처리할 수 없다.

RESTful하다는 말은 단순히 JSON을 반환한다는 뜻이 아니다. URI 설계, 메서드 의미, 상태 코드, 캐시 정책, 계층화된 구조까지 HTTP의 의미를 잘 활용한다는 뜻에 가깝다.

#### 면접 답변

> REST API는 URI로 리소스를 표현하고 HTTP 메서드로 행위를 표현하는 방식입니다. 서버는 클라이언트 상태를 저장하지 않는 무상태성을 따르며, 각 요청은 독립적으로 처리될 수 있어야 합니다. JSON 사용 여부만으로 RESTful하다고 볼 수는 없습니다.

#### 헷갈리는 포인트
- 모든 API를 POST 하나로 처리하면 HTTP 메서드 의미를 살리기 어렵다.
- URI에 동사를 넣기보다 리소스 이름을 중심으로 설계한다.
- 연결을 계속 유지해야 한다는 것은 REST 제약이 아니다.

#### 관련 퀴즈
- `network-c016`
- `network-c030`

### CORS와 Preflight
<!-- article: network-cors-preflight -->

#### 한눈에 보기
- 출처는 프로토콜, 도메인, 포트의 조합이다.
- 브라우저는 동일 출처 정책으로 다른 출처 접근을 기본 제한한다.
- CORS는 서버가 응답 헤더로 교차 출처 요청 허용 여부를 알려주는 메커니즘이다.

#### 핵심 정리

프론트엔드가 `http://localhost:3000`에서 실행되고 API가 `http://localhost:8080`에 있으면 포트가 다르므로 다른 출처다. 브라우저는 보안상 다른 출처 요청을 제한하고, 서버가 CORS 헤더로 허용한 경우에만 응답을 프론트엔드 코드에 노출한다.

단순 요청 조건을 벗어나는 요청은 실제 요청 전에 OPTIONS Preflight를 보낸다. 서버가 `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers` 같은 헤더로 허용하면 브라우저가 실제 요청을 이어서 보낸다.

CORS 문제는 브라우저 보안 정책이다. 서버끼리 직접 통신할 때 같은 방식으로 막히는 것은 아니다. 따라서 프론트엔드에서 우회하기보다 API 서버나 프록시가 올바른 CORS 응답을 내려야 한다.

#### 면접 답변

> CORS는 브라우저의 동일 출처 정책 때문에 제한되는 교차 출처 요청을 서버가 허용할 수 있게 하는 메커니즘입니다. 단순 요청이 아니면 브라우저가 먼저 OPTIONS Preflight를 보내고, 서버가 허용 헤더를 응답하면 실제 요청을 보냅니다.

#### 헷갈리는 포인트
- CORS 오류는 대개 서버 응답 헤더 설정 문제다.
- 모든 GET 요청에 Preflight가 생기는 것은 아니다.
- Origin이 같으려면 프로토콜, 도메인, 포트가 모두 같아야 한다.

#### 관련 퀴즈

### API Gateway와 서비스 메시
<!-- article: network-api-gateway-service-mesh -->

#### 한눈에 보기
- API Gateway는 클라이언트 진입점에서 인증, 라우팅, 제한, 집계를 담당할 수 있다.
- 서비스 메시(Service Mesh)는 서비스 간 통신의 관찰, 보안, 재시도를 인프라 계층에서 다룬다.
- mTLS는 서비스 간 상호 인증과 암호화가 필요할 때 쓰인다.

#### 핵심 정리

마이크로서비스가 많아지면 클라이언트가 각 서비스를 직접 호출하기 어려워진다. API Gateway는 외부 요청의 단일 진입점 역할을 하며 인증, 권한 확인, 요청 라우팅, rate limiting, 응답 집계 같은 공통 기능을 처리한다.

서비스 메시는 서비스 간 내부 통신에 초점을 둔다. 각 서비스 옆의 프록시가 트래픽을 중계하며 재시도, 타임아웃, 회로 차단, 메트릭, 분산 추적, mTLS 같은 기능을 애플리케이션 코드 바깥에서 제공한다.

둘은 경쟁 관계라기보다 관심사가 다르다. API Gateway는 외부 클라이언트와 시스템 사이의 문이고, 서비스 메시는 내부 서비스 간 통신망을 안정적으로 운영하는 도구에 가깝다.

#### 면접 답변

> API Gateway는 외부 요청의 진입점에서 인증, 라우팅, rate limiting, 응답 집계 같은 공통 기능을 처리합니다. 서비스 메시는 내부 서비스 간 통신의 재시도, 타임아웃, 관찰성, mTLS 같은 기능을 인프라 계층에서 제공해 마이크로서비스 운영 복잡도를 줄입니다.

#### 헷갈리는 포인트
- 서비스가 적고 통신이 단순하면 서비스 메시가 과한 복잡도가 될 수 있다.
- mTLS는 서버만 인증하는 일반 TLS와 달리 클라이언트와 서버가 서로 인증한다.
- API Gateway가 내부 서비스 간 모든 통신 문제를 자동으로 해결하지는 않는다.

#### 관련 퀴즈
- `network-t012`
- `network-t013`
- `network-t015`
- `network-s010`

### Circuit Breaker와 장애 격리
<!-- article: network-circuit-breaker-fault-isolation -->

#### 한눈에 보기
- 느린 외부 호출은 호출한 서비스의 스레드와 커넥션을 고갈시킬 수 있다.
- Circuit Breaker는 실패가 계속될 때 호출을 잠시 차단해 장애 전파를 막는다.
- 타임아웃, 재시도, bulkhead, fallback과 함께 설계한다.

#### 핵심 정리

마이크로서비스 A가 B를 호출하는데 B가 3초 이상 지연되면, A의 요청 처리 스레드가 계속 대기한다. 이런 상황이 누적되면 A도 새 요청을 처리하지 못하고 장애가 연쇄적으로 퍼진다.

Circuit Breaker는 실패율이나 지연이 기준을 넘으면 회로를 열어 B 호출을 즉시 실패시키거나 fallback 응답을 반환한다. 일정 시간이 지나면 일부 요청만 보내 회복 여부를 확인하고, 정상으로 판단되면 다시 회로를 닫는다.

중요한 점은 무한 재시도를 피하는 것이다. 재시도는 일시적 오류에는 도움이 되지만, 이미 과부하인 서비스에 더 많은 요청을 보내 장애를 키울 수 있다. 타임아웃, 제한된 재시도, 회로 차단을 함께 둬야 한다.

#### 면접 답변

> Circuit Breaker는 의존 서비스의 실패나 지연이 계속될 때 호출을 잠시 차단해 장애 전파를 막는 패턴입니다. 타임아웃과 제한된 재시도, fallback을 함께 사용하면 호출한 서비스의 스레드 고갈과 연쇄 장애를 줄일 수 있습니다.

#### 헷갈리는 포인트
- Circuit Breaker는 성능 최적화보다 장애 격리에 가깝다.
- 재시도를 많이 하면 항상 좋아지는 것이 아니라 장애 서비스를 더 압박할 수 있다.
- 504 오류는 로드밸런서, 게이트웨이, 백엔드 지연을 함께 진단해야 한다.

#### 관련 퀴즈
- `network-t014`
- `network-s002`
- `network-s004`

### 브라우저에 URL을 입력하면
<!-- article: network-browser-url-flow -->

#### 한눈에 보기
- URL 해석, DNS 조회, TCP/TLS 연결, HTTP 요청, 응답 처리, 렌더링 순서로 이어진다.
- 네트워크 구간에서는 캡슐화, 라우팅, 로드밸런싱, 서버 처리가 함께 일어난다.
- 이 질문은 여러 네트워크 개념을 하나의 흐름으로 엮어 말하는 대표 면접 질문이다.

#### 핵심 정리

브라우저는 먼저 URL을 파싱해 프로토콜, 도메인, 포트, 경로를 확인한다. 이후 캐시와 DNS를 통해 도메인을 IP 주소로 바꾼다. IP를 얻으면 TCP 연결을 만들고, HTTPS라면 TLS 핸드셰이크로 보안 채널을 만든다.

브라우저가 HTTP 요청을 보내면 운영체제의 TCP/IP 스택이 데이터를 캡슐화한다. TCP는 포트와 시퀀스 정보를 붙이고, IP는 목적지 주소를 붙이며, 이더넷은 다음 홉 MAC 주소를 붙여 프레임으로 전송한다. 중간 라우터들은 IP 기반으로 다음 경로를 선택한다.

서버 측에서는 방화벽, 리버스 프록시, 로드밸런서, 웹 서버, 애플리케이션, 데이터베이스가 차례로 관여할 수 있다. 응답을 받은 브라우저는 HTML, CSS, JavaScript를 파싱해 DOM과 CSSOM을 만들고 화면을 그린다.

#### 면접 답변

> 브라우저에 URL을 입력하면 URL을 해석하고 DNS로 IP를 찾은 뒤 TCP 연결과 HTTPS라면 TLS 핸드셰이크를 수행합니다. 이후 HTTP 요청을 보내고 서버가 응답하면 브라우저는 HTML, CSS, JavaScript를 파싱해 렌더링합니다.

#### 헷갈리는 포인트
- DNS 조회가 끝나야 목적지 IP로 TCP 연결을 만들 수 있다.
- HTTPS에서는 TCP 연결 뒤 TLS 핸드셰이크가 필요하다.
- 브라우저 렌더링은 네트워크 응답 이후의 별도 단계다.

#### 관련 퀴즈
- `network-o001`
- `network-o006`

### 로드밸런싱과 헬스 체크
<!-- article: network-load-balancing-health-check -->

#### 한눈에 보기
- 로드밸런싱은 여러 서버에 트래픽을 분산해 가용성과 확장성을 높인다.
- L4는 IP와 포트 중심, L7은 HTTP 헤더와 경로 같은 애플리케이션 정보까지 본다.
- 헬스 체크는 장애 서버로 요청이 가지 않도록 하는 핵심 기능이다.

#### 핵심 정리

로드밸런서는 클라이언트 요청을 여러 백엔드 서버에 나눠 보낸다. Round Robin은 순서대로 분산하고, Least Connections는 현재 연결 수가 적은 서버를 고른다. 연결 시간이 긴 서비스에서는 Least Connections가 Round Robin보다 더 적합할 수 있다.

L4 로드밸런서는 전송 계층 정보인 IP와 포트를 기준으로 빠르게 분산한다. L7 로드밸런서는 HTTP 경로, 헤더, 쿠키 등을 보고 `/api`와 `/static`을 다른 서버로 보내는 식의 세밀한 라우팅이 가능하다.

헬스 체크는 백엔드 서버가 정상인지 주기적으로 확인한다. 응답하지 않거나 오류가 계속되는 서버를 풀에서 제외하고, 회복되면 다시 트래픽을 보낸다. 헬스 체크가 없으면 장애 서버가 계속 요청을 받을 수 있다.

#### 면접 답변

> 로드밸런싱은 여러 서버에 트래픽을 분산해 부하를 낮추고 가용성을 높이는 기술입니다. L4는 IP와 포트 중심으로, L7은 HTTP 경로나 헤더까지 보고 분산합니다. 헬스 체크는 장애 서버를 제외해 정상 서버로만 요청을 보내기 위해 필요합니다.

#### 헷갈리는 포인트
- Round Robin은 단순하지만 서버 성능이나 실제 연결 수를 반영하지 못할 수 있다.
- L7은 세밀한 제어가 가능하지만 L4보다 처리 비용이 클 수 있다.
- 특정 시간대의 504는 백엔드 과부하, 타임아웃, 로드밸런서 설정을 함께 봐야 한다.

#### 관련 퀴즈
- `network-c014`
- `network-c032`

- `network-t006`
- `network-s002`

### CDN과 캐시 전략
<!-- article: network-cdn-cache-strategy -->

#### 한눈에 보기
- CDN은 사용자와 가까운 엣지 서버에서 콘텐츠를 제공해 지연을 줄인다.
- 정적 파일 비중이 높은 서비스에서 효과가 크다.
- 사용자별 민감 응답은 캐시 정책을 엄격히 관리해야 한다.

#### 핵심 정리

CDN(Content Delivery Network)은 이미지, CSS, JavaScript 같은 콘텐츠를 전 세계 엣지 서버에 캐싱한다. 사용자는 원본 서버가 아니라 가까운 엣지 서버에서 응답을 받기 때문에 지연 시간이 줄고 원본 서버 부하도 감소한다.

정적 이미지가 전체 트래픽의 큰 비중을 차지한다면 CDN 적용은 즉각적인 개선책이 될 수 있다. 글로벌 서비스에서 특정 지역만 지연이 크다면 해당 지역에 엣지 캐시나 리전을 추가하는 전략도 고려한다.

반대로 사용자별 민감 정보, 자주 바뀌는 개인화 응답, 캐시 무효화가 어려운 데이터는 CDN 캐시가 위험할 수 있다. 이런 응답은 `private`, `no-store`, 짧은 TTL 같은 정책을 명확히 지정해야 한다.

#### 면접 답변

> CDN은 사용자와 가까운 엣지 서버에 콘텐츠를 캐싱해 응답 속도를 높이고 원본 서버 부하를 줄입니다. 정적 파일이 많거나 글로벌 지연이 문제인 서비스에 효과적이지만, 민감한 사용자별 응답은 캐시되지 않도록 정책을 명확히 해야 합니다.

#### 헷갈리는 포인트
- CDN은 정적 파일에만 한정되지 않지만, 정적 파일에서 효과를 체감하기 쉽다.
- 캐시 히트율이 낮거나 개인화가 강한 데이터는 CDN 이점이 줄어든다.
- 캐시로 DB 부하를 줄일 때는 TTL, 무효화, 일관성 요구사항을 같이 설계한다.

#### 관련 퀴즈
- `network-c013`
- `network-t008`

- `network-s013`

### 프록시와 리버스 프록시
<!-- article: network-forward-reverse-proxy -->

#### 한눈에 보기
- Forward Proxy는 클라이언트 앞에서 클라이언트를 대신한다.
- Reverse Proxy는 서버 앞에서 서버를 대신한다.
- 리버스 프록시는 로드밸런싱, TLS 종료, 캐싱, 보안에 자주 사용된다.

#### 핵심 정리

Forward Proxy는 회사나 학교 네트워크의 프록시처럼 클라이언트 쪽에 가깝다. 클라이언트 요청을 대신 보내며 접근 제어, 로깅, 캐싱, 클라이언트 IP 은닉 같은 목적으로 사용된다.

Reverse Proxy는 Nginx나 CDN처럼 서버 앞에 위치한다. 클라이언트는 리버스 프록시를 서버처럼 인식하고, 리버스 프록시는 내부 서버로 요청을 전달한다. 이를 통해 내부 서버 주소를 숨기고, 로드밸런싱과 TLS 종료, 정적 파일 캐싱을 수행할 수 있다.

보안 사고 중에는 리버스 프록시 캐시에 사용자별 민감 응답이 저장되어 다른 사용자에게 노출되는 경우가 있다. 이때는 캐시 키, 인증 헤더 처리, `Cache-Control` 정책을 반드시 점검해야 한다.

#### 면접 답변

> Forward Proxy는 클라이언트 앞에서 외부 서버 접근을 대신하는 프록시이고, Reverse Proxy는 서버 앞에서 클라이언트 요청을 내부 서버로 전달하는 프록시입니다. 리버스 프록시는 로드밸런싱, TLS 종료, 캐싱, 내부 서버 보호에 많이 사용됩니다.

#### 헷갈리는 포인트
- Forward Proxy와 Reverse Proxy는 위치와 보호 대상이 다르다.
- CDN은 리버스 프록시 성격을 가진 캐시 계층으로 볼 수 있다.
- 리버스 프록시 캐시는 사용자별 응답을 섞어 저장하지 않도록 주의해야 한다.

#### 관련 퀴즈
- `network-c033`
- `network-s008`

### DNS Round Robin과 TTL
<!-- article: network-dns-round-robin-ttl -->

#### 한눈에 보기
- DNS Round Robin은 하나의 도메인에 여러 IP를 번갈아 응답하는 단순 분산 방식이다.
- 구현은 쉽지만 실제 부하, 서버 장애, 클라이언트 캐시를 정교하게 제어하기 어렵다.
- TTL은 변경 전파 속도와 DNS 질의량 사이의 균형이다.

#### 핵심 정리

DNS Round Robin은 `example.com`에 여러 IP를 등록하고 질의마다 다른 IP를 응답하게 한다. 별도 로드밸런서 없이도 분산처럼 동작하므로 구조가 단순하다.

하지만 DNS는 백엔드 서버의 실제 CPU, 연결 수, 응답 시간을 직접 보지 못한다. 어떤 서버가 장애가 나도 캐시에 남아 있는 IP로 요청이 계속 갈 수 있고, 클라이언트나 중간 리졸버가 TTL 동안 응답을 보관하므로 즉시 전환도 어렵다.

TTL을 짧게 하면 장애 전환과 배포 전환에는 유리하지만 DNS 질의량이 증가한다. TTL을 길게 하면 조회 비용은 줄지만 변경 반영이 늦다. 정교한 헬스 체크와 부하 반영이 필요하면 L4/L7 로드밸런서를 쓰는 편이 낫다.

#### 면접 답변

> DNS Round Robin은 하나의 도메인에 여러 IP를 등록해 DNS 응답을 번갈아 주는 방식입니다. 단순하지만 서버 장애 감지, 실제 부하 반영, 균등 분산, 클라이언트 DNS 캐시 제어가 어렵기 때문에 정교한 분산에는 로드밸런서가 더 적합합니다.

#### 헷갈리는 포인트
- DNS가 클라이언트 캐시를 즉시 지울 수는 없다.
- TTL을 무조건 짧게 잡으면 DNS 인프라 부하가 커질 수 있다.
- DNS Round Robin은 헬스 체크가 있는 로드밸런서와 같은 수준의 제어를 제공하지 않는다.

#### 관련 퀴즈
- `network-t005`
- `network-t016`

### 보안 운영: SYN flood, DDoS, mTLS
<!-- article: network-security-operations -->

#### 한눈에 보기
- SYN flood는 TCP 연결 대기 큐를 고갈시키는 공격이다.
- DDoS는 여러 출처의 대량 트래픽으로 서비스 처리 능력을 소모시킨다.
- mTLS는 서비스 간 통신에서 양쪽 신원을 모두 검증하고 암호화한다.

#### 핵심 정리

SYN flood는 공격자가 대량의 SYN 패킷을 보내 서버의 반쯤 열린 연결 큐를 채우는 공격이다. 서버는 정상 클라이언트의 연결을 받을 자원이 부족해진다. SYN cookies, backlog 조정, rate limiting, 방화벽 정책 등이 대응책이 될 수 있다.

DDoS는 여러 IP에서 분산된 공격 트래픽이 들어오는 상황이다. 단일 서버 설정만으로 막기 어렵기 때문에 CDN, WAF, Anycast, rate limiting, 트래픽 필터링, 오토스케일링 같은 조합이 필요하다.

내부 서비스 간 보안을 강화하려면 mTLS를 고려할 수 있다. 일반 TLS가 주로 서버 인증을 제공한다면, mTLS는 클라이언트도 인증서를 제시해 양쪽이 서로를 검증한다. 마이크로서비스 환경에서 서비스 신원 확인과 암호화에 유용하다.

#### 면접 답변

> SYN flood는 대량의 SYN으로 TCP 연결 대기 큐를 고갈시키는 공격이며 SYN cookies나 rate limiting으로 완화할 수 있습니다. DDoS는 분산 트래픽을 CDN, WAF, 필터링, 제한 정책으로 방어하고, 서비스 간 통신 보안은 mTLS로 상호 인증을 강화할 수 있습니다.

#### 헷갈리는 포인트
- HSTS는 브라우저가 이후 요청을 HTTPS로 강제하도록 돕는 보안 헤더다.
- mTLS는 외부 사용자 인증만을 위한 기술이 아니라 서비스 간 인증에도 많이 쓰인다.
- DDoS 방어는 애플리케이션 코드보다 네트워크와 엣지 계층의 대응이 중요할 때가 많다.

#### 관련 퀴즈

- `network-s011`

- `network-t015`

## 면접 질문 정리
<!-- category: network-interview -->

### 네트워크 기본 답변
<!-- article: network-interview-basics -->

#### 한눈에 보기
- 계층, 주소, 프로토콜을 따로 외우기보다 흐름으로 연결한다.
- 기본 질문은 "무엇을 구분하는가"와 "어느 계층에서 동작하는가"로 정리한다.
- 짧은 정의 뒤 예시를 붙이면 답변이 안정적이다.

#### 핵심 정리

OSI 7계층 질문은 순서를 말한 뒤 각 계층의 대표 역할을 붙인다. 물리는 비트 전송, 데이터링크는 MAC 기반 프레임 전달, 네트워크는 IP 라우팅, 전송은 TCP와 UDP, 응용은 HTTP와 DNS처럼 연결하면 된다.

IP와 MAC 차이는 "IP는 논리 주소, MAC은 물리 주소"만 말하고 끝내지 말고, IP는 네트워크 간 라우팅, MAC은 같은 링크에서 다음 홉 전달에 쓰인다고 설명한다.

TCP와 UDP 차이는 신뢰성과 지연 관점으로 정리한다. 정확성, 순서, 재전송이 중요하면 TCP, 빠른 응답과 실시간성이 중요하고 일부 손실을 감수할 수 있으면 UDP라고 말한다.

#### 면접 답변

> 네트워크 기본 질문은 계층별 역할과 주소의 쓰임을 연결해 답하겠습니다. IP는 목적지 네트워크로 가는 논리 주소, MAC은 같은 링크에서 프레임을 전달하는 주소이고, TCP는 신뢰성 중심, UDP는 낮은 지연과 단순성을 우선하는 전송 방식입니다.

#### 헷갈리는 포인트
- DNS는 도메인에서 IP, ARP는 IP에서 MAC을 찾는다.
- 포트는 호스트가 아니라 프로세스를 구분한다.
- 라우터와 스위치는 보는 주소가 다르다.

### 실무 선택 질문
<!-- article: network-interview-tradeoffs -->

#### 한눈에 보기
- 선택형 질문은 요구사항을 먼저 말하고 기술을 고른다.
- 성능, 신뢰성, 운영 복잡도, 보안 요구를 비교한다.
- "항상 좋다"보다 "이 조건이면 적합하다"가 좋은 답변이다.

#### 핵심 정리

TCP와 UDP 선택은 손실 허용 여부와 지연 민감도를 기준으로 잡는다. 실시간 영상은 약간의 손실을 감수하고 지연을 줄이는 것이 중요할 수 있지만, 금융 거래나 파일 전송은 정확성과 순서 보장이 우선이다.

REST, WebSocket, SSE 선택은 통신 방향과 빈도에 따라 달라진다. CRUD 중심이면 REST가 단순하고, 서버가 지속적으로 push해야 하면 WebSocket이나 SSE가 적합하다. 양방향이면 WebSocket, 서버 단방향 이벤트면 SSE가 단순하다.

L4와 L7 로드밸런서는 제어 수준과 비용을 비교한다. 단순하고 빠른 분산이면 L4, 경로와 헤더 기반 라우팅이나 인증 처리가 필요하면 L7을 선택한다. JWT와 세션도 확장성과 즉시 폐기 가능성 사이의 트레이드오프로 설명한다.

#### 면접 답변

> 실무 선택 질문은 요구사항을 먼저 나누겠습니다. 정확성과 순서가 중요하면 TCP, 지연이 중요하면 UDP를 고려합니다. 단순 CRUD는 REST, 실시간 양방향은 WebSocket, 서버 단방향 이벤트는 SSE가 적합합니다. L4와 L7은 필요한 라우팅 제어 수준에 따라 선택합니다.

#### 헷갈리는 포인트
- JWT는 확장성은 좋지만 탈취와 즉시 무효화 문제가 있다.
- HTTP/3 도입은 손실과 모바일 네트워크 전환이 잦은 환경에서 효과를 기대할 수 있다.
- CDN은 정적 콘텐츠와 글로벌 지연에는 좋지만 모든 응답에 무조건 적합하지는 않다.

### 장애 분석 질문
<!-- article: network-interview-troubleshooting -->

#### 한눈에 보기
- 장애 질문은 어느 계층에서 실패했는지 좁혀간다.
- DNS, 연결, TLS, HTTP 상태 코드, 로드밸런서, 백엔드를 순서대로 확인한다.
- 증상만 보고 결론을 확정하지 말고 검증할 관측값을 함께 말한다.

#### 핵심 정리

도메인 접속만 안 되고 IP 직접 접속은 된다면 DNS 레코드, CNAME, TTL, 로컬 캐시를 확인한다. 연결 자체가 안 되면 방화벽, 포트 리스닝, 라우팅, 보안 그룹을 본다. HTTPS만 실패하면 인증서, SNI, TLS 버전, 체인 검증을 확인한다.

504 Gateway Timeout은 게이트웨이나 로드밸런서가 백엔드 응답을 제시간에 받지 못한 상황이다. 특정 시간대에만 발생하면 트래픽 증가, 백엔드 포화, 커넥션 풀 부족, 타임아웃 설정, 특정 서버 편중을 함께 확인한다.

CORS 오류는 브라우저에서 보이는 정책 오류이므로 서버의 허용 Origin, 메서드, 헤더, credentials 설정을 확인한다. SYN flood나 DDoS처럼 트래픽 패턴이 비정상적인 경우는 애플리케이션 로그뿐 아니라 네트워크 계층 지표와 엣지 방어 설정도 봐야 한다.

#### 면접 답변

> 네트워크 장애는 DNS, TCP 연결, TLS, HTTP 응답, 프록시와 로드밸런서, 백엔드 순서로 범위를 좁히겠습니다. 예를 들어 IP 직접 접속은 되는데 도메인이 안 되면 DNS를 의심하고, 504가 특정 시간대에 발생하면 백엔드 지연과 로드밸런서 타임아웃, 서버 편중을 확인합니다.

#### 헷갈리는 포인트
- CORS는 브라우저 정책이라 서버 간 호출과 다르게 보아야 한다.
- 500ms 지연이 특정 지역에만 있으면 네트워크 거리와 CDN 엣지 유무를 확인한다.
- DDoS는 서버 증설만으로 충분하지 않을 수 있고 엣지 방어가 중요하다.
