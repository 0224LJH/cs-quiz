# 개발자 필수 지식

이 문서는 CS 면접 준비를 위해 개발자가 자주 설명해야 하는 공통 개념을 모바일에서 읽기 좋게 정리한 학습 자료입니다.

구현 기술 하나보다 중요한 것은 개념 사이의 경계와 선택 기준을 말할 수 있는 능력입니다. 좋은 코드, 프로그래밍 패러다임, API, 테스트, 배포, Git, 보안 개념을 면접 답변 형태로 연결해 둡니다.

## 좋은 코드와 추상화
<!-- category: general-clean-code-abstraction -->

### 좋은 코드는 읽히고 바뀐다
<!-- article: general-good-code -->

#### 한눈에 보기
- 좋은 코드는 읽기 쉽고, 중복이 적고, 테스트하기 쉽다.
- 함수와 클래스는 가능한 한 하나의 책임에 집중해야 한다.
- 이름, 구조, 테스트가 코드의 유지보수성을 만든다.

#### 핵심 정리

좋은 코드는 멋진 문법을 많이 쓰는 코드가 아니라 다음 사람이 빠르게 이해하고 안전하게 바꿀 수 있는 코드입니다. 요구사항은 계속 바뀌므로, 코드는 처음 작성되는 시간보다 읽히고 수정되는 시간이 더 길다는 점을 기준으로 봐야 합니다.

함수가 여러 일을 동시에 하면 이름이 흐려지고 테스트가 어려워집니다. 예를 들어 사용자 검증, 저장, 이메일 발송, 로그 기록을 한 메서드에 몰아넣으면 변경 이유가 여러 개가 됩니다. 이런 코드는 단일 책임 원칙(SRP)을 어기기 쉽고 작은 수정이 예상하지 못한 부작용으로 번집니다.

중복은 항상 나쁘다고 단정할 수는 없지만, 같은 정책이 여러 곳에 흩어져 있으면 정책 변경 때 누락 위험이 커집니다. 반대로 너무 이른 추상화는 아직 안정되지 않은 차이를 억지로 묶어서 더 복잡한 코드를 만들 수 있습니다.

#### 면접 답변

> 좋은 코드는 읽기 쉽고, 변경하기 쉽고, 테스트하기 쉬운 코드라고 생각합니다. 함수나 클래스가 하나의 책임에 집중하면 이름이 명확해지고 영향 범위가 줄어들어 유지보수가 쉬워집니다.

#### 헷갈리는 포인트
- 짧은 코드가 항상 좋은 코드는 아니다. 의도가 드러나는 구조가 더 중요하다.
- 중복 제거는 실제로 같은 정책이 반복될 때 의미가 있다.
- 주석으로 설명하기 전에 이름과 구조로 표현할 수 있는지 먼저 본다.

### 추상화는 복잡도를 낮추는 경계다
<!-- article: general-abstraction -->

#### 한눈에 보기
- 추상화는 중요한 특징과 인터페이스만 드러내고 내부 구현을 숨기는 설계 방법이다.
- 함수, 클래스, 인터페이스, API, 프레임워크 모두 추상화 계층이 될 수 있다.
- 좋은 추상화는 사용자를 단순하게 만들고, 나쁜 추상화는 내부 복잡도를 밖으로 새게 한다.

#### 핵심 정리

추상화는 복잡한 구현을 감추고 사용자가 알아야 할 약속만 드러내는 것입니다. 데이터베이스 사용자가 저장 엔진의 내부 구조를 몰라도 SQL로 조회할 수 있는 것, 라이브러리 사용자가 내부 알고리즘을 몰라도 함수 이름과 인자로 기능을 쓰는 것이 추상화입니다.

추상화의 핵심은 숨김과 약속입니다. 내부 구현은 바뀔 수 있지만 외부 인터페이스가 안정적이면 사용하는 쪽의 변경을 줄일 수 있습니다. 그래서 API, 인터페이스, 계층 분리는 모두 변경 영향 범위를 줄이기 위한 장치입니다.

다만 추상화는 공짜가 아닙니다. 아직 변화 방향을 모르는 코드를 미리 일반화하면 계층만 늘고 이해는 어려워질 수 있습니다. 반복되는 책임, 교체 가능성이 큰 구현, 외부에 드러내면 안 되는 세부사항이 있을 때 추상화의 이득이 커집니다.

#### 면접 답변

> 추상화는 복잡한 내부 구현을 숨기고 외부에는 필요한 인터페이스만 제공하는 것입니다. 이를 통해 사용하는 쪽은 단순해지고, 내부 구현이 바뀌어도 영향 범위를 줄일 수 있습니다.

#### 헷갈리는 포인트
- 추상화는 어렵게 만드는 것이 아니라 덜 알아도 쓰게 만드는 것이다.
- 인터페이스를 추가한다고 항상 좋은 추상화가 되는 것은 아니다.
- 구현 세부사항이 외부 계약에 새어 나오면 추상화가 약해진다.

### SOLID와 의존성 방향
<!-- article: general-solid-dependency -->

#### 한눈에 보기
- SOLID는 객체지향 설계에서 변경에 강한 구조를 만들기 위한 원칙이다.
- DIP는 고수준 정책이 저수준 구현에 직접 의존하지 않게 한다.
- 의존성 주입은 DIP를 구현하는 대표적인 방법이다.

#### 핵심 정리

SOLID는 단일 책임, 개방 폐쇄, 리스코프 치환, 인터페이스 분리, 의존성 역전 원칙을 묶어 부르는 말입니다. 면접에서는 다섯 가지를 암기하는 것보다 변경이 생겼을 때 어떤 코드가 흔들리는지 설명하는 것이 중요합니다.

의존성 역전 원칙(DIP)은 고수준 모듈과 저수준 모듈이 모두 추상화에 의존해야 한다는 원칙입니다. 예를 들어 주문 서비스가 특정 결제 API 구현체를 직접 생성하면 결제사가 바뀔 때 주문 서비스도 바뀝니다. 반대로 결제 인터페이스에 의존하면 구현체 교체의 영향이 줄어듭니다.

의존성 주입(DI)은 필요한 구현체를 외부에서 넣어주는 방식입니다. 객체가 직접 의존 객체를 만들지 않으므로 테스트에서 가짜 구현체로 바꿔 끼우기 쉽고, 런타임 정책에 따라 구현체를 바꾸기도 쉽습니다. 대신 객체와 설정이 늘어나 복잡도가 증가할 수 있습니다.

#### 면접 답변

> SOLID는 변경에 강한 객체지향 설계를 위한 원칙입니다. 특히 DIP는 비즈니스 로직이 구체적인 DB나 외부 API 구현에 직접 의존하지 않고 인터페이스에 의존하게 만들어 교체와 테스트를 쉽게 합니다.

#### 헷갈리는 포인트
- DI는 DIP를 구현하는 방법 중 하나이지 DIP 그 자체는 아니다.
- LSP는 상위 타입으로 사용할 때 하위 타입이 기대 동작을 깨면 안 된다는 원칙이다.
- SRP는 코드 줄 수가 아니라 변경 이유가 하나인지에 가깝다.

#### 관련 퀴즈

### 코드 리뷰와 주석은 팀의 안전장치다
<!-- article: general-review-comments -->

#### 한눈에 보기
- 코드 리뷰는 버그 발견뿐 아니라 지식 공유와 설계 일관성 유지가 목적이다.
- 주석은 무엇을 하는지보다 왜 그렇게 했는지에 집중할 때 가치가 크다.
- 오래된 주석은 코드보다 더 위험한 잘못된 정보를 줄 수 있다.

#### 핵심 정리

코드 리뷰는 작성자를 평가하는 절차가 아니라 팀이 코드의 책임을 함께 나누는 과정입니다. 자동 테스트가 잡기 어려운 로직 오류, 보안 위험, 예외 상황, 설계 방향을 사람이 확인합니다. 동시에 작성자만 알던 맥락이 팀에 퍼지기 때문에 버스 팩터를 낮추는 효과도 있습니다.

주석은 코드로 표현하기 어려운 이유를 남길 때 유용합니다. 비즈니스 제약, 외부 시스템의 이상한 동작, 성능 때문에 선택한 알고리즘처럼 맥락을 남기면 나중에 코드를 바꾸는 사람이 같은 실수를 피할 수 있습니다.

반대로 코드가 그대로 설명하는 내용을 주석으로 반복하면 유지보수 대상만 늘어납니다. 코드가 바뀌었는데 주석이 그대로 남으면, 읽는 사람은 코드와 주석 중 무엇을 믿어야 할지 모르게 됩니다.

#### 면접 답변

> 코드 리뷰는 버그 조기 발견, 지식 공유, 코드 품질과 팀 표준 유지를 위한 과정입니다. 주석은 코드가 무엇을 하는지보다 왜 그런 선택을 했는지를 설명할 때 가장 유용합니다.

#### 헷갈리는 포인트
- 리뷰는 취향 논쟁보다 동작, 안정성, 유지보수성에 집중해야 한다.
- 주석이 필요하다는 사실이 이름이나 구조를 개선하라는 신호일 수 있다.
- 문서화 주석과 구현 내부 주석은 목적이 다르다.

## OOP와 함수형 프로그래밍
<!-- category: general-oop-functional -->

### 객체지향 프로그래밍은 책임을 객체에 나눈다
<!-- article: general-oop -->

#### 한눈에 보기
- 객체지향 프로그래밍은 현실의 개념을 객체와 책임으로 모델링한다.
- 캡슐화, 상속, 다형성, 추상화가 대표적인 특성이다.
- 상태를 가진 객체가 많아지면 예측이 어려워질 수 있다.

#### 핵심 정리

객체지향 프로그래밍(OOP)은 프로그램을 데이터와 행동을 가진 객체들의 협력으로 구성하는 방식입니다. 사용자는 객체의 내부 필드를 직접 만지는 대신 공개된 메서드를 통해 메시지를 보냅니다. 이 구조는 도메인 개념을 코드로 옮기기 쉽고, 책임을 기준으로 코드를 나누기 좋습니다.

캡슐화는 내부 구현을 숨기고 필요한 인터페이스만 공개하는 특성입니다. 내부 필드를 직접 공개하지 않으면 객체가 스스로 유효한 상태를 지키기 쉬워지고, 구현을 바꿔도 외부 영향이 줄어듭니다.

OOP의 장점은 재사용성, 유지보수성, 디버깅 편의성입니다. 단점은 객체가 상태를 가지기 때문에 상태 변화가 복잡해질수록 버그를 추적하기 어려워질 수 있다는 점입니다. 그래서 불변 객체, 명확한 책임 분리, 테스트 가능한 구조가 중요합니다.

#### 면접 답변

> 객체지향은 데이터와 행동을 가진 객체들이 메시지를 주고받으며 협력하도록 설계하는 방식입니다. 캡슐화를 통해 내부 구현을 숨기고 책임을 나누면 재사용성과 유지보수성이 좋아집니다.

#### 헷갈리는 포인트
- 객체는 단순히 필드 묶음이 아니라 책임을 가진 단위다.
- 상속은 재사용 도구이지만 잘못 쓰면 결합도를 높인다.
- 캡슐화는 getter, setter를 모두 열어두는 것과 다르다.

#### 관련 퀴즈

### 함수형 프로그래밍은 값과 변환에 집중한다
<!-- article: general-functional-programming -->

#### 한눈에 보기
- 함수형 프로그래밍은 불변 데이터와 순수 함수를 중요하게 본다.
- 순수 함수는 같은 입력에 항상 같은 출력을 반환하고 부작용이 없다.
- `map`, `filter`, `reduce`는 컬렉션을 선언적으로 다루는 대표 연산이다.

#### 핵심 정리

함수형 프로그래밍(FP)은 상태를 직접 바꾸기보다 입력값을 새로운 결과값으로 변환하는 방식에 집중합니다. 순수 함수는 외부 상태를 변경하지 않고 같은 입력에 같은 출력을 반환하므로 테스트와 예측이 쉽습니다.

불변성은 이미 만들어진 값을 바꾸지 않는 성질입니다. 값이 바뀌지 않으면 여러 코드가 같은 데이터를 참조해도 예상치 못한 변경이 줄어듭니다. 동시성 상황에서도 공유 상태 변경으로 인한 문제가 줄어드는 장점이 있습니다.

`map`은 각 요소를 변환하고, `filter`는 조건에 맞는 요소만 남기며, `reduce`는 여러 요소를 하나의 값으로 누적합니다. 이 연산들은 원본을 직접 바꾸지 않고 새 결과를 만드는 방향으로 이해하면 쉽습니다.

#### 면접 답변

> 함수형 프로그래밍은 불변 데이터와 순수 함수를 중심으로 프로그램을 구성하는 방식입니다. 부작용을 줄여 테스트와 예측이 쉬워지고, map, filter, reduce 같은 연산으로 데이터를 선언적으로 변환합니다.

#### 헷갈리는 포인트
- 함수형 프로그래밍은 함수를 쓰기만 하면 되는 것이 아니다.
- 순수 함수는 I/O, 전역 상태 변경, DB 접근 같은 부작용이 없어야 한다.
- OOP와 FP는 배타적이지 않고 한 프로젝트 안에서 함께 사용할 수 있다.

### 오버로딩과 오버라이딩은 다형성을 다르게 만든다
<!-- article: general-overloading-overriding -->

#### 한눈에 보기
- 오버로딩은 같은 이름의 메서드를 매개변수 차이로 여러 개 두는 것이다.
- 오버라이딩은 상위 타입의 메서드를 하위 타입에서 재정의하는 것이다.
- 오버로딩은 보통 컴파일 시점, 오버라이딩은 런타임 다형성과 연결된다.

#### 핵심 정리

오버로딩(Overloading)은 같은 클래스 안에서 같은 이름의 메서드를 여러 개 제공하되 매개변수 타입, 개수, 순서를 다르게 하는 방식입니다. 비슷한 동작을 하나의 이름으로 묶어 사용성을 높이는 데 쓰입니다.

오버라이딩(Overriding)은 상속이나 인터페이스 구현 관계에서 상위 타입의 메서드를 하위 타입이 자신의 방식으로 다시 정의하는 것입니다. 변수 타입은 상위 타입이어도 실제 객체 타입에 따라 실행되는 메서드가 달라질 수 있습니다.

면접에서는 두 개념의 발생 위치와 결정 시점을 구분해 말하면 좋습니다. 오버로딩은 같은 이름의 여러 후보 중 어떤 메서드를 호출할지 고르는 문제이고, 오버라이딩은 상위 타입의 계약을 하위 타입이 지키면서 동작을 바꾸는 문제입니다.

#### 면접 답변

> 오버로딩은 같은 이름의 메서드를 매개변수 차이로 여러 개 정의하는 것이고, 오버라이딩은 상위 타입의 메서드를 하위 타입에서 재정의하는 것입니다. 오버라이딩은 런타임 다형성과 직접 연결됩니다.

#### 헷갈리는 포인트
- 리턴 타입만 다른 것은 일반적으로 오버로딩 조건이 아니다.
- 오버라이딩은 상위 타입의 기대 동작을 깨면 LSP 위반으로 이어질 수 있다.
- 이름이 같다는 점만 보고 두 개념을 섞으면 안 된다.

## 라이브러리와 프레임워크
<!-- category: general-library-framework -->

### 라이브러리와 프레임워크의 차이는 제어 흐름이다
<!-- article: general-library-framework -->

#### 한눈에 보기
- 라이브러리는 개발자가 필요할 때 호출하는 도구 모음이다.
- 프레임워크는 실행 흐름과 구조를 제공하고 개발자의 코드를 호출한다.
- 핵심 차이는 제어의 역전(IoC)이 있는지로 설명할 수 있다.

#### 핵심 정리

라이브러리는 특정 기능을 재사용할 수 있게 만든 코드 묶음입니다. 개발자는 필요한 시점에 라이브러리 함수를 직접 호출합니다. 예를 들어 날짜 처리, HTTP 요청, 유틸리티 함수처럼 필요한 기능을 가져다 쓰는 방식입니다.

프레임워크는 애플리케이션의 기본 구조와 실행 흐름을 제공합니다. 개발자는 프레임워크가 정한 규칙 안에 코드를 작성하고, 프레임워크가 적절한 시점에 개발자의 코드를 호출합니다. 이처럼 제어 흐름이 개발자에서 프레임워크로 넘어가는 것을 제어의 역전이라고 설명할 수 있습니다.

실무에서는 둘의 경계가 완전히 칼같이 나뉘지는 않습니다. 중요한 것은 누가 전체 흐름을 쥐고 있는지, 프로젝트 구조와 생명주기를 누가 정하는지를 보고 설명하는 것입니다.

#### 면접 답변

> 라이브러리는 개발자가 필요한 순간 직접 호출하는 도구이고, 프레임워크는 전체 구조와 흐름을 제공한 뒤 개발자의 코드를 정해진 시점에 호출합니다. 핵심 차이는 제어의 역전입니다.

#### 헷갈리는 포인트
- 기능이 많다고 무조건 프레임워크는 아니다.
- 프레임워크를 쓰면 구조의 일관성을 얻지만 규칙과 학습 비용도 생긴다.
- 라이브러리는 자유도가 높지만 팀마다 구조가 달라질 수 있다.

## RESTful API와 API 개념
<!-- category: general-api-rest -->

### API는 소프트웨어 사이의 약속이다
<!-- article: general-api-concept -->

#### 한눈에 보기
- API는 어떤 기능을 외부에서 사용할 수 있도록 제공하는 규칙과 인터페이스다.
- 함수 시그니처, HTTP 엔드포인트, 요청과 응답 형식, 인증 방식이 API에 포함될 수 있다.
- API가 안정적이면 내부 구현을 바꿔도 사용자 영향이 줄어든다.

#### 핵심 정리

API(Application Programming Interface)는 소프트웨어 컴포넌트가 서로 상호작용하기 위한 약속입니다. 웹 API라면 URL, HTTP 메서드, 요청 헤더, 요청 바디, 응답 상태 코드, 응답 데이터 형식이 모두 약속에 포함됩니다.

API를 잘 설계하면 제공자는 내부 구현을 숨길 수 있고, 사용자는 내부 구조를 몰라도 필요한 기능을 사용할 수 있습니다. 데이터베이스 테이블 구조나 내부 서비스 구성을 직접 노출하지 않고도 기능을 제공할 수 있다는 점에서 API는 중요한 추상화 계층입니다.

외부에 공개된 API는 한 번 사용자가 생기면 쉽게 바꾸기 어렵습니다. 그래서 응답 필드 추가, 버전 관리, 하위 호환성, 인증 방식 같은 운영 관점까지 함께 고려해야 합니다.

#### 면접 답변

> API는 소프트웨어가 서로 통신하기 위한 약속입니다. 웹 API에서는 엔드포인트, 메서드, 요청과 응답 형식, 인증 방식이 그 약속에 포함되며 내부 구현을 숨기는 추상화 역할도 합니다.

#### 헷갈리는 포인트
- API는 웹 API만 의미하지 않는다. 함수나 라이브러리 인터페이스도 API다.
- API 문서는 구현 설명보다 사용자가 지켜야 할 계약을 중심으로 해야 한다.
- 공개 API 변경은 기존 클라이언트와 하위 호환성을 고려해야 한다.

### RESTful API는 리소스와 행위를 분리한다
<!-- article: general-rest-api -->

#### 한눈에 보기
- REST는 리소스를 URI로 표현하고 행위는 HTTP 메서드로 표현한다.
- `GET`, `POST`, `PUT`, `PATCH`, `DELETE`는 목적과 멱등성이 다르다.
- URI는 동사보다 명사와 계층 구조로 설계하는 것이 좋다.

#### 핵심 정리

RESTful API는 API 설계의 중심을 리소스에 둡니다. 사용자 목록은 `/users`, 특정 사용자는 `/users/{id}`, 사용자의 주문은 `/users/{id}/orders`처럼 명사와 계층으로 표현합니다. `getUser`, `deleteUser` 같은 동사는 URI에 넣기보다 HTTP 메서드로 표현합니다.

HTTP 메서드는 역할이 다릅니다. `GET`은 조회, `POST`는 생성이나 처리 요청, `PUT`은 전체 수정, `PATCH`는 부분 수정, `DELETE`는 삭제에 사용합니다. 같은 요청을 여러 번 보내도 결과 상태가 같으면 멱등하다고 하며, `GET`, `PUT`, `DELETE`는 보통 멱등하게 설계합니다.

RESTful하게 설계하면 기존 HTTP 인프라, 상태 코드, 캐시를 활용하기 쉽습니다. 다만 복잡한 화면에서 필요한 데이터가 여러 리소스에 흩어져 있으면 요청이 많아질 수 있고, 필요 이상의 데이터를 받는 문제가 생길 수 있습니다.

#### 면접 답변

> RESTful API는 리소스를 URI의 명사로 표현하고, 행위는 HTTP 메서드로 구분하는 설계 방식입니다. 예를 들어 `GET /users`는 조회, `POST /users`는 생성, `PATCH /users/{id}`는 부분 수정처럼 설계합니다.

#### 헷갈리는 포인트
- `POST /getUserInfo`처럼 URI에 동사를 넣으면 RESTful하지 않다.
- `PUT`은 전체 교체, `PATCH`는 부분 수정에 가깝다.
- REST는 HTTP만의 문법이라기보다 리소스 중심 아키텍처 스타일이다.

### REST와 GraphQL은 요구사항에 따라 고른다
<!-- article: general-rest-graphql -->

#### 한눈에 보기
- REST는 단순하고 HTTP 캐싱과 도구 활용이 쉽다.
- GraphQL은 클라이언트가 필요한 필드를 직접 고르기 좋다.
- GraphQL은 캐싱, 권한, 복잡한 쿼리 제어를 더 신중히 다뤄야 한다.

#### 핵심 정리

REST는 리소스 단위의 엔드포인트를 여러 개 두는 방식입니다. 구조가 단순하고 HTTP 메서드, 상태 코드, 캐시 같은 웹 표준과 잘 맞습니다. 대부분의 팀과 도구가 익숙하기 때문에 공개 API나 단순한 CRUD 서비스에 적합합니다.

GraphQL은 하나의 엔드포인트에 쿼리를 보내고 필요한 필드만 요청할 수 있습니다. 모바일 화면처럼 필요한 데이터 모양이 자주 다르거나, REST로는 여러 번 요청해야 하는 데이터를 한 번에 가져오고 싶을 때 장점이 큽니다.

대신 GraphQL은 쿼리 복잡도 제한, N+1 문제, 캐싱 전략, 권한 검사를 세밀하게 설계해야 합니다. 따라서 항상 GraphQL이 더 좋다기보다, 클라이언트 다양성, 데이터 조합 필요성, 팀의 운영 역량을 기준으로 선택해야 합니다.

#### 면접 답변

> REST는 단순하고 HTTP 캐싱과 표준 도구를 활용하기 쉬운 반면, GraphQL은 클라이언트가 필요한 데이터만 요청하기 좋습니다. 다만 GraphQL은 캐싱과 쿼리 복잡도 관리가 어려워 요구사항과 운영 역량에 따라 선택합니다.

#### 헷갈리는 포인트
- GraphQL이 REST보다 항상 빠른 것은 아니다.
- REST에서도 화면 전용 API나 BFF로 요청 수 문제를 줄일 수 있다.
- GraphQL은 스키마가 강력한 만큼 권한과 필드 노출 관리도 중요하다.

#### 관련 퀴즈

### 동기와 비동기 API는 복잡도와 처리량의 선택이다
<!-- article: general-sync-async-api -->

#### 한눈에 보기
- 동기 API는 흐름이 직관적이지만 대기 시간 동안 리소스가 묶일 수 있다.
- 비동기 API는 처리량을 높일 수 있지만 상태 추적과 에러 처리가 어려워진다.
- 작업 시간, 사용자 응답 필요성, 실패 처리 방식을 기준으로 선택한다.

#### 핵심 정리

동기 방식은 요청을 보내고 응답이 올 때까지 기다리는 모델입니다. 코드 흐름이 직관적이고 디버깅하기 쉽습니다. 결제 승인처럼 결과를 즉시 알아야 다음 단계로 갈 수 있는 작업에는 동기 방식이 자연스럽습니다.

비동기 방식은 요청을 보낸 뒤 결과를 나중에 콜백, 이벤트, 메시지, 폴링 등으로 확인하는 모델입니다. 오래 걸리는 작업이나 외부 시스템 연동처럼 기다리는 시간이 긴 경우 서버 리소스를 더 효율적으로 쓸 수 있습니다.

하지만 비동기 설계는 실패 재시도, 중복 처리, 순서 보장, 상태 조회 API가 필요해질 수 있습니다. 단순히 빠르게 만들기 위해 비동기를 선택하기보다 사용자 경험과 장애 처리까지 함께 설계해야 합니다.

#### 면접 답변

> 동기 API는 단순하고 흐름이 명확하지만 대기 시간 동안 리소스가 묶일 수 있습니다. 비동기 API는 높은 처리량에 유리하지만 에러 처리와 상태 추적이 복잡하므로 작업 특성과 실패 처리 기준에 따라 선택합니다.

#### 헷갈리는 포인트
- 비동기는 항상 더 빠르다는 뜻이 아니라 기다리는 방식을 바꾸는 것이다.
- 비동기 작업은 결과 확인과 재시도 전략이 함께 필요하다.
- 사용자에게 즉시 결과가 필요한 작업은 동기가 더 단순할 수 있다.

## TDD와 테스트
<!-- category: general-tdd-testing -->

### TDD는 테스트가 설계를 끌고 가는 방식이다
<!-- article: general-tdd -->

#### 한눈에 보기
- TDD는 실패하는 테스트를 먼저 쓰고, 최소 구현 뒤 리팩토링하는 사이클이다.
- Red, Green, Refactor 순서로 작은 단위를 반복한다.
- 장기적으로 리팩토링 안전망과 설계 품질을 높일 수 있다.

#### 핵심 정리

TDD(Test-Driven Development)는 기능 코드를 먼저 작성하는 대신, 원하는 동작을 검증하는 테스트를 먼저 작성합니다. 아직 기능이 없으므로 첫 테스트는 실패해야 합니다. 이것이 Red 단계입니다.

다음으로 테스트를 통과하는 최소한의 코드를 작성합니다. 이 단계는 Green입니다. 처음부터 완벽한 구조를 만들려고 하기보다 테스트가 말하는 동작을 만족시키는 데 집중합니다. 이후 테스트가 통과하는 상태를 유지하면서 중복 제거와 구조 개선을 진행하는 것이 Refactor입니다.

TDD의 장점은 리팩토링할 때 회귀 버그를 빨리 찾을 수 있고, 테스트 가능한 설계를 유도한다는 점입니다. 단점은 초기 속도가 느려질 수 있고, 테스트 작성 역량이 부족하면 오히려 유지보수 비용이 생긴다는 점입니다.

#### 면접 답변

> TDD는 실패하는 테스트를 먼저 작성하고, 그 테스트를 통과하는 최소 코드를 만든 뒤, 동작을 유지하면서 리팩토링하는 방식입니다. 단기 비용은 있지만 장기적으로 리팩토링 안전망과 설계 품질을 높입니다.

#### 헷갈리는 포인트
- TDD는 100% 무결성을 보장하지 않는다.
- 테스트를 먼저 쓴다고 통합 테스트가 필요 없어지는 것은 아니다.
- 나쁜 테스트는 구현 변경을 어렵게 만드는 족쇄가 될 수 있다.

#### 관련 퀴즈

- `general-t003`
- `general-o002`

### 단위 테스트와 통합 테스트는 확인 범위가 다르다
<!-- article: general-unit-integration-test -->

#### 한눈에 보기
- 단위 테스트는 함수나 클래스 하나를 격리해 빠르게 확인한다.
- 통합 테스트는 DB, API, 외부 서비스와의 상호작용을 확인한다.
- 모킹은 빠르고 안정적이지만 실제 환경 문제를 놓칠 수 있다.

#### 핵심 정리

단위 테스트는 작은 코드 단위를 빠르게 검증합니다. 외부 의존성은 목(mock)이나 스텁으로 대체할 수 있습니다. 덕분에 테스트가 빠르고 실패 원인을 좁히기 쉽지만, 실제 DB나 네트워크 연동에서 생기는 문제는 확인하지 못합니다.

통합 테스트는 여러 컴포넌트가 함께 동작하는지 확인합니다. 예를 들어 서비스 코드가 실제 데이터베이스와 잘 연동되는지, API 응답 형식이 약속대로 나오는지 확인합니다. 단위 테스트보다 느리고 환경 준비가 필요하지만 실제 장애에 가까운 문제를 잡을 수 있습니다.

좋은 테스트 전략은 둘 중 하나만 선택하지 않습니다. 핵심 비즈니스 로직은 단위 테스트로 빠르게 보호하고, 시스템 경계와 외부 연동은 통합 테스트로 확인합니다. E2E 테스트는 가장 실제 사용자 흐름과 가깝지만 비용이 커서 핵심 흐름 위주로 둡니다.

#### 면접 답변

> 단위 테스트는 개별 함수나 클래스를 격리해 빠르게 검증하고, 통합 테스트는 DB나 외부 API 같은 여러 컴포넌트의 상호작용을 검증합니다. 속도와 신뢰 범위가 다르므로 목적에 맞게 함께 사용합니다.

#### 헷갈리는 포인트
- 목을 많이 쓰면 실제 설정 오류나 SQL 문제를 놓칠 수 있다.
- 모든 것을 하나의 큰 테스트로 검증하면 실패 원인 파악이 어려워진다.
- 테스트 이름은 기대 동작을 설명해야 유지보수에 도움이 된다.

### 레거시 리팩토링은 테스트 안전망부터 만든다
<!-- article: general-legacy-refactoring -->

#### 한눈에 보기
- 테스트 없는 레거시 코드는 먼저 현재 동작을 문서화하는 특성화 테스트가 필요하다.
- 작은 단위로 바꾸고 자주 검증해야 한다.
- 새 아키텍처 이전은 한 번에 갈아엎기보다 점진적으로 진행한다.

#### 핵심 정리

레거시 코드에서 가장 위험한 점은 코드가 더럽다는 사실보다 현재 동작을 아무도 확신하지 못한다는 점입니다. 중요한 비즈니스 로직을 바로 고치기보다, 먼저 현재 입력과 출력, 예외 동작을 테스트로 묶어야 합니다. 이 테스트는 이상적인 동작이 아니라 현재 시스템의 실제 동작을 문서화합니다.

그다음 작은 단위로 리팩토링합니다. 이름 변경, 함수 추출, 의존성 분리처럼 동작을 바꾸지 않는 작업부터 시작하고, 각 단계마다 테스트를 실행합니다. 운영 트래픽이 있는 시스템이라면 카나리 배포나 기능 플래그로 영향 범위를 줄일 수 있습니다.

아키텍처 전환도 마찬가지입니다. 현재 시스템 분석과 테스트 커버리지 확보, 새 구조의 파일럿 구현, 일부 트래픽 전환, 레거시 기능의 점진적 제거 순서로 접근하면 위험을 줄일 수 있습니다.

#### 면접 답변

> 테스트 없는 레거시 코드는 먼저 현재 동작을 특성화 테스트로 고정한 뒤 작은 단위로 리팩토링해야 합니다. 새 아키텍처로 옮길 때도 카나리 배포나 Strangler Fig 방식처럼 점진적으로 전환하는 것이 안전합니다.

#### 헷갈리는 포인트
- 레거시 테스트는 처음부터 이상적인 설계를 검증하려고 하면 안 된다.
- 큰 리팩토링과 기능 변경을 한 번에 섞으면 원인 추적이 어렵다.
- 점진적 마이그레이션은 느려 보이지만 운영 위험을 크게 줄인다.

## MVC, MVP, MVVM
<!-- category: general-ui-architecture -->

### MVC, MVP, MVVM은 화면 관심사를 나누는 방식이다
<!-- article: general-mvc-mvp-mvvm -->

#### 한눈에 보기
- MVC는 Model, View, Controller로 관심사를 분리한다.
- MVP는 Presenter가 View 인터페이스를 통해 화면 로직을 더 적극적으로 담당한다.
- MVVM은 ViewModel과 데이터 바인딩으로 UI 상태를 연결한다.

#### 핵심 정리

MVC는 애플리케이션을 Model, View, Controller로 나눕니다. Model은 데이터와 비즈니스 상태를 다루고, View는 사용자에게 보여지는 표현을 담당하며, Controller는 입력을 해석해 모델 변경과 뷰 선택을 조정합니다.

MVP는 Controller 대신 Presenter가 중심이 됩니다. Presenter는 View 인터페이스와 통신하며 화면 표시 로직을 담당합니다. View를 인터페이스로 분리하기 쉬워 테스트가 편해질 수 있지만, View와 Presenter가 1:1로 묶여 코드가 늘어날 수 있습니다.

MVVM은 ViewModel이 화면 상태를 표현하고 View는 데이터 바인딩을 통해 ViewModel과 연결됩니다. UI 상태가 복잡하고 프레임워크가 바인딩을 잘 지원할 때 유리합니다. 선택 기준은 프레임워크 지원, 테스트 전략, View와 로직의 결합도입니다.

#### 면접 답변

> MVC는 Controller가 입력을 조정하고, MVP는 Presenter가 View 인터페이스를 통해 화면 로직을 담당하며, MVVM은 ViewModel과 데이터 바인딩으로 UI 상태를 연결합니다. 프레임워크와 테스트 전략에 맞춰 선택합니다.

#### 헷갈리는 포인트
- 세 패턴은 데이터베이스 설계 방식이 아니라 UI 관심사 분리 방식이다.
- MVC에서도 구현 방식에 따라 Controller와 View의 책임이 달라질 수 있다.
- MVVM은 데이터 바인딩 지원이 약한 환경에서는 장점이 줄어든다.

#### 관련 퀴즈

## 컴파일러, 인터프리터, JIT
<!-- category: general-compiler-interpreter-jit -->

### 실행 방식은 번역 시점과 최적화 방식이 다르다
<!-- article: general-compiler-interpreter-jit -->

#### 한눈에 보기
- 컴파일러는 실행 전 소스 코드를 다른 실행 가능 형태로 번역한다.
- 인터프리터는 실행 시점에 코드를 해석하며 실행한다.
- JIT는 실행 중 자주 쓰이는 코드를 네이티브 코드로 컴파일해 성능을 높인다.

#### 핵심 정리

컴파일러는 소스 코드를 목적 코드, 기계어, 바이트코드 같은 실행 가능한 형태로 미리 번역합니다. 컴파일 과정은 시간이 걸리지만 번역된 코드는 실행이 빠를 수 있습니다. C, C++, Go 같은 언어를 설명할 때 자주 등장합니다.

인터프리터는 실행 시점에 코드를 읽고 해석하며 실행합니다. 별도의 긴 컴파일 과정 없이 바로 실행하기 쉬우나, 같은 코드를 반복 실행할 때는 컴파일 방식보다 느릴 수 있습니다. Python, Ruby, JavaScript를 설명할 때 흔히 예로 듭니다.

JIT(Just-In-Time) 컴파일러는 두 방식의 장점을 섞습니다. 처음에는 해석하거나 바이트코드를 실행하다가, 자주 실행되는 코드가 보이면 런타임에 네이티브 코드로 컴파일해 반복 실행 성능을 높입니다. 대신 워밍업 시간과 런타임 컴파일 오버헤드가 있습니다.

#### 면접 답변

> 컴파일러는 실행 전에 코드를 실행 가능한 형태로 번역하고, 인터프리터는 실행 시점에 코드를 해석합니다. JIT는 런타임에 자주 실행되는 코드를 네이티브 코드로 컴파일해 성능을 높이지만 워밍업 비용이 있습니다.

#### 헷갈리는 포인트
- 현대 런타임은 컴파일과 인터프리팅을 섞는 경우가 많다.
- JIT는 항상 시작 시간이 빠르다는 뜻이 아니다.
- JIT는 메모리 문제를 자동으로 해결하는 기술이 아니다.

#### 관련 퀴즈

## CI/CD
<!-- category: general-cicd -->

### CI/CD는 통합과 배포의 반복을 자동화한다
<!-- article: general-cicd -->

#### 한눈에 보기
- CI는 코드 변경마다 빌드와 테스트를 실행해 통합 문제를 빨리 찾는다.
- CD는 검증된 산출물을 배포 가능한 상태로 만들거나 실제 배포까지 자동화한다.
- 파이프라인은 실패하면 다음 단계로 넘어가지 않게 설계해야 한다.

#### 핵심 정리

CI(Continuous Integration)는 개발자가 코드를 자주 통합하고, 그때마다 자동으로 빌드와 테스트를 실행하는 방식입니다. 목적은 통합 문제를 늦게 발견하지 않도록 하는 것입니다. GitHub Actions, Jenkins, GitLab CI 같은 도구가 대표적입니다.

CD는 Continuous Delivery 또는 Continuous Deployment로 쓰입니다. Delivery는 언제든 배포 가능한 상태까지 자동화하는 데 가깝고, Deployment는 운영 배포까지 자동화하는 의미로 사용됩니다. 팀마다 용어를 다르게 쓰기도 하므로 면접에서는 둘을 구분해 설명하면 좋습니다.

파이프라인은 보통 코드 push, 빌드, 단위 테스트와 정적 분석, 스테이징 배포와 통합 테스트, 운영 배포와 모니터링 순서로 구성됩니다. 테스트 없이 main에 push될 때마다 바로 운영 배포되는 구조는 위험합니다.

#### 면접 답변

> CI/CD는 코드 통합, 빌드, 테스트, 배포 과정을 자동화해 문제를 빠르게 발견하고 배포 실수를 줄이는 방식입니다. CI는 통합 검증, CD는 검증된 산출물을 배포 가능한 상태나 운영까지 보내는 흐름입니다.

#### 헷갈리는 포인트
- CI가 있다고 코드 충돌이 자동으로 해결되는 것은 아니다.
- 테스트가 불안정하면 파이프라인 전체 신뢰도가 떨어진다.
- 운영 배포에는 조건, 승인, 롤백, 모니터링이 함께 필요하다.

#### 관련 퀴즈
- `general-c003`

### 빌드는 배포 가능한 산출물을 만드는 단계다
<!-- article: general-build-artifact -->

#### 한눈에 보기
- 빌드는 소스 코드와 의존성을 실행 가능한 형태로 변환한다.
- 산출물은 JAR, 바이너리, Docker 이미지, 정적 번들 등이 될 수 있다.
- 이후 테스트와 배포는 같은 산출물을 기준으로 진행하는 것이 좋다.

#### 핵심 정리

빌드(Build)는 소스 코드를 실제 실행하거나 배포할 수 있는 산출물로 만드는 과정입니다. 컴파일, 의존성 해결, 번들링, 정적 파일 생성, Docker 이미지 생성 등이 빌드에 포함될 수 있습니다.

빌드 산출물이 명확해야 배포가 재현 가능합니다. 개발자 로컬에서 다시 빌드해서 배포하는 방식은 환경 차이와 실수 가능성이 큽니다. CI에서 만든 산출물을 테스트하고, 같은 산출물을 스테이징과 운영으로 승격시키는 방식이 안정적입니다.

빌드는 단순한 변환 과정처럼 보이지만 보안 스캔, 라이선스 검사, 품질 검사와 연결되기도 합니다. 특히 컨테이너 이미지를 만든다면 베이스 이미지 취약점과 secret 포함 여부도 확인해야 합니다.

#### 면접 답변

> 빌드는 소스 코드와 설정, 의존성을 이용해 실행 가능한 바이너리, JAR, Docker 이미지, 정적 번들 같은 배포 산출물을 만드는 단계입니다. 이후 테스트와 배포는 이 산출물을 기준으로 진행합니다.

#### 헷갈리는 포인트
- 빌드와 배포는 다르다. 빌드는 산출물 생성, 배포는 실행 환경에 반영하는 단계다.
- 로컬에서 만든 산출물과 CI 산출물이 다르면 재현성이 떨어진다.
- 빌드 실패는 코드 문제뿐 아니라 의존성, 환경, 설정 문제일 수 있다.

#### 관련 퀴즈
- `general-c029`
- `general-o004`

### 기능 플래그는 배포와 출시를 분리한다
<!-- article: general-feature-flag -->

#### 한눈에 보기
- 기능 플래그는 코드를 배포해도 사용자에게 기능을 켜지 않을 수 있게 한다.
- 카나리 배포, A/B 테스트, 긴급 비활성화에 유용하다.
- 오래 남은 플래그는 코드 복잡도와 테스트 조합을 크게 늘린다.

#### 핵심 정리

기능 플래그(Feature Flag)는 코드 안에 기능의 활성화 조건을 두는 방식입니다. 코드는 운영에 배포하지만 특정 사용자, 특정 비율, 특정 환경에서만 기능을 켤 수 있습니다. 덕분에 배포와 실제 출시를 분리할 수 있습니다.

운영 위험을 줄이는 데 특히 유용합니다. 새 기능을 1% 사용자에게만 켜고 지표를 확인하거나, 문제가 생기면 재배포 없이 기능을 끌 수 있습니다. 레거시 시스템을 새 구조로 옮길 때도 트래픽을 조금씩 전환하는 데 활용할 수 있습니다.

하지만 플래그가 늘어나면 코드 경로와 테스트 조합이 폭발합니다. 실험이 끝난 플래그는 제거하고, 플래그의 소유자와 만료 시점을 관리해야 합니다.

#### 면접 답변

> 기능 플래그는 배포된 코드의 기능을 조건에 따라 켜고 끌 수 있게 해 배포와 출시를 분리합니다. 카나리 배포나 긴급 롤백에 유용하지만 오래 방치하면 코드 복잡도와 테스트 비용이 커집니다.

#### 헷갈리는 포인트
- 기능 플래그는 롤백 전략을 보완하지만 모든 장애를 해결하지는 않는다.
- 플래그 조건이 권한 정책처럼 변하면 보안 검토가 필요하다.
- 오래된 플래그 제거도 개발 작업의 일부다.

#### 관련 퀴즈

- `general-o005`

## Docker와 컨테이너
<!-- category: general-docker-container -->

### 컨테이너는 애플리케이션 실행 환경을 패키징한다
<!-- article: general-docker-container -->

#### 한눈에 보기
- 컨테이너는 애플리케이션과 의존성을 이미지로 묶어 실행한다.
- 호스트 OS 커널을 공유하고 네임스페이스와 cgroup으로 격리한다.
- VM은 게스트 OS 전체를 실행하므로 일반적으로 더 무겁다.

#### 핵심 정리

Docker 컨테이너는 애플리케이션 코드, 라이브러리, 설정을 이미지로 패키징하고 어디서든 비슷한 방식으로 실행할 수 있게 해줍니다. 개발, 테스트, 운영 환경 차이를 줄이고 배포를 재현하기 쉽게 만드는 것이 큰 장점입니다.

컨테이너와 가상 머신(VM)의 핵심 차이는 운영체제 계층입니다. 컨테이너는 호스트 OS 커널을 공유하면서 프로세스 격리를 제공합니다. VM은 하이퍼바이저 위에서 게스트 OS 전체를 실행하므로 격리는 강하지만 더 무겁고 시작 시간이 길 수 있습니다.

컨테이너는 가볍지만 보안과 운영을 자동으로 해결하지 않습니다. 이미지 취약점, secret 관리, 볼륨 데이터, 네트워크 설정, 권한, 리소스 제한, 로그와 모니터링을 따로 관리해야 합니다.

#### 면접 답변

> Docker 컨테이너는 애플리케이션과 의존성을 이미지로 패키징하고 호스트 커널을 공유해 격리 실행합니다. VM은 게스트 OS 전체를 실행하므로 더 무겁지만 격리 수준이 강한 편입니다.

#### 헷갈리는 포인트
- 컨테이너는 VM처럼 OS 전체를 포함하지 않는다.
- 컨테이너 삭제 시 데이터가 사라질 수 있으므로 영속 데이터는 볼륨으로 관리해야 한다.
- Docker를 쓴다고 보안 패치와 모니터링이 사라지는 것은 아니다.

#### 관련 퀴즈

- `general-t017`

## HTTP 버전
<!-- category: general-http-version -->

### HTTP 상태 코드와 요청 흐름은 웹 API의 기본 언어다
<!-- article: general-http-status-flow -->

#### 한눈에 보기
- HTTP 요청은 DNS 조회, 연결 수립, 요청 전송, 응답 수신의 흐름을 거친다.
- 2xx는 성공, 4xx는 클라이언트 오류, 5xx는 서버 오류 계열이다.
- 상태 코드는 API 사용자가 실패 원인을 판단하는 중요한 힌트다.

#### 핵심 정리

사용자가 URL을 입력하면 클라이언트는 먼저 도메인을 IP로 바꾸기 위해 DNS를 조회합니다. 이후 서버와 TCP 연결을 수립하고, HTTP 요청을 보낸 뒤, 서버가 상태 코드와 바디를 포함한 응답을 돌려줍니다. HTTPS라면 HTTP 데이터를 주고받기 전에 TLS 핸드셰이크가 추가됩니다.

상태 코드는 응답의 의미를 짧게 알려주는 약속입니다. `200 OK`는 성공, `201 Created`는 생성 성공, `400 Bad Request`는 요청 형식 문제, `401 Unauthorized`는 인증 필요, `403 Forbidden`은 인증됐지만 권한 없음, `404 Not Found`는 리소스 없음, `500 Internal Server Error`는 서버 내부 오류를 뜻합니다.

상태 코드를 대충 쓰면 클라이언트가 재시도해야 하는지, 사용자에게 어떤 메시지를 보여줘야 하는지 판단하기 어렵습니다. API 설계에서는 응답 바디만큼 상태 코드도 일관되게 관리해야 합니다.

#### 면접 답변

> HTTP 요청은 DNS 조회, TCP 연결 수립, 요청 전송, 응답 수신 순서로 진행됩니다. 상태 코드는 응답 의미를 나타내며 2xx는 성공, 4xx는 클라이언트 오류, 5xx는 서버 오류로 구분합니다.

#### 헷갈리는 포인트
- `401`은 미인증, `403`은 인증은 됐지만 권한이 없는 상황에 가깝다.
- 모든 실패를 `200`으로 감싸면 HTTP 의미를 활용하기 어렵다.
- `500`은 클라이언트가 고칠 수 없는 서버 쪽 문제를 의미한다.

#### 관련 퀴즈
- `general-c007`
- `general-o003`

### HTTP/2는 하나의 연결을 더 효율적으로 쓴다
<!-- article: general-http2 -->

#### 한눈에 보기
- HTTP/1.1은 텍스트 기반이고 요청 처리에서 비효율이 생길 수 있다.
- HTTP/2는 바이너리 프레이밍, 멀티플렉싱, 헤더 압축을 제공한다.
- HTTP/2도 TCP 위에서는 패킷 손실에 따른 한계가 남을 수 있다.

#### 핵심 정리

HTTP/1.1은 널리 쓰이는 프로토콜이지만 하나의 연결에서 요청과 응답을 효율적으로 섞어 처리하기 어렵고, 매 요청마다 반복되는 헤더가 많습니다. 여러 리소스를 가져오는 웹 페이지에서는 연결과 대기 비용이 커질 수 있습니다.

HTTP/2는 메시지를 바이너리 프레임으로 나누고, 하나의 TCP 연결 안에서 여러 스트림을 동시에 주고받을 수 있게 합니다. 이를 멀티플렉싱이라고 합니다. 또한 헤더 압축을 통해 반복되는 헤더 전송 비용을 줄입니다.

서버 푸시는 서버가 클라이언트가 요청할 것 같은 리소스를 미리 보내는 기능이지만, 실제 운영에서는 캐시와 우선순위 문제로 신중히 사용됩니다. 면접에서는 HTTP/2의 대표 개선점인 멀티플렉싱, 바이너리 프레이밍, 헤더 압축을 중심으로 설명하면 충분합니다.

#### 면접 답변

> HTTP/2는 HTTP/1.1의 비효율을 줄이기 위해 바이너리 프레이밍, 멀티플렉싱, 헤더 압축을 제공합니다. 하나의 TCP 연결에서 여러 요청과 응답을 스트림으로 동시에 처리할 수 있다는 점이 핵심입니다.

#### 헷갈리는 포인트
- HTTP/2가 REST 설계를 대체하는 것은 아니다.
- 멀티플렉싱은 요청을 섞어 보내는 전송 방식의 개선이다.
- 서버 푸시는 HTTP/2의 기능이지만 항상 성능을 높이는 것은 아니다.

#### 관련 퀴즈
- 아직 직접 연결할 퀴즈 없음

## Git과 GitHub
<!-- category: general-git-github -->

### Git 브랜치 워크플로우는 협업 규칙이다
<!-- article: general-git-workflow -->

#### 한눈에 보기
- Git은 변경 이력을 추적하고 브랜치로 독립 작업을 가능하게 한다.
- Feature Branch 흐름은 브랜치 생성, 개발, PR, 리뷰, 병합 순서로 진행된다.
- main 브랜치 보호 규칙은 협업 사고를 줄이는 기본 장치다.

#### 핵심 정리

Git은 버전 관리 시스템입니다. 변경 이력을 커밋으로 남기고, 브랜치를 통해 여러 작업을 독립적으로 진행할 수 있습니다. GitHub는 Git 저장소 위에 Pull Request, Issue, Actions, Projects 같은 협업 기능을 얹은 플랫폼입니다.

Feature Branch 워크플로우에서는 main 또는 develop에서 기능 브랜치를 만들고, 그 브랜치에서 개발과 커밋을 진행합니다. 이후 Pull Request를 열어 코드 리뷰와 CI 검증을 받고, 문제가 없으면 main이나 develop에 병합합니다.

팀 프로젝트에서 main에 직접 push하거나 force push하면 다른 사람의 작업을 덮어쓸 위험이 큽니다. 브랜치 보호 규칙, 필수 리뷰, CI 통과 조건을 설정하면 이런 위험을 줄일 수 있습니다.

#### 면접 답변

> Git은 변경 이력을 관리하고 브랜치로 독립 작업을 가능하게 하는 버전 관리 도구입니다. 팀에서는 feature 브랜치에서 작업하고 PR, 리뷰, CI 검증을 거쳐 main에 병합하는 흐름을 많이 사용합니다.

#### 헷갈리는 포인트
- Git과 GitHub는 다르다. Git은 도구, GitHub는 협업 플랫폼이다.
- force push는 공유 브랜치에서 매우 위험하다.
- 브랜치 전략은 배포 주기와 팀 규모에 맞게 정해야 한다.

#### 관련 퀴즈
- `general-s002`
- `general-d001`
- `general-o001`

### merge와 rebase는 히스토리를 다르게 만든다
<!-- article: general-merge-rebase-stash -->

#### 한눈에 보기
- merge는 병합 커밋으로 두 브랜치의 합류 기록을 남긴다.
- rebase는 커밋을 다른 기준 위에 재적용해 선형 히스토리를 만든다.
- stash는 커밋하지 않은 변경을 잠시 보관하는 기능이다.

#### 핵심 정리

`git merge`는 두 브랜치의 공통 조상을 기준으로 변경 사항을 합치고, 필요하면 병합 커밋을 만듭니다. 언제 어떤 브랜치가 합쳐졌는지 기록이 남아 추적이 쉽지만 히스토리가 복잡해질 수 있습니다.

`git rebase`는 현재 브랜치의 커밋을 다른 브랜치 끝으로 옮겨 다시 적용합니다. 히스토리가 선형으로 정리되어 읽기 쉬워지지만, 이미 공유된 커밋을 rebase하면 다른 사람의 히스토리를 흔들 수 있습니다. 그래서 공개 브랜치에는 조심해야 합니다.

`git stash`는 아직 커밋하지 않은 변경 사항을 임시 저장하고 워킹 디렉터리를 깨끗하게 만드는 기능입니다. 급하게 다른 브랜치로 전환하거나 pull을 받아야 할 때 유용하며, `stash pop`으로 다시 꺼낼 수 있습니다.

#### 면접 답변

> merge는 병합 커밋을 만들어 히스토리를 보존하고, rebase는 커밋을 재적용해 선형 히스토리를 만듭니다. rebase는 깔끔하지만 공유 커밋에 쓰면 위험하고, stash는 작업 중 변경을 잠시 보관할 때 사용합니다.

#### 헷갈리는 포인트
- rebase가 merge보다 항상 좋은 것은 아니다.
- 충돌 해결은 merge와 rebase 모두에서 발생할 수 있다.
- stash는 백업이나 협업 공유 용도가 아니라 임시 보관에 가깝다.

#### 관련 퀴즈
- `general-c002`
- `general-t002`
- `general-c008`

### Git Flow는 릴리스 관리가 엄격한 전략이다
<!-- article: general-git-flow -->

#### 한눈에 보기
- Git Flow는 main, develop, feature, release, hotfix 브랜치 역할을 나눈다.
- 릴리스 주기가 명확하고 운영 hotfix가 잦은 팀에 유리하다.
- 지속 배포 환경에서는 단순 Feature Branch 전략이 더 가벼울 수 있다.

#### 핵심 정리

Git Flow는 운영 배포 기준인 main, 통합 개발 브랜치인 develop, 기능 개발용 feature, 배포 준비용 release, 운영 긴급 수정용 hotfix 브랜치를 나누어 사용합니다. 브랜치 역할이 명확해 릴리스 관리가 체계적입니다.

하지만 브랜치가 많고 흐름이 길어 지속 배포를 자주 하는 팀에는 무겁게 느껴질 수 있습니다. 짧은 feature 브랜치와 PR 중심으로 main에 자주 병합하는 GitHub Flow 방식이 더 적합한 경우도 많습니다.

선택 기준은 팀 규모, 릴리스 주기, 운영 hotfix 필요성, 배포 빈도입니다. 브랜치 전략 자체보다 중요한 것은 main 보호, 코드 리뷰, CI 검증, 롤백 규칙처럼 팀이 실제로 지키는 운영 규칙입니다.

#### 면접 답변

> Git Flow는 main, develop, feature, release, hotfix 브랜치 역할을 나눈 전략입니다. 릴리스 관리에는 유리하지만 복잡하므로, 배포 빈도가 높고 빠른 팀은 단순 Feature Branch 전략을 선택할 수 있습니다.

#### 헷갈리는 포인트
- Git Flow를 쓴다고 품질이 자동으로 보장되지는 않는다.
- 단순 전략일수록 리뷰와 보호 규칙이 더 중요하다.
- 브랜치 이름보다 병합 조건과 배포 규칙이 핵심이다.

#### 관련 퀴즈
- `general-c013`
- `general-t019`

## 암호화와 인증
<!-- category: general-security-authentication -->

### 해싱은 되돌릴 수 없는 검증용 변환이다
<!-- article: general-hashing -->

#### 한눈에 보기
- 해싱은 임의 길이 입력을 고정 길이 해시값으로 바꾸는 단방향 함수다.
- 무결성 검증과 비밀번호 저장에 사용되지만 목적에 따라 알고리즘 선택이 다르다.
- 비밀번호에는 salt와 비용 조절이 가능한 느린 해시가 필요하다.

#### 핵심 정리

해싱(Hashing)은 원문을 고정 길이의 digest로 변환하는 단방향 함수입니다. 같은 입력은 같은 해시값을 만들지만, 해시값만 보고 원문을 복원할 수 없어야 합니다. 입력이 조금만 달라져도 완전히 다른 해시값이 나오는 성질도 중요합니다.

무결성 검증에는 SHA-256 같은 빠른 해시가 유용합니다. 파일이 전송 중 바뀌지 않았는지, 데이터가 변조되지 않았는지 확인할 때 쓰입니다. 하지만 비밀번호 저장에는 빠른 해시를 단독으로 쓰면 무차별 대입 공격에 취약합니다.

비밀번호는 bcrypt, scrypt, Argon2처럼 salt와 비용 조절이 가능한 느린 해시를 사용합니다. salt는 같은 비밀번호도 다른 해시값을 갖게 해 레인보우 테이블 공격을 어렵게 만듭니다.

#### 면접 답변

> 해싱은 원문을 고정 길이 값으로 바꾸는 단방향 함수입니다. 무결성 검증에는 SHA-256 같은 해시를 쓰고, 비밀번호 저장에는 salt와 비용 조절이 가능한 bcrypt 같은 느린 해시를 사용해야 합니다.

#### 헷갈리는 포인트
- 해싱은 복호화가 가능한 암호화가 아니다.
- 충돌 가능성이 이론적으로 없다는 뜻은 아니다.
- SHA-256은 무결성 검증에는 좋지만 비밀번호 저장에는 단독 사용을 피한다.

#### 관련 퀴즈
- `general-c024`

### 대칭키와 비대칭키는 속도와 키 교환의 균형이다
<!-- article: general-symmetric-asymmetric-encryption -->

#### 한눈에 보기
- 대칭키 암호화는 같은 키로 암호화와 복호화를 수행해 빠르다.
- 비대칭키 암호화는 공개키와 개인키 쌍을 사용해 키 교환과 서명에 유리하다.
- 실무에서는 두 방식을 조합한 하이브리드 암호화를 많이 사용한다.

#### 핵심 정리

대칭키 암호화는 하나의 비밀키로 암호화와 복호화를 수행합니다. AES가 대표적이며 속도가 빠르고 대용량 데이터 암호화에 적합합니다. 문제는 통신 상대에게 같은 키를 안전하게 전달해야 한다는 점입니다.

비대칭키 암호화는 공개키와 개인키 한 쌍을 사용합니다. 공개키는 공개해도 되고 개인키는 소유자만 보관합니다. 공개키로 암호화한 데이터는 개인키로 복호화할 수 있고, 개인키로 만든 서명은 공개키로 검증할 수 있습니다. RSA, ECC가 대표적입니다.

비대칭키는 키 교환과 인증에는 유리하지만 계산 비용이 큽니다. 그래서 대용량 데이터를 모두 비대칭키로 암호화하기보다, 비대칭키로 대칭키를 안전하게 합의한 뒤 실제 데이터는 대칭키로 암호화하는 하이브리드 방식을 씁니다.

#### 면접 답변

> 대칭키는 같은 키로 암호화와 복호화를 해서 빠르지만 키 교환이 어렵습니다. 비대칭키는 공개키와 개인키를 사용해 키 교환과 전자서명에 유리하지만 느려서, 실무에서는 둘을 조합한 하이브리드 방식을 많이 씁니다.

#### 헷갈리는 포인트
- 공개키는 공개되어도 되지만 개인키는 절대 유출되면 안 된다.
- 비대칭키가 더 안전하다고 해서 대용량 데이터에 항상 적합한 것은 아니다.
- 디지털 서명은 기밀성보다 인증과 무결성 확인에 가깝다.

#### 관련 퀴즈
- `general-c025`
- `general-c026`
- `general-c027`

### TLS와 CA는 서버를 믿고 안전한 키를 나누기 위한 구조다
<!-- article: general-tls-ca -->

#### 한눈에 보기
- CA는 서버 공개키와 도메인 정보를 검증해 인증서에 서명한다.
- TLS는 서버 인증서 검증 뒤 세션 키를 합의한다.
- 실제 HTTP 데이터는 합의된 대칭키로 암호화되어 전송된다.

#### 핵심 정리

HTTPS는 HTTP에 TLS 보안 계층을 더한 방식입니다. 클라이언트는 먼저 서버가 보낸 인증서를 확인해 이 서버가 정말 해당 도메인의 서버인지 검증합니다. 이때 인증서에는 서버 공개키, 도메인 정보, CA의 서명이 포함됩니다.

CA(Certificate Authority)는 인증 기관입니다. 특정 공개키가 특정 도메인이나 조직에 속한다는 사실을 검증하고 인증서에 서명합니다. 브라우저는 신뢰할 수 있는 루트 CA 체인을 통해 이 서명을 검증합니다.

서버 신뢰성을 확인한 뒤 클라이언트와 서버는 키 교환 과정을 통해 세션 키를 합의합니다. 이후 실제 애플리케이션 데이터는 빠른 대칭키 암호화로 보호됩니다. 이 흐름이 TLS에서 하이브리드 암호화가 쓰이는 이유입니다.

#### 면접 답변

> TLS에서는 먼저 CA가 서명한 서버 인증서를 검증해 서버 신뢰성을 확인합니다. 그 뒤 키 교환으로 세션 키를 합의하고, 실제 HTTP 데이터는 빠른 대칭키로 암호화해 전송합니다.

#### 헷갈리는 포인트
- 인증서는 서버의 신뢰성을 확인하기 위한 것이지 데이터를 모두 직접 암호화하는 파일이 아니다.
- CA 검증이 약하면 중간자 공격 위험이 커진다.
- TLS의 실제 데이터 암호화는 대부분 대칭키로 이뤄진다.

#### 관련 퀴즈
- `general-c027`
- `general-c028`
- `general-o006`

### JWT는 서명된 토큰이지 기본 암호문이 아니다
<!-- article: general-jwt -->

#### 한눈에 보기
- JWT는 Header, Payload, Signature로 구성된 토큰 형식이다.
- 서명은 토큰 위조를 막기 위한 것이며 Payload는 기본적으로 읽을 수 있다.
- 탈취 위험 때문에 짧은 만료와 Refresh Token, HTTPS가 중요하다.

#### 핵심 정리

JWT(JSON Web Token)는 인증 정보를 토큰에 담아 클라이언트가 보관하고, 서버가 서명을 검증해 신뢰하는 방식에 자주 사용됩니다. 서버가 세션 저장소를 유지하지 않아도 되므로 무상태 인증과 확장성 측면에서 장점이 있습니다.

중요한 점은 JWT Payload가 기본적으로 암호화된 것이 아니라는 점입니다. Base64URL로 인코딩되어 있을 뿐 누구나 내용을 읽을 수 있습니다. 따라서 주민번호, 비밀번호, 카드번호 같은 민감 정보는 Payload에 넣으면 안 됩니다.

JWT가 탈취되면 만료 전까지 악용될 수 있습니다. 그래서 Access Token은 짧게 두고, Refresh Token으로 재발급하며, 필요하면 블랙리스트나 토큰 버전 전략을 사용합니다. HTTPS 없이 JWT를 주고받는 것도 위험합니다.

#### 면접 답변

> JWT는 서명된 토큰으로, 서버가 세션 저장소 없이도 토큰의 위조 여부를 검증할 수 있게 합니다. 하지만 Payload는 기본적으로 암호화가 아니므로 민감 정보를 넣으면 안 되고, 탈취에 대비해 짧은 만료와 HTTPS가 필요합니다.

#### 헷갈리는 포인트
- JWT 서명은 위조 방지이지 내용 숨김이 아니다.
- JWT를 쓰면 로그아웃과 강제 만료 처리가 오히려 어려워질 수 있다.
- Refresh Token도 탈취되면 위험하므로 저장 위치와 회전 전략이 중요하다.

#### 관련 퀴즈
- `general-t018`
- `general-d003`

## 아키텍처 선택과 트레이드오프
<!-- category: general-architecture-tradeoffs -->

### 모놀리식과 MSA는 운영 복잡도가 다르다
<!-- article: general-monolith-msa -->

#### 한눈에 보기
- 모놀리식은 개발, 배포, 디버깅이 단순하다.
- MSA는 서비스별 독립 배포와 확장에 유리하지만 운영 복잡도가 크다.
- 처음부터 MSA가 정답은 아니며 필요가 분명할 때 분리하는 접근이 실용적이다.

#### 핵심 정리

모놀리식 아키텍처는 하나의 애플리케이션 안에 기능을 함께 배치합니다. 개발과 배포가 단순하고, 트랜잭션 관리와 디버깅도 상대적으로 쉽습니다. 초기 서비스나 팀 규모가 작은 경우 빠르게 제품을 만들기에 적합합니다.

MSA(Microservices Architecture)는 기능을 여러 서비스로 나누고 각 서비스가 독립적으로 배포, 확장, 운영될 수 있게 합니다. 특정 기능만 확장하거나, 팀별로 기술 스택을 선택하기 쉬운 장점이 있습니다.

대신 서비스 간 통신, 분산 트랜잭션, 장애 전파, 모니터링, 로그 추적, 배포 순서 관리가 어려워집니다. 그래서 단순히 유행이라서 MSA를 선택하기보다, 도메인 경계와 조직 구조, 확장 요구가 분명할 때 신중히 선택해야 합니다.

#### 면접 답변

> 모놀리식은 단순하고 빠르게 개발하기 좋지만 규모가 커지면 배포와 확장에 한계가 생길 수 있습니다. MSA는 독립 배포와 확장에 유리하지만 운영 복잡도가 크므로 요구사항과 팀 역량에 따라 선택해야 합니다.

#### 헷갈리는 포인트
- MSA가 항상 모놀리식보다 좋은 것은 아니다.
- 서비스가 나뉘면 데이터 일관성 문제도 함께 어려워진다.
- 모듈화가 잘 된 모놀리식은 좋은 출발점이 될 수 있다.

#### 관련 퀴즈
- `general-t004`
- `general-o005`

### CQRS와 이벤트 소싱은 읽기와 쓰기의 모델을 분리한다
<!-- article: general-cqrs-event-sourcing -->

#### 한눈에 보기
- CQRS는 명령과 조회의 책임을 분리하는 패턴이다.
- 이벤트 소싱은 상태 변경 결과가 아니라 변경 이벤트를 저장한다.
- 감사 추적과 확장성에 유리하지만 구현과 운영 복잡도가 높다.

#### 핵심 정리

CQRS(Command Query Responsibility Segregation)는 데이터를 변경하는 명령(Command)과 데이터를 읽는 조회(Query)의 모델을 분리하는 방식입니다. 읽기와 쓰기의 부하, 데이터 모양, 성능 요구가 크게 다를 때 유용합니다.

이벤트 소싱(Event Sourcing)은 현재 상태만 저장하는 대신 상태를 바꾼 이벤트들을 순서대로 저장합니다. 현재 상태는 이벤트를 재생해서 만들 수 있습니다. 주문 생성, 결제 완료, 배송 시작 같은 이벤트를 기록하면 감사 추적과 과거 상태 재구성이 쉬워집니다.

하지만 두 패턴은 단순 CRUD보다 훨씬 복잡합니다. 최종 일관성, 이벤트 스키마 변경, 중복 처리, 재처리, 디버깅, 운영 도구가 필요합니다. 복잡한 도메인과 명확한 이득이 있을 때 선택해야 합니다.

#### 면접 답변

> CQRS는 쓰기 모델과 읽기 모델을 분리하고, 이벤트 소싱은 상태 변경 이벤트를 저장해 현재 상태를 재구성하는 방식입니다. 감사 추적과 확장성에 유리하지만 최종 일관성과 운영 복잡도가 커집니다.

#### 헷갈리는 포인트
- CQRS와 이벤트 소싱은 함께 쓰일 수 있지만 같은 개념은 아니다.
- 단순 CRUD 시스템에 적용하면 복잡도만 늘 수 있다.
- 이벤트는 한 번 발행되면 변경하기 어려우므로 스키마 진화 전략이 필요하다.

#### 관련 퀴즈

- `general-t015`

### 개발 방법론과 데이터 저장소 선택도 트레이드오프다
<!-- article: general-methodology-data-choice -->

#### 한눈에 보기
- Agile은 변화 대응에 강하고 Waterfall은 요구사항이 안정적일 때 관리가 쉽다.
- SQL은 ACID, 정규화, 복잡한 쿼리에 강하다.
- NoSQL은 유연한 스키마와 수평 확장에 유리하지만 일관성 모델을 확인해야 한다.

#### 핵심 정리

Agile과 Waterfall은 개발을 진행하는 방식의 차이입니다. Waterfall은 요구사항, 설계, 구현, 테스트를 순차적으로 진행하므로 요구사항이 안정적이고 승인 절차가 명확한 프로젝트에 맞습니다. Agile은 짧은 반복 주기로 피드백을 반영하므로 변화가 잦은 제품 개발에 유리합니다.

데이터 저장소 선택도 비슷합니다. SQL 데이터베이스는 정형 스키마, ACID 트랜잭션, 조인과 복잡한 쿼리에 강합니다. 금융, 주문, 재고처럼 일관성이 중요한 도메인에서 자주 선택됩니다.

NoSQL은 문서, 키-값, 컬럼, 그래프 등 다양한 모델이 있고 수평 확장과 유연한 스키마에 강점이 있습니다. 다만 트랜잭션 범위, 일관성 수준, 쿼리 기능은 제품마다 다르므로 요구사항을 기준으로 선택해야 합니다.

#### 면접 답변

> Agile은 변화 대응과 빠른 피드백에 유리하고 Waterfall은 요구사항이 안정적일 때 관리가 쉽습니다. SQL은 ACID와 복잡한 쿼리에 강하고, NoSQL은 유연한 스키마와 수평 확장에 유리하지만 일관성 보장을 확인해야 합니다.

#### 헷갈리는 포인트
- Agile은 계획이 없다는 뜻이 아니다.
- NoSQL이 항상 SQL보다 빠르다는 뜻은 아니다.
- 저장소 선택은 데이터 구조, 일관성, 쿼리 패턴, 확장 요구를 함께 봐야 한다.

#### 관련 퀴즈

- `general-t014`

### 개발자 필수 질문은 선택 기준으로 답한다
<!-- article: general-interview-checklist -->

#### 한눈에 보기
- 단순 정의형 질문은 핵심 한 문장, 장단점, 예시 순서로 답한다.
- 트레이드오프 질문은 언제 어떤 선택이 맞는지 기준을 제시한다.
- 시나리오 질문은 위험을 줄이는 순서와 검증 방법을 함께 말한다.

#### 핵심 정리

면접에서 "무엇인가요?"라는 질문은 정의만 말하면 짧게 끝납니다. 좋은 답변은 핵심 정의 뒤에 왜 필요한지, 어떤 장단점이 있는지, 실무에서 어떤 기준으로 선택하는지를 붙입니다. 예를 들어 Docker는 컨테이너 도구라고 끝내지 말고 환경 일관성과 이미지 보안까지 연결해야 합니다.

트레이드오프 질문은 정답 하나를 고르는 문제가 아닙니다. REST와 GraphQL, 모놀리식과 MSA, merge와 rebase처럼 각각의 장점과 비용이 다릅니다. 답변에서는 "항상 A가 좋다"보다 요구사항, 팀 규모, 운영 복잡도, 변경 빈도 같은 선택 기준을 말해야 합니다.

시나리오 질문은 순서가 중요합니다. 운영 장애라면 먼저 영향 범위와 지표를 확인하고, 최근 변경과 로그를 보고, 재현과 롤백 가능성을 판단합니다. 리팩토링이라면 테스트 안전망을 먼저 만들고 작게 바꾸는 식으로 위험을 낮추는 흐름을 보여줘야 합니다.

#### 면접 답변

> 개발자 공통 질문은 정의, 장단점, 선택 기준, 실무 예시 순서로 답하면 좋습니다. 특히 트레이드오프 질문에서는 항상 정답을 고르기보다 요구사항과 운영 비용을 기준으로 판단하는 과정을 보여주는 것이 중요합니다.

#### 헷갈리는 포인트
- 암기한 정의만 말하면 깊이 질문에서 막히기 쉽다.
- "항상", "무조건" 같은 표현은 대부분 위험하다.
- 장애나 마이그레이션 질문은 안전한 순서와 검증 방법을 함께 말해야 한다.
