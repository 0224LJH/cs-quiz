### Java 실행 방식과 JVM의 역할
<!-- article: java-runtime-jvm -->

#### 한눈에 보기
- Java 소스 코드는 `javac`로 바이트코드(`.class`)가 된다.
- JVM은 바이트코드를 읽어 인터프리트하거나 JIT 컴파일해서 실행한다.
- Java의 플랫폼 독립성은 바이트코드 덕분이고, JVM 자체는 운영체제별로 다르다.

#### 핵심 정리

Java는 순수 인터프리터 언어도, C/C++처럼 처음부터 기계어로만 컴파일되는 언어도 아닙니다. 개발자가 작성한 `.java` 파일은 `javac`에 의해 JVM이 이해하는 바이트코드로 변환됩니다.

실행 시점에는 JVM이 바이트코드를 읽습니다. 처음에는 인터프리터 방식으로 실행하면서 코드의 실행 빈도를 관찰하고, 반복적으로 많이 실행되는 코드는 JIT 컴파일러가 네이티브 기계어로 바꿔 캐시합니다. 그래서 Java는 컴파일과 인터프리트, JIT 컴파일이 결합된 하이브리드 실행 모델로 설명할 수 있습니다.

면접에서는 "Java는 플랫폼 독립적이고 JVM은 플랫폼 종속적"이라는 구분이 중요합니다. 같은 바이트코드는 여러 운영체제에서 실행될 수 있지만, 실제로 그 바이트코드를 돌리는 JVM 구현체는 각 운영체제와 CPU 환경에 맞게 제공됩니다.

#### 면접 답변

> Java는 소스 코드를 바이트코드로 컴파일한 뒤, JVM이 그 바이트코드를 인터프리트하거나 JIT 컴파일해서 실행하는 하이브리드 방식의 언어입니다. 바이트코드 덕분에 플랫폼 독립성을 얻고, JVM이 실행 환경 차이를 흡수합니다.

#### 헷갈리는 포인트
- `javac`는 소스 코드를 바이트코드로 바꾸는 컴파일러다.
- JVM은 바이트코드를 실행하는 런타임이다.
- JIT는 자주 실행되는 코드를 네이티브 코드로 바꿔 성능을 높인다.

### 기본형과 참조형, 그리고 Call By Value
<!-- article: java-primitive-reference-call-by-value -->

#### 한눈에 보기
- 기본형은 값 자체를 저장한다.
- 참조형 변수는 객체 자체가 아니라 객체를 가리키는 참조 값을 저장한다.
- Java의 메서드 인자 전달은 항상 Call By Value다.

#### 핵심 정리

Java의 기본형은 `byte`, `short`, `int`, `long`, `float`, `double`, `boolean`, `char` 8가지입니다. 기본형 변수에는 실제 값이 들어갑니다. 반면 `String`, 배열, 클래스, 인터페이스 타입은 참조형이며, 변수에는 힙에 있는 객체를 가리키는 참조 값이 들어갑니다.

Java가 항상 Call By Value라고 말하는 이유도 여기에 있습니다. 기본형을 메서드에 넘기면 값이 복사됩니다. 참조형을 넘길 때도 객체 자체가 넘어가는 것이 아니라 참조 값이 복사됩니다.

따라서 메서드 안에서 참조가 가리키는 객체의 내부 상태를 바꾸면 호출자도 그 변경을 볼 수 있습니다. 하지만 매개변수에 새 객체를 대입해도 호출자의 변수 자체가 다른 객체를 가리키게 되지는 않습니다.

#### 면접 답변

> Java는 항상 값에 의한 전달입니다. 기본형은 실제 값이 복사되고, 참조형은 객체가 아니라 참조 값이 복사됩니다. 그래서 객체 내부 상태 변경은 보일 수 있지만, 매개변수 재할당은 호출자 변수에 영향을 주지 않습니다.

#### 헷갈리는 포인트
- 참조형 전달은 "참조에 의한 전달"이 아니다.
- 참조 값이 복사되기 때문에 같은 객체를 가리킬 수 있을 뿐이다.
- `String`도 참조형이지만 불변 객체라 내부 상태가 바뀌지 않는다.

### String과 String Pool
<!-- article: java-string-pool-immutability -->

#### 한눈에 보기
- `String`은 불변 객체다.
- 문자열 리터럴은 String Pool에서 재사용될 수 있다.
- 문자열 비교에서 `==`는 참조 비교, `equals()`는 내용 비교다.

#### 핵심 정리

`String`은 한 번 만들어지면 내부 값을 바꿀 수 없는 불변 객체입니다. 문자열을 수정하는 것처럼 보이는 연산은 기존 객체를 바꾸는 것이 아니라 새 문자열 객체를 만드는 방식으로 동작합니다.

문자열 리터럴은 String Pool을 통해 재사용될 수 있습니다. 예를 들어 `"hello"`라는 리터럴이 여러 번 등장하면 같은 객체를 참조할 수 있습니다. 반면 `new String("hello")`는 별도 객체를 만들 수 있으므로 `==` 비교 결과가 다르게 나올 수 있습니다.

반복문에서 `String`을 `+`로 계속 연결하면 매번 새 객체가 생겨 비효율적입니다. 단일 스레드의 반복 문자열 조립에는 `StringBuilder`를, 공유 mutable 버퍼가 꼭 필요한 멀티스레드 상황에는 `StringBuffer`를 고려합니다.

#### 면접 답변

> String은 불변 객체라서 String Pool에서 안전하게 재사용할 수 있고, 여러 스레드가 공유해도 상태가 바뀌지 않습니다. 내용 비교는 `equals()`로 해야 하며, 반복적인 문자열 조합은 `StringBuilder`를 쓰는 것이 보통 더 효율적입니다.

#### 헷갈리는 포인트
- `==`는 두 변수가 같은 객체를 가리키는지 비교한다.
- `equals()`는 객체가 정의한 동등성 기준으로 비교한다.
- `Integer` 같은 래퍼 타입도 캐싱 범위 때문에 `==` 결과가 헷갈릴 수 있다.

### static과 final
<!-- article: java-static-final -->

#### 한눈에 보기
- `static`은 인스턴스가 아니라 클래스에 속한다.
- `final` 변수는 재할당할 수 없다.
- `final` 참조형 변수는 참조 대상의 내부 변경까지 막지는 못한다.

#### 핵심 정리

`static` 멤버는 객체를 만들지 않아도 클래스 이름으로 접근할 수 있고, 모든 인스턴스가 공유합니다. 상수, 유틸 메서드, 공통 캐시처럼 클래스 수준에서 하나만 존재해야 하는 값이나 기능에 어울립니다.

하지만 `static`은 전역 상태가 되기 쉽습니다. 오래 살아 있는 참조 때문에 메모리 누수가 생길 수 있고, 여러 스레드가 동시에 변경하면 동시성 문제가 발생할 수 있습니다. 테스트에서도 전역 상태는 격리하기 어렵습니다.

`final`은 붙는 위치에 따라 의미가 달라집니다. 변수에 붙으면 재할당 금지, 메서드에 붙으면 오버라이딩 금지, 클래스에 붙으면 상속 금지입니다. 참조형 변수에 `final`을 붙이면 다른 객체로 바꾸는 것은 막지만, 그 객체가 가변이면 내부 값은 바뀔 수 있습니다.

#### 면접 답변

> static은 클래스 레벨 공유가 필요할 때 사용하고, final은 재할당이나 재정의를 막을 때 사용합니다. 다만 static 상태는 전역 공유로 인한 동시성, 테스트, 메모리 유지 문제를 만들 수 있고, final 참조가 객체 내부 불변성까지 보장하지는 않습니다.

#### 헷갈리는 포인트
- `static` 필드는 객체가 아니라 클래스에 귀속된다.
- `obj.x`처럼 null 참조로 static 필드에 접근해도 컴파일상 허용될 수 있지만 좋은 코드가 아니다.
- 진짜 불변 객체를 만들려면 필드 `final`만으로는 부족하고 객체 설계 전체가 불변이어야 한다.

#### 관련 퀴즈
- `java-c004`
- `java-c007`
- `java-t011`
- `java-c031`
- `java-d007`

### 클래스, 객체, 캡슐화
<!-- article: java-class-object-encapsulation -->

#### 한눈에 보기
- 클래스는 객체를 만들기 위한 설계도다.
- 객체는 클래스를 기반으로 메모리에 생성된 실체다.
- 캡슐화는 내부 상태를 보호하고 변경 지점을 통제한다.

#### 핵심 정리

클래스는 필드와 메서드로 객체의 구조와 동작을 정의합니다. 객체는 그 클래스 정의를 기반으로 런타임에 생성된 인스턴스입니다. 같은 클래스로 만든 객체라도 각 객체는 독립적인 상태를 가질 수 있습니다.

캡슐화는 객체 내부 상태를 외부에서 직접 만지지 못하게 하고, 메서드를 통해 의미 있는 변경만 허용하는 방식입니다. 필드를 `private`로 두고 필요한 접근 메서드나 비즈니스 메서드를 제공하면 객체의 불변 조건을 지키기 쉬워집니다.

접근 제어자는 캡슐화를 구현하는 기본 도구입니다. `private`는 같은 클래스, package-private은 같은 패키지, `protected`는 같은 패키지와 하위 클래스, `public`은 어디서든 접근 가능합니다.

#### 면접 답변

> 클래스는 객체의 구조와 동작을 정의하는 설계도이고, 객체는 그 설계도로 생성된 런타임 실체입니다. 캡슐화는 내부 상태를 숨기고 메서드 경계를 통해 변경을 통제해서 객체의 일관성을 지키는 원칙입니다.

#### 헷갈리는 포인트
- getter/setter를 무조건 만드는 것이 캡슐화는 아니다.
- 객체가 스스로 상태 변경 규칙을 갖게 만드는 것이 더 좋은 캡슐화다.
- `protected`는 "상속받으면 항상 어디서든 접근"처럼 단순하지 않고 패키지와 상속 관계 규칙을 함께 본다.

#### 관련 퀴즈
- `java-c021`
- `java-c032`
- `java-c033`

### 인터페이스와 추상 클래스
<!-- article: java-interface-abstract-class -->

#### 한눈에 보기
- 둘 다 직접 인스턴스화할 수 없다.
- 추상 클래스는 상태와 공통 구현을 담기 좋다.
- 인터페이스는 역할과 계약을 표현하고 다중 구현이 가능하다.

#### 핵심 정리

추상 클래스는 상속 계층에서 공통 상태와 공통 동작을 묶고, 일부 동작만 하위 클래스에 맡기고 싶을 때 유용합니다. 생성자와 필드를 가질 수 있고, 일반 메서드와 추상 메서드를 함께 둘 수 있습니다.

인터페이스는 어떤 타입이 제공해야 하는 행위의 계약을 표현합니다. Java 8 이후에는 `default` 메서드와 `static` 메서드를 가질 수 있지만, 중심 역할은 여전히 구현 클래스가 지켜야 할 API를 정의하는 것입니다.

선택 기준은 "is-a 계층과 공통 상태"가 중요한지, 아니면 "역할과 교체 가능성"이 중요한지입니다. 여러 구현체가 같은 역할을 제공해야 하고 다중 구현 가능성이 필요하면 인터페이스가 더 자연스럽습니다.

#### 면접 답변

> 추상 클래스는 공통 상태와 구현을 공유하는 상속 기반 모델에 적합하고, 인터페이스는 구현체가 지켜야 할 역할과 계약을 정의하는 데 적합합니다. Java에서는 다중 구현이 가능한 인터페이스를 통해 더 유연한 설계를 만들 수 있습니다.

#### 헷갈리는 포인트
- Java 8 이후 인터페이스도 default 메서드를 가질 수 있다.
- default 메서드는 편리하지만 인터페이스가 과도하게 구현 세부사항을 갖게 만들 수 있다.
- 공통 코드 재사용만을 이유로 상속을 선택하면 결합도가 높아질 수 있다.

#### 관련 퀴즈
- `java-c006`
- `java-t005`

### 오버로딩, 오버라이딩, 다형성
<!-- article: java-overloading-overriding-polymorphism -->

#### 한눈에 보기
- 오버로딩은 같은 이름의 메서드를 매개변수 목록 다르게 여러 개 정의하는 것이다.
- 오버라이딩은 상위 타입의 메서드를 하위 타입에서 재정의하는 것이다.
- 다형성은 상위 타입으로 하위 객체를 다루고, 실제 객체 기준으로 동작을 바꾸는 능력이다.

#### 핵심 정리

오버로딩은 컴파일 시점에 매개변수 타입과 개수로 호출 대상이 결정됩니다. 반환 타입만 다르게 하는 것은 오버로딩이 아닙니다. 호출하는 쪽에서는 같은 이름의 메서드를 상황에 맞게 사용할 수 있습니다.

오버라이딩은 상속 관계에서 상위 클래스나 인터페이스의 메서드를 하위 타입이 다시 구현하는 것입니다. 인스턴스 메서드는 런타임 실제 객체 타입에 따라 호출되므로 다형성을 가능하게 합니다.

예를 들어 `Animal a = new Dog()`에서 참조 변수 타입은 `Animal`이지만, 오버라이딩된 `sound()` 호출은 실제 객체인 `Dog`의 구현으로 동작합니다. 이 특성 덕분에 코드는 상위 추상화에 의존하면서 구현체를 교체할 수 있습니다.

#### 면접 답변

> 오버로딩은 같은 이름의 메서드를 매개변수 목록으로 구분하는 컴파일 시점 다형성이고, 오버라이딩은 상속 관계에서 상위 메서드를 재정의해 런타임 실제 객체 기준으로 호출되게 하는 기능입니다.

#### 헷갈리는 포인트
- 메서드 이름의 대소문자가 다르면 오버라이딩이 아니다.
- `@Override`를 붙이면 오타로 인한 실수를 컴파일 단계에서 잡을 수 있다.
- 필드와 static 메서드는 인스턴스 메서드와 같은 방식의 동적 디스패치 대상이 아니다.

#### 관련 퀴즈
- `java-c035`
- `java-c036`
- `java-d004`

### equals와 hashCode 계약
<!-- article: java-equals-hashcode-contract -->

#### 한눈에 보기
- `equals()`가 true인 두 객체는 반드시 같은 `hashCode()`를 가져야 한다.
- `hashCode()`가 같다고 `equals()`가 true일 필요는 없다.
- `HashMap`, `HashSet`의 정상 동작은 이 계약에 의존한다.

#### 핵심 정리

객체의 동등성을 직접 정의할 때는 `equals()`와 `hashCode()`를 함께 고려해야 합니다. 두 객체가 논리적으로 같다고 판단되어 `equals()`가 true라면, 해시 기반 컬렉션에서 같은 버킷을 찾을 수 있도록 `hashCode()`도 같아야 합니다.

반대로 해시 충돌은 허용됩니다. 서로 다른 객체라도 같은 해시 값을 가질 수 있습니다. 이때 컬렉션은 같은 버킷 안에서 `equals()`를 다시 호출해 실제로 같은 키인지 판단합니다.

이 계약을 어기면 `HashMap`에 넣은 값을 다시 찾지 못하거나, `HashSet`에 중복 데이터가 들어가는 것처럼 보이는 문제가 생길 수 있습니다.

#### 면접 답변

> equals가 true인 객체들은 반드시 같은 hashCode를 반환해야 합니다. 해시 기반 컬렉션은 먼저 hashCode로 버킷을 찾고, 그다음 equals로 실제 동등성을 확인하기 때문에 두 메서드의 계약을 함께 지켜야 합니다.

#### 헷갈리는 포인트
- `hashCode()`가 같아도 두 객체가 같다는 뜻은 아니다.
- 값 객체를 만들 때는 동등성 기준에 포함할 필드를 명확히 정해야 한다.
- 가변 필드를 해시 기준으로 쓰면 컬렉션에 넣은 뒤 조회가 깨질 수 있다.

#### 관련 퀴즈
- `java-c012`
- `java-s005`

### List 계열 선택 기준
<!-- article: java-list-arraylist-linkedlist-vector -->

#### 한눈에 보기
- `ArrayList`는 인덱스 접근이 빠르다.
- `LinkedList`는 노드 연결 구조지만 임의 접근은 느리다.
- `Vector`는 동기화 비용 때문에 특별한 이유가 없으면 잘 선택하지 않는다.

#### 핵심 정리

`ArrayList`는 내부 배열을 사용합니다. 인덱스로 접근하는 작업은 O(1)에 가깝지만, 중간 삽입이나 삭제는 뒤쪽 요소를 이동해야 하므로 O(n)이 될 수 있습니다. 대부분의 일반적인 목록 처리에서는 `ArrayList`가 기본 선택지입니다.

`LinkedList`는 노드들이 서로 연결된 구조입니다. 이미 위치를 알고 있는 노드의 삽입과 삭제는 포인터 변경으로 처리할 수 있지만, 특정 인덱스까지 찾아가는 데 O(n)이 걸립니다. 그래서 "삽입 삭제가 많으면 무조건 LinkedList"라고 외우면 위험합니다.

`Vector`는 오래된 동기화 컬렉션입니다. 메서드 단위로 동기화되어 있지만 현대 Java에서는 보통 `ArrayList`와 외부 동기화, 또는 동시성 컬렉션을 상황에 맞게 선택합니다.

#### 면접 답변

> 일반적인 조회와 순회 중심 목록은 ArrayList를 우선 고려합니다. LinkedList는 임의 접근이 느리기 때문에 실제 접근 패턴을 보고 선택해야 하고, Vector는 동기화 오버헤드가 있어 레거시 호환 목적이 아니면 자주 쓰지 않습니다.

#### 헷갈리는 포인트
- `Arrays.asList()`가 반환한 리스트는 크기 변경이 제한될 수 있다.
- 향상된 for문 중 컬렉션을 직접 수정하면 `ConcurrentModificationException`이 날 수 있다.
- 반복 중 삭제는 `Iterator.remove()`나 별도 수집 후 제거를 고려한다.

### HashMap과 ConcurrentHashMap
<!-- article: java-hashmap-concurrenthashmap -->

#### 한눈에 보기
- `HashMap`은 일반적으로 빠르지만 스레드 안전하지 않다.
- `ConcurrentHashMap`은 멀티스레드 환경의 동시 접근을 지원한다.
- 해시 기반 컬렉션은 `hashCode()`와 `equals()` 계약에 의존한다.

#### 핵심 정리

`HashMap`은 키의 `hashCode()`로 버킷 인덱스를 정하고, 같은 버킷에 이미 데이터가 있으면 `equals()`로 동일 키인지 확인합니다. 동일 키면 값을 갱신하고, 새 키면 엔트리를 추가합니다. Java 8 이후에는 충돌이 많아질 때 일부 버킷 구조가 트리로 바뀔 수 있습니다.

단일 스레드이거나 외부에서 동기화를 보장하는 상황에서는 `HashMap`이 단순하고 빠릅니다. 하지만 여러 스레드가 동시에 읽고 쓰는 공유 맵이라면 데이터 불일치나 예측하기 어려운 문제가 생길 수 있습니다.

`ConcurrentHashMap`은 동시 접근을 고려해 설계된 컬렉션입니다. 다만 모든 복합 로직이 자동으로 원자적이 되는 것은 아니므로, "없으면 로드해서 넣기" 같은 작업은 `computeIfAbsent` 같은 원자적 API를 검토해야 합니다.

#### 면접 답변

> HashMap은 스레드 안전하지 않은 일반 맵이고, ConcurrentHashMap은 멀티스레드 환경에서 동시 읽기와 쓰기를 지원하는 맵입니다. 공유 캐시처럼 여러 스레드가 접근하는 구조에서는 복합 연산의 원자성까지 고려해 ConcurrentHashMap의 전용 API를 쓰는 것이 좋습니다.

#### 헷갈리는 포인트
- `ConcurrentHashMap`은 `null` 키와 값을 허용하지 않는다.
- `containsKey` 후 `put`은 두 작업 사이에 다른 스레드가 끼어들 수 있다.
- 정렬된 순서가 필요하면 `HashMap` 계열이 아니라 목적에 맞는 다른 맵을 본다.

#### 관련 퀴즈
- `java-t002`
- `java-o004`
- `java-d015`

### Comparable, Comparator, PriorityQueue
<!-- article: java-comparable-comparator-priorityqueue -->

#### 한눈에 보기
- `Comparable`은 객체 자신의 자연 순서를 정의한다.
- `Comparator`는 외부에서 비교 기준을 주입한다.
- `PriorityQueue`는 보통 배열 기반 이진 힙으로 우선순위를 관리한다.

#### 핵심 정리

`Comparable`은 클래스가 기본 정렬 기준을 스스로 갖게 할 때 사용합니다. 예를 들어 이름순, 나이순 중 하나가 그 타입의 자연스러운 순서라면 `compareTo()`로 정의할 수 있습니다.

`Comparator`는 외부 비교 전략입니다. 같은 타입을 상황에 따라 이름순, 생성일순, 점수순으로 다르게 정렬해야 한다면 `Comparator`가 더 적합합니다. 람다와 메서드 참조를 사용하면 비교자를 간결하게 만들 수 있습니다.

`PriorityQueue`는 가장 우선순위가 높은 값을 빠르게 꺼내야 할 때 사용합니다. 내부 구조는 일반적으로 이진 힙이므로 삽입과 삭제는 O(log n), 우선순위가 가장 높은 원소 확인은 O(1)에 가깝습니다.

#### 면접 답변

> Comparable은 타입 내부의 자연 순서를 정의하고, Comparator는 외부에서 정렬 기준을 제공할 때 사용합니다. PriorityQueue는 비교 기준에 따라 힙을 유지해 우선순위가 높은 원소를 효율적으로 꺼낼 수 있게 합니다.

#### 헷갈리는 포인트
- `numbers.sort((a, b) -> b - a)`는 정수 오버플로우 위험이 있다.
- 안전하게는 `Integer.compare(b, a)`처럼 비교 메서드를 사용한다.
- `PriorityQueue`의 순회 결과가 전체 정렬 순서를 보장하는 것은 아니다.

#### 관련 퀴즈
- `java-c018`
- `java-d011`

### 제네릭과 와일드카드
<!-- article: java-generics-wildcard-type-erasure -->

#### 한눈에 보기
- 제네릭은 컴파일 시점 타입 안정성을 높인다.
- 타입 소거 때문에 런타임에는 제네릭 타입 정보가 제한된다.
- `<T>`는 타입 파라미터 선언, `<?>`는 알 수 없는 타입을 받는 와일드카드다.

#### 핵심 정리

제네릭을 사용하면 컬렉션에서 값을 꺼낼 때 매번 형변환하지 않아도 되고, 잘못된 타입을 넣는 실수를 컴파일 단계에서 줄일 수 있습니다. 예를 들어 `List<String>`은 문자열 목록이라는 의도를 타입으로 표현합니다.

하지만 Java 제네릭은 타입 소거 방식을 사용합니다. 컴파일 후 바이트코드에서는 타입 파라미터 정보가 대부분 제거되고 `Object`나 바운드 타입으로 다뤄집니다. 이 때문에 `new T[]` 같은 코드는 만들 수 없고, raw type을 섞으면 런타임 `ClassCastException` 위험이 커집니다.

`<T>`는 클래스나 메서드가 타입 파라미터를 직접 선언할 때 사용합니다. `<?>`는 이미 만들어진 제네릭 타입을 받을 때 구체 타입이 중요하지 않거나 알 수 없음을 표현합니다.

#### 면접 답변

> 제네릭은 컴파일 타임 타입 체크로 형변환 오류를 줄이고 코드 재사용성을 높입니다. 다만 Java는 타입 소거를 사용하므로 런타임에는 제네릭 타입 정보가 제한되고, raw type을 섞으면 타입 안정성이 깨질 수 있습니다.

#### 헷갈리는 포인트
- `List<String>`과 `List<Integer>`는 런타임에 같은 `List` 타입처럼 취급된다.
- raw type은 하위 호환용에 가깝고 새 코드에서는 피하는 것이 좋다.
- 읽기와 쓰기 방향이 있는 API에서는 `extends`와 `super` 와일드카드를 구분한다.

#### 관련 퀴즈
- `java-c015`
- `java-d009`

## 예외 처리와 자원 관리
<!-- category: java-exception-resource -->

### Checked Exception과 Unchecked Exception
<!-- article: java-checked-unchecked-exception -->

#### 한눈에 보기
- Checked Exception은 컴파일러가 처리를 강제한다.
- Unchecked Exception은 `RuntimeException` 계열이며 처리가 강제되지 않는다.
- 예외를 삼키는 코드는 장애 원인을 숨긴다.

#### 핵심 정리

Checked Exception은 호출자가 반드시 `try-catch`로 처리하거나 `throws`로 전파해야 합니다. 파일, 네트워크, 데이터베이스처럼 외부 환경 문제로 복구 가능성이 있는 상황을 표현하는 데 자주 사용됩니다.

Unchecked Exception은 `RuntimeException`과 그 하위 타입입니다. `NullPointerException`, `IllegalArgumentException`, `IndexOutOfBoundsException`처럼 프로그래밍 오류나 잘못된 호출 계약을 표현할 때 많이 사용합니다.

Checked Exception은 API 계약을 명확히 하지만 코드가 장황해질 수 있습니다. Unchecked Exception은 코드 흐름을 단순하게 만들지만 호출자가 예외 상황을 놓칠 수 있습니다. 실무에서는 계층, 복구 가능성, 프레임워크의 트랜잭션 정책까지 함께 고려합니다.

#### 면접 답변

> Checked Exception은 컴파일러가 처리를 강제하는 예외이고, Unchecked Exception은 RuntimeException 계열로 처리가 강제되지 않습니다. 복구 가능한 외부 상황은 Checked로 표현할 수 있지만, 현대 프레임워크에서는 코드 간결성과 트랜잭션 처리를 이유로 Unchecked를 선호하는 경우도 많습니다.

#### 헷갈리는 포인트
- `catch (Exception e) {}`처럼 아무것도 하지 않는 코드는 문제를 숨긴다.
- 예외를 변환할 때는 원인 예외를 함께 담아 추적 가능하게 해야 한다.
- Spring 트랜잭션은 기본적으로 unchecked 예외에서 롤백되는 정책을 갖는다.

#### 관련 퀴즈
- `java-c008`
- `java-t003`
- `java-d008`
- `java-o002`

### try-with-resources와 finally
<!-- article: java-try-with-resources-finally -->

#### 한눈에 보기
- `try-with-resources`는 `AutoCloseable` 자원을 자동으로 닫는다.
- `finally`는 예외 여부와 관계없이 실행된다.
- 자원 해제 누락은 메모리뿐 아니라 파일, 소켓, 커넥션 누수로 이어질 수 있다.

#### 핵심 정리

파일, 스트림, JDBC 커넥션처럼 명시적으로 닫아야 하는 자원은 사용 후 반드시 해제해야 합니다. `try-with-resources`는 `AutoCloseable`을 구현한 객체를 try 괄호 안에 선언하면 블록 종료 시 자동으로 `close()`를 호출합니다.

기존의 `finally` 블록으로도 자원 해제를 할 수 있지만, 여러 자원을 다룰 때 코드가 복잡해지고 close 중 발생한 예외가 원래 예외를 가리는 문제가 생길 수 있습니다. `try-with-resources`는 이런 상황을 더 안전하고 간결하게 다룹니다.

`finally`는 return이 있더라도 실행됩니다. 그래서 정리 작업에는 유용하지만, finally에서 return을 하거나 예외를 덮어쓰는 코드는 피해야 합니다.

#### 면접 답변

> try-with-resources는 AutoCloseable 자원을 블록 종료 시 자동으로 닫아 자원 누수를 줄이는 문법입니다. finally도 항상 실행되지만, 자원 해제 코드가 복잡해지기 쉬워 Java 7 이후에는 try-with-resources를 우선 고려합니다.

#### 헷갈리는 포인트
- `finally`는 `try`에서 return하더라도 실행된다.
- close가 필요한 자원을 GC에만 맡기면 해제 시점을 예측하기 어렵다.
- 자원 해제와 객체 메모리 회수는 같은 개념이 아니다.

#### 관련 퀴즈
- `java-c020`
- `java-s004`

### null 처리와 Optional
<!-- article: java-null-optional -->

#### 한눈에 보기
- `Optional`은 값이 없을 수 있음을 타입으로 드러낸다.
- null 가능성을 무시하면 `NullPointerException`이 발생한다.
- `Optional`은 필드보다 반환 타입에서 더 자주 사용한다.

#### 핵심 정리

Java에서 null은 흔한 장애 원인입니다. 객체가 없을 수 있는 상황을 명확히 표현하지 않으면 호출자는 방어 코드를 빠뜨리기 쉽고, 실행 중 `NullPointerException`이 발생합니다.

`Optional<T>`는 "값이 있을 수도, 없을 수도 있음"을 반환 타입으로 표현합니다. `map`, `flatMap`, `orElse`, `orElseThrow` 같은 메서드를 사용하면 null 체크를 명시적인 흐름으로 바꿀 수 있습니다.

다만 `Optional`이 모든 null 문제의 답은 아닙니다. 메서드 인자나 엔티티 필드에 남용하면 오히려 모델이 복잡해질 수 있습니다. API 경계에서 부재 가능성을 명확히 드러내는 반환 타입으로 쓰는 것이 일반적입니다.

#### 면접 답변

> Optional은 값의 부재 가능성을 타입으로 표현해 null 처리를 명시적으로 만들기 위한 도구입니다. NullPointerException을 줄이는 데 도움이 되지만, 필드나 파라미터에 무분별하게 쓰기보다는 반환 타입 중심으로 사용하는 편이 좋습니다.

#### 헷갈리는 포인트
- `Optional.get()`을 무조건 호출하면 null 체크를 안 한 것과 비슷해진다.
- `orElse()`는 인자가 미리 평가될 수 있고, 지연 평가가 필요하면 `orElseGet()`을 쓴다.
- null 허용 여부는 API 문서나 타입으로 분명히 드러내야 한다.

## JVM과 메모리
<!-- category: java-jvm-memory -->

### Runtime Data Area와 OS 프로세스 메모리
<!-- article: java-runtime-data-area-os-process -->

#### 한눈에 보기
- OS 관점에서 JVM은 하나의 프로세스다.
- JVM은 프로세스 메모리 안에서 Runtime Data Area를 논리적으로 관리한다.
- Heap과 Method Area는 공유 영역, Stack과 PC Register는 스레드별 영역이다.

#### 핵심 정리

Java 애플리케이션을 실행하면 운영체제에는 JVM 프로세스가 올라갑니다. 그 프로세스 안에서 JVM은 Java 실행을 위한 메모리 구역을 논리적으로 나눕니다.

대표적인 구역은 Method Area, Heap, Java Stack, PC Register, Native Method Stack입니다. Method Area에는 클래스 메타데이터, static 변수, 상수 풀, 메서드 바이트코드 등이 저장됩니다. Heap에는 `new`로 만든 객체와 배열이 저장됩니다.

Java Stack은 각 스레드마다 존재하며 메서드 호출 프레임, 지역 변수, 매개변수, 피연산자 스택 등을 담습니다. PC Register는 각 Java 스레드가 현재 실행 중인 바이트코드 위치를 가리키는 논리적 카운터입니다.

#### 면접 답변

> OS 입장에서는 JVM도 하나의 프로세스이고, JVM은 그 프로세스 주소 공간 안에서 Method Area, Heap, Stack, PC Register 같은 런타임 데이터 영역을 논리적으로 나누어 관리합니다.

#### 헷갈리는 포인트
- JVM의 Method Area가 OS의 코드 세그먼트와 완전히 같은 것은 아니다.
- Stack에는 보통 지역 변수와 참조 값이 있고, 실제 객체는 Heap에 있다.
- CPU의 PC는 네이티브 명령 주소, JVM PC Register는 바이트코드 위치를 의미한다.

### GC와 Reachability
<!-- article: java-gc-reachability-stw -->

#### 한눈에 보기
- GC는 힙에서 더 이상 도달할 수 없는 객체를 수거한다.
- 판단 기준은 Reference Counting이 아니라 Reachability다.
- GC 과정 일부에서는 Stop-The-World가 발생할 수 있다.

#### 핵심 정리

Java의 GC는 힙에 있는 객체 중 더 이상 사용할 수 없는 객체를 찾아 메모리를 회수합니다. 기준은 GC Root에서 참조 체인을 따라 도달 가능한지입니다. 스택의 지역 변수, static 변수, JNI 참조 등이 대표적인 GC Root가 될 수 있습니다.

두 객체가 서로를 참조하는 순환 참조가 있어도 GC Root에서 도달할 수 없다면 수거 대상이 됩니다. 이것이 단순 참조 카운팅 방식과 다른 중요한 지점입니다.

GC가 객체 그래프를 분석하거나 객체를 이동하는 동안 참조 관계가 계속 바뀌면 정확한 수집이 어렵습니다. 그래서 일부 단계에서 애플리케이션 스레드를 잠시 멈추는 Stop-The-World가 발생할 수 있습니다. 최신 GC는 이 시간을 줄이기 위해 여러 전략을 사용합니다.

#### 면접 답변

> Java GC는 GC Root에서 도달할 수 없는 힙 객체를 수거합니다. 순환 참조가 있어도 루트에서 닿지 않으면 제거될 수 있고, 객체 참조 관계를 안정적으로 분석하기 위해 일부 단계에서 Stop-The-World가 발생할 수 있습니다.

#### 헷갈리는 포인트
- GC가 있다고 모든 메모리 문제가 사라지는 것은 아니다.
- JNI가 잡고 있는 Java 객체 참조도 GC Root로 취급될 수 있다.
- Minor GC는 Young Generation 중심, Major 또는 Full GC는 Old Generation이나 전체 힙에 더 큰 영향을 준다.

#### 관련 퀴즈

- `java-c026`

- `java-c028`
- `java-c017`
- `java-o003`
- `java-t007`

### 메모리 누수와 OutOfMemoryError
<!-- article: java-memory-leak-oom -->

#### 한눈에 보기
- GC는 도달 불가능한 객체만 회수한다.
- 사용이 끝난 객체도 계속 참조되면 GC 대상이 아니다.
- 무한히 커지는 컬렉션은 OutOfMemoryError로 이어질 수 있다.

#### 핵심 정리

Java에서도 메모리 누수는 발생합니다. 객체를 더 이상 사용하지 않지만 어딘가에서 계속 참조하고 있으면 GC는 그 객체를 살아 있다고 판단합니다. 대표적인 예는 static 컬렉션에 객체를 계속 쌓는 코드, 해제하지 않은 이벤트 리스너, 람다가 큰 객체를 캡처한 채 오래 살아 있는 경우입니다.

자료구조 구현에서도 누수가 생길 수 있습니다. 예를 들어 직접 만든 스택에서 `pop()`으로 크기만 줄이고 배열 슬롯의 참조를 null로 지우지 않으면, 논리적으로 제거된 객체가 배열을 통해 계속 도달 가능할 수 있습니다.

`OutOfMemoryError`는 힙이나 메타스페이스 등 필요한 메모리를 더 이상 확보할 수 없을 때 발생합니다. 단순히 메모리를 많이 쓰는 것뿐 아니라, 오래 살아남는 참조가 계속 쌓이는 구조인지 확인해야 합니다.

#### 면접 답변

> Java의 메모리 누수는 사용이 끝난 객체가 여전히 참조되어 GC가 회수하지 못할 때 발생합니다. static 컬렉션, 해제하지 않은 리스너, 닫지 않은 자원, 오래 살아 있는 람다 캡처 등이 대표적이며, 누적되면 OutOfMemoryError로 이어질 수 있습니다.

#### 헷갈리는 포인트
- 순환 참조 자체는 GC가 처리할 수 있다.
- 문제는 GC Root에서 계속 도달 가능한 참조다.
- `StackOverflowError`는 보통 깊은 재귀나 순환 생성처럼 스택이 고갈될 때 발생한다.

#### 관련 퀴즈

## 동시성
<!-- category: java-concurrency -->

### Thread-Safe와 공유 상태
<!-- article: java-thread-safe-shared-state -->

#### 한눈에 보기
- Thread-Safe는 여러 스레드가 동시에 접근해도 올바르게 동작한다는 뜻이다.
- `count++`는 읽기, 증가, 쓰기가 합쳐진 복합 연산이다.
- 공유 mutable 상태는 동기화 전략이 필요하다.

#### 핵심 정리

멀티스레드 환경에서 여러 스레드가 같은 변수를 동시에 수정하면 경쟁 상태가 발생할 수 있습니다. `count++`는 한 줄처럼 보이지만 실제로는 값을 읽고, 증가시키고, 다시 쓰는 여러 단계로 이루어집니다.

두 스레드가 동시에 값을 읽으면 둘 다 같은 이전 값을 기준으로 증가시킬 수 있습니다. 이 경우 increment를 여러 번 호출했는데도 기대한 값보다 작은 결과가 나옵니다.

Thread-Safe하게 만들려면 공유 상태를 줄이거나, 불변 객체를 사용하거나, `synchronized`, `Lock`, 원자 클래스, 동시성 컬렉션 같은 도구로 접근을 통제해야 합니다. 가장 좋은 출발점은 공유 mutable 상태 자체를 최소화하는 것입니다.

#### 면접 답변

> Thread-Safe하다는 것은 여러 스레드가 동시에 접근해도 데이터가 깨지지 않고 의도한 결과를 보장한다는 뜻입니다. 공유 가변 상태가 있다면 동기화, 원자 연산, 불변 객체, 동시성 컬렉션 등을 사용해 경쟁 상태를 막아야 합니다.

#### 헷갈리는 포인트
- `volatile`은 가시성을 보장하지만 `count++`의 원자성을 보장하지 않는다.
- 지역 변수는 보통 스레드마다 독립적인 스택에 있어 공유 필드보다 안전하다.
- 싱글톤 객체 자체가 하나라는 사실이 내부 상태의 스레드 안전성을 보장하지는 않는다.

#### 관련 퀴즈
- `java-c013`
- `java-s002`
- `java-s010`

### synchronized, volatile, ReentrantLock
<!-- article: java-synchronized-volatile-reentrantlock -->

#### 한눈에 보기
- `synchronized`는 모니터 락으로 상호 배제와 가시성을 제공한다.
- `volatile`은 가시성 중심이고 복합 연산의 원자성은 보장하지 않는다.
- `ReentrantLock`은 타임아웃, 공정성, 조건 변수 같은 고급 제어가 가능하다.

#### 핵심 정리

`synchronized` 메서드는 인스턴스 메서드라면 `this`, static 메서드라면 클래스 객체를 락으로 사용합니다. `synchronized` 블록은 특정 객체를 락으로 지정할 수 있어서 락 범위를 더 작게 잡을 수 있습니다.

`volatile`은 한 스레드의 쓰기가 다른 스레드에 보이도록 가시성을 보장합니다. 하지만 `i++` 같은 복합 연산은 여러 단계이므로 `volatile`만으로는 안전하지 않습니다.

`ReentrantLock`은 명시적으로 `lock()`과 `unlock()`을 호출합니다. 코드가 조금 장황해지는 대신, 락 획득 타임아웃, 인터럽트 가능한 락, 공정성 옵션, 여러 `Condition` 같은 기능이 필요할 때 유용합니다. 단순한 임계 구역은 `synchronized`가 더 간결합니다.

#### 면접 답변

> synchronized는 간단한 상호 배제와 가시성 보장에 적합하고, ReentrantLock은 타임아웃이나 공정성 같은 세밀한 제어가 필요할 때 사용합니다. volatile은 가시성 보장용이며 복합 연산의 원자성까지 해결하지는 못합니다.

#### 헷갈리는 포인트
- `synchronized` 블록은 락 대상을 직접 지정할 수 있다.
- Double-Checked Locking은 `volatile` 없이 안전하지 않을 수 있다.
- 데드락은 여러 스레드가 서로가 가진 락을 기다리며 진행하지 못하는 상태다.

#### 관련 퀴즈
- `java-c016`
- `java-t004`
- `java-s007`
- `java-s008`
- `java-d012`

### Future, CompletableFuture, ExecutorService
<!-- article: java-future-completablefuture-executorservice -->

#### 한눈에 보기
- `ExecutorService`는 스레드 생성과 작업 실행을 분리한다.
- `Future`는 비동기 작업의 결과를 나중에 받는 객체다.
- `CompletableFuture`는 비동기 작업을 연결하고 조합하기 쉽다.

#### 핵심 정리

스레드를 직접 계속 생성하면 비용이 크고 관리가 어렵습니다. `ExecutorService`는 스레드풀을 통해 작업 제출과 실행을 분리하고, 재사용 가능한 스레드로 작업을 처리합니다.

`Future`는 비동기 작업의 결과를 표현합니다. `get()`을 호출하면 결과가 준비될 때까지 기다리며, 작업 완료 여부나 취소 여부를 확인할 수 있습니다. 다만 여러 비동기 작업을 자연스럽게 연결하는 표현력은 제한적입니다.

`CompletableFuture`는 결과 변환, 후속 작업 연결, 여러 작업 조합을 더 선언적으로 다룰 수 있습니다. 스레드풀 크기는 작업 성격에 따라 달라집니다. CPU 중심 작업은 코어 수를 기준으로, I/O 대기 시간이 긴 작업은 더 많은 동시성을 고려할 수 있습니다.

#### 면접 답변

> ExecutorService는 스레드풀로 작업 실행을 관리하고, Future는 비동기 결과를 나중에 조회하는 객체입니다. CompletableFuture는 비동기 결과를 변환하고 조합하는 API를 제공해 복잡한 비동기 흐름을 더 표현력 있게 만들 수 있습니다.

#### 헷갈리는 포인트
- 스레드풀은 크다고 항상 좋은 것이 아니다.
- CPU 작업과 I/O 작업은 적정 스레드 수 기준이 다르다.
- 작업이 끝난 ExecutorService는 적절히 종료해야 한다.

#### 관련 퀴즈
- `java-t009`

## Java 8+ 핵심 기능

### 람다와 함수형 인터페이스
<!-- article: java-lambda-functional-interface -->

#### 한눈에 보기
- 람다는 익명 함수를 간결하게 표현하는 문법이다.
- 함수형 인터페이스는 추상 메서드가 하나인 인터페이스다.
- 람다는 외부 지역 변수를 사용할 때 effectively final 규칙을 따른다.

#### 핵심 정리

람다는 기존의 익명 클래스 문법을 더 짧고 읽기 쉽게 표현합니다. 정렬 기준, 필터 조건, 콜백처럼 "동작"을 값처럼 전달해야 하는 상황에서 코드가 간결해집니다.

람다의 타입은 함수형 인터페이스입니다. `Runnable`, `Callable`, `Comparator`, `Function`, `Predicate`, `Consumer`, `Supplier` 등이 대표적입니다. `@FunctionalInterface`를 붙이면 추상 메서드가 하나인지 컴파일러가 확인해줍니다.

람다에서 외부 지역 변수를 사용할 때는 그 변수가 final 또는 effectively final이어야 합니다. 지역 변수는 스택에 있고 람다는 더 오래 살아남을 수 있기 때문에, 값 캡처 방식으로 안정성을 확보합니다.

#### 면접 답변

> 람다는 함수형 인터페이스의 구현을 간결하게 표현하는 문법입니다. 추상 메서드가 하나인 인터페이스를 대상으로 하며, 외부 지역 변수는 final 또는 effectively final이어야 안전하게 캡처할 수 있습니다.

#### 헷갈리는 포인트
- default 메서드나 static 메서드는 함수형 인터페이스의 추상 메서드 개수에 포함되지 않는다.
- 람다가 캡처한 객체가 큰 객체이고 람다가 오래 살아 있으면 메모리 유지 문제가 생길 수 있다.
- 모든 익명 클래스를 람다로 바꾸는 것이 항상 더 읽기 좋은 것은 아니다.

#### 관련 퀴즈
- `java-t008`

### Stream과 병렬 스트림
<!-- article: java-stream-parallel-stream -->

#### 한눈에 보기
- Stream은 컬렉션 데이터를 선언적으로 처리하는 API다.
- 중간 연산은 lazy하게 동작하고 Stream을 반환한다.
- 최종 연산이 호출될 때 파이프라인이 실행된다.

#### 핵심 정리

Stream은 데이터를 "어떻게 반복할지"보다 "무엇을 얻을지" 중심으로 표현하게 해줍니다. `filter`, `map`, `sorted` 같은 중간 연산을 연결하고, `collect`, `forEach`, `count` 같은 최종 연산으로 결과를 만듭니다.

중간 연산은 최종 연산 전까지 실제로 실행되지 않습니다. 이 lazy 특성 덕분에 필요한 만큼만 계산하거나, 파이프라인 전체를 최적화할 여지가 생깁니다.

병렬 스트림은 CPU를 많이 쓰는 독립 작업과 큰 데이터셋에서 효과가 있을 수 있습니다. 하지만 작은 데이터, I/O 작업, 공유 상태 변경이 있는 작업에서는 오히려 느리거나 위험할 수 있습니다.

#### 면접 답변

> Stream은 컬렉션 처리를 선언적으로 표현하는 API이고, 중간 연산은 lazy하게 쌓였다가 최종 연산이 호출될 때 실행됩니다. 병렬 스트림은 큰 독립 CPU 작업에는 도움이 될 수 있지만, 작은 데이터나 공유 상태가 있는 작업에는 신중해야 합니다.

#### 헷갈리는 포인트
- Stream은 한 번 소비되면 재사용할 수 없다.
- `parallelStream()`이 항상 빠른 것은 아니다.
- `forEach` 안에서 공유 리스트에 추가하는 방식은 병렬 환경에서 위험할 수 있다.

#### 관련 퀴즈

### default method와 직렬화
<!-- article: java-default-method-serialization -->

#### 한눈에 보기
- default method는 인터페이스에 기본 구현을 추가할 수 있게 한다.
- 인터페이스 진화에는 도움이 되지만 설계 책임이 흐려질 수 있다.
- Java 직렬화는 편리하지만 보안, 호환성, 성능 이슈가 있다.

#### 핵심 정리

default method는 기존 인터페이스에 새 메서드를 추가하면서도 기존 구현체를 모두 깨뜨리지 않기 위해 도입되었습니다. 공통 기본 동작을 제공할 수 있지만, 인터페이스가 지나치게 많은 구현 세부사항을 갖게 되면 역할이 모호해집니다.

직렬화는 객체를 바이트 형태로 저장하거나 전송하는 기능입니다. 단순해 보이지만 클래스 구조 변경 시 호환성 문제가 생길 수 있고, 역직렬화 과정에서 보안 취약점이 발생할 수 있습니다.

현대 애플리케이션에서는 외부 통신이나 저장 형식으로 JSON, Protocol Buffers 같은 명시적 포맷을 선택하는 경우가 많습니다. Java 기본 직렬화는 꼭 필요한 상황인지 먼저 검토해야 합니다.

#### 면접 답변

> default method는 기존 인터페이스를 깨지 않고 기본 구현을 추가하는 장점이 있지만, 인터페이스 책임이 커질 수 있습니다. Java 직렬화는 객체 저장과 전송을 쉽게 하지만 보안, 버전 호환성, 성능 이슈 때문에 신중하게 사용해야 합니다.

#### 헷갈리는 포인트
- default method는 다중 구현 충돌이 생길 수 있어 명시적 재정의가 필요할 때가 있다.
- 직렬화 가능한 클래스의 구조 변경은 역직렬화 호환성에 영향을 준다.
- 신뢰할 수 없는 데이터의 역직렬화는 보안상 위험하다.

#### 관련 퀴즈

## 면접 질문 정리

### Java 면접 답변을 구조화하는 방법
<!-- article: java-interview-answer-structure -->

#### 한눈에 보기
- 정의만 말하지 말고 동작 방식과 선택 기준을 함께 말한다.
- 메모리, 동시성, 성능 질문은 "왜 문제가 되는지"를 설명한다.
- 코드 문제는 실행 결과보다 원인을 먼저 잡는다.

#### 핵심 정리

Java 면접에서는 단순 암기보다 개념의 경계가 중요합니다. 예를 들어 `==`와 `equals()`는 "주소와 값"으로 끝내지 말고, String Pool이나 해시 컬렉션 문제까지 연결하면 답변이 단단해집니다.

JVM과 GC 질문은 운영체제 메모리와 JVM 논리 영역을 구분하고, Stack과 Heap의 역할을 분리해서 설명해야 합니다. GC가 자동으로 메모리를 관리한다는 말 뒤에는 Reachability, GC Root, STW, 메모리 누수 가능성을 함께 붙이면 좋습니다.

동시성 질문은 "여러 스레드가 동시에 접근한다"는 상황을 먼저 그린 뒤, 가시성, 원자성, 상호 배제 중 어떤 문제가 있는지 나누어 답변하면 흔들리지 않습니다.

#### 면접 답변

> Java 답변은 정의, 동작 방식, 장단점, 실무 주의점 순서로 정리하면 좋습니다. 특히 JVM, GC, 동시성, 컬렉션은 내부 동작과 선택 기준을 함께 설명해야 면접에서 깊이 있는 답변이 됩니다.

#### 헷갈리는 포인트
- 결과를 외우기보다 JVM과 언어 규칙으로 결과가 나온 이유를 설명한다.
- 트레이드오프 문제는 "항상 A"보다 "상황에 따라 A/B"로 답한다.
- 디버깅 문제는 예외 종류, 발생 위치, 수정 방향을 함께 말한다.
