# 프로세스와 스레드의 차이

## 프로세스(Process)

프로세스는 **실행 중인 프로그램**으로, 디스크로부터 메모리에 적재되어 CPU의 할당을 받을 수 있는 것을 말한다. 운영체제로부터 주소 공간, 파일, 메모리 등을 할당받으며 이것들을 총칭하여 프로세스라고 한다.

![image.png](%EC%9A%B4%EC%98%81%EC%B2%B4%EC%A0%9C/image.png)

[[운영체제] -  프로세스 메모리 구조(코드, 데이터, 힙, 스택)](https://jenlvestory.tistory.com/19)

프로세스는 다음과 같은 메모리 영역으로 구성된다:
- **코드(Code) 영역**: 실행할 프로그램의 코드가 저장
- **데이터(Data) 영역**: 전역 변수, 정적 변수가 저장
- **힙(Heap) 영역**: 동적 할당되는 메모리 공간
- **스택(Stack) 영역**: 함수 호출 시 지역 변수, 매개 변수, 복귀 주소 저장

### 프로세스 제어 블록(Process Control Block, PCB)

PCB는 특정 **프로세스에 대한 중요한 정보를 저장**하고 있는 운영체제의 자료 구조이다. 운영체제는 프로세스를 관리하기 위해 **프로세스의 생성과 동시에 고유한 PCB를 생성**한다.

프로세스는 CPU를 할당받아 작업을 처리하다가도 프로세스 전환이 발생하면 진행하던 작업을 저장하고 CPU를 반환해야 하는데, 이때 작업의 진행 상황을 모두 PCB에 저장하게 된다. 그리고 다시 CPU를 할당받게 되면 PCB에 저장되어 있던 내용을 불러와 이전에 종료됐던 시점부터 다시 작업을 수행한다.

**PCB에 저장되는 정보**

- 프로세스 식별자(Process ID, PID): 프로세스 식별 번호
- 프로세스 상태: new, ready, running, waiting, terminated 등의 상태를 저장
- 프로그램 카운터(PC): 프로세스가 다음에 실행할 명령어의 주소
- CPU 레지스터: 누산기, 인덱스 레지스터, 범용 레지스터 등
- CPU 스케줄링 정보: 프로세스의 우선순위, 스케줄 큐에 대한 포인터 등
- 메모리 관리 정보: 페이지 테이블 또는 세그먼트 테이블 등과 같은 정보를 포함
- 입출력 상태 정보: 프로세스에 할당된 입출력 장치들과 열린 파일 목록
- 어카운팅 정보: 사용된 CPU 시간, 시간제한, 계정 번호 등

**연관 개념**: [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD), [프로세스 상태](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4%EC%9D%98-%EC%83%81%ED%83%9C)

### 프로세스의 상태

프로세스는 생성부터 종료까지 여러 상태를 거치며 변화한다:

- **New(생성)**: 프로세스가 생성된 상태
- **Ready(준비)**: 프로세스가 CPU 할당을 기다리는 상태
- **Running(실행)**: 프로세스가 CPU를 할당받아 실행 중인 상태
- **Waiting(대기)**: 프로세스가 I/O 작업 등의 이벤트를 기다리는 상태
- **Terminated(종료)**: 프로세스의 실행이 완료된 상태
- **Suspended(중단)**: 중기 스케줄러에 의해 메모리에서 디스크로 스왑 아웃된 상태

**연관 개념**: [스케줄러](about:blank#%EC%8A%A4%EC%BC%80%EC%A4%84%EB%9F%AC), [CPU 스케줄러](about:blank#cpu-%EC%8A%A4%EC%BC%80%EC%A4%84%EB%9F%AC)

## 컨텍스트 스위칭(Context Switching)

컨텍스트 스위칭은 **PCB를 기반으로 프로세스의 상태를 저장하고 다시 복원시키는 과정**이다. 이는 프로세스가 종료되거나 인터럽트에 의해 발생한다.

**컨텍스트 스위칭의 과정**
1. 현재 실행 중인 프로세스의 상태(레지스터, PC 등)를 PCB에 저장
2. 다음 실행할 프로세스의 PCB를 메모리에서 로드
3. PCB에 저장된 상태를 복원하여 프로세스 실행 재개

**컨텍스트 스위칭의 비용**

1. **유휴 시간 발생**: 컨텍스트 스위칭 동안 CPU는 실제 작업을 수행하지 못함
2. **캐시 미스**: 프로세스가 바뀌면 캐시 메모리를 클리어해야 하므로 캐시 미스 발생
3. **오버헤드**: 상태 저장 및 로드 과정에서 발생하는 시간적 비용

**스레드에서의 컨텍스트 스위칭**

스레드는 스택 영역을 제외한 모든 메모리(코드, 데이터, 힙)를 공유하기 때문에 프로세스 컨텍스트 스위칭보다 비용이 적고 시간도 더 적게 걸린다.

**연관 개념**: [PCB](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4-%EC%A0%9C%EC%96%B4-%EB%B8%94%EB%A1%9D), [멀티스레드](about:blank#%EB%A9%80%ED%8B%B0%EC%8A%A4%EB%A0%88%EB%93%9C), [캐시](about:blank#%EC%BA%90%EC%8B%9C%EC%9D%98-%EC%A7%80%EC%97%AD%EC%84%B1)

## 스레드(Thread)

스레드는 **프로세스의 실행 단위**라고 할 수 있다. 한 프로세스 내에서 동작하는 여러 실행 흐름으로, 프로세스 내의 주소 공간이나 자원을 공유할 수 있다.

스레드는 다음을 포함한다:
- 스레드 ID
- 프로그램 카운터(PC)
- 레지스터 집합
- 스택

같은 프로세스에 속한 다른 스레드와 **코드, 데이터 섹션, 힙 영역, 그리고 열린 파일이나 신호와 같은 운영체제 자원들을 공유**한다.

하나의 프로세스를 다수의 실행 단위로 구분하여 자원을 공유하고 자원의 생성과 관리의 중복성을 최소화하여 수행 능력을 향상하는 것을 멀티스레딩이라고 한다. 이 경우 각각의 스레드는 독립적인 작업을 수행해야 하기 때문에 각자의 스택과 PC 레지스터 값을 갖고 있다.

**연관 개념**: [멀티스레드](about:blank#%EB%A9%80%ED%8B%B0%EC%8A%A4%EB%A0%88%EB%93%9C), [프로세스](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4)

### 스택을 스레드마다 독립적으로 할당하는 이유

스택은 함수 호출 시 전달되는 인자, 복귀 주소 및 함수 내에서 선언하는 지역 변수 등을 저장하기 위해 사용되는 메모리 공간이다.

스택 메모리 공간이 독립적이라는 것은 **독립적인 함수 호출이 가능**하다는 것이고, 이는 **독립적인 실행 흐름**이 추가되는 것이다. 따라서 스레드의 정의에 따라 독립적인 실행 흐름을 추가하기 위한 최소 조건으로 독립된 스택을 할당한다.

### PC Register를 스레드마다 독립적으로 할당하는 이유

PC 값은 스레드가 명령어의 어디까지 수행하였는지를 나타낸다. 스레드는 CPU를 할당받았다가 스케줄러에 의해 다시 선점당한다.

그렇기 때문에 명령어가 연속적으로 수행되지 못하고, **어느 부분까지 수행했는지 기억할 필요**가 있다. 따라서 PC 레지스터를 독립적으로 할당한다.

**연관 개념**: [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD), [CPU 스케줄러](about:blank#cpu-%EC%8A%A4%EC%BC%80%EC%A4%84%EB%9F%AC)

# 멀티스레드

## 멀티스레딩의 장점

프로세스를 이용하여 동시에 처리하던 일을 스레드로 구현할 경우 다음과 같은 장점이 있다:

1. **메모리 공간과 시스템 자원 소모 감소**: 스레드는 코드, 데이터, 힙 영역을 공유하므로 메모리 효율적
2. **빠른 통신**: 스레드 간 통신 시 별도의 IPC 없이 전역 변수나 힙 영역을 통해 데이터 공유 가능
3. **빠른 컨텍스트 스위칭**: 스레드 간 전환 시 캐시 메모리를 비울 필요가 없어 프로세스 전환보다 빠름
4. **시스템 처리량 향상**: 여러 스레드로 작업을 분산하여 처리 가능

이러한 장점 때문에 여러 프로세스로 할 수 있는 작업들을 하나의 프로세스에서 스레드로 나눠 수행하는 것이다.

**연관 개념**: [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD), [IPC](about:blank#ipc), [캐시](about:blank#%EC%BA%90%EC%8B%9C%EC%9D%98-%EC%A7%80%EC%97%AD%EC%84%B1)

## 멀티스레딩의 문제점

멀티프로세스 기반으로 프로그래밍할 때는 프로세스 간 공유하는 자원이 없기 때문에 동일한 자원에 동시에 접근하는 일이 없었지만, 멀티스레딩을 기반으로 프로그래밍할 때는 이 부분을 신경 써야 한다. → 하나가 문제 생기면, 그게 전체의 문제가 될 수 있음.

서로 다른 스레드가 데이터와 힙 영역을 공유하기 때문에 **어떤 스레드가 다른 스레드에서 사용 중인 변수나 자료 구조에 접근하여 엉뚱한 값을 읽어오거나 수정**할 수 있다. 이를 **경쟁 상태(Race Condition)**라고 한다.

그렇기 때문에 멀티스레딩 환경에서는 **동기화 작업**이 필요하다. 동기화를 통해 작업 처리 순서를 컨트롤하고 공유 자원에 대한 접근을 컨트롤하는 것이다.

하지만 이로 인해 **병목 현상**이 발생하여 성능이 저하될 가능성이 높다. 그러므로 과도한 락(Lock)으로 인한 병목 현상을 줄여야 한다.

**연관 개념**: [프로세스 동기화](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4-%EB%8F%99%EA%B8%B0%ED%99%94), [임계 영역](about:blank#critical-section), [교착 상태](about:blank#%EA%B5%90%EC%B0%A9-%EC%83%81%ED%83%9C)

## 멀티스레드 vs 멀티프로세스

**멀티스레드의 장점**
- 적은 메모리 공간 차지
- 빠른 컨텍스트 스위칭
- 스레드 간 통신이 간단

**멀티스레드의 단점**
- 하나의 스레드에 문제가 생기면 전체 프로세스에 영향
- 동기화 문제 발생 가능
- 디버깅이 어려움

**멀티프로세스의 장점**
- 하나의 프로세스가 죽어도 다른 프로세스에 영향 없음
- 안정성과 신뢰성이 높음
- 격리성 보장

**멀티프로세스의 단점**
- 많은 메모리 공간과 CPU 시간 소요
- 프로세스 간 통신(IPC)이 복잡
- 컨텍스트 스위칭 오버헤드가 큼

이 두 가지는 동시에 여러 작업을 수행한다는 점에서 같지만 **적용해야 하는 시스템에 따라 적합/부적합이 구분**된다. 따라서 대상 시스템의 특징에 따라 적합한 동작 방식을 선택하고 적용해야 한다.

**선택 기준**
- 안정성이 중요하고 작업 간 독립성이 필요한 경우: 멀티프로세스
- 성능과 자원 효율이 중요하고 데이터 공유가 빈번한 경우: 멀티스레드

**연관 개념**: [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD), [IPC](about:blank#ipc), [프로세스 동기화](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4-%EB%8F%99%EA%B8%B0%ED%99%94)

# 스케줄러

프로세스를 스케줄링하기 위한 Queue에는 세 가지 종류가 존재한다:

- **Job Queue**: 현재 시스템 내에 있는 모든 프로세스의 집합
- **Ready Queue**: 현재 메모리 내에 있으면서 CPU를 할당받아 실행되기를 기다리는 프로세스의 집합
- **Device Queue**: Device I/O 작업을 대기하고 있는 프로세스의 집합

각각의 Queue에 프로세스들을 넣고 빼주는 스케줄러에도 크게 **세 가지 종류**가 존재한다.

**연관 개념**: [프로세스의 상태](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4%EC%9D%98-%EC%83%81%ED%83%9C), [CPU 스케줄러](about:blank#cpu-%EC%8A%A4%EC%BC%80%EC%A4%84%EB%9F%AC)

운영체제에서 말하는 **장기 / 중기 / 단기 스케줄러**는

“어떤 프로세스를 언제, 어디에 둘 것인지”를 단계별로 나눠서 관리하는 개념이야.

큰 흐름부터 보면:

> 디스크(작업 대기) → 메모리(ready) → CPU(실행 중)
> 
> 
> 이 사이사이를 각각 다른 스케줄러가 담당한다고 보면 됨
>

## 1. 장기 스케줄러 (Long-term Scheduler, Job Scheduler)

**역할: “시스템에 어떤 작업을 ‘들여보낼지’ 결정”**

- 위치: **디스크에 쌓여 있는 작업들 → 메모리의 준비(Ready) 큐**
- 하는 일:
    - 새로 들어온 작업들 중에서 **어떤 프로세스를 메모리에 올릴지 선택**
    - 즉, “이 작업을 지금 시스템에 참여시킬까, 나중에 할까?”를 결정
- 목적:
    - *다중 프로그래밍 정도(degree of multiprogramming)**를 조절
        - 동시에 메모리에 너무 많이 올리면 오히려 문맥 교환만 늘어나서 느려짐
        - 너무 적게 올리면 CPU가 노는 시간이 많아짐
    - **CPU-bound / I/O-bound 비율 조절**
        - CPU만 빡세게 쓰는 애들만 잔뜩 있으면 I/O 장치가 놀고
        - I/O만 하는 애들만 많으면 CPU가 놈
            
            → 둘을 적절히 섞어서 전체 자원 활용률을 높임
            
- 호출 빈도: **매우 드묾**
    - 새로운 작업이 시스템에 들어올 때만 가끔 호출됨

---

## 2. 중기 스케줄러 (Medium-term Scheduler, Swapper)

**역할: “메모리가 벅찰 때, 누구를 잠시 ‘시스템 밖으로 내보낼지’ 결정”**

- 위치: **메모리에 있는 프로세스 ↔ 디스크(스왑 영역)**
    - `Ready / Blocked` 상태의 프로세스 중 일부를 **통째로 디스크로 내보내거나(swap out)**
    - 다시 **메모리로 불러들임(swap in)**
- 하는 일:
    - 메모리가 부족하거나, 현재 프로세스가 너무 많을 때
        
        → 우선순위 낮거나, 오래 Blocked 상태인 프로세스를 **중지(suspend) + swap out**
        
    - 나중에 여유가 생기면 다시 **swap in** 해서 Ready 상태로 복귀시킴
- 목적:
    - 장기 스케줄러가 정한 다중 프로그래밍 정도를 **동적으로 조절**
    - 메모리 부족 시 **시스템 전체가 느려지지 않게** 프로세스 수를 줄이기
- 특징:
    - *스와핑(swapping)**이 있는 시스템에서 주로 강조되는 개념
    - 실제 상용 OS에서는 장기 스케줄러와 기능이 섞여 있거나, 명시적으로 구분 안 하기도 함

---

## 3. 단기 스케줄러 (Short-term Scheduler, CPU Scheduler)

**역할: “당장 다음에 CPU를 누가 쓸지” 결정**

- 위치: **메모리 안의 Ready Queue → CPU**
- 하는 일:
    - Ready 상태에 있는 프로세스들 중에서 **다음에 실행할 프로세스를 선택**
    - 선택한 프로세스로 **문맥 교환(context switch)** 수행
- 목적:
    - CPU를 **항상 바쁘게** 유지하면서도,
    - 스케줄링 알고리즘에 따라
        - 평균 응답 시간 최소화
        - 대기 시간 최소화
        - 우선순위 보장
        - 공정성(Fairness) 보장 등 목표 달성
- 호출 빈도: **아주 자주**
    - 프로세스가 종료될 때
    - 프로세스가 I/O 요청으로 Blocked 될 때
    - 타임퀀텀(time slice)이 끝났을 때(라운드 로빈 등에서)
    - 우선순위 높은 프로세스가 Ready가 되었을 때(선점 스케줄링) 등

---

| --- | --- | --- | --- | --- | --- |
| 장기 스케줄러 | Job Scheduler | **새 작업** | 디스크(작업 큐) → 메모리(Ready 큐) | 시스템에 입장시킬 작업 선택, 다중 프로그래밍 정도 조절 | 낮음 |

## 현대 OS에서는?

- 실제 리눅스/윈도우에서 **“장기/중기/단기”를 코드 레벨로 딱 나눠놓진 않음.**
    
    개념적으로 설명할 때 나누는 것.
    
- 현실에서는:
    - **장기 + 중기 스케줄링** 역할은
        
        → 작업 시작, 프로세스 생성/종료, 메모리 관리자, OOM Killer, cgroup/컨테이너 리소스 제한 등이 같이 수행
        
    - **단기 스케줄러**는 우리가 흔히 말하는 **CPU 스케줄러(CFS, O(1) scheduler 등)** 가 담당

# CPU 스케줄러

스케줄링 대상은 Ready Queue에 있는 프로세스들이다. CPU 스케줄링 알고리즘은 크게 **선점형(Preemptive)**과 **비선점형(Non-Preemptive)**으로 나뉜다.

**비선점형(Non-Preemptive)**: 프로세스가 CPU를 할당받으면 작업이 완료될 때까지 CPU를 빼앗기지 않음
**선점형(Preemptive)**: 우선순위가 높은 프로세스가 도착하면 현재 실행 중인 프로세스의 CPU를 빼앗을 수 있음

**연관 개념**: [스케줄러](about:blank#%EC%8A%A4%EC%BC%80%EC%A4%84%EB%9F%AC), [프로세스의 상태](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4%EC%9D%98-%EC%83%81%ED%83%9C)

## 비선점형 스케줄링

### FCFS(First Come First Served)

**특징**
- 먼저 온 고객을 먼저 서비스해주는 방식, 즉 **먼저 온 순서대로 처리**
- **비선점형(Non-Preemptive) 스케줄링**
- 일단 CPU를 잡으면 CPU burst가 완료될 때까지 CPU를 반환하지 않음
- 할당되었던 CPU가 반환될 때만 스케줄링이 이루어짐

**문제점**
- **Convoy Effect(호위 효과)**: 소요시간이 긴 프로세스가 먼저 도달하여 뒤에 있는 짧은 프로세스들이 오래 기다려야 하므로 평균 대기 시간이 길어지고 효율성이 낮아지는 현상

**연관 개념**: [SJF](about:blank#sjf), [프로세스의 상태](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4%EC%9D%98-%EC%83%81%ED%83%9C)

### SJF(Shortest Job First)

**특징**
- 다른 프로세스가 먼저 도착했어도 **CPU burst time이 짧은 프로세스에게 선 할당**
- **비선점형(Non-Preemptive) 스케줄링**
- 평균 대기 시간이 가장 짧음 (이론적으로 최적)

**문제점**
- **Starvation(기아 현상)**: CPU 사용 시간이 긴 프로세스는 거의 영원히 CPU를 할당받을 수 없음
- **CPU burst time을 정확히 예측하기 어려움**: 과거 CPU burst time을 통해 예측하지만 완벽하지 않음

**연관 개념**: [SRTF](about:blank#srtf), [우선순위 스케줄링](about:blank#priority-scheduling)

### Priority Scheduling (우선순위 스케줄링)

**특징**
- **우선순위가 가장 높은 프로세스에게 CPU를 할당**하는 스케줄링
- 우선순위는 정수로 표현하며, 일반적으로 작은 숫자가 우선순위가 높음
- **선점형(Preemptive) 방식**: 더 높은 우선순위의 프로세스가 도착하면 실행 중인 프로세스를 멈추고 CPU를 선점
- **비선점형(Non-Preemptive) 방식**: 더 높은 우선순위의 프로세스가 도착하면 Ready Queue의 Head에 넣음

**문제점**
- **Starvation(기아 현상)**: 낮은 우선순위의 프로세스가 무한정 대기할 수 있음
- **무기한 봉쇄(Indefinite blocking)**: 실행 준비는 되어있으나 CPU를 사용하지 못하는 프로세스가 무기한 대기하는 상태

**해결책**
- **Aging(에이징)**: 시간이 지남에 따라 대기 중인 프로세스의 우선순위를 점진적으로 높여주는 방법

**연관 개념**: [SJF](about:blank#sjf), [다단계 큐](about:blank#%EB%8B%A4%EB%8B%A8%EA%B3%84-%ED%81%90)

## 선점형 스케줄링

### RR(Round Robin)

**특징**
- 각 프로세스는 동일한 크기의 할당 시간(**time quantum**)을 가짐
- 할당 시간이 지나면 프로세스는 선점당하고 ready queue의 제일 뒤로 가서 다시 줄을 섬
- **선점형(Preemptive) 스케줄링**
- 현대 운영체제에서 가장 많이 사용되는 알고리즘
- Response time이 빨라짐

**장점**
- 모든 프로세스가 공평하게 CPU를 할당받음
- n개의 프로세스가 있을 때 어떤 프로세스도 (n-1) * time quantum 이상 대기하지 않음

**단점**
- 평균 대기 시간이 길어질 수 있음
- 설정한 time quantum이 크면 FCFS와 같아짐
- 설정한 time quantum이 너무 작으면 컨텍스트 스위칭이 잦아져 오버헤드 증가

**Time Quantum 설정**
- 일반적으로 10-100ms 정도
- 너무 크면 FCFS처럼 동작
- 너무 작으면 컨텍스트 스위칭 오버헤드 증가

**연관 개념**: [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD), [FCFS](about:blank#fcfs)

### SRTF(Shortest Remaining Time First)

**특징**
- **새로운 프로세스가 도착할 때마다 새로운 스케줄링**이 이루어짐
- **선점형(Preemptive) 스케줄링**
- 현재 수행 중인 프로세스의 남은 burst time보다 더 짧은 CPU burst time을 가지는 새로운 프로세스가 도착하면 CPU를 빼앗김
- SJF의 선점형 버전

**장점**
- 평균 대기 시간이 가장 짧음 (이론적으로 최적)

**문제점**
- **Starvation**: 긴 프로세스는 계속 CPU를 할당받지 못할 수 있음
- 새로운 프로세스가 도달할 때마다 스케줄링을 다시하기 때문에 CPU burst time을 측정할 수가 없음
- 잦은 컨텍스트 스위칭으로 오버헤드 증가

**연관 개념**: [SJF](about:blank#sjf), [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD)

### 다단계 큐(Multilevel Queue)

**특징**
- **우선순위에 따른 준비 큐를 여러 개 사용**하고, 큐마다 라운드 로빈이나 FCFS 등 다른 스케줄링 알고리즘을 적용
- 큐 간의 프로세스 이동이 불가능
- 스케줄링 부담이 적지만 유연성이 떨어짐
- 우선순위가 높은 큐부터 처리

**큐의 종류 (일반적)**
1. System processes (최우선)
2. Interactive processes
3. Interactive editing processes
4. Batch processes
5. Student processes (최하위)

**문제점**
- 낮은 큐의 프로세스가 처리되지 않는 **기아 현상(Starvation)** 발생 가능

**해결책: 다단계 피드백 큐(Multilevel Feedback Queue)**
- 프로세스가 큐 간 이동 가능
- Aging 기법 적용 가능
- 현대 운영체제에서 많이 사용

**연관 개념**: [RR](about:blank#rr), [우선순위 스케줄링](about:blank#priority-scheduling)

# 동기와 비동기의 차이

## 동기(Synchronous)

**특징**
- 작업을 순차적으로 실행
- 하나의 작업이 완료될 때까지 다음 작업이 대기
- 호출한 함수의 결과를 기다림

**장점**
- 설계가 간단하고 직관적
- 실행 순서가 보장됨

**단점**
- 하나의 작업이 지연되면 전체 시스템 성능 저하
- 블로킹(Blocking) 발생

## 비동기(Asynchronous)

**특징**
- 작업을 병렬적으로 실행
- 하나의 작업이 완료되지 않아도 다음 작업 수행 가능
- 호출한 함수의 결과를 기다리지 않음

**장점**
- 시스템 자원을 효율적으로 사용
- 응답 시간 단축

**단점**
- 설계가 복잡
- 디버깅이 어려움
- 동기화 문제 발생 가능

**사용 예시**
- I/O 작업
- 네트워크 통신
- 파일 읽기/쓰기
- 데이터베이스 쿼리

**연관 개념**: [프로세스 동기화](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4-%EB%8F%99%EA%B8%B0%ED%99%94), [인터럽트](about:blank#%EC%9D%B8%ED%84%B0%EB%9F%BD%ED%8A%B8)

# 프로세스 동기화

멀티프로세스 또는 멀티스레드 환경에서 여러 프로세스/스레드가 **공유 자원에 동시에 접근**할 때 발생하는 문제를 해결하기 위한 메커니즘이다.

## Critical Section (임계 영역)

**임계 영역**은 여러 프로세스가 공유하는 데이터에 접근하는 코드 영역을 말한다. 한 번에 하나의 프로세스만 진입하여 실행할 수 있어야 한다.

### 임계영역 문제

: 프로세스들이 Critical Section 을 함께 사용할 수 있는 프로토콜을 설계하는 것.

**임계 영역 문제의 해결 조건**

1. **상호 배제(Mutual Exclusion)**
    - 한 프로세스가 임계 영역에 들어갔을 때 다른 프로세스는 들어갈 수 없음
2. **진행(Progress)**
    - 임계 영역에 진입한 프로세스가 없고, 진입하려는 프로세스가 있다면 진입을 허용해야 함
    - 프로세스끼리 서로 방해하지 않아야 함
3. **한정 대기(Bounded Waiting)**
    - 프로세스가 임계 영역 진입을 요청한 후 승인되기 전까지 다른 프로세스가 임계 영역에 진입하는 횟수를 제한
    - 특정 프로세스가 영원히 임계 영역에 들어가지 못하는 것을 방지(Starvation 방지)

**경쟁 상태(Race Condition):** 여러 프로세스/스레드가 공유 자원에 동시에 접근할 때, 실행 순서에 따라 결과가 달라지는 상황을 말한다.

## 임계 영역(Critical Section) 관리 기법

임계 영역 문제를 해결하는 대표적인 세 가지 도구:

1. **Mutex**
2. **Semaphore**
3. **Monitor**

그리고 이들을 이해하기 위해 쓰이는 **Lock** 개념이 있다.

---

### Mutex (뮤텍스)

> MUTual EXclusion: “한 번에 하나만 들어와라”를 강제하는 상호 배제 기법
> 

**역할**

- 여러 스레드가 공유 자원에 동시에 접근하지 못하도록
    
    **한 순간에 단 하나의 스레드만 임계 영역에 들어오게 하는 잠금 객체**
    

**동작**

- `lock()` : 뮤텍스 획득 → 임계 영역 진입
- `unlock()` : 뮤텍스 해제 → 다른 스레드가 진입 가능
- **소유권**: `lock()`한 스레드만 `unlock()` 가능

**특징**

- 가장 전형적인 **상호 배제(Mutual Exclusion)** 도구
- OS / 라이브러리 차원에서 제공 (`pthread_mutex`, `std::mutex`, Java의 내부 락 등)
- 비유: **변기 1칸짜리 화장실**
    - 들어간 사람이 문 잠그고, 나올 때만 열 수 있음

**한 줄 요약**

👉 “임계 영역에 **동시에 한 명만** 들어오게 하는, 소유권 있는 락”

---

### Semaphore (세마포어)

> “쓸 수 있는 자원이 몇 개 남았는지”를 정수 값으로 관리하면서,
> 
> 
> **wait / signal 연산으로 접근을 제어**하는 도구
> 

**역할**

- 정수 값 **S**와 두 연산 **wait(P), signal(V)** 로
    
    **자원의 개수 + 접근 순서를 제어**하는 동기화 기법
    

**동작 원리**

- **S** : 현재 사용 가능한 자원의 수
- `wait()` (P 연산)
    - `S--`
    - `S < 0` 이 되면 → 스레드를 **대기 큐에 넣고 블록**
- `signal()` (V 연산)
    - `S++`
    - `S ≤ 0` 이면 → **대기 큐에서 하나 깨워서 실행 재개**

**종류**

1. **바이너리 세마포어**
    - S ∈ {0, 1}
    - 모양은 뮤텍스와 비슷하지만, 개념은 **“신호”**
    - 누구나 `signal()` 호출 가능 → **소유권 개념 없음**
    - 상호 배제에 사용 가능
2. **카운팅 세마포어**
    - S ≥ 0, 여러 개의 자원 관리
    - 여러 스레드가 **동시에 임계 영역 / 자원**을 사용할 수 있음
    - 예: DB 커넥션 풀, 동시 접속 가능한 최대 인원 등

**장단점**

- 장점
    - **자원 개수까지** 함께 관리 가능 (뮤텍스보다 일반화된 모델)
- 단점
    - 잘못 쓰면 Deadlock / Starvation 등 버그 유발
    - `wait()` / `signal()` 쌍을 **프로그래머가 직접 맞춰야 해서** 실수 가능성 큼
        
        → 모니터보다 안전성이 떨어짐
        

**한 줄 요약**

👉 “자원이 **몇 개 남았는지도 같이 관리**하는 동기화 도구”

---

### Monitor (모니터)

> 공유 데이터와 그 데이터를 다루는 연산을 하나의 모듈로 묶고,
> 
> 
> 내부에 들어올 수 있는 스레드를 **한 번에 하나로 제한**하는 고급 추상화
> 

**역할**

- 공유 데이터를 **캡슐화**하고,
- 그 데이터에 접근하는 함수(메서드)들을 **모니터 안에만 정의**해서
- **언어/런타임이 자동으로 상호 배제를 보장**하도록 하는 구조

**특징**

- 모니터 내부의 코드(메서드)는 **동시에 하나의 스레드만 실행**
- 락 획득/해제, 조건변수 관리 등을 언어/런타임이 **자동 처리**
- 예:
    - Java의 `synchronized` 블록 / 메서드
    - C#의 `lock`

**세마포어와 비교**

| 구분 | 모니터 | 세마포어 |
| --- | --- | --- |
| 난이도 | 쉽다 (언어가 자동으로 관리) | 어렵다 (wait/signal 직접 관리) |
| 상호 배제 | 자동 보장 | 프로그래머가 보장해야 함 |
| 표현 방식 | 메서드/인터페이스 단위 | 정수 S + wait/signal 함수 |
| 용도 | 고수준 동기화(언어 문법 차원) | 저수준 동기화(원시 동기화 도구) |

**한 줄 요약**

👉 “**언어가 뮤텍스+세마포어를 포장해서** 자동으로 상호 배제까지 챙겨주는 고급 동기화 추상화”

### Lock 이란? (개념 정리)

> “락(lock)”은 위에서 본 뮤텍스, 스핀락, RW락 등 ‘잠그는 메커니즘 전체’를 가리키는 말이다.
> 

**Lock = 우산 개념**

- 임계 영역을 보호하기 위해 “잠근다(lock)”고 표현할 때의 **큰 범주**
- 구체적인 종류:
    - **Mutex Lock** (뮤텍스)
    - **Spin Lock**
    - **Read-Write Lock**
    - Reentrant Lock 등

그래서 관계를 정리하면:

- **Lock**: 임계 영역 보호용 **동기화 메커니즘의 큰 개념**
- **Mutex**: 그 중에서도
    
    → “**한 번에 한 스레드만, 소유권을 가진 상태로 잠그는 락**”의 구체적인 형태
    

**한 줄 요약**

👉 “Lock은 큰 말, Mutex는 그 안에 들어가는 **한 종류의 Lock**”

# 교착 상태(Deadlock)

**교착 상태**는 두 개 이상의 프로세스들이 **서로가 가진 자원을 기다리며 중단된 상태**를 말한다. 각 프로세스는 서로가 원하는 자원을 점유한 채 다른 프로세스의 자원을 얻기를 기다린다.

## 교착 상태 발생의 4가지 필요 조건

교착 상태가 발생하기 위해서는 다음 **4가지 조건이 모두 동시에 충족**되어야 한다:

1. **상호 배제(Mutual Exclusion)**
    - 자원은 한 번에 하나의 프로세스만 사용할 수 있음
    - 다른 프로세스는 접근 불가능
2. **점유 대기(Hold and Wait)**
    - 프로세스가 최소한 하나의 자원을 점유하고 있으면서
    - 다른 프로세스가 점유한 자원을 추가로 요구하며 대기
3. **비선점(No Preemption)**
    - 다른 프로세스의 자원을 강제로 빼앗을 수 없음
    - 자원을 점유한 프로세스만이 자발적으로 해제 가능
4. **순환 대기(Circular Wait)**
    - 프로세스들이 순환적으로 다음 프로세스가 요구하는 자원을 점유
    - 예: A→B→C→A의 순환 구조

**주의**: 4가지 조건이 모두 충족되어도 교착 상태가 반드시 발생하는 것은 아니지만, 4가지 조건이 모두 충족되어야만 교착 상태가 발생할 **가능성**이 있다.

**연관 개념**: 프로세스 동기화, 자원 할당 그래프

## 교착 상태의 해결 방법

### 1. 예방(Prevention)

교착 상태 발생의 4가지 필요 조건 중 **하나라도 발생하지 않도록** 사전에 차단한다.

**방법**
- **상호 배제 부정**: 여러 프로세스가 동시에 자원 사용 가능하도록 (현실적으로 어려움)
- **점유 대기 부정**: 프로세스 시작 시 필요한 모든 자원을 한 번에 할당
- **비선점 부정**: 자원을 선점 가능하도록 만듦
- **순환 대기 부정**: 자원에 순서를 정하여 오름차순으로만 할당

**단점**: 자원 활용률 저하, 시스템 처리율 감소

### 2. 회피(Avoidance)

교착 상태가 발생할 가능성이 있는 자원 할당을 피한다. 가장 유명한 알고리즘은 **은행원 알고리즘(Banker’s Algorithm)**이다.

**은행원 알고리즘**
- 각 프로세스가 필요로 하는 최대 자원량을 미리 선언
- 시스템은 **안전 상태(Safe State)**를 유지하는 선에서만 자원 할당
- 안전 상태: 모든 프로세스가 정상적으로 종료될 수 있는 상태

**장점**: 교착 상태를 확실히 예방
**단점**
- 프로세스의 최대 자원 요구량을 미리 알아야 함
- 프로세스 수와 자원 수가 고정되어야 함
- 자원 활용률이 낮음

### 3. 탐지(Detection)

교착 상태 발생을 허용하되, 주기적으로 교착 상태를 탐지한다.

**방법**
- **자원 할당 그래프(Resource Allocation Graph)**를 사용하여 순환 대기 탐지
- 그래프에 사이클이 존재하면 교착 상태

**장점**: 자원 활용률이 높음
**단점**: 탐지 오버헤드, 복구 비용

### 4. 복구(Recovery)

교착 상태를 탐지한 후 복구한다.

**방법**
- **프로세스 종료**: 교착 상태에 있는 프로세스를 하나씩 또는 모두 종료
- **자원 선점**: 교착 상태가 해소될 때까지 자원을 선점하여 다른 프로세스에게 할당

**단점**: 프로세스 종료로 인한 작업 손실

### 5. 무시(Ignore)

교착 상태를 무시하고 발생하지 않는다고 가정한다.

**사용 이유**
- 교착 상태 발생 빈도가 매우 낮음
- 예방, 회피, 탐지의 비용이 더 큼
- 사용자가 직접 처리하도록 함

**대부분의 현대 운영체제(Windows, Linux 등)에서 사용하는 방식**

**연관 개념**: [프로세스 동기화](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4-%EB%8F%99%EA%B8%B0%ED%99%94), [자원 할당](about:blank#%EB%A9%94%EB%AA%A8%EB%A6%AC-%EA%B4%80%EB%A6%AC-%EC%A0%84%EB%9E%B5)

# IPC (Inter-Process Communication)

**IPC**는 프로세스 간 데이터를 주고받고 공유 데이터를 관리하는 메커니즘을 뜻한다. 프로세스는 기본적으로 독립된 메모리 공간을 가지므로, 다른 프로세스의 메모리에 직접 접근할 수 없다. 따라서 IPC를 통해 프로세스 간 통신이 이루어진다.

**IPC가 필요한 이유**
- 정보 공유
- 계산 속도 향상 (병렬 처리)
- 모듈화
- 편의성

**연관 개념**: [프로세스](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4), [멀티프로세스](about:blank#%EB%A9%80%ED%8B%B0%EC%8A%A4%EB%A0%88%EB%93%9C-vs-%EB%A9%80%ED%8B%B0%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4)

## IPC의 종류

### 1. 공유 메모리(Shared Memory)

**특징**
- 여러 프로세스가 동일한 메모리 영역을 공유
- **IPC 방식 중 가장 빠름** (커널의 도움 없이 직접 메모리 접근)
- 불필요한 데이터 복사 오버헤드가 없음

**장점**
- 빠른 속도
- 대용량 데이터 전송에 유리

**단점**
- **동기화 문제**: 같은 메모리 영역을 여러 프로세스가 공유하므로 Race Condition 발생 가능
- 동기화 메커니즘(세마포어, 뮤텍스 등) 필요

**사용 예시**: 멀티미디어 애플리케이션, 대용량 데이터 처리

**연관 개념**: [프로세스 동기화](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4-%EB%8F%99%EA%B8%B0%ED%99%94), [세마포어](about:blank#semaphores)

### 2. 파이프(Pipe)

**특징**
- **단방향 통신** (한쪽에서 쓰고 다른 쪽에서 읽음)
- FIFO (First In First Out) 구조
- 부모-자식 프로세스 간 통신에 주로 사용

**종류**
- **익명 파이프(Anonymous Pipe)**: 부모-자식 관계에서만 사용
- **명명된 파이프(Named Pipe, FIFO)**: 이름이 있어 관계없는 프로세스 간에도 통신 가능

**장점**
- 간단한 구조
- 동기화가 자동으로 이루어짐

**단점**
- 단방향 통신 (양방향 통신을 위해서는 2개의 파이프 필요)
- 제한된 버퍼 크기

**사용 예시**: 쉘에서 | (파이프) 사용

### 3. 메시지 큐(Message Queue)

**특징**
- 메시지(데이터) 단위로 통신
- **FIFO 구조**
- 비동기 통신 가능
- 커널에서 관리

**장점**
- 데이터 경계가 명확함 (메시지 단위)
- 비동기 통신 가능
- 여러 프로세스가 동시에 읽고 쓸 수 있음

**단점**
- 메시지 크기 제한
- 커널을 거치므로 공유 메모리보다 느림

**사용 예시**: 작업 큐, 비동기 작업 처리

### 4. 소켓(Socket)

**특징**
- **네트워크를 통한 프로세스 간 통신**
- 양방향 통신
- 같은 컴퓨터뿐만 아니라 **다른 컴퓨터의 프로세스와도 통신 가능**

**종류**
- **스트림 소켓(TCP)**: 연결 지향, 신뢰성 보장
- **데이터그램 소켓(UDP)**: 비연결, 빠르지만 신뢰성 낮음

**장점**
- 원격 통신 가능
- 범용성이 높음

**단점**
- 네트워크 오버헤드
- 복잡한 구현

**사용 예시**: 웹 서버, 채팅 애플리케이션, 분산 시스템

### 5. 파일(File)

**특징**
- 디스크에 저장된 데이터를 통해 통신
- 영구 저장 가능

**장점**
- 간단한 구현
- 데이터 영구 보존

**단점**
- 가장 느림 (디스크 I/O)
- 동기화 필요

**비교표**

| IPC 방식 | 속도 | 복잡도 | 용도 |
| --- | --- | --- | --- |
| 공유 메모리 | 가장 빠름 | 높음 (동기화 필요) | 대용량 데이터, 고속 통신 |
| 파이프 | 중간 | 낮음 | 부모-자식 간 단순 통신 |
| 메시지 큐 | 중간 | 중간 | 비동기 메시지 전달 |
| 소켓 | 느림 | 높음 | 네트워크 통신 |
| 파일 | 가장 느림 | 낮음 | 영구 저장 필요 시 |

**연관 개념**: [멀티프로세스](about:blank#%EB%A9%80%ED%8B%B0%EC%8A%A4%EB%A0%88%EB%93%9C-vs-%EB%A9%80%ED%8B%B0%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4), [프로세스 동기화](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4-%EB%8F%99%EA%B8%B0%ED%99%94)

# 인터럽트(Interrupt)

**인터럽트**는 CPU가 프로그램을 실행하고 있을 때, **어떤 신호가 들어오면 현재 작업을 중단하고 발생한 상황을 처리**한 후 다시 원래 작업으로 돌아가는 것을 말한다.

**인터럽트가 발생하는 경우**
- 0으로 나누는 산술 연산 오류
- 프로세스 오류
- I/O 디바이스의 입출력 요청
- 우선순위가 높은 프로세스 발생
- 타이머 만료

**인터럽트 처리 과정**
1. CPU가 메모리에 있는 명령어를 순차적으로 실행
2. 인터럽트 발생
3. 현재 실행 중인 프로세스의 상태를 PCB에 저장
4. 인터럽트 벡터를 읽어 인터럽트 핸들러(ISR) 주소 확인
5. 인터럽트 핸들러 함수 실행
6. 인터럽트 처리 완료 후 저장된 상태를 복원하여 원래 프로세스로 복귀

**연관 개념**: [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD), [PCB](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4-%EC%A0%9C%EC%96%B4-%EB%B8%94%EB%A1%9D)

## 인터럽트의 종류

### 1. 하드웨어 인터럽트(Hardware Interrupt)

**외부 인터럽트**라고도 하며, **하드웨어 장치에 의해 발생**하는 인터럽트이다.

**발생 원인**
- **I/O 인터럽트**: 키보드, 마우스, 디스크 등 입출력 장치의 작업 완료
- **타이머 인터럽트**: 일정 시간 간격으로 발생 (CPU 스케줄링에 사용)
- **전원 이상 인터럽트**: 정전, 전원 공급 이상

**특징**
- 비동기적 발생 (예측 불가능)
- 외부 장치가 CPU에게 신호를 보냄

### 2. 소프트웨어 인터럽트(Software Interrupt)

**내부 인터럽트** 또는 **트랩(Trap)**이라고도 하며, **프로그램 실행 중 발생**하는 인터럽트이다.

**발생 원인**
- **예외(Exception)**: 0으로 나누기, 오버플로우, 언더플로우 등 프로그램 오류
- **시스템 콜(System Call)**: 프로그램이 운영체제의 서비스를 요청

**특징**
- 동기적 발생 (프로그램 실행 중 예측 가능)
- 프로그램 내부에서 발생

**인터럽트 우선순위**

인터럽트는 우선순위가 있어, 높은 우선순위의 인터럽트가 발생하면 낮은 우선순위의 인터럽트 처리를 중단하고 먼저 처리한다.

**일반적인 우선순위 (높음 → 낮음)**
1. 전원 이상 (Power fail)
2. 기계 착오 (Machine check)
3. 외부 신호 (External)
4. 입출력 (I/O)
5. 명령어 잘못
6. 프로그램 검사
7. SVC (Supervisor Call)

**연관 개념**: [시스템 콜](about:blank#%EC%8B%9C%EC%8A%A4%ED%85%9C-%EC%BD%9C), [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD)

# 시스템 콜(System Call)

**시스템 콜**은 **응용 프로그램이 운영체제의 커널이 제공하는 서비스를 이용**하기 위한 인터페이스이다. 사용자 프로그램은 직접 커널에 접근할 수 없고, 시스템 콜을 통해서만 커널 함수를 호출할 수 있다.

**시스템 콜이 필요한 이유**
- 사용자 프로그램과 운영체제의 분리로 **시스템 보안 강화**
- 하드웨어 자원에 대한 **안전한 접근 제어**
- 복잡한 저수준 작업을 **추상화하여 프로그래밍 편의성 제공**

**시스템 콜의 종류**
- **프로세스 제어**: fork(), exec(), exit(), wait() 등
- **파일 관리**: open(), read(), write(), close() 등
- **장치 관리**: ioctl(), read(), write() 등
- **정보 유지**: getpid(), alarm(), sleep() 등
- **통신**: pipe(), shmget(), mmap() 등

**연관 개념**: [인터럽트](about:blank#%EC%9D%B8%ED%84%B0%EB%9F%BD%ED%8A%B8), [Mode Bit](about:blank#mode-bit)

## Mode Bit

**Mode Bit**는 **현재 CPU가 실행하는 코드가 사용자 모드인지 커널 모드인지를 구분**하는 플래그이다.

**Mode Bit 값**
- **0**: 커널 모드 (Kernel Mode) - 모든 권한 있음
- **1**: 사용자 모드 (User Mode) - 제한된 권한

**시스템 콜 실행 과정과 Mode Bit 변화**
1. 사용자 프로그램 실행 (Mode Bit = 1)
2. 시스템 콜 발생 (소프트웨어 인터럽트)
3. Mode Bit를 0으로 변경 (커널 모드로 전환)
4. 커널 함수 실행
5. 작업 완료 후 Mode Bit를 1로 변경 (사용자 모드로 복귀)
6. 사용자 프로그램 재개

**예시: 파일 읽기**

```
사용자 프로그램: fs.readFile() 호출
↓
시스템 콜 발생 (Mode Bit = 1 → 0)
↓
커널: 파일 시스템 접근, 디스크 I/O
↓
커널: 데이터 반환 (Mode Bit = 0 → 1)
↓
사용자 프로그램: 데이터 처리
```

## 커널 모드 vs 사용자 모드

### 사용자 모드(User Mode)

**특징**
- **제한된 권한**으로 실행
- 일반 응용 프로그램이 실행되는 모드
- 하드웨어나 중요 시스템 자원에 직접 접근 불가

**할 수 있는 일**
- 일반적인 연산
- 할당된 메모리 영역 내에서의 작업
- 시스템 콜을 통한 커널 서비스 요청

### 커널 모드(Kernel Mode)

**특징**
- **모든 권한**으로 실행
- 운영체제의 핵심 기능이 실행되는 모드
- 모든 하드웨어와 시스템 자원에 접근 가능

**할 수 있는 일**
- 모든 메모리 영역 접근
- 모든 CPU 명령어 실행
- I/O 장치 제어
- 인터럽트 처리

**시스템 콜의 장점**

1. **보안성**: 사용자 프로그램이 직접 하드웨어에 접근하는 것을 방지
2. **안정성**: 잘못된 접근으로 인한 시스템 충돌 방지
3. **추상화**: 복잡한 하드웨어 작업을 간단한 인터페이스로 제공
4. **이식성**: 하드웨어와 독립적인 프로그래밍 가능

**예시: 카메라 접근**

시스템 콜이 없다면:
- 악의적인 앱이 직접 카메라 하드웨어에 접근
- 사용자 동의 없이 카메라 작동 가능
- 사생활 침해

시스템 콜이 있다면:
- 앱은 시스템 콜을 통해서만 카메라 접근 가능
- 운영체제가 권한 확인 및 제어
- 사용자의 명시적 동의 필요

**연관 개념**: [인터럽트](about:blank#%EC%9D%B8%ED%84%B0%EB%9F%BD%ED%8A%B8), [프로세스](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4)

# 메모리 관리 전략

## 메모리 관리 배경

각각의 **프로세스**는 독립된 메모리 공간을 갖고, 운영체제 혹은 다른 프로세스의 메모리 공간에 접근할 수 없는 제한이 걸려있다. 단지, **운영체제**만이 운영체제 메모리 영역과 사용자 메모리 영역의 접근에 제약을 받지 않는다.

**연관 개념**: [가상 메모리](about:blank#%EA%B0%80%EC%83%81-%EB%A9%94%EB%AA%A8%EB%A6%AC), [프로세스](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4)

## Swapping (스와핑)

**스와핑**은 메모리 관리를 위해 사용되는 기법으로, CPU 할당 시간이 끝난 프로세스의 메모리를 보조 기억장치(하드디스크)로 내보내고 다른 프로세스의 메모리를 불러들이는 것이다.

**용어**
- **Swap-in**: 보조 기억장치에서 주 기억장치(RAM)로 불러오는 과정
- **Swap-out**: 주 기억장치에서 보조 기억장치로 내보내는 과정

**특징**
- Swap에는 큰 디스크 전송 시간이 필요
- 현재에는 메모리 공간이 부족할 때 주로 사용
- Round-robin 같은 스케줄링과 함께 사용

**연관 개념**: [중기 스케줄러](about:blank#%EC%A4%91%EA%B8%B0-%EC%8A%A4%EC%BC%80%EC%A4%84%EB%9F%AC), [가상 메모리](about:blank#%EA%B0%80%EC%83%81-%EB%A9%94%EB%AA%A8%EB%A6%AC)

## 단편화(Fragmentation)

**단편화**는 프로세스들이 메모리에 적재되고 제거되는 일이 반복되면서, 프로세스들이 차지하는 메모리 틈 사이에 사용하지 못할 만큼의 작은 자유공간들이 늘어나게 되는 현상이다.

**단편화의 종류**

1. **외부 단편화(External Fragmentation)**
    - 메모리 공간 중 사용하지 못하게 되는 일부분
    - 물리 메모리에서 사이사이 남는 공간들이 분산되어 있을 때 발생
    - 총 여유 공간은 충분하지만 **연속되지 않아** 프로세스를 할당할 수 없음
    
    예시:
    
    ```
    [Process A] [free:2KB] [Process B] [free:3KB] [Process C] [free:4KB]
    → 총 9KB의 여유 공간이 있지만, 6KB 프로세스는 적재 불가
    ```
    
2. **내부 단편화(Internal Fragmentation)**
    - 프로세스가 사용하는 메모리 공간에 포함된 남는 부분
    - 메모리 할당 단위(페이지, 블록)가 프로세스가 실제 필요한 크기보다 클 때 발생
    
    예시:
    
    ```
    페이지 크기: 10,000B
    프로세스 요구: 9,998B
    → 2B의 내부 단편화 발생
    ```
    

**연관 개념**: [Paging](about:blank#paging), [Segmentation](about:blank#segmentation)

## 압축(Compaction)

**압축**은 외부 단편화를 해소하기 위해 프로세스가 사용하는 공간들을 한쪽으로 몰아 자유공간을 확보하는 방법이다.

**문제점**
- 작업 효율이 좋지 않음
- 프로세스를 이동시키는 데 많은 시간과 자원 소요
- 실행 중인 프로세스를 이동시키기 어려움

**예시**

```
압축 전: [Process A] [free] [Process B] [free] [Process C] [free]
압축 후: [Process A] [Process B] [Process C] [free free free]
```

**현대적 해결책**: 페이징(Paging), 세그멘테이션(Segmentation)

**연관 개념**: [단편화](about:blank#%EB%8B%A8%ED%8E%B8%ED%99%94), [Paging](about:blank#paging)

## Paging (페이징)

**페이징**은 프로세스의 **물리 주소 공간이 비연속적**이어도 되도록 하는 메모리 관리 기법이다. 외부 단편화와 압축 작업을 해소하기 위해 고안되었다.

**기본 개념**
- **물리 메모리**: **Frame**이라는 고정 크기 블록으로 분할
- **논리 메모리(프로세스)**: **Page**라는 고정 크기 블록으로 분할
- 페이지와 프레임의 크기는 동일 (보통 4KB 또는 16KB)

**동작 원리**
- 프로세스의 페이지들을 물리 메모리의 빈 프레임에 순서에 상관없이 적재
- **페이지 테이블**을 사용하여 논리 주소를 물리 주소로 변환
- CPU가 논리 주소를 생성하면 MMU(Memory Management Unit)가 페이지 테이블을 참조하여 물리 주소로 변환

**페이지 테이블 구조**
- 각 프로세스마다 별도의 페이지 테이블 보유
- 페이지 번호 → 프레임 번호 매핑
- 페이지 테이블은 메모리에 저장 (PTBR: Page Table Base Register)

**장점**
- **외부 단편화 해결**: 페이지를 순서에 상관없이 빈 프레임에 할당 가능
- 메모리 관리가 간단
- 프로세스 간 메모리 보호 용이

**단점**
- **내부 단편화 발생**: 프로세스 크기가 페이지 크기의 배수가 아니면 마지막 페이지에서 내부 단편화 발생
- 예: 페이지 크기 1,024B, 프로세스 요구 3,172B → 4개 페이지 필요 → 924B 낭비
- **페이지 테이블을 위한 추가 메모리 필요**
- 주소 변환 오버헤드 (TLB로 완화 가능)

**페이지 크기 선택의 트레이드오프**
- **큰 페이지**: 페이지 테이블 크기 감소, 내부 단편화 증가, I/O 시간 증가
- **작은 페이지**: 페이지 테이블 크기 증가, 내부 단편화 감소, I/O 시간 감소

**연관 개념**: [페이지 테이블](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%ED%85%8C%EC%9D%B4%EB%B8%94), [가상 메모리](about:blank#%EA%B0%80%EC%83%81-%EB%A9%94%EB%AA%A8%EB%A6%AC), [TLB](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%ED%85%8C%EC%9D%B4%EB%B8%94)

## Segmentation (세그멘테이션)

**세그멘테이션**은 프로세스를 **논리적 단위인 세그먼트**로 분할하는 메모리 관리 기법이다. 페이징과 달리 세그먼트의 크기는 일정하지 않다.

**기본 개념**
- 프로세스를 **논리적 의미 단위**로 분할
- 예: 코드 세그먼트, 데이터 세그먼트, 스택 세그먼트, 힙 세그먼트
- 각 세그먼트는 서로 다른 크기를 가짐

**세그먼트 테이블**
- **Segment Number**: 세그먼트 번호
- **Base (시작 주소)**: 세그먼트의 시작 물리 주소
- **Limit (한계)**: 세그먼트의 길이

**주소 변환**

```
논리 주소 = (세그먼트 번호, 오프셋)
물리 주소 = Base[세그먼트 번호] + 오프셋
(단, 오프셋 < Limit[세그먼트 번호])
```

**장점**
- **논리적 단위로 분할**: 프로그램의 구조를 반영하여 이해하기 쉬움
- **공유와 보호 용이**: 세그먼트 단위로 공유 및 접근 권한 설정 가능
- **내부 단편화 없음**: 필요한 크기만큼만 할당

**단점**
- **외부 단편화 발생**: 서로 다른 크기의 세그먼트들이 메모리에 적재되고 제거되면서 빈 공간이 분산됨
- 세그먼트의 크기가 커지면 메모리에 적재하기 어려움

**페이징 vs 세그멘테이션 비교**

| 구분 | 페이징 | 세그멘테이션 |
| --- | --- | --- |
| 분할 기준 | 고정 크기 (물리적) | 가변 크기 (논리적) |
| 단편화 | 내부 단편화 | 외부 단편화 |
| 사용자 인식 | 투명 (사용자가 모름) | 가시적 (사용자가 인식) |
| 공유/보호 | 어려움 | 쉬움 |

**페이지드 세그멘테이션(Paged Segmentation)**

# 가상 메모리

**가상 메모리**는 프로세스 전체가 메모리 내에 올라오지 않더라도 실행이 가능하도록 하는 기법이다. 프로그램이 물리 메모리보다 커도 실행할 수 있다는 주요 장점이 있다.

**연관 개념**: [Paging](about:blank#paging), [페이지 교체](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%EA%B5%90%EC%B2%B4-%EC%95%8C%EA%B3%A0%EB%A6%AC%EC%A6%98)

## 가상 메모리 개발 배경

전통적인 시스템의 문제점:
- 실행되는 **코드의 전부를 물리 메모리에 존재**시켜야 함
- **메모리 용량보다 큰 프로그램은 실행 불가**
- 여러 프로그램을 동시에 메모리에 올리기에는 용량의 한계
- 가끔만 사용되는 코드도 메모리를 차지

**프로그램의 일부분만 메모리에 올릴 수 있다면?**

- **물리 메모리 크기에 제약받지 않음**: 가상 메모리 크기만큼 프로그램 작성 가능
- **더 많은 프로그램을 동시에 실행** 가능: 응답시간은 유지되고, CPU 이용률과 처리율은 높아짐
- **Swap에 필요한 I/O 감소**: 프로그램들이 빠르게 실행됨
- **메모리 추상화**: 프로그래머가 물리 메모리 크기를 신경 쓸 필요 없음

**연관 개념**: [메모리 계층](about:blank#%EB%A9%94%EB%AA%A8%EB%A6%AC-%EA%B4%80%EB%A6%AC-%EB%B0%B0%EA%B2%BD), [Demand Paging](about:blank#demand-paging)

## 가상 메모리가 하는 일

가상 메모리는 **실제의 물리 메모리 개념과 사용자의 논리 메모리 개념을 분리**한 것으로 정리할 수 있다. 이로써 작은 메모리를 가지고도 얼마든지 큰 **가상 주소 공간**을 프로그래머에게 제공할 수 있다.

## 가상 주소 공간(Virtual Address Space)

**가상 주소 공간**은 한 프로세스가 메모리에 저장되는 논리적인 모습을 가상 메모리에 구현한 공간이다.

**특징**
- 프로세스가 요구하는 메모리 공간을 가상 메모리에서 제공
- 현재 직접적으로 필요하지 않은 메모리 공간은 실제 물리 메모리에 올리지 않음
- 물리 메모리 절약

**예시**

```
프로그램이 논리 메모리로 100KB 요구
실제 필요한 메모리(Heap, Stack, Code, Data) = 40KB
→ 물리 메모리에는 40KB만 적재
→ 나머지 60KB는 필요할 때 적재
```

**가상 주소 공간 구조**

```
[Stack]
[Free Space (60KB)]
[Heap]
[Data]
[Code]
```

**연관 개념**: [페이지 테이블](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%ED%85%8C%EC%9D%B4%EB%B8%94), [MMU](about:blank#demand-paging)

## 메모리 보호

가상 메모리는 각 프로세스에 독립적인 가상 주소 공간을 제공하여 **프로세스 간 메모리 보호**를 실현한다.

**보호 메커니즘**
- 각 프로세스는 독립적인 가상 주소 공간을 가짐
- 동일한 가상 주소라도 서로 다른 물리 메모리 영역에 매핑
- 한 프로세스의 오류나 악의적 접근이 다른 프로세스에 영향을 주지 않음

**메모리 접근 제어**
- 페이지 단위로 읽기/쓰기/실행 권한 설정 가능
- 권한 없는 접근 시도는 즉시 감지되고 차단
- 버퍼 오버플로우, 코드 인젝션 등의 보안 위협 예방

**연관 개념**: [Paging](about:blank#paging), [프로세스](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4)

## 프로세스 간 페이지 공유

가상 메모리는 다음과 같은 공유를 가능하게 한다:

1. **시스템 라이브러리 공유**
    - 여러 프로세스가 동일한 라이브러리를 사용할 때
    - 각 프로세스는 자신의 가상 주소 공간에 라이브러리가 있는 것처럼 인식
    - 실제로는 하나의 물리 메모리 페이지를 공유
2. **공유 메모리를 통한 IPC**
    - 프로세스들이 메모리를 공유하여 통신
    - 각 프로세스는 자신의 주소 공간처럼 인식
    - 실제 물리 메모리는 공유
3. **fork()를 통한 프로세스 생성**
    - Copy-On-Write(COW) 기법 사용
    - 초기에는 부모 프로세스와 페이지를 공유
    - 수정이 발생할 때만 복사

**연관 개념**: [IPC](about:blank#ipc), [프로세스](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4)

## 페이지 테이블

**페이지 테이블**은 논리 주소(가상 주소)를 물리 주소로 변환하는 매핑 정보를 저장하는 자료 구조이다.

**구성 요소**
- **페이지 번호**: 논리 주소의 페이지 인덱스
- **프레임 번호**: 해당 페이지가 적재된 물리 메모리 프레임 번호
- **Valid/Invalid Bit**: 페이지가 메모리에 있는지 여부
- Valid (1): 페이지가 물리 메모리에 있음
- Invalid (0): 페이지가 물리 메모리에 없음 (디스크에 있음)
- **보호 비트(Protection Bit)**: 읽기/쓰기/실행 권한
- **참조 비트(Reference Bit)**: 최근 접근 여부 (페이지 교체 알고리즘에 사용)
- **수정 비트(Modified Bit)**: 페이지 수정 여부 (Dirty Bit)

**주소 변환 과정**
1. CPU가 논리 주소 생성 (페이지 번호 + 오프셋)
2. 페이지 번호로 페이지 테이블 인덱싱
3. Valid bit 확인
- Valid: 프레임 번호 획득 → 물리 주소 = 프레임 번호 + 오프셋
- Invalid: 페이지 폴트 발생
4. 물리 주소로 메모리 접근

**페이지 테이블의 문제점과 해결**

1. **크기 문제**
    - 32bit 시스템, 페이지 크기 4KB: 페이지 테이블 크기 = 2^20 entries = 4MB (프로세스당)
    - 해결: 계층적 페이지 테이블, 해시 페이지 테이블, 역 페이지 테이블
2. **접근 시간 문제**
    - 메모리 접근마다 2번의 메모리 접근 필요 (페이지 테이블 + 실제 데이터)
    - 해결: TLB (Translation Lookaside Buffer)

**TLB (Translation Lookaside Buffer)**

페이지 테이블의 일부를 캐시하는 고속 하드웨어 캐시이다.

**특징**
- MMU 내부에 위치한 연관 메모리(Associative Memory)
- 최근 사용된 페이지 테이블 엔트리를 저장
- 병렬 검색 지원 (모든 엔트리 동시 비교)

**동작 과정**
1. CPU가 논리 주소 생성
2. TLB에서 페이지 번호 검색
- **TLB Hit**: TLB에서 바로 프레임 번호 획득 (빠름)
- **TLB Miss**: 페이지 테이블 접근 필요 (느림)
3. TLB Miss 시: 페이지 테이블에서 엔트리를 가져와 TLB에 저장

**TLB 적중률 (Hit Ratio)**
- 일반적으로 80% ~ 99%
- 지역성(Locality)이 좋을수록 적중률 높음

**연관 개념**: [Paging](about:blank#paging), [가상 메모리](about:blank#%EA%B0%80%EC%83%81-%EB%A9%94%EB%AA%A8%EB%A6%AC), [캐시](about:blank#%EC%BA%90%EC%8B%9C%EC%9D%98-%EC%A7%80%EC%97%AD%EC%84%B1)

## Demand Paging (요구 페이징)

**요구 페이징**은 프로그램 실행 시작 시 프로그램 전체를 메모리에 적재하지 않고, **초기에 필요한 것들만 적재**하는 전략이다. 가상 메모리 시스템에서 많이 사용된다.

**기본 원리**
- 프로세스 실행 시 필요한 페이지만 메모리에 적재
- **한 번도 접근되지 않은 페이지는 물리 메모리에 적재되지 않음**
- 프로세스 내의 개별 페이지들은 **페이저(Pager)**에 의해 관리됨

**장점**
- I/O 양 감소
- 메모리 사용량 감소
- 빠른 응답 시간
- 더 많은 프로세스 수용 가능

**연관 개념**: [페이지 폴트](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%ED%8F%B4%ED%8A%B8), [가상 메모리](about:blank#%EA%B0%80%EC%83%81-%EB%A9%94%EB%AA%A8%EB%A6%AC)

## 페이지 폴트(Page Fault)

**페이지 폴트**는 CPU가 접근하려는 페이지가 현재 물리 메모리에 없을 때 발생하는 인터럽트이다.

**페이지 폴트 처리 과정**
1. CPU가 페이지에 접근 시도
2. 페이지 테이블 확인 → Invalid bit
3. **페이지 폴트 트랩 발생**
4. 운영체제가 페이지 폴트 핸들러 실행
5. 디스크에서 필요한 페이지 위치 확인
6. 빈 프레임 검색
- 빈 프레임이 있으면: 해당 프레임에 페이지 적재
- 빈 프레임이 없으면: **페이지 교체 알고리즘** 실행하여 희생 페이지 선택
7. 디스크에서 페이지를 메모리로 읽어옴
8. 페이지 테이블 업데이트 (Valid bit = 1, 프레임 번호 설정)
9. 중단되었던 명령어 재실행

**성능에 미치는 영향**

페이지 폴트가 자주 발생하면 성능이 크게 저하된다.

**유효 접근 시간(Effective Access Time)**

```
EAT = (1 - p) × 메모리 접근 시간 + p × 페이지 폴트 처리 시간
```

- p: 페이지 폴트 발생 확률
- 메모리 접근 시간: 약 200ns
- 페이지 폴트 처리 시간: 약 8ms (디스크 I/O 포함)

**예시**
- p = 0.001 (0.1% 페이지 폴트율)
- EAT = 0.999 × 200ns + 0.001 × 8ms = 8,199.8ns
- 약 40배 느림!

**연관 개념**: [Demand Paging](about:blank#demand-paging), [페이지 교체 알고리즘](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%EA%B5%90%EC%B2%B4-%EC%95%8C%EA%B3%A0%EB%A6%AC%EC%A6%98)

## 페이지 교체 알고리즘

물리 메모리가 모두 사용 중인 상황에서 페이지 폴트가 발생하면, **어떤 페이지를 메모리에서 제거할지 결정**해야 한다. 이를 위한 알고리즘이 페이지 교체 알고리즘이다.

**목표**: 페이지 폴트 발생 횟수를 최소화

**연관 개념**: [페이지 폴트](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%ED%8F%B4%ED%8A%B8), [가상 메모리](about:blank#%EA%B0%80%EC%83%81-%EB%A9%94%EB%AA%A8%EB%A6%AC)

### FIFO (First-In-First-Out)

**특징**
- 가장 먼저 메모리에 들어온 페이지를 먼저 교체
- 구현이 가장 간단
- 큐(Queue)를 사용하여 관리

**장점**
- 이해하기 쉽고 구현이 간단

**단점**
- 오래된 페이지가 항상 불필요한 것은 아님 (초기 변수, 자주 사용되는 페이지 등)
- 초기부터 활발하게 사용되는 페이지를 교체하여 페이지 부재율 증가
- **Belady’s Anomaly(벨라디의 모순)**: 프레임 수를 늘렸는데 오히려 페이지 폴트가 증가하는 현상

**Belady’s Anomaly 예시**

```
참조 문자열: 1 2 3 4 1 2 5 1 2 3 4 5
3 프레임: 9회 페이지 폴트
4 프레임: 10회 페이지 폴트 (!)
```

**연관 개념**: [LRU](about:blank#lru), [OPT](about:blank#opt)

### OPT (Optimal Page Replacement, LFD Longest Forward Distance)

**특징**
- **앞으로 가장 오랫동안 사용되지 않을 페이지를 교체**
- 이론적으로 가장 낮은 페이지 폴트율 보장
- Belady’s Anomaly 발생하지 않음

**장점**
- 모든 알고리즘 중 가장 낮은 페이지 폴트율

**단점**
- **구현 불가능**: 미래의 페이지 참조를 미리 알 수 없음
- 주로 다른 알고리즘의 성능 비교 기준으로 사용

**용도**: 이론적 연구, 다른 알고리즘의 성능 비교

**연관 개념**: [LRU](about:blank#lru), [FIFO](about:blank#fifo)

### LRU (Least Recently Used)

**특징**
- **가장 오랫동안 사용되지 않은 페이지를 교체**
- OPT 알고리즘의 근사 알고리즘
- 과거 사용 패턴을 기반으로 미래를 예측

**구현 방법**

1. **카운터 방식**
    - 각 페이지에 참조 시간(타임스탬프) 기록
    - 교체 시 가장 오래된 타임스탬프를 가진 페이지 선택
2. **스택 방식**
    - 페이지 참조 시마다 스택의 최상위로 이동
    - 교체 시 스택 하단의 페이지 선택

**장점**
- FIFO보다 우수한 성능
- Belady’s Anomaly 발생하지 않음
- 현실적으로 구현 가능

**단점**
- 구현 복잡도가 높음
- 모든 페이지 참조마다 타임스탬프 업데이트 필요 (오버헤드)

**성능**
- 일반적으로 OPT에 근접한 성능
- FIFO보다 우수

**연관 개념**: [OPT](about:blank#opt), [LFU](about:blank#lfu), [캐시](about:blank#%EC%BA%90%EC%8B%9C%EC%9D%98-%EC%A7%80%EC%97%AD%EC%84%B1)

### LFU (Least Frequently Used)

**특징**
- **참조 횟수가 가장 적은 페이지를 교체**
- 활발하게 사용되는 페이지는 참조 횟수가 많을 것이라는 가정

**구현**
- 각 페이지에 참조 카운터 유지
- 교체 시 카운터가 가장 작은 페이지 선택
- 동일한 카운터 값: 추가 기준 필요 (FIFO, LRU 등)

**장점**
- 장기적으로 사용되는 페이지 보호

**단점**
- **초기 사용 패턴에 영향**: 초기에 많이 사용되었지만 이후 사용되지 않는 페이지가 메모리에 계속 남음
- OPT를 제대로 근사하지 못함
- 참조 카운터 유지 오버헤드

**예시 문제**

```
페이지 A: 초기에 100번 참조, 이후 사용 안 함
페이지 B: 최근 10번 참조, 현재 활발히 사용 중
→ LFU는 B를 교체 (잘못된 선택)
```

**사용**: 잘 쓰이지 않음

**연관 개념**: [LRU](about:blank#lru), [MFU](about:blank#mfu)

### MFU (Most Frequently Used)

**특징**
- **참조 횟수가 가장 많은 페이지를 교체**
- 참조 횟수가 가장 적은 페이지가 최근에 메모리에 올라왔고, 앞으로 계속 사용될 것이라는 가정

**단점**
- OPT를 제대로 근사하지 못함
- 직관적이지 않음
- 잘 쓰이지 않음

**연관 개념**: [LFU](about:blank#lfu)

### NUR (Not Used Recently) / Clock Algorithm

**특징**
- LRU의 근사 알고리즘
- **참조 비트(Reference Bit)**와 **수정 비트(Modified Bit)** 사용
- 하드웨어 지원으로 구현 간단

**동작 원리**
- 각 페이지에 두 개의 비트:
- **참조 비트(R)**: 페이지가 참조되었는지 (0 또는 1)
- **수정 비트(M)**: 페이지가 수정되었는지 (0 또는 1)

**교체 우선순위** (낮을수록 우선)
1. (R=0, M=0): 최근 참조도 안 되고 수정도 안 됨 → 최우선 교체
2. (R=0, M=1): 최근 참조는 안 됐지만 수정됨
3. (R=1, M=0): 최근 참조됐지만 수정 안 됨
4. (R=1, M=1): 최근 참조되고 수정됨 → 교체 최후 순위

**Clock Algorithm**
- 페이지들을 원형 리스트로 관리
- 포인터가 시계처럼 회전하며 교체할 페이지 탐색
- R=0인 페이지 발견 시 교체
- R=1인 페이지는 R=0으로 설정하고 다음으로 이동

**장점**
- LRU보다 구현이 간단
- 하드웨어 지원으로 오버헤드 적음
- 실용적인 성능

**사용**: 많은 현대 운영체제에서 사용

**연관 개념**: [LRU](about:blank#lru), [페이지 테이블](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%ED%85%8C%EC%9D%B4%EB%B8%94)

## 스레싱(Thrashing)

**스레싱**은 프로세스가 실제 작업보다 **페이징에 더 많은 시간을 소비**하는 현상이다. 페이지 폴트가 빈번하게 발생하여 CPU 이용률이 급격히 저하된다.

**발생 과정**
1. 프로세스들이 필요한 최소 페이지 수보다 적은 프레임을 할당받음
2. 페이지 폴트가 빈번하게 발생
3. 페이지 교체를 위한 스왑 작업(디스크 I/O)이 과도하게 발생
4. CPU는 대부분 대기 상태 → CPU 이용률 낮아짐
5. 운영체제는 낮은 CPU 이용률을 보고 다중 프로그래밍 정도를 높임 (프로세스 추가)
6. 메모리 압박이 더 심해짐 → 스레싱 악화

**악순환**

```
메모리 부족 → 페이지 폴트 증가 → 디스크 I/O 증가
→ CPU 이용률 감소 → 프로세스 추가 → 메모리 더 부족
```

**연관 개념**: 페이지 폴트, 다중 프로그래밍

## 스레싱 방지 방법

### 1. 작업 세트(Working Set)

**개념**
- 프로세스가 일정 시간 동안 자주 참조하는 페이지들의 집합
- 프로세스의 과거 사용 이력을 기반으로 많이 사용하는 페이지 집합을 만들어 **한꺼번에 미리 메모리에 로드**

**동작 원리**
- 작업 세트 크기만큼 프레임 할당
- 작업 세트가 메모리에 올라가지 않으면 프로세스 실행 중단 (swap out)
- 충분한 프레임이 확보되면 프로세스 재개

**장점**
- 페이지 폴트 감소
- 프로세스의 지역성(Locality) 활용

**단점**
- 작업 세트 크기 결정이 어려움
- 과거 참조 이력 추적 오버헤드

### 2. PFF (Page Fault Frequency)

**개념**
- **페이지 폴트 발생 비율**을 모니터링하여 프레임 할당 조절
- 상한선과 하한선 설정

**동작 원리**
- **페이지 폴트율이 상한선 초과**: 프로세스에 더 많은 프레임 할당
- 프레임 추가 할당 불가 시: 일부 프로세스 swap out
- **페이지 폴트율이 하한선 미달**: 프로세스가 너무 많은 프레임 보유
- 일부 프레임 회수하여 다른 프로세스에 할당

**장점**
- 직관적이고 구현이 간단
- 실시간으로 조절 가능

**단점**
- 상한선/하한선 설정이 중요
- 변동이 심한 환경에서는 불안정

### 3. 하드웨어적 해결

- **메모리 증설**: 물리 메모리 크기 증가
- **SSD 사용**: HDD보다 빠른 스왑 성능

**연관 개념**: [페이지 폴트](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%ED%8F%B4%ED%8A%B8), [캐시의 지역성](about:blank#%EC%BA%90%EC%8B%9C%EC%9D%98-%EC%A7%80%EC%97%AD%EC%84%B1)

# 캐시의 지역성

## 캐시의 지역성 원리(Locality)

**캐시 메모리**는 속도가 빠른 장치(CPU)와 느린 장치(메모리) 간의 속도 차이로 인한 병목 현상을 줄이기 위한 메모리이다.

캐시의 성능은 **CPU가 어떤 데이터를 원할 것인가를 얼마나 잘 예측**하느냐에 따라 좌우된다. 이를 위해 **지역성(Locality)의 원리**를 사용한다.

**지역성의 전제 조건**
- 프로그램은 모든 코드나 데이터를 균등하게 접근하지 않는다
- **Locality**: 어느 한 순간에 특정 부분을 집중적으로 참조하는 특성

**적중률(Hit Rate)**
- 캐시에서 원하는 데이터를 찾은 비율
- 지역성이 좋을수록 적중률이 높아짐

**연관 개념**: [메모리 계층](about:blank#%EB%A9%94%EB%AA%A8%EB%A6%AC-%EA%B4%80%EB%A6%AC-%EB%B0%B0%EA%B2%BD), [TLB](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%ED%85%8C%EC%9D%B4%EB%B8%94), [페이지 교체](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%EA%B5%90%EC%B2%B4-%EC%95%8C%EA%B3%A0%EB%A6%AC%EC%A6%98)

## 지역성의 종류

### 1. 시간 지역성(Temporal Locality)

**정의**: 최근에 참조된 주소의 내용은 곧 다음에 다시 참조되는 특성

**예시**
- **반복문(Loop)**`for (int i = 0; i < 100; i++) {       sum += i;  // 'sum'과 'i'가 반복 참조됨   }`
- **함수 호출**: 자주 호출되는 함수
- **변수 재사용**: 같은 변수를 여러 번 사용

**활용**: LRU 페이지 교체 알고리즘, 캐시 메모리

### 2. 공간 지역성(Spatial Locality)

**정의**: 참조된 주소와 인접한 주소의 내용이 다시 참조되는 특성

**예시**
- **배열 순회**`int arr[100];   for (int i = 0; i < 100; i++) {       arr[i] = i;  // 연속된 메모리 위치 접근   }`
- **순차적 코드 실행**: CPU는 다음 명령어를 순차적으로 실행
- **구조체 접근**: 구조체의 멤버들은 메모리에 연속 배치

**활용**: 프리페칭(Prefetching), 캐싱 라인

### 3. 순차 지역성(Sequential Locality)

**정의**: 데이터가 순차적으로 접근되는 특성 (공간 지역성의 특수한 경우)

**예시**
- 프로그램 카운터의 순차적 증가
- 순차적 파일 읽기

**지역성을 고려한 프로그래밍**

좋은 예 (지역성 높음):

```
// 공간 지역성 활용 (행 우선)
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        sum += arr[i][j];
    }
}
```

나쁜 예 (지역성 낮음):

```
// 공간 지역성 활용 못함 (열 우선)
for (int j = 0; j < N; j++) {
    for (int i = 0; i < N; i++) {
        sum += arr[i][j];  // 캐시 미스 빈번
    }
}
```

**연관 개념**: [캐싱 라인](about:blank#caching-line), [LRU](about:blank#lru), [프리페칭](about:blank#demand-paging)

## Caching Line

**캐싱 라인**은 캐시에 데이터를 저장할 때 특정 자료구조를 사용하여 **묶음**으로 저장하는 단위이다.

**캐싱 라인이 필요한 이유**
- 프로세스는 다양한 주소의 데이터를 사용
- 빈번하게 사용하는 데이터의 주소도 흩어져 있음
- 빠른 검색을 위해 태그(Tag)와 함께 저장

**구성 요소**
- **Tag**: 메모리 주소의 일부 (데이터 식별용)
- **Data**: 실제 캐시된 데이터
- **Valid Bit**: 캐시 라인이 유효한지 여부
- **Dirty Bit**: 캐시 데이터가 수정되었는지 여부

**연관 개념**: [캐시의 지역성](about:blank#%EC%BA%90%EC%8B%9C%EC%9D%98-%EC%A7%80%EC%97%AD%EC%84%B1), [메모리 계층](about:blank#%EB%A9%94%EB%AA%A8%EB%A6%AC-%EA%B4%80%EB%A6%AC-%EB%B0%B0%EA%B2%BD)

### 캐싱 라인의 종류

### 1. Full Associative (완전 연관)

**특징**
- 캐시의 어느 위치에나 데이터 저장 가능
- 가장 유연한 방식

**장점**
- 캐시 활용률이 높음
- 충돌(Conflict) 최소화

**단점**
- 모든 캐시 라인을 검색해야 함 (느림)
- 하드웨어 비용이 높음

**사용**: 작은 크기의 캐시 (TLB 등)

### 2. Direct Mapped (직접 매핑)

**특징**
- 각 메모리 블록이 캐시의 정해진 한 위치에만 매핑
- **캐시 인덱스 = (메모리 주소) % (캐시 라인 수)**

**장점**
- 검색이 매우 빠름 (한 번만 확인)
- 하드웨어 구현이 간단

**단점**
- 충돌(Conflict Miss) 빈번
- 캐시 활용률이 낮을 수 있음

**예시**

```
메모리 블록 0, 8, 16 → 캐시 라인 0
메모리 블록 1, 9, 17 → 캐시 라인 1
...
```

### 3. Set Associative (세트 연관)

**특징**
- Full Associative와 Direct Mapped의 절충안
- 캐시를 여러 세트로 나누고, 각 세트 내에서는 Full Associative
- **N-way Set Associative**: 각 세트에 N개의 캐시 라인

**동작 원리**
1. 메모리 주소로 세트 결정 (Direct Mapped 방식)
2. 해당 세트 내에서 검색 (Full Associative 방식)

**장점**
- Direct Mapped보다 충돌 적음
- Full Associative보다 빠른 검색

**단점**
- 하드웨어 복잡도 증가

**사용**: 대부분의 현대 CPU 캐시 (L1, L2, L3)

**일반적인 구성**
- L1 캐시: 8-way Set Associative
- L2 캐시: 16-way Set Associative
- L3 캐시: 16-way 또는 그 이상

**비교표**

| 구분 | Direct Mapped | Set Associative | Full Associative |
| --- | --- | --- | --- |
| 속도 | 가장 빠름 | 중간 | 가장 느림 |
| 충돌 | 많음 | 중간 | 적음 |
| 하드웨어 비용 | 낮음 | 중간 | 높음 |
| 캐시 활용률 | 낮음 | 중간 | 높음 |

**연관 개념**: [캐시의 지역성](about:blank#%EC%BA%90%EC%8B%9C%EC%9D%98-%EC%A7%80%EC%97%AD%EC%84%B1), [TLB](about:blank#%ED%8E%98%EC%9D%B4%EC%A7%80-%ED%85%8C%EC%9D%B4%EB%B8%94)

### 교착 상태

1. **교착 상태가 발생하는 4가지 필요 조건은 무엇인가요?**
    - 상호 배제, 점유 대기, 비선점, 순환 대기
2. **교착 상태를 해결하는 방법에는 어떤 것들이 있나요?**
    - 예방, 회피(은행원 알고리즘), 탐지, 복구, 무시
3. **은행원 알고리즘의 안전 상태란 무엇인가요?**
    - 모든 프로세스가 정상 종료 가능한 상태
4. **대부분의 운영체제가 교착 상태를 무시하는 이유는?**
    - 발생 빈도가 낮고, 예방/회피 비용이 더 큼

### IPC

1. **IPC의 종류와 각각의 특징을 설명해주세요.**
    - 공유 메모리, 파이프, 메시지 큐, 소켓, 파일
2. **공유 메모리가 가장 빠른 IPC 방식인 이유는?**
    - 커널을 거치지 않고 직접 메모리 접근
3. **파이프와 메시지 큐의 차이점은 무엇인가요?**
    - 단방향 vs 양방향, 바이트 스트림 vs 메시지 단위
4. **공유 메모리 사용 시 주의해야 할 점은?**
    - Race Condition, 동기화 필요

### 인터럽트와 시스템 콜

1. **인터럽트란 무엇이고, 어떻게 처리되나요?**
    - CPU 실행 중단, 인터럽트 핸들러 실행, 복귀
2. **하드웨어 인터럽트와 소프트웨어 인터럽트의 차이는?**
    - 외부 장치 vs 프로그램 내부, 비동기 vs 동기
3. **시스템 콜이 필요한 이유는 무엇인가요?**
    - 보안, 안정성, 추상화
4. **Mode Bit의 역할은 무엇인가요?**
    - 사용자 모드와 커널 모드 구분
5. **시스템 콜 발생 시 Mode Bit는 어떻게 변화하나요?**
    - 1(사용자) → 0(커널) → 1(사용자)

### 메모리 관리

1. **외부 단편화와 내부 단편화의 차이를 설명해주세요.**
    - 메모리 사이의 빈 공간 vs 할당된 메모리 내의 빈 공간
2. **페이징 기법이 외부 단편화를 해결하는 원리는?**
    - 고정 크기 블록, 비연속 할당
3. **페이징에서 내부 단편화가 발생하는 이유는?**
    - 프로세스 크기가 페이지 크기의 배수가 아닐 때
4. **페이징과 세그멘테이션의 차이는 무엇인가요?**
    - 고정 크기(물리적) vs 가변 크기(논리적)
5. **세그멘테이션의 장점은 무엇인가요?**
    - 논리적 단위 분할, 공유와 보호 용이

### 가상 메모리

1. **가상 메모리가 필요한 이유는 무엇인가요?**
    - 물리 메모리 크기 제약 극복, 다중 프로그래밍
2. **페이지 테이블의 역할과 구성 요소를 설명해주세요.**
    - 논리 주소 → 물리 주소 변환, Valid bit, 프레임 번호 등
3. **TLB란 무엇이고, 왜 필요한가요?**
    - 페이지 테이블 캐시, 주소 변환 속도 향상
4. **Demand Paging이란 무엇인가요?**
    - 필요한 페이지만 메모리에 적재
5. **페이지 폴트가 발생했을 때의 처리 과정은?**
    - 인터럽트 발생, 디스크에서 페이지 로드, 페이지 테이블 업데이트
6. **페이지 폴트율이 성능에 미치는 영향은?**
    - 디스크 I/O로 인한 큰 지연

### 페이지 교체 알고리즘

1. **FIFO의 Belady’s Anomaly란 무엇인가요?**
    - 프레임 수 증가 시 페이지 폴트 증가하는 현상
2. **LRU 알고리즘을 구현하는 방법은?**
    - 카운터 방식, 스택 방식
3. **OPT 알고리즘이 실제로 구현 불가능한 이유는?**
    - 미래의 페이지 참조를 알 수 없음
4. **LRU와 LFU의 차이점은 무엇인가요?**
    - 최근 사용 시점 vs 사용 빈도
5. **Clock 알고리즘(NUR)의 동작 원리를 설명해주세요.**
    - 참조 비트와 수정 비트, 원형 리스트

### 스레싱

1. **스레싱이란 무엇이고, 왜 발생하나요?**
    - 페이징이 실제 작업보다 많아지는 현상, 메모리 부족
2. **스레싱을 방지하는 방법은?**
    - 작업 세트, PFF, 메모리 증설
3. **작업 세트(Working Set)란 무엇인가요?**
    - 프로세스가 자주 참조하는 페이지 집합
4. **PFF에서 상한선과 하한선의 역할은?**
    - 프레임 할당 조절

### 캐시

1. **캐시의 지역성 원리를 설명해주세요.**
    - 시간 지역성, 공간 지역성
2. **시간 지역성과 공간 지역성의 예시를 들어주세요.**
    - 반복문의 변수, 배열 순회
3. **캐싱 라인의 3가지 방식과 각각의 특징은?**
    - Direct Mapped, Set Associative, Full Associative
4. **LRU 페이지 교체 알고리즘이 지역성과 어떤 관련이 있나요?**
    - 시간 지역성을 활용하여 과거 참조 패턴으로 미래 예측

### 통합 및 심화 질문

1. **프로세스 생성 시(fork) Copy-On-Write를 사용하는 이유는?**
    - 메모리 효율성, 불필요한 복사 방지
2. **컨텍스트 스위칭 시 캐시 메모리는 어떻게 되나요?**
    - 프로세스: 클리어, 스레드: 유지
3. **가상 메모리를 사용할 때 보안상 어떤 이점이 있나요?**
    - 프로세스 간 메모리 격리, 접근 권한 제어
4. **TLB Miss가 발생했을 때의 처리 과정은?**
    - 페이지 테이블 접근, TLB 업데이트
5. **페이지 크기를 크게 할 때와 작게 할 때의 장단점은?**
    - 페이지 테이블 크기, 내부 단편화, I/O 효율
6. **멀티코어 시스템에서 캐시 일관성(Cache Coherence) 문제는 무엇인가요?**
    - 여러 코어의 캐시에 같은 데이터가 다르게 저장되는 문제
7. **커널 스택과 사용자 스택의 차이는 무엇인가요?**
    - 접근 권한, 저장 정보
8. **메모리 계층 구조에서 각 계층의 역할은?**
    - 레지스터, 캐시, 메인 메모리, 보조 기억장치
9. **DMA(Direct Memory Access)란 무엇이고, 왜 필요한가요?**
    - CPU 개입 없이 I/O 장치와 메모리 간 직접 전송
10. **가상 메모리의 페이지 크기를 결정하는 요인은 무엇인가요?**
    - 내부 단편화, 페이지 테이블 크기, I/O 효율
11. **Real-time 시스템에서 페이지 폴트가 문제가 되는 이유는?**
    - 예측 불가능한 지연 시간
12. **페이지 교체 시 Dirty Bit가 중요한 이유는?**
    - 수정된 페이지는 디스크에 쓰기 필요, 미수정 페이지는 버려도 됨
13. **가상 메모리에서 Shared Library는 어떻게 공유되나요?**
    - 여러 프로세스의 가상 주소 공간이 같은 물리 메모리를 가리킴
14. **임계 영역 문제를 하드웨어적으로 해결하는 방법은?**
    - Test-and-Set, Compare-and-Swap 명령어
15. **프로세스 간 통신에서 동기 vs 비동기의 차이와 선택 기준은?**
    - 블로킹 여부, 응답 시간 요구사항
