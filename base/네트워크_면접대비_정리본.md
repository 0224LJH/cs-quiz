# OSI 7계층과 TCP/IP 4계층

## OSI 7계층

OSI(Open Systems Interconnection) 7계층 모델: 네트워크 통신을 표준화하기 위해 만들어진 개념적 모델

### 각 계층의 역할

| **계층** | **설명** | **역할** | **프로토콜 & 장비** | PDU |
| --- | --- | --- | --- | --- |
| **7계층: 어플리케이션** | 사용자와 직접 상호작용 하는 계층 |  | **HTTP**, **HTTPS**, **FTP**, **SMTP**, **DNS**, **SSH** 등 | 데이터 |
| **6계층: 표현**  | 데이터 형식을 변환하고 암호화/복호화 수행 | 데이터 인코딩, 암호화, 압축 | JPEG 등을 여기서.. |  |
| **5계층: 세션**  | 통신 세션을 구성하고 관리 | 세션 설정, 유지, 종료 | **NetBIOS**, **SSH**, **TLS** |  |
| **4계층: 전송**  | 종단간 신뢰성 있는 데이터 전송 담당,  ****포트 번호를 사용하여 프로세스 식별 | 흐름 제어, 오류 제어, 혼잡 제어 | **TCP**, **UDP** | 세그먼트, 데이터그램 |
| **3계층: 네트워크**  | 패킷을 목적까지 라우팅 | 논리적 주소(IP 주소) 지정, 경로 선택 |  **IP**, ICMP, ARP, RARP | 라우터 | 패킷(Packet) |
| **2계층: 데이터**  | 물리적 주소(MAC)를 사용하여 인접 노드 간 데이터 전송 | 프레임 구성, 오류 검출 | Ethernet, PPP | 스위치, 브리지 | 프레임(Frame) |
| **1계층: 물리**  | 실제 비트 스트림을 전기 신호로 변환 | 물리적 연결, 전기적 신호 전송 | 허브, 리피터, 케이블 | 비트(Bit) |

### TCP/IP 4계층

실제 인터넷에서 사용되는 프로토콜 스택

**4계층 - 애플리케이션 계층 (Application)**- OSI 5~7계층에 해당
→ HTTP, HTTPS, FTP, SMTP, DNS 등

**3계층 - 전송 계층 (Transport)** - OSI 4계층에 해당
→TCP, UDP

**2계층 - 인터넷 계층 (Internet)** - OSI 3계층에 해당
→ IP, ICMP, ARP

**1계층 - 네트워크 액세스 계층 (Network Access)** - OSI 1~2계층에 해당
→ Ethernet, Wi-Fi

## 캡슐화와 역캡슐화

- **캡슐화**: 상위 계층에서 하위 계층으로 데이터 전달 시 각 계층의 헤더를 추가
- **역캡슐화**: 하위 계층에서 상위 계층으로 데이터 전달 시 각 계층의 헤더를 제거

## 계층별 PDU (Protocol Data Unit)

- 애플리케이션: 데이터(Data)
- 전송: 세그먼트(Segment) - TCP, 데이터그램(Datagram) - UDP
- 네트워크: 패킷(Packet)
- 데이터링크: 프레임(Frame)
- 물리: 비트(Bit)

### 관련 개념

- **소켓(Socket)**: 네트워크 인터페이스(TCP, UDP, HTTP 등)를 기반으로 통신하는 것
- **포트 번호**: 애플리케이션을 구분하는 논리적 단위 (0~65535)
    - Well-known ports: 0~1023 (HTTP:80, HTTPS:443, FTP:21, SSH:22 등)
    - Registered ports: 1024~49151
    - Dynamic ports: 49152~65535

## HTTP의 GET과 POST 비교

둘 다 HTTP 프로토콜을 이용해서 서버에 무엇인가를 요청할 때 사용하는 방식입니다.

### GET

- 요청 **데이터가 HTTP Request Message의 Header**에 URL 형태로 담겨 전송됩니다
- URL 뒤에 `?`를 붙여 key=value 쌍으로 데이터가 노출됩니다
- **데이터 크기 제한**: URL 길이에 제약이 있어 전송 데이터 크기가 제한적입니다
- **보안**: **URL에 데이터가 그대로 노출**되므로 민감한 정보 전송에 부적절합니다
- **용도**: 서버로부터 데이터를 조회(SELECT)하는 용도
- **멱등성(Idempotent)**: 동일한 요청을 여러 번 해도 결과가 같습니다
- **캐싱**: 브라우저가 요청을 캐싱할 수 있어, 동일 요청 시 빠른 응답이 가능합니다

### POST

- 요청 데이터가 **HTTP Request Message의 Body**에 담겨 전송됩니다
- URL에 데이터가 노출되지 않습니다
- **데이터 크기**: GET보다 큰 데이터를 전송할 수 있으며, 바이너리 데이터도 가능합니다
- **보안**: URL에 노출되지 않지만, 암호화하지 않으면 여전히 보안에 취약합니다
- **용도**: 서버의 값이나 상태를 변경(INSERT, UPDATE, DELETE)하는 용도
- **멱등성 X**: 동일한 요청을 여러 번 하면 다른 결과가 나올 수 있습니다
- **캐싱 X**: 기본적으로 캐싱되지 않습니다

### GET vs POST 선택 기준

- 데이터를 조회만 한다면 → **GET**
- 서버의 상태를 변경한다면 → **POST**
- 보안이 중요한 데이터라면 → **POST** (단, HTTPS 사용 권장)
- 데이터가 크다면 → **POST**

### 주의사항

GET 방식으로 요청할 것을 POST로 잘못 사용하면 캐싱된 데이터가 응답될 수 있으므로 목적에 맞는 메서드를 사용해야 합니다.

### 기타 HTTP 메서드

- **PUT**: 리소스 전체 수정 (전체 교체)
- **PATCH**: 리소스 일부 수정
- **DELETE**: 리소스 삭제
- **HEAD**: GET과 동일하지만 응답 본문 없이 헤더만 반환
- **OPTIONS**: 서버가 지원하는 메서드 확인 (CORS preflight에서 사용)

# TCP와 UDP의 비교

## TCP (Transmission Control Protocol)

**연결 지향적 프로토콜**로 신뢰성 있는 데이터 전송을 보장합니다.

### 특징

- **연결 지향**: 3-way handshake로 연결 수립
- **신뢰성**: 데이터 전송 보장 (ACK, 재전송 메커니즘)
- **순서 보장**: 패킷이 순서대로 도착하도록 보장
- **흐름 제어**: 송신자와 수신자의 데이터 처리 속도 조절
- **혼잡 제어**: 네트워크 혼잡 상황 대응
- **전이중(Full-Duplex)**: 양방향 동시 통신 가능
- **점대점(Point-to-Point)**: 1:1 연결만 가능 (멀티캐스팅, 브로드캐스팅 불가)
- **헤더 크기**: 20~60 bytes (가변)

### 사용 사례

- 파일 전송 (FTP)
- 이메일 (SMTP)
- 웹 브라우징 (HTTP/HTTPS)
- 원격 접속 (SSH)

## UDP (User Datagram Protocol)

**비연결형 프로토콜**로 빠른 데이터 전송에 중점을 둡니다.

### 특징

- **비연결**: 연결 수립 과정 없음
- **비신뢰성**: 데이터 전송 보장 안 함 (ACK 없음, 재전송 없음)
- **순서 보장 안 함**: 패킷이 순서대로 도착하지 않을 수 있음
- **흐름 제어 없음**
- **혼잡 제어 없음**
- **빠른 전송**: 오버헤드가 적어 속도가 빠름
- **1:1, 1:N, N:N 통신**: 브로드캐스팅, 멀티캐스팅 가능
- **헤더 크기**: 8 bytes (고정)

### 사용 사례

- DNS (빠른 응답 필요, 재요청 가능)
- 실시간 스트리밍 (비디오, 음성)
- 온라인 게임
- IoT 센서 데이터 전송

## TCP vs UDP 선택 기준

| 구분 | TCP | UDP |
| --- | --- | --- |
| 신뢰성 | 필요 | 불필요 |
| 순서 | 중요 | 상관없음 |
| 속도 | 느림 | 빠름 |
| 용도 | 정확한 데이터 전송 | 실시간 전송 |

### 체크섬

- **역할**: 데이터 전송 중 오류 검출
- TCP/UDP 모두 체크섬을 사용하여 데이터 무결성 확인
- **알고리즘**: 1의 보수합 사용
    - IPv4 헤더: 1의 보수합
    - TCP/UDP 헤더: 1의 보수합
    - Ethernet 프레임: CRC-32

# TCP 3-way Handshake와 4-way Handshake

## TCP 3-way Handshake (연결 설정)

TCP는 신뢰성 있는 연결을 위해 **3-way handshake** 과정을 거쳐 연결을 수립합니다.

상태 초기

- **클라이언트**: `CLOSED`
- **서버**: `LISTEN`

---

1단계: 클라이언트 → 서버 (연결 요청) : 클라이언트가 먼저 연결을 요청.

- 플래그: **SYN = 1**
- **SEQ = c_isn** (client initial sequence number)
- **ACK 필드는 사용되지 않거나 0**
- 이 패킷을 보내면서:
    - 클라이언트 상태: `CLOSED → SYN-SENT`

---

2단계: 서버 → 클라이언트 (요청 수락 + 자기 ISN 전달) : 서버가 SYN을 받고 응답:

- 플래그: **SYN = 1, ACK = 1**
- **SEQ = s_isn** (server initial sequence number)
- **ACK = c_isn + 1**
    - “네가 보낸 SYN의 시퀀스(c_isn)를 잘 받았고,
        
        이제 그 다음 바이트(c_isn+1)부터 기대할게” 라는 뜻
        
- 이 패킷을 보내면서:
    - 서버 상태: `LISTEN → SYN-RECEIVED`

---

3단계: 클라이언트 → 서버 (최종 확인 ACK): 클라이언트가 서버의 SYN+ACK를 받고 마지막 ACK 전송

- 플래그: **ACK = 1**
- **SEQ = c_isn + 1**
    - 이제 클라이언트 쪽 시퀀스는 **자신의 ISN 다음 바이트부터** 시
- **ACK = s_isn + 1**
    - “네가 보낸 SYN(s_isn)을 잘 받았고,
        
        이제 그 다음 바이트(s_isn+1)부터 기대할게”
        
- 이 패킷을 보내는 순간:
    - 클라이언트 상태: `SYN-SENT → ESTABLISHED`
    - 서버는 이 ACK를 **수신한 후**: `SYN-RECEIVED → ESTABLISHED`
    
    (3번째 ACK에 데이터를 같이 실을 수도 있지만, 필수는 아님)
    

---

### 이 과정을 통해 보장되는 것

1. **서로의 발신/수신 능력 확인**
    - 내가 보낸 걸 상대가 받고, 그에 대한 ACK를 다시 나에게 보내는 왕복 검증 완료
2. **양쪽의 시퀀스 번호 동기화**
    - 클라이언트는 “서버 다음 시퀀스 = `s_isn + 1`”
    - 서버는 “클라이언트 다음 시퀀스 = `c_isn + 1`”
3. 그래서 이 3단계가 끝난 시점부터:
    - 양쪽 모두 **신뢰할 수 있는, 순서 있는 바이트 스트림** 위에서
        
        데이터 전송을 시작할 수 있게 됨.
        

## TCP 4-way Handshake (연결 종료)

TCP 연결을 안전하게 종료하기 위한 과정입니다.

### 과정

초기과정

- **클라이언트**: `ESTABLISHED`
- **서버**: `ESTABLISHED`

---

1단계: 클라이언트 → 서버 : 클라이언트가 더 이상 보낼 데이터가 없어서 종료 요청:

- 플래그: **FIN = 1, ACK = 1**
- **SEQ = 클라이언트의 현재 시퀀스 번호 (c_seq)**
- **ACK = 서버 쪽에서 마지막으로 받은 다음 바이트 번호**
- 이 패킷을 보내면서:
    - 클라이언트 상태: `ESTABLISHED → FIN-WAIT-1`

---

2단계: 서버 → 클라이언트 (FIN에 대한 ACK): 서버가 클라이언트의 FIN을 받으면:

- 플래그: **ACK = 1**
- **SEQ = 서버의 현재 시퀀스 번호 (s_seq)**
- **ACK = c_seq + 1**
    - 클라이언트가 보낸 FIN 1바이트를 잘 받았음을 의미
- 이때:
    - 서버 상태: `ESTABLISHED → CLOSE-WAIT`
    - 클라이언트 상태: `FIN-WAIT-1 → FIN-WAIT-2`

※ 이 상태에서 서버는 **애플리케이션 레벨에서 마지막 데이터 전송/정리**를 할 수 있음.

---

3단계: 서버 → 클라이언트 (서버도 이제 끊을게 – FIN)

- 서버도 할 일 다 끝났다고 판단되면:
    - 플래그: **FIN = 1, ACK = 1**
    - **SEQ = s_fin_seq** (서버가 FIN을 보낼 때의 시퀀스 번호)
    - **ACK = c_seq + 1** (보통 그대로 유지)
- 이 패킷을 보내면서:
    - 서버 상태: `CLOSE-WAIT → LAST-ACK`

---

4단계: 클라이언트 → 서버 (서버의 FIN에 대한 ACK): 

- 클라이언트가 서버의 FIN을 받으면
    - 플래그: **ACK = 1**
    - **SEQ = c_seq + 1** (또는 그 이후, 실제 구현에 따라)
    - **ACK = s_fin_seq + 1**
- 이 ACK를 보낸 뒤:
    - 클라이언트 상태: `FIN-WAIT-2 → TIME-WAIT`
    - 서버는 이 ACK를 받는 순간: `LAST-ACK → CLOSED`

클라이언트는 **TIME-WAIT 상태**에서 일정 시간(2MSL) 대기 후 최종적으로 `CLOSED`로 간다.

---

## 한 줄 요약 흐름

1. **클라이언트 → 서버 : FIN** → 나 더 안 보낼게
2. **서버 → 클라이언트 : ACK** → 알겠다, 네 쪽 송신 종료 인정
3. **서버 → 클라이언트 : FIN** → 나도 이제 더 안 보낼게
4. **클라이언트 → 서버 : ACK** → 알겠다, 우리 둘 다 끝

이렇게 해서:

- **양쪽 송신 방향을 각각 한 번씩 종료(FIN)**
- 각 FIN마다 **ACK 한 번씩** → 총 4번 왕복 → **4-way handshake**

### TIME_WAIT 상태

- 클라이언트는 마지막 ACK 전송 후 일정 시간(2*MSL, Maximum Segment Life Time) 동안 TIME_WAIT 상태 유지
- 이유:
    - 마지막 ACK가 유실될 경우를 대비
    - 지연 패킷 문제 방지

# HTTP와 HTTPS

## HTTP의 문제점

1. **평문 통신**: 도청 가능
2. **통신 상대 미확인**: 위장 가능
3. **완전성 미보장**: 변조 가능

## HTTPS (HTTP Secure)

HTTP에 **SSL/TLS** 계층을 추가하여 보안을 강화한 프로토콜입니다.

### 동작 원리

1. HTTP는 원래 TCP와 직접 통신
2. HTTPS는 HTTP와 TCP 사이에 SSL/TLS 추가
3. **HTTP → SSL/TLS → TCP** 순서로 통신

### SSL/TLS 동작 과정

### 1. (구) TLS 1.0/1.1/1.2 – RSA 기반 키 교환 흐름

1. **ClientHello**
    - 클라이언트 → 서버
    - 지원하는 TLS 버전, 암호 스위트 목록, `Client Random` 전송
2. **ServerHello + Certificate**
    - 서버 → 클라이언트
    - 선택한 TLS 버전, 암호 스위트, `Server Random` 전송
    - 서버 인증서(Certificate, 서버 공개키 포함) 전송
3. **인증서 검증**
    - 클라이언트가 인증서의 서명, 유효기간, 도메인 등을 검증
    - 신뢰 가능하면 서버의 공개키를 사용 가능 상태로 둠
4. **Pre-Master Secret 전송**
    - 클라이언트가 `Pre-Master Secret`(랜덤 값) 생성
    - 서버의 공개키로 `Pre-Master Secret` 암호화 후 서버로 전송
5. **세션키(대칭키) 생성**
    - 클라이언트/서버 모두:
        - `Client Random`, `Server Random`, `Pre-Master Secret`을 이용해
        - 동일한 `Master Secret` 및 세션용 대칭키들 파생
6. **Finished 교환 후 암호화 통신**
    - 양측이 `Finished` 메시지를 주고받아 핸드셰이크 무결성 확인
    - 이후 애플리케이션 데이터는 세션키로 암호화/복호화하며 통신

---

### 2. (현행) TLS 1.2(ECDHE) / TLS 1.3 – ECDHE 기반 키 교환 흐름

1. **ClientHello**
    - 클라이언트 → 서버
    - 지원 TLS 버전, 암호 스위트, `Client Random` 전송
    - 사용할 ECDHE 그룹 및 **클라이언트 ECDHE 공개키** 전송
2. **ServerHello + Certificate + ServerKeyShare**
    - 서버 → 클라이언트
    - 선택한 TLS 버전, 암호 스위트, `Server Random` 전송
    - 서버 인증서(Certificate) 전송
    - **서버 ECDHE 공개키** 전송
3. **인증서 검증**
    - 클라이언트가 서버 인증서를 검증(CA 서명, 유효기간, 도메인 등)
    - 서버의 정당성을 확인
4. **공통 비밀(Shared Secret) 계산**
    - 클라이언트:
        - 자신의 ECDHE 비밀키 + 서버 ECDHE 공개키 → 공통 비밀 K 계산
    - 서버:
        - 자신의 ECDHE 비밀키 + 클라이언트 ECDHE 공개키 → 동일한 K 계산
5. **세션키(대칭키) 파생**
    - 양측 모두:
        - 공통 비밀 K와 랜덤 값들(`Client Random`, `Server Random` 등)을 사용
        - 동일한 세션용 대칭키들(암호화 키, 무결성 키 등) 파생
6. **Finished 교환 후 암호화 통신**
    - 양측이 `Finished` 메시지 교환으로 핸드셰이크 무결성 검증
    - 이후 모든 애플리케이션 데이터는 세션키로 암호화/복호화하며 통신

→ 즉 기존에는 직접 Pre-master secret이라는 공통 비밀 값을 클라이언트가 보냈다면, 지금은 Diffie-Hellman이라는 수학적 성질을 통해 각자의 시크릿키와, 받은 공통키를 통해 같은 값을 만들어내는 것!

- **공개키 암호화**: 세션키 교환에만 사용 (느리지만 안전)
- **대칭키 암호화**: 실제 데이터 전송에 사용 (빠름)

### 인증서 (Certificate)

- **발급 기관**: CA (Certificate Authority)
    - 예: Let’s Encrypt, DigiCert, Comodo
- **포함 정보**: 서버 도메인, 서버 공개키, 유효기간, CA 서명
    
    

### HTTPS의 장점

1. **암호화**: 데이터 도청 방지
2. **인증**: 통신 상대 확인
3. **무결성**: 데이터 변조 방지 (다이제스트 기능)

## 모든 웹에서 HTTPS를 사용해야 할까?

- **과거**: CPU/메모리 부담으로 민감한 데이터만 HTTPS 사용
- **현재**: 하드웨어 발달로 성능 저하 거의 없음
- **HTTP/2.0과 함께 사용 시**: HTTPS가 HTTP보다 더 빠를 수 있음
- **결론**: 모든 웹 페이지에서 HTTPS 사용 권장

### 포트 번호

- HTTP: 80번 포트 (기본)
- HTTPS: 443번 포트 (기본)

# HTTP 버전별 차이

## HTTP/1.0

- **특징**: 연결당 하나의 요청과 응답 처리 
(Keep-alive라는 옵션이 있긴했지만, 기본값이  OFF)
- **문제점**:
    - 매 요청마다 새로운 TCP 연결 필요
    - 3-way handshake 반복으로 성능 저하
    - RTT(Round Trip Time) 증가

## HTTP/1.1

- **Persistent Connection**: 한 번의 TCP 연결로 여러 요청 처리
- **Pipelining**: 응답을 기다리지 않고 연속으로 요청 전송
- **HOL Blocking 문제**:
    - Head Of Line Blocking
    - 앞선 요청의 응답이 지연되면 뒤의 요청도 대기

## HTTP/2.0 (2015년)

### 주요 개선사항

**1. 멀티플렉싱 (Multiplexing)**
- 단일 TCP 연결에서 여러 HTTP 요청과 응답을 비동기로 처리
- 스트림 기반 프레임 전송
- HOL Blocking 해결(완벽히는 아님. 결국 순서가 있기에)

**2. 바이너리 프레이밍 (Binary Framing)**
- HTTP/1.1: 텍스트 메시지 전송
- HTTP/2.0: 바이너리 데이터로 변환
- 메시지 → HEADERS frame + DATA frame

**3. 헤더 압축 (Header Compression)**
- HPACK 알고리즘 사용
- 중복 헤더 제거, 헤더 테이블 유지

**4. 서버 푸시 (Server Push)**
- 클라이언트 요청 없이 서버가 리소스를 미리 전송
- 예: HTML 요청 시 CSS, JS 파일을 함께 전송

**5. 스트림 우선순위**
- 각 스트림에 가중치와 의존성 설정
- 중요한 리소스 우선 전송

## HTTP/3.0

### QUIC 프로토콜 기반

- **전송 계층**: TCP 대신 **UDP** 사용
- **장점**:
    - 연결 수립 시간 단축 (0-RTT, 1-RTT)
    - HOL Blocking 완전 해결
    - 패킷 손실에 강함
    - 연결 마이그레이션 (IP 변경 시에도 연결 유지)

### 주요 특징

- TLS 1.3 기본 내장
- 독립적인 스트림 (스트림별 흐름 제어)
- 향상된 혼잡 제어

# IP주소, MAC주소, ARP

## IP 주소 (Internet Protocol Address)

**논리적 주소**로 네트워크 계층(Layer 3)에서 사용됩니다.

### 특징

- 컴퓨터 네트워크에서 장치들이 서로 인식하고 통신하기 위한 번호
- **변경 가능**: 네트워크 환경에 따라 동적/정적 할당
- **32비트 (IPv4)** 또는 **128비트 (IPv6)**

### IPv4

- 형식: `192.168.0.1` (8비트.8비트.8비트.8비트)
- 범위: 0.0.0.0 ~ 255.255.255.255
- **클래스 구분**:
    - A 클래스: 0.0.0.0 ~ 127.255.255.255
    - B 클래스: 128.0.0.0 ~ 191.255.255.255
    - C 클래스: 192.0.0.0 ~ 223.255.255.255

### 특수 IP 주소

- **127.0.0.1**: 루프백(Loopback) IP, localhost
- **0.0.0.0**: 모든 IP 주소 (서버 바인딩 시)
- **사설 IP**:
    - 10.0.0.0 ~ 10.255.255.255
    - 172.16.0.0 ~ 172.31.255.255
    - 192.168.0.0 ~ 192.168.255.255

## MAC 주소 (Media Access Control Address)

**물리적 주소**로 데이터링크 계층(Layer 2)에서 사용됩니다.

### 특징

- 네트워크 인터페이스(NIC)에 할당된 고유 식별자
- **48비트** (6 octet)
- 형식: `04:42:1A:0E:17:85`
- **변경 불가**: 제조사에서 할당 (하드웨어 주소)

### 구성

- **앞 24비트(3 octet)**: OUI (조직 고유 식별자, 제조사 코드)
- **뒤 24비트(3 octet)**: UAA (제조사에서 구별되는 코드)

## ARP (Address Resolution Protocol)

**IP 주소를 MAC 주소로 변환**하는 프로토콜입니다.

### 동작 과정

1. **ARP Request (브로드캐스트)**
    - “IP 주소 141.23.56.23을 가진 장치의 MAC 주소를 찾습니다”
    - 네트워크의 모든 장치에 브로드캐스트
2. **ARP Reply (유니캐스트)**
    - 해당 IP를 가진 장치만 응답
    - “제 MAC 주소는 A4:6E:F4:59:83:AB입니다”
3. **ARP 테이블 캐싱**
    - IP-MAC 주소 매핑 정보를 캐시에 저장
    - 일정 시간 후 만료 (보통 수분)

### ARP 테이블 확인

```bash
# Windowsarp -a# Linux/Macarp -n
```

### RARP (Reverse ARP)

**MAC 주소를 IP 주소로 변환**하는 프로토콜입니다.
- 디스크가 없는 시스템이 자신의 IP 주소를 찾을 때 사용
- 현재는 DHCP로 대체

### ICMP (Internet Control Message Protocol)

**네트워크 진단 및 오류 보고** 프로토콜입니다.

### 역할

- 네트워크와 노드 사이에서 통신이 잘되나를 확인
- 오류 메시지 전달 (Destination Unreachable, Time Exceeded 등)

### Ping 명령어

- ICMP Echo Request/Reply 사용
- 네트워크 연결 상태 확인
- RTT(Round Trip Time) 측정

```bash
ping www.google.com
```

# 쿠키와 세션

HTTP는 **Stateless** 프로토콜이므로 상태 정보를 저장하지 않습니다. 이를 보완하기 위해 쿠키와 세션을 사용합니다.

## 쿠키 (Cookie)

**클라이언트 측**에 저장되는 작은 데이터 조각입니다.

### 특징

- 클라이언트(브라우저)에 저장
- 서버가 `Set-Cookie` 헤더로 설정
- 클라이언트가 자동으로 HTTP 요청 시 Cookie 헤더에 포함
- **크기 제한**: 최대 4KB
- **만료 시간**: 설정 가능

### 쿠키 속성

- **Domain**: 쿠키가 전송될 도메인
- **Path**: 쿠키가 전송될 경로
- **Expires/Max-Age**: 만료 시간
- **Secure**: HTTPS에서만 전송
- **HttpOnly**: JavaScript에서 접근 불가 (XSS 공격 방지)
- **SameSite**: CSRF 공격 방지

### 용도

- 로그인 정보 유지
- 장바구니
- 사용자 커스터마이징 (언어 설정 등)
- 방문 기록 추적

## 세션 (Session)

**서버 측**에 저장되는 상태 정보입니다.

### 특징

- 서버 메모리 또는 DB에 저장
- **Session ID**만 클라이언트에 쿠키로 저장
- 보안성이 쿠키보다 우수
- 서버 자원 사용

### 동작 과정

1. **로그인 성공**
    - 서버가 Session ID 생성
    - Session ID를 DB에 저장 (사용자 정보와 매핑)
    - `Set-Cookie`로 Session ID 전달
2. **이후 요청**
    - 클라이언트가 Cookie에 Session ID 포함하여 요청
    - 서버가 Session ID로 사용자 확인
3. **로그아웃**
    - 서버에서 Session 삭제

## 쿠키 vs 세션

| 구분 | 쿠키 | 세션 |
| --- | --- | --- |
| 저장 위치 | 클라이언트 | 서버 |
| 보안 | 취약 | 상대적으로 안전 |
| 속도 | 빠름 | 느림 |
| 용량 제한 | 4KB | 제한 없음 |

## JWT (JSON Web Token)

세션의 대안으로 **토큰 기반 인증**을 사용합니다.

### 구조

- **Header**: 토큰 타입, 알고리즘
- **Payload**: 사용자 정보 (claim)
- **Signature**: 위변조 방지

### 장점

- 서버에 상태 저장 불필요 (Stateless)
- 확장성 우수
- 다른 도메인 간 인증 가능

### 단점

- 토큰 크기가 큼
- Payload 정보 노출 (암호화 필요)
- 토큰 탈취 시 무효화 어려움

# REST API

## REST (Representational State Transfer)

**자원을 이름으로 구분하여 해당 자원의 상태를 주고받는 아키텍처 스타일**입니다.

### REST의 구성 요소

1. **자원 (Resource)**: URI로 표현
2. **행위 (Verb)**: HTTP 메서드로 표현
3. **표현 (Representation)**: JSON, XML 등

## REST API 설계 원칙

### 1. URI는 리소스를 표현

```
✅ GET /users/1
❌ GET /getUser/1
```

### 2. 리소스에 대한 행위는 HTTP 메서드로 표현

```
✅ POST /users
✅ GET /users/1
✅ PUT /users/1
✅ DELETE /users/1
❌ GET /users/delete/1
```

### 3. 계층 관계 표현

```
GET /users/1/orders
GET /users/1/orders/100
```

### 4. URI는 소문자 사용

```
✅ /users
❌ /Users
```

### 5. 하이픈(-) 사용, 언더스코어(_) 지양

```
✅ /user-info
❌ /user_info
```

### 6. 파일 확장자 포함 금지

```
✅ GET /users/1
❌ GET /users/1.json
```

## HTTP 메서드별 역할

- **GET**: 조회 (Read)
- **POST**: 생성 (Create)
- **PUT**: 전체 수정 (Update - Replace)
- **PATCH**: 일부 수정 (Update - Modify)
- **DELETE**: 삭제 (Delete)

## HTTP 상태 코드

### 2xx (성공)

- **200 OK**: 요청 성공
- **201 Created**: 리소스 생성 성공
- **204 No Content**: 요청 성공, 응답 본문 없음

### 3xx (리다이렉션)

- **301 Moved Permanently**: 영구 이동
- **302 Found**: 임시 이동
- **304 Not Modified**: 캐시된 데이터 사용

### 4xx (클라이언트 오류)

- **400 Bad Request**: 잘못된 요청
- **401 Unauthorized**: 인증 필요
- **403 Forbidden**: 권한 없음
- **404 Not Found**: 리소스 없음
- **405 Method Not Allowed**: 허용되지 않은 메서드
- **409 Conflict**: 리소스 충돌

### 5xx (서버 오류)

- **500 Internal Server Error**: 서버 내부 오류
- **502 Bad Gateway**: 게이트웨이 오류
- **503 Service Unavailable**: 서비스 이용 불가

# CORS

## CORS (Cross-Origin Resource Sharing)

**교차 출처 리소스 공유** - 다른 출처의 리소스에 접근할 수 있는 권한을 부여하는 메커니즘입니다.

### 출처 (Origin)

`프로토콜 + 도메인 + 포트`의 조합입니다.

```
https://example.com:443/path?query=value

프로토콜: https
도메인: example.com
포트: 443
```

### 동일 출처 정책 (SOP: Same-Origin Policy)

브라우저는 보안상의 이유로 **다른 출처로의 요청을 제한**합니다.

### 예시

```
현재 페이지: https://example.com

✅ https://example.com/api (동일 출처)
❌ http://example.com (프로토콜 다름)
❌ https://api.example.com (도메인 다름)
❌ https://example.com:3000 (포트 다름)
```

## CORS 동작 과정

### 1. Simple Request

다음 조건을 모두 만족하면 Preflight 없이 바로 요청합니다.
- 메서드: GET, HEAD, POST
- Content-Type:
- application/x-www-form-urlencoded
- multipart/form-data
- text/plain
- 커스텀 헤더 없음

### 2. Preflight Request

Simple Request 조건을 만족하지 않으면 **OPTIONS 메서드로 사전 요청**을 보냅니다.

**과정:**
1. **OPTIONS 요청 (Preflight)**`OPTIONS /api/data HTTP/1.1    Origin: http://127.0.0.1:3000    Access-Control-Request-Method: POST    Access-Control-Request-Headers: content-type`

1. **서버 응답**
    
    ```
    HTTP/1.1 200 OK
    Access-Control-Allow-Origin: http://127.0.0.1:3000
    Access-Control-Allow-Methods: POST, GET, OPTIONS
    Access-Control-Allow-Headers: content-type
    Access-Control-Max-Age: 86400
    ```
    
2. **실제 요청**
    - Preflight 승인되면 실제 요청 전송

## CORS 헤더

### 요청 헤더

- **Origin**: 요청 출처
- **Access-Control-Request-Method**: 실제 요청의 메서드
- **Access-Control-Request-Headers**: 실제 요청의 추가 헤더

### 응답 헤더

- **Access-Control-Allow-Origin**: 허용할 출처
    - 특정 출처: `https://example.com`
    - 모든 출처:
- **Access-Control-Allow-Methods**: 허용할 메서드
- **Access-Control-Allow-Headers**: 허용할 헤더
- **Access-Control-Allow-Credentials**: 쿠키 전송 허용 여부
- **Access-Control-Max-Age**: Preflight 캐싱 시간

## CORS 해결 방법

### 1. 서버 측

```jsx
// Node.js Expressapp.use((req, res, next) => {
  res.header('Access-Control-Allow-Origin', 'http://127.0.0.1:3000');  res.header('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE');  res.header('Access-Control-Allow-Headers', 'Content-Type');  next();});
```

### 2. 프록시 서버 사용

- 개발 환경에서 Webpack Dev Server 등의 프록시 활용

### 3. CORS 미들웨어 사용

```jsx
const cors = require('cors');app.use(cors({
  origin: 'http://127.0.0.1:3000',  credentials: true}));
```

# 로드밸런싱

## 로드밸런싱 (Load Balancing)

**여러 서버에 트래픽을 분산**하여 서버의 부하를 분산시키는 기술입니다.

## 로드밸런서의 종류

### L4 로드밸런서 (Transport Layer)

- **전송 계층(Layer 4)**에서 동작
- **IP 주소와 포트 번호**로 트래픽 분산
- TCP/UDP 패킷 정보 기반
- 빠른 처리 속도

### L7 로드밸런서 (Application Layer)

- **응용 계층(Layer 7)**에서 동작
- **HTTP 헤더, URL, 쿠키** 등으로 트래픽 분산
- 콘텐츠 기반 라우팅 가능
- 세밀한 트래픽 제어

## 로드밸런싱 알고리즘

### 1. Round Robin

- 순차적으로 서버에 요청 분배
- 가장 단순하고 널리 사용
- 서버 성능이 동일할 때 효과적

### 2. Weighted Round Robin (WRR)

- 서버마다 가중치를 부여
- 성능이 좋은 서버에 더 많은 요청 분배

### 3. Least Connection

- 현재 연결 수가 가장 적은 서버에 분배
- 세션이 긴 경우 유용
- 실시간 connection 수 관리 필요

### 4. Weighted Least Connection

- Least Connection + 가중치
- 서버 성능 차이와 연결 수를 모두 고려

### 5. IP Hash

- 클라이언트 IP 주소를 해싱하여 서버 할당
- 같은 클라이언트는 항상 같은 서버로 연결
- 세션 유지 가능

### 6. Least Response Time

- 응답 시간이 가장 빠른 서버에 분배
- 서버 상태를 동적으로 반영

### 로드밸런서의 장점

1. **가용성 향상**: 서버 장애 시 다른 서버로 트래픽 전환
2. **확장성**: 서버 추가로 처리량 증가
3. **유연성**: 유지보수 시 무중단 서비스 가능

## 헬스 체크 (Health Check)

- 로드밸런서가 정기적으로 서버 상태 확인
- 장애 서버는 트래픽 분배에서 제외
- **Active Health Check**: 주기적으로 요청 전송
- **Passive Health Check**: 실제 요청의 응답으로 판단

# 프록시 서버

## 프록시 서버 (Proxy Server)

클라이언트와 서버 사이에서 **중계 역할**을 하는 서버입니다.

## Forward Proxy

**클라이언트 앞**에 위치하여 클라이언트를 대신해 요청을 전달합니다.

### 특징

- 클라이언트가 프록시 서버를 통해 인터넷 접속
- 서버는 프록시 서버의 IP만 확인 가능

### 용도

1. **캐싱**: 자주 요청되는 콘텐츠 캐싱으로 속도 향상
2. **보안**: 클라이언트 IP 숨김
3. **접근 제어**: 특정 사이트 차단
4. **로깅**: 사용자 활동 기록

### 예시

- 회사/학교 네트워크의 프록시
- VPN

## Reverse Proxy

**서버 앞**에 위치하여 서버를 대신해 요청을 받습니다.

### 특징

- 클라이언트는 프록시 서버만 인식
- 실제 서버의 존재를 숨김

### 용도

1. **로드밸런싱**: 여러 서버로 트래픽 분산
2. **캐싱**: 정적 콘텐츠 캐싱
3. **보안**: 서버 IP 숨김, DDoS 방어
4. **SSL 종료**: HTTPS 암호화/복호화 처리
5. **압축**: 응답 데이터 압축

### 예시

- Nginx, Apache (리버스 프록시 모드)
- CDN (Content Delivery Network)
- Cloudflare

### Forward Proxy vs Reverse Proxy

| 구분 | Forward Proxy | Reverse Proxy |
| --- | --- | --- |
| 위치 | 클라이언트 앞 | 서버 앞 |
| 보호 대상 | 클라이언트 | 서버 |
| 숨기는 대상 | 클라이언트 IP | 서버 IP |
| 사용 목적 | 접근 제어, 캐싱 | 로드밸런싱, 보안 |

# DNS Round Robin 방식

## DNS (Domain Name System)

도메인 이름을 IP 주소로 변환하는 시스템입니다.

## DNS Round Robin

**하나의 도메인에 여러 IP 주소를 매핑**하여 부하를 분산하는 방식입니다.

### 동작 방식

1. DNS 서버에 하나의 도메인에 여러 IP 등록
    
    ```
    example.com → 192.168.0.1
    example.com → 192.168.0.2
    example.com → 192.168.0.3
    ```
    
2. DNS 쿼리마다 IP 주소를 순환하여 반환
3. 클라이언트는 받은 IP로 접속

## DNS Round Robin의 문제점

### 1. 서버 수만큼 공인 IP 필요

- 부하 분산을 위해 서버를 늘리려면 공인 IP도 늘려야 함
- 비용 증가

### 2. 균등 분산 불가

- **DNS 캐싱**:
    - 클라이언트와 중간 DNS 서버가 응답을 캐싱
    - 같은 클라이언트는 계속 같은 서버로 접속
- **TTL 설정 단축으로 어느 정도 해결 가능하지만 완벽하지 않음**

### 3. 서버 다운 감지 불가

- DNS는 서버의 상태를 모름
- 장애 서버로도 트래픽이 계속 분배됨
- **헬스 체크 기능 없음**

## DNS 스케줄링 알고리즘

### Weighted Round Robin (WRR)

- 각 서버에 가중치 부여
- 성능 좋은 서버에 더 많은 트래픽 분배
- DNS 응답 비율 조정

### Least Connection

- 연결 수가 가장 적은 서버 선택
- 실시간 connection 관리 필요
- DNS 단독으로는 구현 어려움 (로드밸런서와 함께 사용)

### DNS Round Robin의 대안

- **로드밸런서 사용**: L4/L7 로드밸런서로 정교한 분산
- **헬스 체크**: 장애 서버 자동 제외
- **세션 유지**: Sticky Session 지원

# 웹 통신의 큰 흐름

*사용자가 Chrome 브라우저 주소창에 `www.google.com`을 입력하면 어떤 일이 일어날까?*

### 1. 브라우저 단계

1. **URL 파싱**
    - 프로토콜, 도메인, 포트, 경로 분석
    - `https://www.google.com:443/search?q=hello`
2. **HTTP Request 메시지 생성**
    - 요청 라인: `GET /search?q=hello HTTP/1.1`
    - 헤더: Host, User-Agent, Accept 등
3. **OS에 메시지 전송 의뢰**
    - 브라우저는 네트워크 송출 기능이 없음
    - OS의 TCP/IP 프로토콜 스택 사용

### 2. DNS 조회

1. **브라우저 DNS 캐시 확인**
2. **OS DNS 캐시 확인**
3. **로컬 hosts 파일 확인**
4. **DNS 서버에 쿼리**
    - 재귀적 쿼리 (Recursive Query)
    - 루트 DNS → TLD DNS → 권한 있는 DNS
5. **IP 주소 획득**
    - `www.google.com` → `142.250.196.46`

### 3. 프로토콜 스택 & LAN 어댑터

1. **TCP 연결 수립 (3-way handshake)**
    - 프로토콜 스택이 SYN, SYN-ACK, ACK 교환
2. **데이터 분할 및 패킷 생성**
    - HTTP 메시지를 MSS 크기로 분할
    - TCP 헤더 추가 (출발지/목적지 포트, 시퀀스 번호 등)
3. **IP 패킷 생성**
    - IP 헤더 추가 (출발지/목적지 IP, TTL 등)
4. **MAC 주소 확인**
    - ARP 프로토콜로 다음 홉(라우터)의 MAC 주소 획득
5. **프레임 생성**
    - Ethernet 헤더 추가 (출발지/목적지 MAC 주소)
6. **전기 신호 변환**
    - LAN 어댑터가 프레임을 전기 신호로 변환
    - 케이블로 송출

### 4. 스위치와 라우터

1. **스위치**
    - MAC 주소 기반 포워딩
    - 동일 네트워크 내 전달
2. **라우터**
    - IP 주소 기반 라우팅
    - 라우팅 테이블 참조
    - 최적 경로 선택
    - TTL 감소
3. **다음 홉으로 전달**
    - 인터넷 핵심부로 진입

### 5. 인터넷 핵심부

1. **ISP(인터넷 서비스 제공자)**
    - KT, SK브로드밴드, LG U+ 등
2. **다수의 라우터 통과**
    - BGP(Border Gateway Protocol) 기반 라우팅
    - 목적지까지 최적 경로 선택
3. **목적지 ISP 도착**

### 6. 방화벽 & 캐시 서버

1. **방화벽 (Firewall)**
    - 패킷 검사
    - 악의적 트래픽 차단
    - 정책 기반 필터링
2. **캐시 서버**
    - 자주 요청되는 콘텐츠 캐싱
    - 캐시 히트 시 즉시 응답 (웹 서버 접근 불필요)
    - 캐시 미스 시 웹 서버로 전달

### 7. 로드밸런서

- 여러 웹 서버 중 하나로 트래픽 분산
- 헬스 체크로 정상 서버만 선택
- 알고리즘에 따라 서버 선택

### 8. 웹 서버

1. **프로토콜 스택이 패킷 수신**
    - TCP 체크섬 검증
    - ACK 전송
    - HTTP 메시지 재조립
2. **웹 서버 애플리케이션 처리**
    - 요청 라인 파싱
    - 라우팅 (요청 URL에 맞는 핸들러 찾기)
    - 비즈니스 로직 실행
3. **데이터베이스 접근**
    - 필요 시 DB 쿼리
    - 데이터 조회/변경
4. **HTTP Response 생성**
    - 상태 라인: `HTTP/1.1 200 OK`
    - 헤더: Content-Type, Content-Length 등
    - 본문: HTML, JSON 등
5. **응답 전송**
    - 동일한 경로(역순)로 클라이언트에게 전달

### 9. 브라우저 렌더링

1. **HTML 파싱**
    - DOM 트리 생성
2. **CSS 파싱**
    - CSSOM 트리 생성
3. **렌더 트리 구성**
    - DOM + CSSOM 결합
4. **레이아웃 (리플로우)**
    - 각 요소의 위치와 크기 계산
5. **페인팅**
    - 화면에 픽셀 그리기
6. **화면 표시**
    - 사용자에게 최종 결과 표시

### 전체 과정 요약

```
사용자 입력
  ↓
브라우저 (URL 파싱, HTTP 요청 생성)
  ↓
DNS 조회 (도메인 → IP)
  ↓
TCP 연결 (3-way handshake)
  ↓
HTTP 요청 전송
  ↓
라우터 (패킷 라우팅)
  ↓
인터넷 (여러 라우터 경유)
  ↓
목적지 네트워크 (방화벽, 로드밸런서)
  ↓
웹 서버 (요청 처리, DB 접근)
  ↓
HTTP 응답 생성
  ↓
역경로로 전송
  ↓
브라우저 (렌더링)
  ↓
화면 표시
```

# 추가 중요 개념

### 흐름 제어 (Flow Control)

**송신자와 수신자의 데이터 처리 속도 차이를 조절**하는 메커니즘입니다.

### Stop-and-Wait

- 데이터 하나 전송 후 ACK 대기
- 비효율적

### Sliding Window

- 윈도우 크기만큼 ACK 없이 전송 가능
- 수신자의 버퍼 크기에 따라 윈도우 크기 조절

### 혼잡 제어 (Congestion Control)

**네트워크 혼잡 상황을 감지하고 전송률을 조절**하는 메커니즘입니다.

### AIMD (Additive Increase / Multiplicative Decrease)

- 성공 시 윈도우 크기 +1 (선형 증가)
- 실패 시 윈도우 크기 1/2 (지수 감소)

### Slow Start

- 윈도우 크기를 1부터 지수적으로 증가
- 임계값 도달 시 선형 증가로 전환

### Fast Retransmit

- 중복 ACK 3개 수신 시 즉시 재전송
- 타임아웃 대기 불필요

### Fast Recovery

- Fast Retransmit 후 혼잡 회피 단계로 진입
- 윈도우 크기를 절반으로 줄이고 선형 증가
