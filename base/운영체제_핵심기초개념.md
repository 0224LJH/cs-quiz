# 운영체제(Operating System, OS)란?

## 정의

운영체제는 **컴퓨터 시스템의 자원을 효율적으로 관리하고, 사용자가 컴퓨터를 편리하게 사용할 수 있도록 하는 시스템 소프트웨어**이다. 하드웨어와 응용 프로그램 사이에서 중재자 역할을 수행한다.

## 운영체제의 목적

1. **사용 편의성**: 사용자가 컴퓨터를 쉽게 사용할 수 있도록 인터페이스 제공
2. **효율성**: 한정된 시스템 자원(CPU, 메모리, I/O 장치)을 효율적으로 관리
3. **안정성과 보안**: 시스템의 안정적인 동작과 데이터 보호

## 운영체제의 역할

운영체제의 핵심 역할은 다음과 같으며, 이는 **커널**이 담당한다:

### 1. CPU 스케줄링과 프로세스 관리

- 여러 프로세스 간 CPU 할당을 관리
- 프로세스의 생성, 종료, 상태 관리
- 프로세스 간 동기화 및 통신 제어

**연관 개념**: CPU 스케줄러, 프로세스 상태, 컨텍스트 스위칭

### 2. 메모리 관리

- 프로세스에 메모리 할당 및 회수
- 가상 메모리 관리
- 메모리 보호 및 공유

**연관 개념**: 가상 메모리, 페이징, 메모리 계층 구조

### 3. 파일 시스템 관리

- 디스크의 파일을 관리
- 파일의 생성, 삭제, 읽기, 쓰기 제공
- 디렉토리 구조 관리

### 4. I/O 디바이스 관리

- 입출력 장치와 응용 프로그램 간의 인터페이스 제공
- 디바이스 드라이버를 통한 하드웨어 제어
- 버퍼링, 캐싱, 스풀링 등을 통한 성능 최적화

**연관 개념**: 인터럽트, DMA 컨트롤러

## 운영체제의 구조

운영체제는 계층적 구조로 이루어져 있다:

```
┌─────────────────────────────┐
│      유저 프로그램           │  ← 응용 프로그램
├─────────────────────────────┤
│  인터페이스 (GUI/CUI)        │  ← 사용자 인터페이스
├─────────────────────────────┤
│      시스템 콜               │  ← 커널 접근 인터페이스
├─────────────────────────────┤
│        커널                  │  ← OS의 핵심
│  (I/O 드라이버, 파일시스템)  │
├─────────────────────────────┤
│       하드웨어               │  ← 물리적 자원
└─────────────────────────────┘
```

### 인터페이스의 종류

**GUI (Graphical User Interface)**
- 그래픽 기반의 사용자 인터페이스
- 마우스와 키보드로 조작
- 예시: Windows, macOS, Ubuntu Desktop

**CUI (Character User Interface)**
- 문자 기반의 사용자 인터페이스
- 명령어를 입력하여 조작
- 예시: Linux 터미널, Windows CMD, ChatGPT

**연관 개념**: [시스템 콜](about:blank#%EC%8B%9C%EC%8A%A4%ED%85%9C-%EC%BD%9Csystem-call), [커널](about:blank#%EC%BB%A4%EB%84%90kernel)

# 컴퓨터 시스템의 구조

컴퓨터 시스템은 다음과 같은 구성 요소로 이루어져 있다:

```
┌──────────────────────────────────────────────┐
│                  CPU                         │
│  (인터럽트에 의해 명령어 실행)                │
└──────────────────────────────────────────────┘
           ↕ (메모리 버스)
┌──────────────────────────────────────────────┐
│                 메모리                        │
│     (데이터와 명령어 저장)                     │
└──────────────────────────────────────────────┘
           ↕                    ↕
    ┌──────────┐          ┌──────────┐
    │   타이머  │          │ DMA 컨트롤러│
    └──────────┘          └──────────┘
                    ↕
    ┌────────────────────────────────┐
    │      I/O 디바이스 컨트롤러      │
    │    (각 디바이스마다 존재)       │
    └────────────────────────────────┘
              ↕ (로컬 버퍼)
    ┌────────────────────────────────┐
    │     I/O 디바이스               │
    │  (키보드, 마우스, 디스크 등)     │
    └────────────────────────────────┘
```

## 주요 구성 요소

### CPU (Central Processing Unit)

**인터럽트에 의해 메모리에 존재하는 명령어를 해석하고 실행하는 일꾼**

**연관 개념**: [CPU의 구성 요소](about:blank#cpu%EC%9D%98-%EA%B5%AC%EC%84%B1-%EC%9A%94%EC%86%8C), [인터럽트](about:blank#%EC%9D%B8%ED%84%B0%EB%9F%BD%ED%8A%B8interrupt)

### 메모리 (Memory)

전자회로에서 데이터와 상태를 기록하는 장치로, CPU가 작업할 수 있는 **작업장** 역할을 한다.

**연관 개념**: [메모리 계층 구조](about:blank#%EB%A9%94%EB%AA%A8%EB%A6%AC-%EA%B3%84%EC%B8%B5-%EA%B5%AC%EC%A1%B0), [가상 메모리](about:blank#%EA%B0%80%EC%83%81-%EB%A9%94%EB%AA%A8%EB%A6%AC)

### DMA 컨트롤러 (Direct Memory Access Controller)

CPU의 일을 보조하는 일꾼으로, **CPU 개입 없이 I/O 디바이스와 메모리 간 직접 데이터 전송**을 담당한다.

**왜 필요한가?**
- I/O 작업마다 CPU가 개입하면 CPU가 낭비됨
- DMA가 대신 데이터를 전송하고, 완료되면 CPU에 인터럽트로 알림
- CPU는 다른 중요한 작업에 집중 가능

**연관 개념**: [인터럽트](about:blank#%EC%9D%B8%ED%84%B0%EB%9F%BD%ED%8A%B8interrupt), [I/O 관리](about:blank#%EC%9A%B4%EC%98%81%EC%B2%B4%EC%A0%9C%EC%9D%98-%EC%97%AD%ED%95%A0)

### 타이머 (Timer)

특정 프로그램이 CPU를 독점하는 것을 방지하기 위해 **시간 제한**을 설정하는 장치

- 일정 시간이 지나면 타이머 인터럽트 발생
- CPU 스케줄러가 다음 프로세스에 CPU 할당

**연관 개념**: [CPU 스케줄러](about:blank#cpu-%EC%8A%A4%EC%BC%80%EC%A4%84%EB%9F%AC), [인터럽트](about:blank#%EC%9D%B8%ED%84%B0%EB%9F%BD%ED%8A%B8interrupt)

### 디바이스 컨트롤러 (Device Controller)

각 I/O 디바이스를 제어하는 **작은 CPU** 역할

- 디바이스의 동작을 관리
- 로컬 버퍼를 통해 데이터 임시 저장
- CPU에 인터럽트를 통해 작업 완료 알림

### 로컬 버퍼 (Local Buffer)

각 디바이스 컨트롤러에 달려 있는 **작은 메모리**

- I/O 작업 중 데이터를 임시로 저장
- CPU와 I/O 디바이스 간의 속도 차이를 완충

# CPU의 구성 요소

CPU는 다음 세 가지 핵심 구성 요소로 이루어져 있다:

## 1. 산술논리연산장치 (ALU: Arithmetic and Logical Unit)

**역할**: 산술 연산과 논리 연산을 수행하는 회로 장치

**연산 종류**
- **산술 연산**: 덧셈, 뺄셈, 곱셈, 나눗셈
- **논리 연산**: AND, OR, NOT, XOR 등

## 2. 제어장치 (CU: Control Unit)

**역할**: 프로세스의 조작을 지시하는 두뇌 역할

**주요 기능**
- 메모리에서 명령어를 읽어옴
- 명령어를 해석(디코딩)
- 데이터 처리를 위한 순서 결정
- ALU와 레지스터, 메모리 등을 제어

## 3. 레지스터 (Register)

**역할**: CPU 내부에서 처리할 명령어나 데이터를 **임시로 저장**하는 매우 빠른 기억 장치

**주요 레지스터 종류**

### 프로그램 카운터 (PC: Program Counter)

- **다음에 실행할 명령어의 주소**를 저장
- CPU가 명령어를 순차적으로 실행하도록 함
- 분기 명령어(if, for, while 등)가 실행되면 PC 값이 변경됨

**연관 개념**: [PCB](about:blank#pcb), [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD)

### 명령어 레지스터 (IR: Instruction Register)

- 현재 실행 중인 명령어를 저장

### 누산기 (Accumulator)

- 연산 결과를 임시로 저장

### 범용 레지스터 (General Purpose Register)

- 다양한 용도로 사용되는 레지스터

### 스택 포인터 (SP: Stack Pointer)

- 스택의 최상단 주소를 가리킴

**연관 개념**: [스택 영역](about:blank#%ED%94%84%EB%A1%9C%EC%84%B8%EC%8A%A4-%EB%A9%94%EB%AA%A8%EB%A6%AC-%EA%B5%AC%EC%A1%B0), [함수 호출](about:blank#%EC%8A%A4%ED%83%9D-%EC%98%81%EC%97%AD%EC%9D%98-%EC%97%AD%ED%95%A0)

# 커널 (Kernel)

## 정의

커널은 **운영체제의 핵심 부분**으로, 시스템의 모든 것을 제어하는 중추적 역할을 한다.

## 커널의 역할

1. **시스템 콜 인터페이스 제공**: 사용자 프로그램이 하드웨어 자원에 접근할 수 있는 유일한 통로
2. **보안**: 하드웨어 자원에 대한 무단 접근 차단
3. **메모리 관리**: 프로세스별 메모리 할당 및 보호
4. **프로세스 관리**: 프로세스 생성, 종료, 스케줄링
5. **파일 시스템 관리**: 파일 읽기/쓰기, 디렉토리 관리
6. **I/O 디바이스 관리**: 입출력 장치 제어
7. **I/O 요청 처리**: 입출력 작업 관리

## 커널 함수 (Kernel Function)

커널 안에 있는 여러 함수들을 의미하며, **Native Function**이라고도 한다.

**예시**
- 프로세스 생성: `fork()`
- 프로세스 종료: `kill()`
- 파일 읽기: `read()`
- 파일 쓰기: `write()`
- 메모리 할당: `malloc()` (내부적으로 `brk()` 시스템 콜 사용)

**연관 개념**: [시스템 콜](about:blank#%EC%8B%9C%EC%8A%A4%ED%85%9C-%EC%BD%9Csystem-call), [유저 모드와 커널 모드](about:blank#%EC%9C%A0%EC%A0%80-%EB%AA%A8%EB%93%9C%EC%99%80-%EC%BB%A4%EB%84%90-%EB%AA%A8%EB%93%9C)

# 유저 모드와 커널 모드

프로세스는 두 가지 실행 모드를 가진다:

## 유저 모드 (User Mode)

**정의**: 사용자가 접근할 수 있는 영역을 **제한적**으로 두며, 컴퓨터 자원에 함부로 침범하지 못하는 모드

**특징**
- 일반 응용 프로그램이 실행되는 모드
- 하드웨어 자원에 직접 접근 불가
- 메모리의 제한된 영역만 접근 가능
- 잘못된 동작이 전체 시스템에 영향을 주지 않음

## 커널 모드 (Kernel Mode)

**정의**: 운영체제가 실행되는 모드로, **모든 컴퓨터 자원에 접근**할 수 있는 모드

**특징**
- 운영체제의 커널이 실행되는 모드
- 모든 하드웨어 자원에 직접 접근 가능
- 모든 메모리 영역에 접근 가능
- 모든 명령어 실행 가능

## Mode Bit

**정의**: 시스템 콜이 작동될 때 유저 모드와 커널 모드를 구분하는 **플래그 변수**

**값**
- **1**: 유저 모드
- **0**: 커널 모드

**동작 원리**
1. 응용 프로그램 실행 중 (Mode Bit = 1)
2. 시스템 콜 발생
3. Mode Bit를 0으로 변경 → 커널 모드 진입
4. 커널 함수 실행
5. Mode Bit를 1로 변경 → 유저 모드 복귀

**연관 개념**: [시스템 콜](about:blank#%EC%8B%9C%EC%8A%A4%ED%85%9C-%EC%BD%9Csystem-call), [인터럽트](about:blank#%EC%9D%B8%ED%84%B0%EB%9F%BD%ED%8A%B8interrupt)

# 시스템 콜 (System Call)

## 정의

시스템 콜은 **운영체제의 커널에 접근하기 위한 인터페이스**이며, 유저 프로그램이 운영체제의 서비스를 받기 위해 커널 함수를 호출할 때 거쳐야 하는 통로이다.

## 시스템 콜이 필요한 이유

### 1. 보안 (Security)

사용자 프로그램이 하드웨어에 직접 접근하면 악의적인 프로그램이 시스템을 손상시킬 수 있다.

**예시**: 악의적인 카메라 앱이 커널에 제약 없이 접근하면 사용자 의도와 무관하게 카메라를 켜서 사생활을 침해할 수 있다.

### 2. 안정성 (Stability)

운영체제의 관리 하에 프로그램이 실행되므로 **시스템의 안정성**이 강화된다.

### 3. 추상화 (Abstraction)

유저 프로그램은 **복잡한 파일 시스템, 프로세스 생성 등의 내부 동작을 신경 쓸 필요 없이** 시스템 콜만 호출하면 된다.

## 시스템 콜의 과정

유저 프로그램이 I/O 요청을 위한 트랩(Trap)을 발동하면 다음 과정이 발생한다:

```
1. 유저 프로그램에서 I/O 요청 발생 (예: 파일 읽기)
   ↓
2. 트랩(Trap) 발생 - 소프트웨어 인터럽트
   ↓
3. 올바른 I/O 요청인지 검증
   ↓
4. Mode Bit: 1 → 0 (유저 모드 → 커널 모드)
   ↓
5. 커널 함수 실행 (예: fs.readFile())
   ↓
6. I/O 작업 완료
   ↓
7. Mode Bit: 0 → 1 (커널 모드 → 유저 모드)
   ↓
8. 유저 프로그램으로 결과 반환
```

## 시스템 콜의 종류

### 1. 프로세스 제어 (Process Control)

- `fork()`: 새로운 프로세스 생성
- `exit()`: 프로세스 종료
- `wait()`: 자식 프로세스 대기
- `exec()`: 프로그램 실행

### 2. 파일 관리 (File Management)

- `open()`: 파일 열기
- `read()`: 파일 읽기
- `write()`: 파일 쓰기
- `close()`: 파일 닫기

### 3. 디바이스 관리 (Device Management)

- `ioctl()`: 디바이스 제어
- `read()`: 디바이스로부터 읽기
- `write()`: 디바이스에 쓰기

### 4. 정보 유지 (Information Maintenance)

- `getpid()`: 프로세스 ID 얻기
- `time()`: 현재 시간 얻기

### 5. 통신 (Communication)

- `pipe()`: 파이프 생성
- `socket()`: 소켓 생성
- `send()`, `recv()`: 데이터 송수신

**연관 개념**: [Mode Bit](about:blank#mode-bit), [인터럽트](about:blank#%EC%9D%B8%ED%84%B0%EB%9F%BD%ED%8A%B8interrupt), [IPC](about:blank#ipc)

# 인터럽트 (Interrupt)

## 정의

인터럽트는 **CPU가 프로그램을 실행하고 있을 때, 입출력 장치 등의 장치나 예외 상황이 발생하여 처리가 필요한 경우 CPU에게 알려 처리**하도록 하는 기술이다.

## 인터럽트가 필요한 이유

만약 인터럽트가 없다면:
- CPU가 I/O 작업이 완료될 때까지 **계속 확인**(polling)해야 함
- I/O 작업은 CPU보다 훨씬 느리기 때문에 CPU가 낭비됨
- 시스템 전체의 효율성이 크게 저하됨

인터럽트를 사용하면:
- I/O 작업을 시작하고 CPU는 다른 작업 수행
- I/O 완료 시 디바이스 컨트롤러가 인터럽트로 CPU에 알림
- CPU는 필요할 때만 I/O 작업 처리

## 인터럽트 처리 과정

```
1. 프로세스 A 실행 중
   ↓
2. 인터럽트 발생 (예: 키보드 입력, 타이머)
   ↓
3. 현재 실행 중인 명령어 완료
   ↓
4. 현재 프로세스의 상태를 PCB에 저장
   ↓
5. 인터럽트 벡터를 통해 인터럽트 핸들러 주소 확인
   ↓
6. 인터럽트 핸들러(ISR: Interrupt Service Routine) 실행
   ↓
7. 인터럽트 처리 완료
   ↓
8. 저장된 상태를 복원하여 프로세스 A 재개
```

## 인터럽트의 종류

### 하드웨어 인터럽트 (Hardware Interrupt)

**정의**: 하드웨어 장치가 CPU에 보내는 신호

**특징**
- **비동기적(Asynchronous)**: 언제 발생할지 예측 불가
- CPU 외부에서 발생

**예시**
- **I/O 인터럽트**: 키보드 입력, 마우스 클릭, 디스크 읽기/쓰기 완료
- **타이머 인터럽트**: 일정 시간 경과 시 발생
- **전원 이상**: 정전, 전압 이상

### 소프트웨어 인터럽트 (Software Interrupt) = 트랩(Trap)

**정의**: 프로그램 내부에서 발생하는 인터럽트

**특징**
- **동기적(Synchronous)**: 명령어 실행 결과로 발생
- CPU 내부에서 발생

**예시**
- **시스템 콜**: 프로그램이 운영체제 서비스를 요청
- **예외(Exception)**: 0으로 나누기, 잘못된 메모리 접근(Segmentation Fault), 오버플로우

**하드웨어 인터럽트 vs 소프트웨어 인터럽트 비교**

| 구분 | 하드웨어 인터럽트 | 소프트웨어 인터럽트 |
| --- | --- | --- |
| 발생 위치 | CPU 외부 (I/O 장치) | CPU 내부 (프로그램) |
| 발생 시점 | 비동기적 (예측 불가) | 동기적 (명령어 실행 시) |
| 예시 | 키보드, 마우스, 타이머 | 시스템 콜, 예외 |
| 처리 | 인터럽트 핸들러 | 예외 핸들러 또는 시스템 콜 |

**연관 개념**: [컨텍스트 스위칭](about:blank#%EC%BB%A8%ED%85%8D%EC%8A%A4%ED%8A%B8-%EC%8A%A4%EC%9C%84%EC%B9%AD), [시스템 콜](about:blank#%EC%8B%9C%EC%8A%A4%ED%85%9C-%EC%BD%9Csystem-call), [DMA](about:blank#dma-%EC%BB%A8%ED%8A%B8%EB%A1%A4%EB%9F%AC)

# 메모리 계층 구조

컴퓨터의 메모리는 **속도와 용량, 비용**의 트레이드오프에 따라 계층적으로 구성된다.

```
        빠름, 작음, 비쌈
              ↑
    ┌──────────────────┐
    │   레지스터        │  ← CPU 내부, 가장 빠름 (1 cycle)
    ├──────────────────┤
    │   L1 캐시         │  ← ~1-3 cycles
    ├──────────────────┤
    │   L2 캐시         │  ← ~10 cycles
    ├──────────────────┤
    │   L3 캐시         │  ← ~40 cycles
    ├──────────────────┤
    │  메인 메모리(RAM) │  ← ~100-300 cycles
    ├──────────────────┤
    │   SSD/HDD        │  ← ~10,000+ cycles
    ├──────────────────┤
    │  네트워크 저장소  │  ← 가장 느림
    └──────────────────┘
              ↓
       느림, 큼, 저렴
```

## 각 계층의 특징

### 1. 레지스터 (Register)

- **용량**: 수십~수백 바이트
- **속도**: 가장 빠름 (CPU 클럭과 동일)
- **위치**: CPU 내부
- **용도**: 현재 실행 중인 명령어와 데이터 저장

### 2. 캐시 (Cache)

- **용량**: KB ~ MB 단위
- **속도**: 매우 빠름
- **위치**: CPU 내부 또는 가까이
- **용도**: 자주 사용되는 데이터를 임시 저장
- **계층**: L1, L2, L3 캐시로 구분

**연관 개념**: 캐시의 지역성, 캐싱 라인

### 3. 메인 메모리 (Main Memory, RAM)

- **용량**: GB 단위
- **속도**: 빠름
- **특징**: 휘발성 (전원이 꺼지면 데이터 소실)
- **용도**: 실행 중인 프로그램과 데이터 저장

### 4. 보조 기억장치 (Secondary Storage)

- **용량**: TB 단위 이상
- **속도**: 느림
- **특징**: 비휘발성 (전원이 꺼져도 데이터 유지)
- **종류**: SSD, HDD
- **용도**: 프로그램과 파일을 영구 저장

## 메모리 계층 구조의 원리

### 지역성의 원리 (Principle of Locality)

메모리 계층 구조가 효과적인 이유는 프로그램이 **지역성**을 가지기 때문이다.

**시간 지역성 (Temporal Locality)**
- 최근에 접근한 데이터는 **가까운 미래에 다시 접근**할 가능성이 높음
- 예시: 반복문의 카운터 변수

**공간 지역성 (Spatial Locality)**
- 접근한 데이터 근처의 데이터에 **곧 접근**할 가능성이 높음
- 예시: 배열 순회

**연관 개념**: 캐시의 지역성, 가상 메모리, 페이지 교체 알고리즘

# 커널 스택과 사용자 스택

프로세스는 두 종류의 스택을 가진다:

## 사용자 스택 (User Stack)

**정의**: 사용자 모드에서 실행되는 코드의 함수 호출을 관리하는 스택

**특징**
- 유저 모드에서만 접근 가능
- 프로세스의 가상 메모리 공간에 위치
- 일반 함수 호출, 지역 변수, 매개변수 저장

## 커널 스택 (Kernel Stack)

**정의**: 커널 모드에서 실행되는 코드의 함수 호출을 관리하는 스택

**특징**
- 커널 모드에서만 접근 가능
- 가상 메모리 주소 공간의 상위 영역에 위치
- 시스템 콜 처리, 인터럽트 핸들러 실행 시 사용
- 각 프로세스마다 별도의 커널 스택 존재

**왜 분리되어 있는가?**
1. **보안**: 사용자 프로그램이 커널 스택에 접근하면 시스템 정보 유출 또는 손상 가능
2. **안정성**: 사용자 스택 오버플로우가 커널에 영향을 주지 않도록 격리
3. **권한 관리**: Mode Bit에 따라 접근 가능한 스택이 자동으로 전환됨

**연관 개념**: [유저 모드와 커널 모드](about:blank#%EC%9C%A0%EC%A0%80-%EB%AA%A8%EB%93%9C%EC%99%80-%EC%BB%A4%EB%84%90-%EB%AA%A8%EB%93%9C), [시스템 콜](about:blank#%EC%8B%9C%EC%8A%A4%ED%85%9C-%EC%BD%9Csystem-call), [PCB](about:blank#pcb)

# 프로세스 메모리 구조

프로세스가 메모리에 적재될 때의 구조는 다음과 같다:

```
높은 주소
    ↑
┌─────────────┐
│    커널      │  ← 커널 영역 (모든 프로세스 공유)
├─────────────┤
│    스택      │  ← 아래 방향으로 성장 ↓
│      ↓      │     함수 호출, 지역 변수
│             │
│   (빈 공간)  │
│             │
│      ↑      │
│    힙       │  ← 위 방향으로 성장 ↑
├─────────────┤     동적 할당 메모리
│  데이터 영역 │  ← 전역 변수, 정적 변수
├─────────────┤
│  코드 영역   │  ← 실행 코드 (읽기 전용)
└─────────────┘
낮은 주소
    ↓
```

## 1. 코드 영역 (Code Segment, Text Segment)

**내용**: 실행할 **프로그램의 코드**(기계어)가 저장

**특징**
- **읽기 전용(Read-Only)**: 코드가 실행 중 변경되지 않도록 보호
- **공유 가능**: 같은 프로그램을 여러 프로세스가 실행하면 코드 영역은 공유
- **컴파일 타임에 크기 결정**

## 2. 데이터 영역 (Data Segment)

**내용**: **전역 변수와 정적(static) 변수**가 저장

**세부 구분**
- **Data Segment**: 초기화된 전역/정적 변수
- **BSS Segment**: 초기화되지 않은 전역/정적 변수 (0으로 초기화됨)

**특징**
- 프로그램 시작 시 할당, 종료 시 해제
- **컴파일 타임에 크기 결정**
- 읽기/쓰기 가능

## 3. 힙 영역 (Heap Segment)

**내용**: **동적으로 할당**되는 메모리 공간

**특징**
- **런타임에 크기 결정** (동적)
- 프로그래머가 명시적으로 할당(`malloc()`, `new`)과 해제(`free()`, `delete`) 관리
- **낮은 주소에서 높은 주소 방향**으로 성장
- 메모리 누수(Memory Leak) 가능성: 할당한 메모리를 해제하지 않으면 발생

**사용 예시**

```
동적 배열 할당, 연결 리스트, 트리 등의 자료구조
```

## 4. 스택 영역 (Stack Segment)

**내용**: **함수 호출 시의 지역 변수, 매개변수, 복귀 주소** 저장

**특징**
- **런타임에 크기 결정** (동적)
- **높은 주소에서 낮은 주소 방향**으로 성장
- 함수 호출 시 자동으로 할당, 함수 종료 시 자동으로 해제
- **LIFO(Last In First Out)** 구조
- 스택 오버플로우 가능성: 재귀 호출이 너무 깊거나 지역 변수가 너무 크면 발생

**스택 프레임 (Stack Frame)**
- 하나의 함수 호출에 대한 정보를 담는 단위
- 지역 변수, 매개변수, 복귀 주소(Return Address), 이전 프레임 포인터 포함

## 왜 스택과 힙이 반대 방향으로 성장하는가?

1. **메모리 효율성**: 두 영역이 같은 방향으로 성장하면 중간에 큰 빈 공간이 낭비됨
2. **유연성**: 스택이 많이 필요한 프로그램과 힙이 많이 필요한 프로그램 모두 수용 가능
3. **충돌 탐지**: 두 영역이 만나면 메모리 부족 상태를 쉽게 감지 가능

**연관 개념**: 스레드와 스택, 메모리 관리, 가상 메모리

# 면접 대비 핵심 질문

## 운영체제 기초

1. **운영체제란 무엇이고, 왜 필요한가요?**
    - 정의, 역할, 목적 설명
2. **운영체제의 주요 역할 4가지는 무엇인가요?**
    - CPU 스케줄링, 메모리 관리, 파일 시스템 관리, I/O 디바이스 관리
3. **GUI와 CUI의 차이는 무엇인가요?**
    - 그래픽 vs 문자 기반 인터페이스

## 컴퓨터 시스템 구조

1. **CPU의 구성 요소 3가지를 설명해주세요.**
    - ALU, 제어장치, 레지스터
2. **레지스터 중 프로그램 카운터(PC)의 역할은 무엇인가요?**
    - 다음 실행할 명령어의 주소 저장
3. **DMA 컨트롤러가 필요한 이유는 무엇인가요?**
    - CPU 부담 감소, I/O 효율 향상
4. **타이머의 역할은 무엇인가요?**
    - CPU 독점 방지, 시간 제한

## 커널과 모드

1. **커널이란 무엇이고, 어떤 역할을 하나요?**
    - OS의 핵심, 자원 관리, 시스템 콜 제공
2. **유저 모드와 커널 모드의 차이를 설명해주세요.**
    - 권한, 접근 가능한 자원
3. **Mode Bit의 역할과 값의 의미는 무엇인가요?**
    - 1: 유저 모드, 0: 커널 모드
4. **커널 스택과 사용자 스택이 분리된 이유는 무엇인가요?**
    - 보안, 안정성, 권한 관리

## 시스템 콜

1. **시스템 콜이란 무엇이고, 왜 필요한가요?**
    - 커널 접근 인터페이스, 보안, 안정성, 추상화
2. **시스템 콜 처리 과정에서 Mode Bit는 어떻게 변화하나요?**
    - 1 → 0 → 1 (유저 → 커널 → 유저)
3. **시스템 콜의 종류에는 어떤 것들이 있나요?**
    - 프로세스 제어, 파일 관리, 디바이스 관리, 정보 유지, 통신

## 인터럽트

1. **인터럽트란 무엇이고, 왜 필요한가요?**
    - CPU에 이벤트 알림, 폴링 방지, 효율성 향상
2. **하드웨어 인터럽트와 소프트웨어 인터럽트의 차이는 무엇인가요?**
    - 외부 vs 내부, 비동기 vs 동기
3. **인터럽트 처리 과정을 설명해주세요.**
    - 발생 → 상태 저장 → 핸들러 실행 → 상태 복원
4. **타이머 인터럽트의 역할은 무엇인가요?**
    - 일정 시간마다 CPU 스케줄링 트리거

## 메모리

1. **메모리 계층 구조를 설명해주세요.**
    - 레지스터 → 캐시 → 메인 메모리 → 보조 기억장치
2. **메모리 계층 구조가 효과적인 이유는 무엇인가요?**
    - 지역성의 원리 (시간 지역성, 공간 지역성)
3. **프로세스 메모리 구조의 4가지 영역을 설명해주세요.**
    - 코드, 데이터, 힙, 스택
4. **스택과 힙이 반대 방향으로 성장하는 이유는?**
    - 메모리 효율성, 유연성, 충돌 탐지
5. **힙과 스택의 차이점은 무엇인가요?**
    - 동적 할당 vs 자동 관리, 성장 방향, 수명

## 통합 질문

1. **프로세스가 시스템 콜을 통해 파일을 읽을 때 전체 과정을 설명해주세요.**
    - 시스템 콜 → Mode Bit 변경 → 커널 함수 → I/O 요청 → 인터럽트 → 데이터 반환
2. **컨텍스트 스위칭 시 커널 스택은 어떻게 되나요?**
    - 각 프로세스마다 별도의 커널 스택 존재, PCB에 저장된 정보로 전환
3. **사용자 프로그램이 하드웨어에 직접 접근하지 못하도록 막는 메커니즘은?**
    - 유저 모드/커널 모드 분리, 시스템 콜만을 통한 접근
4. **메모리 계층 구조와 캐시의 지역성이 어떻게 연관되나요?**
    - 지역성 원리를 활용하여 상위 계층에서 hit rate 향상
5. **인터럽트와 시스템 콜의 공통점과 차이점은?**
    - 공통점: 커널 모드 전환, 차이점: 하드웨어 vs 소프트웨어, 비동기 vs 동기

## 1. 개념 vs 메커니즘부터 나누자

### ① 시스템 콜 (System Call) = “서비스 이름 / API 개념”

- 유저 입장에서 보면:
    
    > “OS야, 파일 좀 읽어줘”, “프로세스 하나 만들어줘” 같은 공식 서비스 요청
    > 
- 즉, `read()`, `write()`, `fork()` 같은 **OS 기능을 부르는 인터페이스 이름**이야.
- “무언가 커널 권한이 필요한 일을 **공식적으로 요청하는 창구**” = 시스템 콜.

### ② 트랩 / 소프트웨어 인터럽트 (Trap / Software Interrupt) = “진입 메커니즘”

- 시스템 콜을 실제로 구현할 때,
    - CPU에게 **“커널 모드로 넘어가!”**라고 시키는 **특수 명령**이 필요함.
    - x86 기준: `int 0x80`, `syscall` 이런 애들
- 이 명령을 실행하면:
    - 지금 실행 중이던 유저 코드 흐름을 끊고
    - **커널 모드로 전환 + 시스템 콜 핸들러로 점프**
- 이 “명령 하나로 커널로 점프하는 이벤트”를
    
    보통 **트랩(trap)** / **소프트웨어 인터럽트**라고 부름.
    

즉,

> 시스템 콜 = “OS 서비스 이름/기능”
> 
> 
> **트랩(소프트웨어 인터럽트) = 그 서비스로 들어가는 CPU-level “문 열기 동작”**
> 

라고 보면 편해.

---

## 2. “커널 권한 필요한 건 다 소프트웨어 인터럽트다?” → 반쯤 맞고, 표현만 조심

정리하면:

- **유저 프로그램이 의도적으로 커널에게 부탁하는 경우**
    
    → 시스템 콜 호출
    
    → 내부적으로 **트랩(소프트웨어 인터럽트 명령)** 실행
    
    → 커널 모드 진입
    
- **유저 프로그램이 실수로 이상한 짓 한 경우** (0으로 나누기, 잘못된 주소 접근 등)
    
    → CPU가 “이건 안 됨” 하고 **예외(exception)** 발생
    
    → 이것도 넓게 보면 **소프트웨어 기인 이벤트**지만
    
    우리가 이걸 “시스템 콜”이라고 부르진 않지
    

그래서 네 문장을 교과서스럽게 고치면:

> “유저 모드에서 커널 권한이 필요한 일을 합법적으로 요청할 때는
> 
> 
> 시스템 콜이라는 인터페이스를 사용하고,
> 
> 이 시스템 콜은 내부적으로 보통 트랩/소프트웨어 인터럽트 같은 메커니즘을 사용해
> 
> 커널 모드로 진입한다.”
> 

이게 제일 깔끔한 버전이야.

---

## 3. 한방에 외우는 관계 정리

이렇게 3단계로 외우면 좋아:

1. **System Call**
    - “OS야, 이 기능 좀 해줘”
    - → **API/서비스 이름**
2. **Trap (Software Interrupt)**
    - “지금 이 시점에서 커널로 점프해!”
    - → **커널 진입 메커니즘 (특수 명령 실행 결과)**
    - 시스템 콜을 구현할 때 주로 사용
3. **Hardware Interrupt**
    - “밖에서 누가 나 불렀냐?” (키보드, 디스크, 타이머…)
    - → 유저 코드가 원하지 않아도 끼어드는 외부 이벤트

그래서 네 질문에 짧게 답하면:

> “소프트웨어를 통해 커널 권한이 필요한 일을 요청할 때, 그 진입 메커니즘이 보통 소프트웨어 인터럽트(트랩)이고,
그걸 통해 호출되는 OS 서비스 묶음을 우리가 ‘시스템 콜’이라고 부른다.”
> 

라고 이해하면 딱 맞게 잡은 거야 😎
