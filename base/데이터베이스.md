# 데이터베이스

## 데이터베이스를 사용하는 이유

데이터베이스가 존재하기 이전에는 파일 시스템을 이용하여 데이터를 관리하였다. 데이터를 각각의 파일 단위로 저장하며 이러한 일들을 처리하기 위한 독립적인 애플리케이션과 상호 연동이 되어야 한다. 이 때의 문제점은 **데이터 종속성 문제**와 **중복성**, **데이터 무결성** 문제이다.

## 데이터베이스의 특징

1. **데이터의 독립성**
    - 물리적 독립성 : 데이터베이스 사이즈를 늘리거나 성능 향상을 위해 데이터 파일을 늘리거나 새롭게 추가하더라도 관련된 응용 프로그램을 수정할 필요가 없다.
    - 논리적 독립성 : 데이터베이스는 논리적인 구조로 다양한 응용 프로그램의 논리적 요구를 만족시켜줄 수 있다.
2. **데이터의 무결성**
    
    여러 경로를 통해 잘못된 데이터가 발생하는 경우의 수를 방지하는 기능으로 데이터의 유효성 검사를 통해 데이터의 무결성을 구현하게 된다.
    
3. **데이터의 보안성**
    
    인가된 사용자들만 데이터베이스나 데이터베이스 내의 자원에 접근할 수 있도록 계정 관리 또는 접근 권한을 설정함으로써 모든 데이터에 보안을 구현할 수 있다.
    
4. **데이터의 일관성**
    
    연관된 정보를 논리적인 구조로 관리함으로써 어떤 하나의 데이터만 변경했을 경우 발생할 수 있는 데이터의 불일치성을 배제할 수 있다.
    
5. **데이터 중복 최소화**
    
    데이터베이스는 데이터를 통합해서 관리함으로써 파일 시스템의 단점 중 하나인 자료의 중복과 데이터의 중복성 문제를 해결할 수 있다.
    

## 데이터베이스의 성능

데이터베이스의 성능 이슈는 **디스크 I/O 를 어떻게 줄이느냐**에서 시작된다. 디스크 I/O 란 디스크 드라이브의 플래터(원판)을 돌려서 읽어야 할 데이터가 저장된 위치로 디스크 헤더를 이동시킨 다음 데이터를 읽는 것을 의미한다.

데이터를 읽는데 걸리는 시간은 디스크 헤더를 움직여서 읽고 쓸 위치로 옮기는 단계에서 결정된다. 즉 **디스크의 성능은 디스크 헤더의 위치 이동 없이 얼마나 많은 데이터를 한 번에 기록하느냐에 따라 결정된다**.

그렇기 때문에 **순차 I/O 가 랜덤 I/O 보다 빠를 수 밖에 없다**. 하지만 현실에서는 대부분의 I/O 작업이 랜덤 I/O 이다. 랜덤 I/O 를 순차 I/O 로 바꿔서 실행할 수는 없을까? 이러한 생각에서부터 시작되는 데이터베이스 쿼리 튜닝은 **랜덤 I/O 자체를 줄여주는 것이 목적**이라고 할 수 있다.

## 데이터베이스 기본 개념

### 엔터티 (Entity)

사람, 장소, 물건, 사건, 개념 등 여러 개의 속성을 지닌 명사이자 데이터베이스에서 실제 세계의 객체나 개념을 모델링한 것. 데이터베이스 내에서 하나의 고유하게 식별될 수 있는 속성들을 가진다.

**약한 엔터티와 강한 엔터티**
- **강한 엔터티(Strong Entity)**: 자체적인 주요 키(Primary Key)를 가지고 다른 엔터티에 의존하지 않는 독립적인 엔터티
- **약한 엔터티(Weak Entity)**: 독립적으로 존재할 수 없고 다른 강한 엔터티에 의존하는 엔터티. 자체적인 주요 키를 가지지 않으며, 관련 강한 엔터티의 키를 참조

예: 학생(강한 엔터티) - 강의등록(약한 엔터티)

### 릴레이션 (Relation)

데이터베이스에서 정보를 구분하여 저장하는 기본 단위이자 엔터티의 집합과 그 엔터티의 속성을 표현하는 구조. 엔터티가 데이터베이스에 올라가면 릴레이션이 된다.

- **관계형 데이터베이스**: ‘테이블’
- **NoSQL 데이터베이스**: ‘컬렉션’

### 속성 (Attribute)과 도메인 (Domain)

- **속성**: 릴레이션에서 관리하는 구체적이며 고유한 이름을 갖는 정보
- **도메인**: 릴레이션에 포함된 각각의 속성들이 가질 수 있는 값의 집합

예: ’성별’이라는 속성의 도메인은 {남, 여}

### 필드 (Field)와 레코드 (Record)

엔터티가 데이터베이스에 들어갈 때 엔터티의 속성은 테이블의 **필드**로 변환된다. 이러한 필드를 가진 데이터들이 행 단위로 쌓이는 것을 **레코드** 또는 **튜플**이라고 한다.

### 타입 (Type)

데이터베이스마다 지원되는 타입이 다르다. MySQL은 세분화된 타입을 지원하지만, MongoDB는 string, number 등 단순한 타입만 지원한다.

**MySQL 주요 타입**:
- **숫자**: TINYINT(1바이트), SMALLINT(2바이트), MEDIUMINT(3바이트), INT(4바이트), BIGINT(8바이트)
- **문자열**:
- CHAR: 고정 길이 (0-255)
- VARCHAR: 가변 길이 (0-65,535)
- TEXT: 긴 텍스트 (최대 65,535), 디스크에 저장
- **날짜**: DATE, DATETIME(8바이트), TIMESTAMP(4바이트)

**CHAR vs VARCHAR**:
- CHAR: 고정 길이, 유동적이지 않은 데이터에 적합
- VARCHAR: 가변 길이, 유동적인 데이터에 적합, 길이 기록용 1-2바이트 추가

**VARCHAR vs TEXT**:
- VARCHAR: 행 내부(메모리)에 저장, 인덱스 가능, 읽기 성능 우수
- TEXT: 행 외부(디스크)에 저장, 인덱스 시 크기 제한 필요, 검색/수정이 빈번하지 않은 데이터에 적합

# Index

## 인덱스(Index)란 무엇인가?

인덱스는 책의 색인과 같은 개념으로, 데이터는 책의 내용이고 데이터가 저장된 레코드의 주소는 인덱스 목록에 있는 페이지 번호가 된다.

DBMS는 칼럼의 값과 해당 레코드가 저장된 주소를 키와 값의 쌍으로 인덱스를 만들어 둔다. DBMS의 인덱스는 **항상 정렬된 상태를 유지**하기 때문에 원하는 값을 탐색하는데는 빠르지만 새로운 값을 추가하거나 삭제, 수정하는 경우에는 쿼리문 실행 속도가 느려진다.

결론적으로 DBMS에서 인덱스는 **데이터의 저장 성능을 희생하고 그 대신 데이터의 읽기 속도를 높이는 기능**이다.

## Index 자료구조

### B+-Tree 인덱스 알고리즘

일반적으로 사용되는 인덱스 알고리즘은 **B+-Tree 알고리즘**이다. B+-Tree는 이진검색트리를 일반화한 자료구조로:
- 2개 이상의 자식을 가진 노드를 허용
- 탐색에 평균 O(logN) 시간 소요
- **대수확장성**: 트리 깊이가 리프 노드 수에 비해 매우 느리게 성장
- 칼럼의 값을 변형하지 않고 원래의 값을 이용해 인덱싱

**B+Tree의 주요 특징**:
1. 모든 리프노드들은 같은 레벨에 있음 (균형잡힌 트리)
2. B+트리의 차수는 노드가 가질 수 있는 자식의 최대 수
3. 노드 내에 여러 키를 저장하고, 각 키는 자식 노드들 사이ㅇ의 값을 구분
4. 노드가 꽉 찼을 때 분할, 사용량이 줄어들면 병합

**대수확장성의 효율**:
- 트리 깊이가 리프 노드 수에 비해 매우 느리게 성장
- 예: 트리 깊이 10개로 100만개의 레코드 검색 가능
- 4차 B-트리의 경우 깊이가 한 단계 증가할 때마다 최대 인덱스 항목 수는 4배 증가

### Hash 인덱스 알고리즘

칼럼의 값으로 해시 값을 계산해서 인덱싱하는 알고리즘:
- 매우 빠른 검색 지원 (O(1))
- 값을 변형해서 인덱싱하므로, 전방 일치나 부분 검색 불가
- 주로 메모리 기반의 데이터베이스에서 사용

## 왜 index 를 생성하는데 B+tree 를 사용하는가?

해시 테이블은 등호(=) 연산에 특화되어 있다. 하지만 데이터베이스 질의에는 부등호(<, >, <=, >=) 연산도 포함된다. 동등 연산에만 특화된 해시 테이블은 **데이터베이스의 자료구조로 적합하지 않다**.

## Primary Index vs Secondary Index

- **Primary Index**: Primary Key에 대해서 생성된 인덱스. 테이블 당 하나만 존재
- **Secondary Index**: Primary Key가 아닌 다른 칼럼에 대해서 생성된 인덱스. 테이블 당 여러 개 생성 가능

## Clustered Index vs Non-clustered Index

### Clustered Index (클러스터드 인덱스)

- 인덱스가 적용된 속성 값에 의해 **레코드의 물리적 저장 위치가 결정**되는 인덱스
- 비슷한 값들을 동시에 조회하는 경우가 많다는 점에서 착안
- 일반적으로 Primary Key에 대해 자동으로 생성
- 테이블 당 **한 개만** 생성 가능
- **데이터 페이지가 정렬되어 저장**됨
- 인덱스 페이지의 리프노드에 실제 **데이터 페이지**가 들어있음
- **장점**: 탐색 속도가 빠름 (정렬되어 있어 범위 검색에 유리)
- **단점**: 삽입, 삭제, 수정이 느림 (재정렬 필요)
- 인덱스의 순서와 데이터의 순서가 일치

### Non-clustered Index (논클러스터드 인덱스)

- 데이터를 **물리적으로 정렬하지 않음**
- 별도의 인덱스 테이블을 만들어 실제 데이터 테이블의 행을 참조
- 인덱스 페이지 리프노드에 실제 데이터가 아닌 **데이터 페이지에 관한 포인터**가 있음
- 테이블 당 **여러 개** 생성 가능
- **장점**: 삽입, 삭제, 수정이 빠름 (재정렬 불필요)
- **단점**: 탐색이 상대적으로 느림 (포인터를 통한 간접 참조)
- 인덱스의 순서와 데이터의 순서가 불일치

**MySQL에서의 인덱스 생성**:

```sql
-- Clustered Index (Primary Key)ALTER TABLE table_name ADD PRIMARY KEY (column);
-- Non-clustered IndexCREATE INDEX index_name ON table_name (column);
ALTER TABLE table_name ADD INDEX index_name (column);
```

## Composite Index

복합 인덱스를 설정할 때 **인덱스 필드의 순서가 중요**하다. 예를 들어 (title, author) 순서로 인덱스를 설정한다면:
- title을 검색하는 경우: 인덱스 효과를 볼 수 있음
- author만으로 검색하는 경우: 인덱스 효과 없음

**복합 인덱스 설정 원칙**:
1. 등호(=) 비교 쿼리가 있다면 제일 먼저 인덱스로 설정
2. 정렬에 쓰는 필드를 그 다음 인덱스로 설정
3. 다중 값 출력 필드(<, >, BETWEEN 등)를 마지막 인덱스로 설정
4. **카디널리티가 높은 순서**대로 설정 (유니크한 값의 정도)

예: (age, email) 중 email을 먼저 설정해야 함 (카디널리티가 높음)

## Index 의 성능과 고려해야할 사항

### 인덱스의 비용

**1. INSERT/DELETE/UPDATE 시 추가 작업 발생**
- INSERT: 인덱스에 대한 데이터도 추가해야 함
- DELETE: 인덱스에 존재하는 값은 삭제하지 않고 사용 안한다는 표시만 함
- 반복되면 실제 데이터는 10만건인데 인덱스는 100만건이 되어 성능 저하
- UPDATE: INSERT와 DELETE의 문제점을 동시에 수반

**2. 인덱스 자체의 유지 비용**
- 두 번 탐색 (인덱스 리스트 → 컬렉션)
- B-트리 높이 균형 조절 비용
- 데이터 효율적 분산 비용

### 효율적인 인덱스 사용

**데이터의 형식에 따른 인덱스 효율**

`이름`, `나이`, `성별` 필드가 있는 테이블에서:
- 이름: 다양한 경우의 수 → 인덱스 효율적
- 성별: 남/여 두 가지만 → 인덱스 비효율적

이유: **값의 range가 적은 경우** 인덱스를 읽고 다시 디스크 I/O가 발생하기 때문

**인덱스 성능 최적화 방법**:
1. **항상 테스팅**: EXPLAIN 함수를 통해 쿼리 실행 계획 분석
2. **사용되지 않거나 중복되는 인덱스 제거**
3. **컬럼 크기를 작게 유지**
4. **선택도(Selectivity)가 높은 컬럼**에 인덱스 생성

**연관 개념**: [디스크 I/O](about:blank#%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4%EC%9D%98-%EC%84%B1%EB%8A%A5), [Composite Index](about:blank#composite-index)

# 정규화에 대해서

## 정규화 탄생 배경

한 릴레이션에 여러 엔티티의 애트리뷰트들을 혼합하게 되면:
- 정보가 중복 저장됨
- 저장 공간을 낭비
- **갱신 이상(Anomaly)** 발생

갱신 이상이란 동일한 정보를 한 릴레이션에는 변경하고, 나머지 릴레이션에서는 변경하지 않은 경우 어느 것이 정확한지 알 수 없게 되는 것이다.

## 갱신 이상의 종류

- **삽입 이상(Insertion Anomalies)**
    - 원하지 않는 자료가 삽입되거나, 삽입하는데 자료가 부족해 삽입이 되지 않아 발생하는 문제
    - 예: 신규 회원 가입 시 이벤트 참여가 필수인 경우
- **삭제 이상(Deletion Anomalies)**
    - 하나의 자료만 삭제하고 싶지만, 그 자료가 포함된 튜플 전체가 삭제됨으로 원하지 않는 정보 손실이 발생
    - 예: 이벤트 참여 취소 시 회원 정보까지 삭제되는 경우
- **수정(갱신) 이상(Modification Anomalies)**
    - 정확하지 않거나 일부의 튜플만 갱신되어 정보가 모호해지거나 일관성이 없어져 정확한 정보 파악이 되지 않는 문제
    - 예: 회원 등급 변경 시 일부 레코드만 수정되는 경우

## 정규화란 무엇인가?

관계형 데이터베이스에서 **중복을 최소화하기 위해 데이터를 구조화하는 작업**. 불만족스러운 “나쁜” 릴레이션의 애트리뷰트들을 나누어서 “좋은” 작은 릴레이션으로 분해하는 작업.

정규화 과정을 거치게 되면 **정규형(Normal Form)**을 만족하게 된다. 정규형이란 특정 조건을 만족하는 릴레이션의 스키마의 형태를 말하며 제1정규형, 제2정규형, 제3정규형, BCNF 등이 존재한다.

## ‘나쁜’ 릴레이션은 어떻게 파악하는가?

엔티티를 구성하고 있는 애트리뷰트 간에 **함수적 종속성(Functional Dependency)**을 판단한다. 판단된 함수적 종속성은 좋은 릴레이션 설계의 정형적 기준으로 사용된다.

## 함수적 종속성이란 무엇인가?

X와 Y를 임의의 애트리뷰트 집합이라고 할 때, **X의 값이 Y의 값을 유일하게(unique) 결정**한다면 “X는 Y를 함수적으로 결정한다”라고 한다. 이를 `X → Y`로 표기한다.

- X: 결정자(Determinant)
- Y: 종속자(Dependent)

**완전 함수 종속**: `X → Y`라고 가정했을 때, X의 어떠한 애트리뷰트라도 제거하면 더 이상 함수적 종속성이 성립하지 않는 경우

**부분 함수 종속**: 키의 일부분만으로도 다른 속성을 결정할 수 있는 경우

**이행적 함수 종속**: `X → Y`, `Y → Z`의 경우에 의해서 추론될 수 있는 `X → Z`의 종속관계

## 정규화의 종류

### 제 1 정규형 (1NF)

애트리뷰트의 도메인이 오직 **원자값(Atomic Value)**만을 포함하고, 튜플의 모든 애트리뷰트가 도메인에 속하는 하나의 값을 가져야 한다. 즉, 복합 애트리뷰트, 다중값 애트리뷰트, 중첩 릴레이션 등 비 원자적인 애트리뷰트들을 허용하지 않는 릴레이션 형태.

**위반 예시**:
- 한 칼럼에 여러 값 저장 (예: 취미 필드에 “독서, 영화, 운동”)
- 반복 그룹 존재

### 제 2 정규형 (2NF)

모든 비주요 애트리뷰트들이 주요 애트리뷰트(PK)에 대해서 **완전 함수적 종속**이면 제2정규형을 만족한다. 즉, **부분 함수 종속을 제거한 형태**.

**조건**:
- 제1정규형을 만족
- 키가 아닌 모든 속성이 기본키에 완전 함수 종속

### 제 3 정규형 (3NF)

어떠한 비주요 애트리뷰트도 기본키에 대해서 **이행적으로 종속되지 않으면** 제3정규형을 만족한다. 즉, 비주요 애트리뷰트가 비주요 애트리뷰트에 의해 종속되는 경우가 없는 릴레이션 형태.

**조건**:
- 제2정규형을 만족
- 이행적 함수 종속 제거

### BCNF (Boyce-Codd Normal Form)

여러 후보 키가 존재하는 릴레이션에 해당하는 정규화. 복잡한 식별자 관계에 의해 발생하는 문제를 해결하기 위해 제3정규형을 보완.

**조건**:
- 제3정규형을 만족
- **모든 결정자가 후보키**

## 정규화의 장단점

### 장점

1. 데이터베이스 변경 시 이상 현상(Anomaly) 제거
2. 데이터베이스 구조 확장 시 재 디자인 최소화
3. 사용자에게 데이터 모델을 더욱 의미있게 제공
4. 다양한 질의를 지원

### 단점

1. 릴레이션의 분해로 인해 릴레이션 간의 JOIN 연산이 많이 발생
2. 질의에 대한 응답 시간이 느려질 수 있음
3. 성능 저하 가능

## 반(역)정규화(Denormalization)

반정규화는 정규화된 엔티티, 속성, 관계를 시스템의 성능 향상 및 개발과 운영의 단순화를 위해 **의도적으로 정규화 원칙을 위배**하는 행위.

### 반정규화를 하는 이유

1. 정규화에 충실하면 종속성, 활용성은 향상되지만 수행속도가 느려지는 경우
2. 다량의 범위를 자주 처리해야 하는 경우
3. 테이블에 지나치게 조인을 많이 사용하게 되어 데이터를 조회하는 것이 기술적으로 어려울 경우

### 반정규화 시 주의점

- 데이터의 무결성이 깨질 수 있음
- 입력, 수정, 삭제의 질의문에 대한 응답 시간이 늦어질 수 있음

**연관 개념**: [JOIN](about:blank#%EC%A1%B0%EC%9D%B8), [트랜잭션 격리수준](about:blank#%ED%8A%B8%EB%9E%9C%EC%9E%AD%EC%85%98-%EA%B2%A9%EB%A6%AC%EC%88%98%EC%A4%80)

# Transaction

## 트랜잭션(Transaction)이란 무엇인가?

트랜잭션은 작업의 **완전성(Completeness)**을 보장해주는 것이다. 즉, 논리적인 작업 셋을 모두 완벽하게 처리하거나 또는 처리하지 못할 경우에는 원 상태로 복구해서 작업의 일부만 적용되는 현상이 발생하지 않게 만들어주는 기능.

- **사용자 입장**: 작업의 논리적 단위
- **시스템 입장**: 데이터들을 접근 또는 변경하는 프로그램의 단위

**주요 개념**:
- **커밋(Commit)**: 여러 쿼리가 성공적으로 처리되었다고 확정하는 명령어. 변경된 내용이 영구적으로 저장
- **롤백(Rollback)**: 트랜잭션으로 처리한 하나의 묶음 과정을 일어나기 전으로 되돌리는 일(취소)
- **트랜잭션 전파**: 여러 트랜잭션 관련 메서드의 호출을 하나의 트랜잭션에 묶이도록 하는 것

## 트랜잭션과 Lock

**잠금(Lock)**과 **트랜잭션**은 서로 다른 개념:
- **잠금**: 동시성을 제어하기 위한 기능. 여러 커넥션에서 동시에 동일한 자원을 요청할 경우 순서대로 한 시점에는 하나의 커넥션만 변경할 수 있게 해주는 역할
- **트랜잭션**: 데이터의 정합성을 보장하기 위한 기능. 하나의 논리적인 작업 셋 자체가 100% 적용되거나 아무것도 적용되지 않아야 함을 보장

**연관 개념**: [교착상태](about:blank#%EA%B5%90%EC%B0%A9%EC%83%81%ED%83%9C), [트랜잭션 격리수준](about:blank#%ED%8A%B8%EB%9E%9C%EC%9E%AD%EC%85%98-%EA%B2%A9%EB%A6%AC%EC%88%98%EC%A4%80)

## 트랜잭션의 특성 (ACID)

Transaction은 다음의 **ACID**라는 4가지 특성을 만족해야 한다.

### 원자성 (Atomicity) - “All or Nothing”

트랜잭션과 관련된 일이 **모두 수행되었거나 되지 않았거나**를 보장하는 특징. 트랜잭션 중간에 어떠한 문제가 발생한다면 트랜잭션에 해당하는 어떠한 작업 내용도 수행되어서는 안 되며, 아무런 문제가 발생되지 않았을 경우에만 모든 작업이 수행되어야 한다.

### 일관성 (Consistency)

**’허용된 방식’으로만 데이터를 변경**해야 하는 것. 트랜잭션이 완료된 다음의 상태에서도 트랜잭션이 일어나기 전의 상황과 동일하게 데이터의 일관성을 보장해야 한다.

### 고립성 (Isolation)

각각의 트랜잭션은 **서로 간섭없이 독립적으로 수행**되어야 한다. 여러 트랜잭션이 동시에 실행될 때 한 트랜잭션의 실행이 다른 트랜잭션에 영향을 미치지 않아야 한다.

### 지속성 (Durability)

트랜잭션이 정상적으로 종료된 다음에는 **영구적으로 데이터베이스에 작업의 결과가 저장**되어야 한다.

## 트랜잭션 격리수준 (Isolation Level)

트랜잭션의 격리성(Isolation)은 여러 단계가 있으며, 해당 단계에 따라 격리성과 동시성의 정도가 다르다.

**격리성과 동시성은 반비례 관계**:
- 격리성이 높으면 → 동시성이 낮아짐 (성능 저하)
- 격리성이 낮으면 → 동시성이 높아짐 (데이터 정합성 문제 발생 가능)

### 격리수준 설정 (MySQL)

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

## 격리수준에 따른 현상

**1. 팬텀 리드 (Phantom Read)**
- 한 트랜잭션 내에서 동일한 쿼리를 2번 이상 보냈을 때 해당 조회 결과가 다른 것
- 유령(phantom)처럼 새로운 행이 생기거나 사라지는 현상

**2. 반복 가능하지 않은 조회 (Non-repeatable Read)**
- 한 트랜잭션 내의 같은 행에 두 번 이상 조회가 발생했는데, 그 값이 다른 것
- 다른 트랜잭션이 해당 행을 수정하여 발생

**3. 더티 리드 (Dirty Read)**
- 하나의 트랜잭션이 다른 트랜잭션의 **아직 커밋되지 않은 데이터를 읽는** 현상
- 롤백될 수 있는 데이터를 읽어 데이터 정합성 문제 발생

## 격리수준의 종류

**1. SERIALIZABLE (직렬화 가능)**
- 가장 엄격한 격리 수준
- 커밋 완료된 데이터에 대해서만 조회 가능
- 트랜잭션을 순차적으로 진행
- 여러 트랜잭션이 동시에 같은 행에 접근 불가
- **현상**: 팬텀 리드, 반복 가능하지 않은 조회, 더티 리드 모두 발생 안 함
- **장점**: 데이터 정합성 완벽 보장
- **단점**: 동시성 처리 성능 최저

**2. REPEATABLE READ (반복 가능한 읽기)**
- 커밋 완료된 데이터에 대해서만 조회 가능
- 반복해서 행을 조회하더라도 똑같은 행을 보장
- 하나의 트랜잭션이 수정한 행을 다른 트랜잭션이 수정 불가
- 새로운 행을 추가하는 것은 막지 않음
- **현상**: 팬텀 리드 발생 가능, 반복 가능하지 않은 조회/더티 리드 발생 안 함
- **MySQL 8.0 InnoDB의 기본값**
- **장점**: 일반적인 경우에 적절한 성능과 정합성

**3. READ COMMITTED (커밋된 읽기)**
- 커밋 완료된 데이터에 대해서만 조회 가능
- 커밋이 되지 않은 정보는 읽지 못함
- **현상**: 팬텀 리드, 반복 가능하지 않은 조회 발생 가능, 더티 리드 발생 안 함
- **PostgreSQL, SQL Server, Oracle의 기본값**
- **가장 많이 사용되는 격리 수준**
- **장점**: 높은 동시성과 적절한 정합성

**4. READ UNCOMMITTED (커밋되지 않은 읽기)**
- 가장 낮은 격리 수준이며 가장 빠름
- 다른 트랜잭션이 커밋하지 않은 정보를 읽을 수 있음
- **현상**: 팬텀 리드, 반복 가능하지 않은 조회, 더티 리드 모두 발생 가능
- **단점**: 데이터 정합성 문제 발생 가능성 높음

| 격리수준 | Dirty Read | Non-repeatable Read | Phantom Read |
| --- | --- | --- | --- |
| READ UNCOMMITTED | O | O | O |
| READ COMMITTED | X | O | O |
| REPEATABLE READ | X | X | O |
| SERIALIZABLE | X | X | X |

**연관 개념**: [교착상태](about:blank#%EA%B5%90%EC%B0%A9%EC%83%81%ED%83%9C), [Lock](about:blank#%ED%8A%B8%EB%9E%9C%EC%9E%AD%EC%85%98%EA%B3%BC-lock)

## 트랜잭션의 상태

![image.png](%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4/image.png)

- **Active**: 트랜잭션의 활동 상태. 트랜잭션이 실행중이며 동작중인 상태
- **Failed**: 트랜잭션 실패 상태. 트랜잭션이 더이상 정상적으로 진행 할 수 없는 상태
- **Partially Committed**: 트랜잭션의 `Commit` 명령이 도착한 상태. 트랜잭션의 commit 이전 sql문이 수행되고 commit만 남은 상태
- **Committed**: 트랜잭션 완료 상태. 트랜잭션이 정상적으로 완료된 상태
- **Aborted**: 트랜잭션이 취소 상태. 트랜잭션이 취소되고 트랜잭션 실행 이전 데이터로 돌아간 상태

**Partially Committed 와 Committed의 차이점**:
- Commit 요청이 들어오면 `Partial Committed` 상태가 된다
- Commit을 문제없이 수행할 수 있으면 `Committed` 상태로 전이
- 오류가 발생하면 `Failed` 상태가 됨

### 트랜잭션을 사용할 때 주의할 점

트랜잭션은 **꼭 필요한 최소의 코드에만 적용**하는 것이 좋다. 즉, **트랜잭션의 범위를 최소화**하라는 의미.

**이유**:
- 데이터베이스 커넥션은 개수가 제한적
- 각 단위 프로그램이 커넥션을 소유하는 시간이 길어지면 사용 가능한 여유 커넥션의 개수가 줄어듦
- 어느 순간에는 각 단위 프로그램에서 커넥션을 가져가기 위해 기다려야 하는 상황 발생 가능

**연관 개념**: [데이터베이스 성능](about:blank#%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4%EC%9D%98-%EC%84%B1%EB%8A%A5), [교착상태](about:blank#%EA%B5%90%EC%B0%A9%EC%83%81%ED%83%9C)

# 교착상태

## 교착상태란 무엇인가

복수의 트랜잭션을 사용하다보면 교착상태가 일어날 수 있다. 교착상태(**Deadlock**)란 **두 개 이상의 트랜잭션이 특정 자원(테이블 또는 행)의 잠금(Lock)을 획득한 채 다른 트랜잭션이 소유하고 있는 잠금을 요구하면 아무리 기다려도 상황이 바뀌지 않는 상태**.

## 교착상태 발생 조건 (4가지 모두 충족 시 발생)

1. **상호 배제 (Mutual Exclusion)**: 자원은 한 번에 하나의 트랜잭션만 사용할 수 있음
2. **점유 및 대기 (Hold and Wait)**: 트랜잭션이 최소한 하나의 자원을 점유하고 있으면서 다른 트랜잭션이 점유한 자원을 추가로 요구
3. **비선점 (No Preemption)**: 한 트랜잭션이 점유한 자원은 그 트랜잭션이 명시적으로 해제할 때까지 다른 트랜잭션에 의해 강제로 빼앗길 수 없음
4. **순환 대기 (Circular Wait)**: 각 트랜잭션이 순환적으로 다음 트랜잭션에 의해 점유된 자원을 대기

**연관 개념**: [운영체제 교착상태](관련개념)

### 교착상태의 예(MySQL)

![image.png](%EB%8D%B0%EC%9D%B4%ED%84%B0%EB%B2%A0%EC%9D%B4%EC%8A%A4/image%201.png)

MySQL의 MVCC(Multi-Version Concurrency Control)에 따른 특성 때문에 트랜잭션에서 갱신 연산(INSERT, UPDATE, DELETE)를 실행하면 잠금을 획득한다. (기본은 행에 대한 잠금)

**예시**:

```sql
-- Transaction 1CREATE TABLE B (i1 int not null primary key) engine = innodb;
START TRANSACTION;
INSERT INTO B VALUES(1);
-- Transaction 2CREATE TABLE A (i1 int not null primary key) engine = innodb;
START TRANSACTION;
INSERT INTO A VALUES(1);
-- Transaction 1이 A의 잠금을 요청INSERT INTO A VALUES(1);  -- 대기 상태-- Transaction 2가 B의 잠금을 요청INSERT INTO B VALUES(1);  -- ERROR 1213: Deadlock found
```

일반적인 DBMS는 교착상태를 독자적으로 검출해 보고한다.

## 교착상태 빈도를 낮추는 방법

### 1. 교착상태 탐지 (Detection)

**Timeout 방식**:
- 일정시간(timeout) 이후 트랜잭션이 실행되지 않았을 경우 롤백
- 장점: 구현이 간단하고, 데드락 상황을 빠르게 해결
- 단점: 타임아웃 시간 설정이 어려움, 실제 데드락이 아닌 상황에서도 트랜잭션 중단 가능

**Wait-for Graph 방식**:
- 트랜잭션에 관한 대기 그래프를 만들고 해당 그래프에 사이클이 발생됨을 주기적으로 검사
- 단점: 대규모 데이터베이스의 경우 모든 트랜잭션의 사이클을 찾는 비용이 큼

### 2. 교착상태 방지 (Prevention)

**격리수준 변경**:
- 데드락을 방지하기 위해 격리 수준을 조정
- 예: REPEATABLE_READ에서 SERIALIZABLE로 변경
- 단점: SERIALIZABLE이 아닌 이상 모든 데드락을 제거할 수 없음

**자원 할당 순서 지정 (Ordering of Resource Allocation)**:
- 서비스 로직을 구축할 때 자원할당이 교착상태가 발생하지 않도록 논리구조 변경
- 모든 트랜잭션이 동일한 테이블 순으로 접근하도록 함

**Wait-Die 방식 (비선점 기법)**:
- 타임스탬프를 기반으로 오래된 트랜잭션을 우선순위가 높다고 판정
- Wait: 우선순위가 높은 트랜잭션이 자원 요청 시 대기
- Die: 우선순위가 낮은 트랜잭션이 자원 요청 시 롤백 후 재시도
- 예: T5(오래된)가 T10(최신)의 자원 요청 → T5는 대기
- 예: T10(최신)이 T5(오래된)의 자원 요청 → T10은 롤백

**Wound-Wait 방식 (선점 기법)**:
- 타임스탬프를 기반으로 오래된 트랜잭션을 우선순위가 높다고 판정
- Wound: 우선순위가 높은 트랜잭션이 자원 요청 시 낮은 우선순위 트랜잭션의 자원을 강제 해제
- Wait: 우선순위가 낮은 트랜잭션이 자원 요청 시 높은 우선순위 트랜잭션이 해제될 때까지 대기
- 예: T5(오래된)가 T10(최신)의 자원 요청 → T10 강제 해제, T5가 자원 획득
- 예: T10(최신)이 T5(오래된)의 자원 요청 → T10은 T5 종료까지 대기

### 일반적인 교착상태 방지 팁

- 트랜잭션을 자주 커밋한다
- 정해진 순서로 테이블에 접근한다
- 읽기 잠금 획득 (SELECT ~ FOR UPDATE)의 사용을 피한다
- 한 테이블의 복수 행을 복수의 연결에서 순서 없이 갱신하면 교착상태가 발생하기 쉬움 → 테이블 단위의 잠금을 획득해 갱신을 직렬화 (동시성은 떨어지지만 교착상태 회피)

# 조인

## 조인(JOIN)의 종류

조인(Join)은 두 개 이상의 테이블을 연결하여 데이터를 탐색하는 방법.

### 내부 조인 (Inner Join)

- 두 테이블에서 일치하는 데이터만 반환 (교집합)

```sql
SELECT E.Name, D.DepartmentName
FROM Employees E
INNER JOIN Departments D ON E.DepartmentID = D.DepartmentID;
```

### 외부 조인 (Outer Join)

**왼쪽 외부 조인 (Left Outer Join)**:
- 첫 번째(왼쪽) 테이블의 모든 레코드와 두 번째(오른쪽) 테이블의 일치하는 레코드 반환
- 오른쪽 테이블에 일치하는 레코드가 없으면 NULL 값

**오른쪽 외부 조인 (Right Outer Join)**:
- 두 번째(오른쪽) 테이블의 모든 레코드와 첫 번째(왼쪽) 테이블의 일치하는 레코드 반환
- 왼쪽 테이블에 일치하는 레코드가 없으면 NULL 값

**전체 외부 조인 (Full Outer Join)**:
- 두 테이블의 모든 레코드 반환
- 한 테이블에만 존재하는 레코드는 다른 테이블 부분이 NULL 값

### 교차 조인 (Cross Join)

- 두 개 이상의 테이블에서 가능한 모든 행의 조합을 생성
- 카테시안 곱(Cartesian Product) 형성
- 예: 3개 행 × 3개 행 = 9개 행
- 성능이 좋지 않지만, 테스팅 시 사용

### 자연 조인 (Natural Join)

- 두 테이블 간의 동일한 이름을 가진 모든 열을 기반으로 자동 조인
- 명시적으로 조인 조건을 지정할 필요 없음
- 내부 조인(Inner Join)을 자동으로 수행

**성능 비교**:
- 일반적으로 Inner Join이 Outer Join보다 빠름
- 결과 집합이 양쪽 테이블에서 일치하는 행만 필요로 하기 때문
- 실제 성능은 쿼리, 데이터베이스 구조, 인덱스 구조 등에 따라 다름

**연관 개념**: [조인 알고리즘](about:blank#%EC%A1%B0%EC%9D%B8-%EC%95%8C%EA%B3%A0%EB%A6%AC%EC%A6%98), [인덱스](about:blank#index)

## 조인 알고리즘

데이터베이스의 Query Optimizer가 자동으로 최적의 조인 알고리즘을 선택한다.

### 1. 중첩 루프 조인 (Nested Loop Join)

한 테이블의 각 행에 대해 다른 테이블의 모든 행을 순차적으로 검사하여 조인.

**특징**:
- 시간 복잡도: O(M × N)
- 소규모 데이터셋에는 적합
- 테이블 크기가 커지면 비효율적
- 외부 테이블의 각 행에 대해 내부 테이블 전체 스캔

**블록 중첩 루프 조인 (BNL, Block Nested Loop)**:
- 중첩루프조인의 개선 버전
- 한 테이블의 블록(여러 행의 집합)을 조인 버퍼(메모리)에 한번에 로드
- 해당 블록에 대해 다른 테이블의 각 행을 순회하여 조인

### 2. 정렬 병합 조인 (Sort Merge Join)

두 테이블을 조인 필드를 기준으로 정렬한 후, 정렬된 순서에 따라 병합하여 조인.

**특징**:
- 시간 복잡도: O(M log M + N log N)
- 데이터가 이미 정렬되어 있을 때 더 효율적
- 등가조인이 아닌 비등가조인(<, >, <=, >=)에 효율적
- 동등(=) 조인과 비동등 조인 모두 사용 가능

### 3. 해시 조인 (Hash Join)

한 테이블(보통 더 작은 테이블)의 행을 사용하여 해시 테이블을 만든 다음, 다른 테이블을 순회하며 해시 테이블을 사용하여 조인.

**작동 방식**:
1. **빌드 단계(Build Phase)**: 바이트가 더 작은 테이블로부터 해시 테이블 생성. 각 행의 조인 키에 해시 함수를 적용하여 해시 테이블에 저장
2. **프로브 단계(Probe Phase)**: 다른 테이블의 각 행에 대한 조인키를 해싱하여 해시 테이블에서 매칭되는 행을 찾아 조인

**특징**:
- 시간 복잡도: O(M + N)
- 큰 데이터셋에 대해 매우 효율적
- 두 테이블 사이에 큰 크기 차이가 있을 때 효율적
- **동등(=) 조인에만 사용 가능**
- 하나의 테이블이 서버 메모리에 온전히 들어간다면 중첩루프조인보다 더 효율적
- 메모리가 부족할 경우 성능 저하

**MySQL 힌트 설정**:

```sql
-- 해시조인 비활성화SET optimizer_switch="hash_join=off";
```

# 스토리지 엔진

## InnoDB vs MyISAM

데이터베이스의 스토리지 엔진은 데이터의 저장, 검색, 트랜잭션 관리, 동시성 제어, 캐싱 등 핵심 기능을 담당하는 필수 구성 요소.

**InnoDB**는 MySQL 8.0의 기본 스토리지 엔진이며, **MyISAM**으로도 변경이 가능.

| 특성 | InnoDB | MyISAM |
| --- | --- | --- |
| 트랜잭션 지원 | 지원 | 미지원 |
| 락 및 동시성 | 행 레벨 락(Row-level Lock) | 테이블 레벨 락(Table-level Lock) |
| 외래 키 제약 | 지원 | 미지원 |
| 복구 기능 | 크래시 복구 지원 | 제한적 |
| MVCC | 지원 | 미지원 |
| 전문 검색 | 지원 (5.6+) | 지원 |
| 사용 사례 | 트랜잭션이 필요한 애플리케이션 | 읽기 위주 애플리케이션 |

**InnoDB 특징**:
- 행 레벨 락으로 동시성 높음
- ACID 속성 완벽 지원
- 외래 키를 통한 참조 무결성 보장
- MVCC(Multi-Version Concurrency Control)로 읽기 작업이 쓰기 작업을 차단하지 않음

**MyISAM 특징**:
- 테이블 레벨 락으로 동시성 낮음
- 트랜잭션 미지원 (ROLLBACK 불가)
- 빠른 SELECT 성능
- 전문 검색(Full-text search)에 유리

**언제 MyISAM을 사용하는가?**
- 트랜잭션이 필요 없는 경우
- 주로 SELECT만 수행하는 경우
- 데이터 변경이 거의 없는 경우

**연관 개념**: [트랜잭션](about:blank#transaction), [Lock](about:blank#%ED%8A%B8%EB%9E%9C%EC%9E%AD%EC%85%98%EA%B3%BC-lock), [MVCC](관련개념)

# Statement vs PreparedStatement

**속도**: `PreparedStatement`가 빠르다. 쿼리를 수행하기 전에 이미 쿼리가 컴파일 되어 있으며, 반복 수행의 경우 프리 컴파일된 쿼리를 통해 수행이 이루어지기 때문.

**SQL 타입**:
- `Statement`: 변수를 설정하고 바인딩하는 Static SQL 사용
- `PreparedStatement`: 쿼리 자체에 조건이 들어가는 Dynamic SQL 사용 

→ `dynamic sql`을 사용하는데 따르는 퍼포먼스 저하를 고려하지 않을 수 없다.

**성능 고려사항**:
허나 SQL 문을 파싱하는 시간은 전체 실행 시간의 10분의 1에 불과. 가장 큰 비중을 차지하는 것은 테이블에서 레코드(row)를 가져오는 과정.

**보안**: `PreparedStatement`는 **SQL Injection** 등의 문제를 보완. 따라서 `PreparedStatement`를 사용하는 것이 더 안전하고 권장됨.

# NoSQL

## 정의

관계형 데이터 모델을 **지양**하며 대량의 분산된 데이터를 저장하고 조회하는 데 특화되었으며 스키마 없이 사용 가능하거나 느슨한 스키마를 제공하는 저장소.

**특징**:
- 대량의 데이터를 빠르게 처리하기 위해 메모리에 임시 저장하고 응답
- 동적인 스케일 아웃 지원
- 가용성을 위한 데이터 복제
- 관계형 데이터베이스가 제공하지 못하는 성능과 특징 제공

## CAP 이론

분산 시스템에서 다음 세 가지 속성을 모두 만족하는 것은 불가능. 두 가지만 만족할 수 있다.

### 1. 일관성 (Consistency)

동시성 또는 동일성. 다중 클라이언트에서 같은 시간에 조회하는 데이터는 항상 동일한 데이터임을 보증.

**데이터 동기화 방법**:
- **동기식**: 데이터 저장 결과를 클라이언트로 응답하기 전에 모든 노드에 데이터 저장. 느린 응답시간, 데이터 정합성 보장
- **비동기식**: 메모리나 임시 파일에 기록 후 클라이언트에 먼저 응답, 이후 노드로 데이터 동기화. 빠른 응답시간, 쓰기 노드 장애 시 데이터 손실 가능

**최종 일관성 (Eventual Consistency)**: 데이터의 변경을 시간의 흐름에 따라 여러 노드에 전파

### 2. 가용성 (Availability)

모든 클라이언트의 읽기와 쓰기 요청에 대하여 항상 응답이 가능해야 함을 보증. 내고장성(Fault Tolerance)이라고도 함.

**보장 방법 - 데이터 복제 (Replication)**:
- **Master-Slave 복제**: 동일한 데이터를 가진 저장소를 하나 더 생성
- **Peer-to-Peer 복제**: 데이터 단위로 중복 저장

### 3. 네트워크 분할 허용성 (Partition Tolerance)

지역적으로 분할된 네트워크 환경에서 두 지역 간의 네트워크가 단절되거나 데이터 유실이 일어나더라도 각 지역 내의 시스템은 정상적으로 동작해야 함.

## 저장방식에 따른 NoSQL 분류

### 1. Key-Value Model

가장 기본적인 형태. 키 하나로 데이터 하나를 저장하고 조회할 수 있는 단일 키-값 구조.

**특징**:
- 단순한 저장구조로 복잡한 조회 연산 미지원
- 고속 읽기와 쓰기에 최적화
- 트랜잭션 처리 불가능
- 데이터 정합성 보장 어려움

**사용 사례**:
- 사용자 프로필 정보
- 웹 서버 클러스터의 세션 정보
- 장바구니 정보
- URL 단축 정보

**예시**: Redis

### 2. Document Model

키-값 모델을 개념적으로 확장. 하나의 키에 하나의 구조화된 문서를 저장하고 조회.

**특징**:
- 논리적 데이터 저장과 조회 방법이 관계형 데이터베이스와 유사
- 키는 문서에 대한 ID로 표현
- 저장된 문서를 컬렉션으로 관리
- 문서 ID에 대한 인덱스를 생성하여 O(1) 시간에 문서 조회
- B-트리 인덱스를 사용하여 2차 인덱스 생성
- 읽기와 쓰기 비율이 7:3일 때 가장 좋은 성능

**사용 사례**:
- 중앙 집중식 로그 저장
- 타임라인 저장
- 통계 정보 저장

**예시**: MongoDB

### 3. Column Model

하나의 키에 여러 개의 컬럼 이름과 컬럼 값의 쌍으로 이루어진 데이터를 저장하고 조회. 모든 컬럼은 항상 타임 스탬프 값과 함께 저장.

**구조**:
- **컬럼**: 컬럼 이름 + 컬럼 값 + 타임스탬프
- **로우(Row)**: 컬럼들의 집합
- **로우키(Row Key)**: 각 로우를 유일하게 식별
- **키 스페이스(Key Space)**: 로우들의 집합

**특징**:
- 쓰기에 더 특화 (읽기보다)
- 데이터를 먼저 커밋로그와 메모리에 저장 후 응답 → 빠른 응답속도
- 쓰기 연산이 많은 서비스에 적합
- 대량의 데이터 입력 및 조회에 적합

**사용 사례**:
- 채팅 내용 저장
- 실시간 분석을 위한 데이터 저장소

**예시**: Google Bigtable, Apache Cassandra, HBase

## MongoDB의 특징

### 1. 기본키: ObjectID (_id)

도큐먼트 생성 시마다 다른 컬렉션에서 중복된 값을 지니기 힘든 유니크한 값이며 기본키인 ObjectID가 자동 생성.

**구성**:
- 유닉스 시간 기반 타임스탬프(4바이트)
- 랜덤 값(5바이트)
- 카운트(3바이트)

### 2. BSON (Binary JSON)

- 클라이언트와 서버 간 데이터 교환: JSON 형식
- 내부적 데이터 저장 및 처리: BSON(Binary JSON) 형식
- BSON은 추가적인 데이터 타입을 지원하고 바이너리 형식으로 효율적인 저장 가능

### 3. 확장성

샤딩(Sharding)을 통한 수평적 확장 지원. 데이터를 여러 서버에 분산시켜 단일 서버 부하를 줄이고 대규모 데이터셋에 걸쳐 쿼리 성능 향상.

### 4. 2차원 좌표 인덱싱

2D sphere index를 사용하여 2차원 좌표를 인덱싱 가능. 지리적 위치 기반 쿼리에 유용.

### 5. 트랜잭션

- **체크포인트**: 60초 간격으로 생성, 마지막 유효한 체크포인트에서 복구 가능
- **저널링(Journaling)**: 체크포인트 간의 모든 데이터 수정 사항 유지, 체크포인트 사이에 종료되면 저널을 사용하여 복구

### MongoDB 사용 시 주의점

**1. 크기**:
- 스키마 없이 유동적으로 데이터 삽입 가능
- key-value 형태로 저장될 때 key의 길이도 매번 저장됨
- RDBMS보다 더 많은 저장 공간 필요

**2. 메모리**:
- WiredTiger 스토리지 엔진은 서버 총 RAM의 약 50%를 캐시로 사용
- 정확히는 (총 RAM - 1GB)의 50%
- 최소 캐시 할당량: 256MB

## 연관 개념 정리

### 인덱스 관련

- B-Tree 자료구조 ↔︎ 디스크 I/O 최적화
- Clustered Index ↔︎ 물리적 정렬 ↔︎ 범위 검색 성능
- Composite Index ↔︎ 쿼리 패턴 ↔︎ 카디널리티

### 트랜잭션 관련

- ACID ↔︎ 격리수준 ↔︎ 동시성 제어
- Lock ↔︎ 교착상태 ↔︎ 성능
- 격리수준 ↔︎ Phantom Read/Non-repeatable Read/Dirty Read

### 정규화 관련

- 함수적 종속성 ↔︎ 정규형
- 정규화 ↔︎ JOIN 증가 ↔︎ 성능 저하
- 반정규화 ↔︎ 중복 허용 ↔︎ 조회 성능 향상

### 조인 관련

- 조인 알고리즘 ↔︎ 인덱스 활용 ↔︎ 쿼리 최적화
- 중첩 루프 조인 ↔︎ 인덱스 스캔
- 해시 조인 ↔︎ 메모리 사용 ↔︎ 동등 조인

### NoSQL 관련

- CAP 이론 ↔︎ 분산 시스템 ↔︎ 일관성 vs 가용성
- RDBMS vs NoSQL ↔︎ ACID vs BASE
- Document Model ↔︎ 유연한 스키마 ↔︎ 빠른 개발
