# 암호화 방법/방식

# 대표적인 암호화 방법/방식 비교

암호화 기술은 **목적**에 따라 크게 **단방향(해싱)**과 **양방향(암호화/복호화)**로 나뉩니다.

---

## 1. 해싱 (Hashing) - 단방향 암호화

### **핵심 개념**

- **복호화 불가능**: 원본 데이터로 되돌릴 수 없음
- **고정 길이 출력**: 입력 크기와 무관하게 일정한 길이의 해시값 생성
- **동일 입력 → 동일 출력**: 같은 데이터는 항상 같은 해시값
- **눈사태 효과**: 입력의 작은 변화도 완전히 다른 해시값 생성

### **대표적인 해싱 알고리즘**

**MD5 (Message Digest 5)**

- **출력 크기**: 128bit (32자리 16진수)
- **상태**: **취약 - 사용 지양**
- **문제점**:
    - 레인보우 테이블 공격에 취약
    - 충돌 발생 가능 (서로 다른 입력이 같은 해시값)
- **사용**: 과거 비밀번호 저장 (현재는 권장하지 않음)

**SHA 계열 (Secure Hash Algorithm)**

- **SHA-1**: 160bit - 취약, 2017년 충돌 발견
- **SHA-256**: 256bit - **현재 표준**, 블록체인에 사용
- **SHA-512**: 512bit - 더 강력한 보안
- **특징**:
    - MD5보다 안전
    - 인증서 검증, 데이터 무결성 확인에 사용
    - 비트코인 등 블록체인의 기반

**bcrypt**

- **목적**: 비밀번호 저장 전용
- **핵심 특징**:
    - **Salt 자동 생성**: 같은 비밀번호도 다른 해시값
    - **의도적으로 느림** (Cost Factor 조절 가능)
    - 무차별 대입 공격(brute force) 방어
- **현대 표준**: 데이터베이스 비밀번호 저장에 가장 많이 사용

### **사용 사례**

- 비밀번호 저장 (bcrypt)
- 데이터 무결성 검증 (SHA-256)
- 디지털 서명
- 블록체인

---

## 2. 대칭키 암호화 (Symmetric Encryption) - 양방향

### **핵심 개념**

- **암호화와 복호화에 같은 키 사용**
- 빠른 속도
- 키 관리가 핵심 문제

### **대표 알고리즘**

**AES (Advanced Encryption Standard)**

- **키 크기**: 128, 192, 256bit
- **현재 표준**: 정부 기관, 금융권에서 사용
- **특징**:
    - 매우 빠름
    - 하드웨어 가속 지원
    - 블록 암호화 방식
- **사용**: HTTPS, VPN, 파일 암호화

**DES / 3DES**

- **DES**: 56bit - **취약, 사용 중단**
- **3DES**: DES를 3번 적용 - 느려서 AES로 대체됨

### **장단점**

| 장점 | 단점 |
| --- | --- |
| 속도가 매우 빠름 | **키 배포 문제** - 안전하게 키를 전달하기 어려움 |
| 대용량 데이터 암호화에 적합 | 통신 상대방 수만큼 키 필요 (N명이면 N개) |
| 구현이 비교적 간단 | 키 유출 시 전체 시스템 위험 |

### **핵심 문제: 키 교환**

→ 이 문제를 해결하기 위해 **비대칭키 암호화** 등장

## 3. 비대칭키 암호화 (Asymmetric Encryption) - 양방향

### **핵심 개념**

- *공개키(Public Key)**와 **개인키(Private Key)** 한 쌍 사용
- **공개키로 암호화 → 개인키로 복호화**
- **개인키로 서명 → 공개키로 검증**

### **대표 알고리즘**

**RSA**

- **키 크기**: 2048, 4096bit
- **원리**: 큰 소수의 곱셈은 쉽지만 소인수분해는 어렵다는 수학적 난제 이용
- **특징**:
    - 매우 안전
    - **대칭키보다 수백~수천 배 느림**
    - 주로 대칭키를 암호화하는데 사용 (하이브리드 방식)

**ECC (Elliptic Curve Cryptography)**

- **타원곡선 암호화**
- RSA보다 짧은 키로 같은 수준의 보안
- 모바일 기기에 적합

### **장단점**

| 장점 | 단점 |
| --- | --- |
| **키 배포 문제 해결** - 공개키는 공개 가능 | 속도가 매우 느림 |
| 디지털 서명 가능 | 대용량 데이터 암호화 부적합 |
| 키 관리 용이 | 복잡한 수학적 연산 |

### **사용 방식**

**1. 암호화/복호화**

```
발신자: 수신자의 공개키로 암호화 → 전송
수신자: 자신의 개인키로 복호화

```

**2. 디지털 서명 (무결성 + 인증)**

```
발신자: 데이터를 해싱 → 개인키로 서명 → 전송
수신자: 발신자의 공개키로 서명 검증 → 무결성/신원 확인

```

## 4. 하이브리드 암호화 (실제 사용 방식)

### **HTTPS/TLS의 암호화 과정**

**문제**:

- 대칭키는 빠르지만 키 교환이 어려움
- 비대칭키는 안전하지만 느림

**해결**: **둘을 조합**

**TLS Handshake 과정**:

1. **비대칭키로 대칭키 교환** (RSA/ECDH)
    - 클라이언트가 서버의 공개키(인증서)로 세션키(대칭키) 암호화
    - 서버는 개인키로 복호화하여 세션키 획득
2. **이후 모든 통신은 대칭키(AES)로 암호화**
    - 빠른 속도로 대용량 데이터 암호화

**장점**:

- 안전한 키 교환 (비대칭키)
- 빠른 통신 (대칭키)

## 5. 인증 관련 암호화

### **CA (Certificate Authority) - 인증 기관**

- **역할**: 디지털 인증서 발급 및 검증
- **인증서 내용**:
    - 서버의 공개키
    - 도메인 정보
    - CA의 디지털 서명 (CA 개인키로 서명)

**TLS 인증서 검증 과정**:

1. 서버가 인증서 전송 (공개키 + CA 서명 포함)
2. 브라우저가 인증서 내용 해싱 (해시값 A)
3. CA의 공개키로 CA 서명 복호화 (해시값 B)
4. A == B 비교 → 위조 여부 판단

### **JWT (JSON Web Token) - 토큰 기반 인증**

- **구조**: Header.Payload.Signature
- **Signature 생성**:
    
    ```
    HMAC-SHA256(  base64(header) + "." + base64(payload),  secret_key)
    
    ```
    
- **특징**:
    - 서버가 비밀키로 서명
    - 클라이언트는 토큰을 저장하여 인증에 사용
    - 서버는 서명 검증으로 위조 여부 확인

**Access Token / Refresh Token**:

- **Access Token**: 짧은 수명 (15분~1시간), 실제 API 호출에 사용
- **Refresh Token**: 긴 수명 (2주~수개월), Access Token 재발급용
- 이유: **보안과 편의성 균형** - Access Token 탈취 피해 최소화

## 6. 선택 기준 정리

| 목적 | 사용 방식 | 이유 |
| --- | --- | --- |
| **비밀번호 저장** | bcrypt, scrypt | 복호화 불필요, Salt, 느린 속도로 brute force 방어 |
| **데이터 무결성 검증** | SHA-256 | 빠르고 안전한 해싱 |
| **대용량 데이터 암호화** | AES | 빠른 속도 |
| **안전한 키 교환** | RSA, ECDH | 키 배포 문제 해결 |
| **디지털 서명** | RSA + SHA-256 | 신원 확인 + 무결성 |
| **웹 통신 보안** | HTTPS (TLS) | 하이브리드 방식 |
| **API 인증** | JWT + bcrypt | 무상태 인증 |

## 7. 다른 개념과의 연결

**해시 테이블과의 관계**

- 해시 테이블: 빠른 탐색을 위한 해싱 (보안 목적 아님)
- 암호화 해싱: 보안이 목적, 충돌 저항성 중요

**데이터베이스**

- 비밀번호는 bcrypt로 해싱하여 저장
- 암호화 필드는 AES로 암호화 (복호화 필요 시)
- 전송 중 데이터는 TLS로 보호

**네트워크 프로토콜**

- **HTTP**: 평문 전송 (포트 80)
- **HTTPS**: TLS 암호화 (포트 443)
- **SSH**: 비대칭키 인증 + 대칭키 통신

**블록체인**

- SHA-256 해싱으로 블록 연결
- 디지털 서명으로 트랜잭션 인증

## 8. 면접 꼬리질문 대비

**"왜 HTTPS는 RSA와 AES를 함께 사용하나?"**
→ RSA는 안전하지만 느려서 대용량 데이터 암호화 부적합. RSA로 AES 세션키를 안전하게 교환한 뒤, 빠른 AES로 실제 데이터 암호화

**"bcrypt가 의도적으로 느린 이유는?"**
→ 무차별 대입 공격(brute force) 방어. 1초에 수억 번 시도를 1초에 수백 번으로 제한

**"Salt가 무엇이고 왜 필요한가?"**
→ 같은 비밀번호도 다른 해시값을 갖게 하는 랜덤 문자열. 레인보우 테이블 공격 방어

**"JWT는 왜 서명만 하고 암호화하지 않나?"**
→ Payload는 base64 인코딩(암호화 아님)으로 누구나 읽을 수 있음. **목적은 위조 방지이지 내용 숨김이 아님**. 민감 정보는 Payload에 넣지 않음

**"대칭키와 비대칭키를 언제 각각 사용하나?"**
→ 대칭키는 대용량 데이터 암호화, 비대칭키는 키 교환 및 디지털 서명. 실무에서는 하이브리드 사용

---

## 9. 실무 관점

- **절대 자체 암호화 알고리즘을 만들지 말 것**
- **검증된 라이브러리 사용**: OpenSSL, Bouncy Castle 등
- **키 관리가 가장 중요**: 코드에 하드코딩 금지, 환경변수나 Key Management Service 사용
- **정기적인 키 교체**: 특히 대칭키
- **최신 보안 표준 준수**: TLS 1.3, AES-256 등

이 정도로 정리하시면 암호화의 핵심 개념과 실전 활용, 그리고 면접에서의 깊이 있는 대응이 가능합니다.
