# 자료구조

# Array vs Linked List

## Array (배열)

**핵심 특징:**
- **논리적 저장 순서 = 물리적 저장 순서**: 메모리 상에서 연속적으로 할당
- **Random Access 가능**: 인덱스로 O(1)에 접근
- **Cache-Friendly**: 연속된 메모리 공간으로 인한 높은 캐시 효율

**시간복잡도:**
- 접근(참조): O(1) - 인덱스를 통한 직접 접근
- 탐색: O(n) - 순차 탐색 필요
- 맨 끝에 삽입/삭제: O(1)
- 중간 삽입/삭제: O(n) - 요소들을 shift 해야 함

**장점:**
- 인덱스를 통한 빠른 접근
- 구현이 간단하고 메모리 오버헤드가 적음
- 캐시 지역성으로 인한 성능 이점

**단점:**
- 크기가 고정되어 있음 (동적 배열의 경우 resize 비용 발생)
- 중간 삽입/삭제 시 많은 요소 이동 필요
- 메모리 낭비 가능성 (미사용 공간)

## Linked List (연결 리스트)

**핵심 특징:**
- **논리적 저장 순서 ≠ 물리적 저장 순서**: 메모리 상에서 불연속적
- **Sequential Access**: 순차적 접근만 가능
- **동적 크기 조정**: 필요에 따라 노드 추가/제거 용이

**시간복잡도:**
- 접근(참조): O(n) - 처음부터 순차 탐색
- 탐색: O(n)
- 삽입/삭제: O(1) - 삽입/삭제 위치를 알고 있을 때
- 하지만 위치 탐색에 O(n) 추가 소요 → 실질적으로 O(n)

**장점:**
- 동적 크기 조정 용이
- 삽입/삭제 시 요소 이동 불필요 (포인터만 조정)

**단점:**
- 인덱스를 통한 직접 접근 불가
- 포인터를 위한 추가 메모리 필요
- 캐시 효율 낮음 (메모리 산재)

**Doubly Linked List (이중 연결 리스트):**
- 양방향 탐색 가능
- 이전 노드로의 접근이 용이
- 추가 포인터로 인한 메모리 오버헤드

## Array vs Linked List 선택 기준

**Array를 사용해야 하는 경우:**
- 데이터 크기가 고정적이거나 예측 가능
- 빠른 인덱스 접근이 빈번히 필요
- 메모리를 효율적으로 사용해야 할 때
- 캐시 성능이 중요한 경우

**Linked List를 사용해야 하는 경우:**
- 데이터 크기가 가변적
- 삽입/삭제가 빈번
- 메모리 단편화를 피하고 싶을 때

**면접 포인트:**
- Array는 탐색이 많고 크기가 고정적일 때, Linked List는 삽입/삭제가 빈번할 때 유리
- Tree의 기본 구조가 Linked List 기반
- 실제로는 대부분 Array 기반의 동적 배열(ArrayList, Vector) 사용

# Stack and Queue

## Stack (스택)

**핵심 특징:**
- **LIFO (Last In First Out)** 또는 FILO (First In Last Out)
- 가장 최근에 추가된 데이터가 먼저 제거됨

**시간복잡도:**
- Push (삽입): O(1)
- Pop (삭제): O(1)
- Top/Peek (최상단 조회): O(1)
- Search (탐색): O(n)
- n번째 원소 접근: O(n)

**주요 연산:**
- `push(value)`: 스택 최상단에 요소 추가
- `pop()`: 스택 최상단 요소 제거 및 반환
- `top()` / `peek()`: 최상단 요소 조회 (제거하지 않음)
- `isEmpty()`: 스택이 비어있는지 확인
- `size()`: 스택의 크기 반환

**활용 사례:**
- 함수 호출 스택 (Call Stack)
- 괄호 유효성 검사
- 후위 표기법 계산
- DFS (깊이 우선 탐색)
- 브라우저 뒤로 가기
- 문자열 역순 출력
- Undo/Redo 기능

## Queue (큐) + PriorityQueue

**핵심 특징:**
- **FIFO (First In First Out)**
- 가장 먼저 추가된 데이터가 먼저 제거됨

**시간복잡도:**
- Enqueue (삽입): O(1)
- Dequeue (삭제): O(1)
- Front (맨 앞 조회): O(1)
- Search (탐색): O(n)
- n번째 원소 접근: O(n)

**주요 연산:**
- `enqueue(value)` / `push(value)`: 큐의 뒤에 요소 추가
- `dequeue()` / `pop()`: 큐의 앞 요소 제거 및 반환
- `front()`: 큐의 앞 요소 조회 (제거하지 않음)
- `isEmpty()`: 큐가 비어있는지 확인
- `size()`: 큐의 크기 반환

**활용 사례:**
- BFS (너비 우선 탐색)
- 프린터 작업 대기열
- 프로세스 스케줄링
- 캐시 구현 (LRU Cache)
- 이벤트 처리

**Priority Queue (우선순위 큐):**
- 일반 큐와 달리 우선순위가 높은 요소가 먼저 나감
- 주로 Heap으로 구현
- 삽입/삭제: O(log n)
- 최대값/최소값 조회: O(1)
- 다익스트라, 프림 알고리즘 등에 활용

**Deque (Double-ended Queue):**
- 양쪽 끝에서 삽입/삭제 가능
- 앞뒤 모두 O(1)
- Stack과 Queue 기능을 모두 수행 가능

**면접 포인트:**
- Stack 두 개로 Queue 구현, Queue 두 개로 Stack 구현 방법
- 괄호 유효성 체크는 Stack으로 효율적으로 해결
- Array로 구현 시 Circular Queue 개념 필요 (효율적 공간 활용)

# Tree

트리는 **비선형(Non-linear) 자료구조**로, **계층적 관계(Hierarchical Relationship)**를 표현합니다.

## 트리 기본 용어

- **Node (노드)**: 트리를 구성하는 각각의 요소
- **Edge (간선)**: 노드와 노드를 연결하는 선

- **Root Node (루트 노드)**: 트리의 최상위 노드 (부모가 없는 노드)
- **Leaf Node (단말 노드)**: 자식이 없는 노드
- **Internal Node (내부 노드)**: 단말 노드가 아닌 노드 (루트 포함)

- **Parent Node (부모 노드)**: 특정 노드의 상위 노드
- **Child Node (자식 노드)**: 특정 노드의 하위 노드
- **Sibling (형제 노드)**: 같은 부모를 가진 노드들
- **Ancestor (조상 노드)**: 루트 노드까지의 경로에 있는 모든 노드
- **Descendant (자손 노드)**: 특정 노드의 하위에 있는 모든 노드

## 트리의 속성

- **Level (레벨)**: 트리의 각 층 (보통 루트를 0 또는 1로 시작)
- **Height (높이)**: 루트 노드부터 가장 깊은 리프 노드까지의 거리
    - 트리 전체의 높이는 루트에서 가장 먼 리프까지의 거리
- **Depth (깊이)**: 루트 노드부터 특정 노드까지의 거리
    - 각 노드마다 다름
- **Degree (차수)**: 한 노드가 가지는 자식 노드의 개수
- **Subtree (서브트리)**: 트리 내의 부분 집합

## Binary Tree (이진 트리)

**정의:** 각 노드가 최대 2개의 자식을 가지는 트리

**이진 트리의 종류:**

1. **Full Binary Tree (정 이진 트리 / 전 이진 트리)**
    - 모든 노드가 0개 또는 2개의 자식을 가짐
    - 1개의 자식만 가지는 노드가 없음
2. **Complete Binary Tree (완전 이진 트리)**
    - 마지막 레벨을 제외한 모든 레벨이 완전히 채워짐
    - 마지막 레벨은 왼쪽부터 채워짐
    - **Heap의 기본 구조**
    - 배열로 효율적 구현 가능
3. **Perfect Binary Tree (포화 이진 트리)**
    - 모든 내부 노드가 2개의 자식을 가짐
    - 모든 리프 노드가 같은 레벨에 있음
    - 노드 개수: 2^(h+1) - 1 (h는 높이)
4. **Degenerate Binary Tree (변질 이진 트리 / 편향 이진 트리)**
    - 모든 노드가 하나의 자식만 가짐
    - 사실상 연결 리스트와 동일한 구조
    - 최악의 경우 탐색 시간 O(n)
5. **Balanced Binary Tree (균형 이진 트리)**
    - 모든 노드의 왼쪽과 오른쪽 서브트리 높이 차이가 1 이하
    - AVL Tree, Red-Black Tree가 대표적
    - 탐색, 삽입, 삭제 O(log n) 보장

**배열 기반 이진 트리 인덱싱:**
- 루트 노드: index 1
- 노드 i의 부모: i/2
- 노드 i의 왼쪽 자식: 2i
- 노드 i의 오른쪽 자식: 2i+1

## BST (Binary Search Tree)

**정의:** 다음 규칙을 만족하는 이진 트리

**규칙:**
1. 노드에 저장된 키는 유일하다
2. 왼쪽 서브트리의 모든 키 < 부모 노드의 키
3. 오른쪽 서브트리의 모든 키 > 부모 노드의 키
4. 왼쪽과 오른쪽 서브트리도 이진 탐색 트리다 (재귀적 정의)

**시간복잡도:**
- 평균: O(log n) - 균형 잡힌 경우
- 최악: O(n) - 편향 트리의 경우 (Skewed Tree)

**문제점:**
- 삽입 순서에 따라 편향 트리가 될 수 있음
- 예: 1, 2, 3, 4, 5 순서로 삽입 시 완전히 우측으로 편향
- 이 경우 연결 리스트와 동일한 성능 (O(n))

**해결책:**
- **Self-Balancing Tree** 사용
- AVL Tree, Red-Black Tree 등이 자동으로 균형 유지
- 모든 연산을 O(log n)으로 보장

**면접 포인트:**
- BST는 정렬된 데이터를 효율적으로 저장/탐색하기 위한 구조
- Inorder 순회 시 정렬된 순서로 출력됨
- 편향 트리 문제 인식과 해결책 제시 중요

## AVL Tree

**정의:** 스스로 균형을 잡는 이진 탐색 트리 (Self-Balancing BST)

**핵심 특징:**
- 모든 노드에서 왼쪽과 오른쪽 서브트리의 높이 차이가 1 이하
- **Balance Factor (BF)** = 왼쪽 서브트리 높이 - 오른쪽 서브트리 높이
- BF는 항상 -1, 0, 1 중 하나

**균형 유지 방법 (Rotation):**
1. **LL (Left-Left) Case**: Right Rotation
2. **RR (Right-Right) Case**: Left Rotation
3. **LR (Left-Right) Case**: Left Rotation → Right Rotation
4. **RL (Right-Left) Case**: Right Rotation → Left Rotation

**시간복잡도:**
- 탐색, 삽입, 삭제: O(log n) 보장
- Rotation 연산: O(1)

**장점:**
- 엄격한 균형 유지로 탐색 성능 우수
- 예측 가능한 성능

**단점:**
- Red-Black Tree에 비해 삽입/삭제 시 더 많은 회전 필요
- 구현이 복잡함

**Red-Black Tree와의 비교:**
- AVL은 더 엄격하게 균형 유지 → 탐색 성능 우수
- RB Tree는 느슨한 균형 → 삽입/삭제 성능 우수
- 대부분의 라이브러리는 RB Tree 채택 (균형 유지 비용 < 탐색 효율)

## B-Tree & B+Tree

**B-Tree:**
- 하나의 노드에 여러 개의 데이터를 저장할 수 있는 트리
- 균형 트리 (Balanced Tree)
- **데이터베이스 인덱스**의 기본 구조
- 디스크 I/O 최소화에 최적화

**특징:**
- 한 노드에 최대 M개의 자식 (M-way tree)
- 모든 리프 노드는 같은 레벨
- 노드는 최소 M/2개의 자식 유지 (루트 제외)

**B+Tree:**
- B-Tree의 변형
- **리프 노드에만 실제 데이터 저장**
- 내부 노드는 인덱스 역할만
- 리프 노드들이 연결 리스트로 연결됨

**B-Tree vs B+Tree:**
- B+Tree는 범위 탐색에 유리 (리프 노드 연결)
- B+Tree는 내부 노드 크기가 작아 더 많은 키 저장 가능
- 대부분의 DBMS는 B+Tree 사용 (MySQL, PostgreSQL 등)

**면접 포인트:**
- 데이터베이스가 B-Tree 계열을 사용하는 이유: 디스크 I/O 최소화
- 한 번의 디스크 읽기로 여러 키 비교 가능
- 트리 높이가 낮아 탐색 횟수 감소

## Binary Heap

**정의:** Complete Binary Tree 기반의 자료구조

**종류:**
1. **Max Heap (최대 힙)**
- 부모 노드 ≥ 자식 노드
- 루트 노드가 최댓값

1. **Min Heap (최소 힙)**
    - 부모 노드 ≤ 자식 노드
    - 루트 노드가 최솟값

**시간복잡도:**
- 최댓값/최솟값 조회: O(1)
- 삽입: O(log n) - heapify up
- 삭제 (루트): O(log n) - heapify down
- 임의의 원소 검색: O(n)

**힙 구현:**
- 배열로 구현 (Complete Binary Tree 특성 활용)
- 인덱스 1부터 시작
- 부모: i/2
- 왼쪽 자식: 2i
- 오른쪽 자식: 2i+1

**Heapify 과정:**

1. **삽입 시 (Heapify Up)**
    - 배열의 마지막에 요소 추가
    - 부모와 비교하며 위로 이동
    - 힙 속성이 만족될 때까지 반복
2. **삭제 시 (Heapify Down)**
    - 루트 노드 제거
    - 마지막 노드를 루트로 이동
    - 자식들과 비교하며 아래로 이동
    - 힙 속성이 만족될 때까지 반복

**활용:**
- Priority Queue 구현
- Heap Sort
- 다익스트라 알고리즘
- 프림 알고리즘
- 중앙값 찾기 (두 개의 힙 활용)

**BST vs Heap:**
- BST: 정렬된 데이터 유지 (inorder 순회 시 정렬)
- Heap: 최댓값/최솟값에 빠른 접근 (부분적 정렬)
- Heap은 BST보다 느슨한 정렬 상태 유지

**면접 포인트:**
- Complete Binary Tree여야 하는 이유: 배열로 효율적 구현
- Heap은 우선순위가 중요한 상황에서 사용
- K번째로 큰/작은 원소 찾기: Min/Max Heap 활용

# Red Black Tree

**정의:** 스스로 균형을 잡는 이진 탐색 트리

**목적:** BST의 편향 트리 문제 해결 → 모든 연산을 O(log n)으로 보장

## Red-Black Tree의 규칙

1. 모든 노드는 Red 또는 Black
2. **루트 노드는 Black**
3. **모든 리프 노드(NIL)는 Black**
4. **Red 노드의 자식은 모두 Black** (Red 노드 연속 불가)
5. **모든 경로의 Black 노드 개수는 동일** (Black-Height)

**Black-Height:**
- 특정 노드에서 리프까지의 경로상 Black 노드의 개수
- 자기 자신은 제외

## Red-Black Tree의 특징

1. BST의 특성을 모두 만족
2. **최소 경로 ≥ 최대 경로 / 2**
    - Red 노드가 연속될 수 없으므로
    - 완벽한 균형은 아니지만 느슨한 균형 유지
3. NIL 노드를 리프로 간주
4. AVL보다 삽입/삭제 시 회전 횟수 적음

**시간복잡도:**
- 탐색, 삽입, 삭제: O(log n) 보장

### 삽입 (Insertion)

**기본 전략:**
1. 일반 BST처럼 삽입
2. **새 노드를 Red로 지정** (Black-Height 변경 최소화)
3. 규칙 위반 시 조정

**삽입 후 조정:**
- **Uncle이 Red**: Recoloring
- **Uncle이 Black**: Rotation + Recoloring
- LL, RR, LR, RL 케이스에 따라 회전

### 삭제 (Deletion)

**기본 전략:**
1. 일반 BST처럼 삭제
2. 삭제된 노드가 Black이면 Black-Height 위반
3. 위반 시 조정

**삭제 후 조정:**
- Sibling 노드의 색과 자식 색에 따라 케이스 분류
- Rotation과 Recoloring으로 균형 복구

### AVL vs Red-Black Tree

| 특성 | AVL Tree | Red-Black Tree |
| --- | --- | --- |
| 균형 조건 | 엄격 (높이 차이 ≤ 1) | 느슨 (최대/최소 경로 비율 ≤ 2) |
| 탐색 성능 | 더 빠름 | 약간 느림 |
| 삽입/삭제 | 더 많은 회전 | 적은 회전 |
| 사용 예 | 탐색 집약적 | 삽입/삭제 빈번 |
| 구현 | Java TreeMap | C++ map/set |

**면접 포인트:**
- Red 노드로 삽입하는 이유: Black-Height를 변경하지 않기 위함
- Red-Black Tree는 완벽한 균형이 아닌 “충분히 균형 잡힌” 트리
- 실무에서는 Red-Black Tree를 더 많이 사용 (구현 복잡도 대비 성능)

# Hash Table

**정의:** Key를 해시 함수를 통해 해시값으로 변환하고, 이를 인덱스로 사용하여 데이터를 저장/검색하는 자료구조

**핵심 개념:**
- **Hash (해시)**: 다양한 길이의 데이터를 고정된 길이의 데이터로 매핑한 값
- **Hashing (해싱)**: 임의의 데이터를 해시로 변환하는 과정
- **Hash Function (해시 함수)**: 입력값을 고정된 길이의 해시로 변환하는 함수

**시간복잡도:**
- **평균:** O(1) - 이상적인 경우
- **최악:** O(n) - 모든 키가 같은 인덱스에 충돌

## 좋은 Hash Function의 조건

1. **키 전체를 참조**하여 해시값 생성
2. **충돌을 최소화**
3. **계산이 빠름**
4. **해시값이 고르게 분포**

**중요한 인사이트:**
- 1:1 매핑은 사실상 불가능하며 비효율적 (메모리 낭비)
- 충돌을 최소화하고, 발생한 충돌을 효과적으로 해결하는 것이 핵심

## Collision Resolution (충돌 해결)

### 1. Open Addressing (개방 주소법)

**정의:** 충돌 발생 시 다른 빈 버킷을 찾아 저장

**방법:**

1. **Linear Probing (선형 탐사)**
    - 충돌 시 다음 버킷을 순차적으로 탐색
    - h(k) → h(k)+1 → h(k)+2 → …
    - 단점: Primary Clustering (연속된 데이터 뭉침)
2. **Quadratic Probing (이차 탐사)**
    - 충돌 시 제곱수만큼 이동하여 탐색
    - h(k) → h(k)+1² → h(k)+2² → h(k)+3² → …
    - Linear보다 clustering 완화
3. **Double Hashing (이중 해싱)**
    - 두 번째 해시 함수를 사용하여 탐사 간격 결정
    - 가장 효과적이지만 계산 비용 증가

**특징:**
- 연속된 메모리 공간 사용 → **캐시 효율 높음**
- 데이터가 많아질수록 (Load Factor ↑) 성능 급격히 저하
- 테이블 크기를 늘리기 어려움

**Load Factor (적재율):**
- (저장된 데이터 수) / (테이블 크기)
- 0.7~0.8 이상 시 성능 저하

### 2. Separate Chaining (분리 연결법)

**정의:** 각 버킷을 연결 리스트나 트리로 구성하여 충돌된 데이터를 체인으로 연결

**구현 방법:**

1. **Linked List 사용**
    - 각 버킷을 연결 리스트로 구성
    - 삽입/삭제 간단
    - 작은 데이터에서는 연결 리스트 오버헤드 부담
2. **Tree 사용 (Red-Black Tree)**
    - Java 8부터 HashMap이 채택
    - 한 버킷의 데이터가 8개 이상 시 Tree로 전환
    - 6개 이하 시 다시 List로 전환 (Switching 비용 고려)
    - 최악의 경우 O(log n) 보장

**특징:**
- Open Addressing보다 충돌에 강함
- **테이블 확장을 늦출 수 있음**
- 캐시 효율은 낮음
- Java HashMap, C++ unordered_map 등에서 사용

## Open Addressing vs Separate Chaining

Open Addressing의 경우 해시 버킷을 채운 밀도가 높아질수록 Worst Case 발생 빈도가 높아지기에, 일반적으로 Separate Chaining보다 느리다.

| 특성 | Open Addressing | Separate Chaining |
| --- | --- | --- |
| 메모리 | 테이블 내부만 사용 | 추가 포인터 필요 |
| 캐시 효율 | 높음 | 낮음 |
| Load Factor | 0.7~0.8 권장 | 1.0 이상도 가능 |
| 구현 | 간단 | 복잡 |
| 성능 | 데이터 적을 때 유리 | 데이터 많을 때 유리 |

**적절한 데이터 개수:**
- 데이터가 적고 메모리 효율이 중요: Open Addressing
- 데이터가 많고 안정적 성능 필요: Separate Chaining

## 보조 해시 함수 (Supplement Hash Function)

**목적:** 키의 해시값을 변형하여 충돌 가능성을 줄임

**사용:**
- Separate Chaining과 함께 사용
- 최악의 경우를 피하기 위한 추가 방어선

## Hash Table Resize

**필요성:**
- 충돌 증가로 인한 성능 저하 방지
- 메모리 효율과 성능의 균형

**Resize 시점:**
- Load Factor가 임계값(보통 0.75) 초과 시
- Java HashMap의 경우 0.75 초과 시 크기를 2배로 확장

**Resize 과정:**
1. 새로운 크기의 테이블 생성 (보통 2배)
2. 모든 데이터를 새 테이블에 재해싱 (Rehashing)
3. 시간복잡도: O(n)

**주의사항:**
- Resize는 비용이 큰 작업
- 예상 데이터 크기를 알면 초기 크기 설정으로 Resize 최소화

**면접 포인트:**
- 해시 테이블의 평균 O(1)은 충돌이 적을 때만 성립
- Load Factor 관리가 성능의 핵심
- Java HashMap은 Separate Chaining + Tree 변환으로 최악의 경우 대비
- 좋은 해시 함수는 충돌을 최소화하고 해시값을 고르게 분포시킴

# Map and Set

## Map

**정의:** Key-Value 쌍으로 데이터를 저장하는 자료구조 (중복 키 불가)

### Ordered Map (일반적인 `map`, `TreeMap`)

**구현:**

- 대부분 **Red-Black Tree** 기반 (C++ `map`, Java `TreeMap`)
- Balanced BST로 인해 **키가 정렬된 순서**로 저장됨

**시간복잡도:**

- 탐색: `O(log n)`
- 삽입: `O(log n)`
- 삭제: `O(log n)`

**주요 연산:**

- `insert({key, value})`: 키-값 쌍 삽입
- `erase(key)`: 키에 해당하는 항목 삭제
- `find(key)`: 키 탐색
- `[key]` / `at(key)`: 키로 값 접근 (언어마다 다름, C++의 `operator[]`는 없으면 생성)
- `size()`: 크기 반환
- `clear()`: 전체 삭제
- (`lower_bound`, `upper_bound`: 범위 검색용)

**특징:**

- **키가 자동으로 정렬됨** (오름차순이 기본)
- `lower_bound`, `upper_bound` 연산 가능
- 키 범위를 기준으로 하는 **범위 기반 탐색**이 용이
- 순회 시 항상 **정렬된 순서로 순회**됨

### Unordered Map (Hash Map: `unordered_map`, `HashMap`)

**정의:**

키-값 쌍을 저장하지만, **키의 순서를 전혀 보장하지 않는** Map.

정렬 대신 **빠른 평균 접근 속도**에 집중한 자료구조.

**구현:**

- **Hash Table 기반**
    - 내부적으로 “버킷 배열 + 해시 함수” 사용
    - 같은 버킷에 여러 키가 모이면 체이닝(연결 리스트 등)으로 충돌 처리
- C++: `unordered_map`
    
    Java: `HashMap` (기본 Map 구현)
    

**시간복잡도:**

- 평균:
    - 탐색: `O(1)`
    - 삽입: `O(1)`
    - 삭제: `O(1)`
- 최악:
    - 해시가 한쪽으로 몰릴 경우 `O(n)` 까지 갈 수 있음
- 버킷이 꽉 차면 **rehash**가 일어나며, 이때는 일시적으로 많은 시간이 걸리지만 **분할 상환 평균은 여전히 O(1)**로 보는 것이 일반적

**주요 연산:**

- `insert({key, value})` / `put(key, value)`: 키-값 삽입
- `erase(key)` / `remove(key)`: 키 삭제
- `find(key)` / `get(key)`: 값 탐색
- `[key]` / `operator[]` / `getOrDefault`: 키로 값 접근
- `count(key)` / `containsKey(key)`: 키 존재 여부 확인
- `size()`, `empty()`, `clear()`: 크기 및 전체 삭제
- (C++의 경우 `bucket_count`, `load_factor` 등 해시 관련 부가 연산도 있음)

**특징:**

- **정렬되지 않음**
    - 순회 시 순서 보장 X (컴파일러/환경에 따라 순서가 달라질 수 있음)
- 정렬이 필요 없는 대신, **단일 키에 대한 접근이 매우 빠름**
- 키에 대한 **범위 쿼리(예: L ≤ key ≤ R)**를 직접적으로 지원하지 않음
    
    → 필요하다면 키를 따로 모아 정렬해야 함
    
- 해시 함수와 `==` 정의가 중요
    - 커스텀 타입을 키로 쓰려면 직접 해시 함수/Comparator를 구현해야 할 수 있음

**활용:**

- **가장 많이 쓰이는 용도**
    - 빈도 수 세기 (frequency counter)
    - 방문 여부 체크(visited)
    - 캐시, 메모이제이션 테이블
- “정렬은 필요 없고, 특정 키에 대해 빨리 찾고/넣고/지우는 게 목적”일 때 선택

## Set

**정의:** 중복을 허용하지 않는 데이터의 집합 (Key만 존재하는 Map 느낌)

### Ordered Set (`set`, `TreeSet`)

**구현:**

- Map과 동일하게 **Red-Black Tree 기반 Balanced BST**
- **Key만 저장** (Value는 없음, 사실상 `map<Key, bool>` 같은 느낌)

**시간복잡도:**

- 탐색: `O(log n)`
- 삽입: `O(log n)`
- 삭제: `O(log n)`

**주요 연산:**

- `insert(value)`: 값 삽입
- `erase(value)`: 값 삭제
- `find(value)`: 값 탐색
- `count(value)`: 값의 존재 여부 (0 또는 1)
- `size()`: 크기 반환
- (`lower_bound`, `upper_bound`: 값 기준 범위 탐색)

**활용:**

- **중복 제거**
- 값의 **존재 여부 확인**
- 항상 **정렬된 상태 유지** → 정렬된 순서로 순회 가능
- 구간 쿼리, “이 값 이상/이하 중 가장 가까운 값” 찾기 등에 유용

---

### Unordered Set (`unordered_set`, `HashSet`)

**정의:**

중복 없는 집합이라는 점은 그대로지만,

**정렬 대신 해시 기반으로 빠른 평균 접근**에 집중한 Set.

**구현:**

- **Hash Table 기반**
    - 내부적으로 `unordered_map<T, bool>` 같은 느낌
- C++: `unordered_set`
    
    Java: `HashSet` (내부적으로는 거의 `HashMap` 사용)
    

**시간복잡도:**

- 평균:
    - 탐색: `O(1)`
    - 삽입: `O(1)`
    - 삭제: `O(1)`
- 최악: `O(n)` (해시 충돌이 심할 경우)

**주요 연산:**

- `insert(value)`: 값 삽입
- `erase(value)`: 값 삭제
- `find(value)` / `count(value)` / `contains(value)`: 존재 여부 확인
- `size()`, `empty()`, `clear()`: 크기 및 초기화
- (마찬가지로 `bucket_count`, `load_factor` 등 해시 관련 연산 존재)

**특징:**

- **정렬되지 않음** → 순회 순서 보장 X
- 같은 원소를 여러 번 넣어도 **중복 저장되지 않음**
- 해시 기반이라, 단순 **“이 값 있냐 없냐”**를 체크할 때 가장 빠른 구현 중 하나
- 값의 대소 관계, 범위 조건과는 어울리지 않음
    
    → 그런 경우엔 `set`(Ordered Set)을 쓰는 게 맞음
    

**활용:**

- 어떤 값의 **존재 여부만 빠르게 확인**하면 될 때
    - 예: 방문 체크, 이미 처리한 값 체크, 중복 제거용 필터
- 정렬이 필요 없는 대규모 집합

---

### Multiset / Multimap

**정의:**

- **Multiset**: 같은 값을 여러 번 넣을 수 있는 Set
- **Multimap**: 같은 키에 여러 Value를 매달 수 있는 Map

**특징:**

- 중복 허용 (같은 키/값을 여러 개 저장 가능)
- 구현은 보통 Ordered 버전 기준 (Balanced BST)
- C++에는 `multiset`, `multimap`뿐 아니라
    
    `unordered_multiset`, `unordered_multimap`도 존재 (해시 기반, 중복 허용)
    

---

### 면접 포인트 정리

- **구조**
    - C++ `map` / `set` → Red-Black Tree (Balanced BST, Ordered)
    - C++ `unordered_map` / `unordered_set` → Hash Table (Unordered)
    - Java `TreeMap` / `TreeSet` → Tree 기반 (Ordered)
        
        Java `HashMap` / `HashSet` → Hash Table 기반 (Unordered)
        
- **언제 무엇을 쓰는가**
    - **Ordered (map/set)**:
        - 정렬이 필요하거나,
        - `lower_bound`, `upper_bound`, 범위 쿼리 등 “대소 비교 기반 연산”이 중요할 때
    - **Unordered (unordered_map/set, HashMap/HashSet)**:
        - 정렬 필요 X
        - “이 키(또는 값)가 있냐/없냐”를 **최대한 빠르게** 처리하고 싶을 때

# Graph

**정의:** 정점(Vertex)과 간선(Edge)의 집합

**특징:**
- 사이클 허용 (Tree는 사이클 없는 그래프)
- 방향성 유무에 따라 분류
- 네트워크, 관계, 경로 등을 표현

## Graph 용어

### Undirected Graph vs Directed Graph

**Undirected Graph (무방향 그래프):**
- 간선에 방향이 없음
- (u, v) = (v, u)
- 예: 친구 관계, 도로망

**Directed Graph (방향 그래프, Digraph):**
- 간선에 방향이 있음
- (u, v) ≠ (v, u)
- 예: 웹 링크, 작업 순서

### Degree

**Undirected Graph:**
- Degree: 한 정점에 연결된 간선의 개수

**Directed Graph:**
- Indegree: 정점으로 들어오는 간선의 개수
- Outdegree: 정점에서 나가는 간선의 개수

### 기타 용어

- **Weighted Graph (가중치 그래프)**: 간선에 비용/거리 정보가 있는 그래프
- **Unweighted Graph**: 모든 간선의 가중치가 동일
- **Connected Graph (연결 그래프)**: 모든 정점 쌍이 경로로 연결
- **Disconnected Graph**: 연결되지 않은 정점 쌍이 존재
- **Complete Graph (완전 그래프)**: 모든 정점이 서로 연결
- **Cyclic Graph**: 사이클이 존재
- **Acyclic Graph (DAG)**: 사이클이 없음 (방향 그래프)
- **Subgraph (부분 그래프)**: 그래프의 일부 정점과 간선

## Graph 구현

### 1. Adjacency Matrix (인접 행렬)

**구현:** V × V 크기의 2차원 배열

**특징:**
- `matrix[i][j] = 1` (또는 가중치): i와 j 사이에 간선 존재
- `matrix[i][j] = 0`: 간선 없음

**시간복잡도:**
- 간선 존재 확인: O(1)
- 특정 정점의 모든 인접 정점 찾기: O(V)
- 모든 간선 찾기: O(V²)

**공간복잡도:** O(V²)

**장점:**
- 두 정점 간 연결 확인이 빠름
- 구현이 간단

**단점:**
- 간선 수에 관계없이 O(V²) 메모리 필요
- Sparse Graph(간선이 적은 그래프)에서 메모리 낭비
- 모든 간선 탐색에 O(V²) 소요

**적합한 경우:**
- Dense Graph (간선이 많은 그래프)
- 간선 존재 여부 확인이 빈번
- 정점 수가 적음

### 2. Adjacency List (인접 리스트)

**구현:** 각 정점마다 연결된 정점들의 리스트 유지

**자료구조 선택:**
- **Vector (동적 배열)**
- 메모리 효율적
- 캐시 친화적
- 대부분의 경우 더 빠름
- **Linked List**
- 삽입/삭제가 빈번할 때
- 메모리 단편화 방지

**시간복잡도:**
- 간선 존재 확인: O(degree) - 해당 정점의 차수만큼
- 특정 정점의 모든 인접 정점 찾기: O(degree)
- 모든 간선 찾기: O(V + E)

**공간복잡도:** O(V + E)

**장점:**
- 실제 간선 수만큼만 메모리 사용
- 특정 정점의 인접 정점 빠르게 탐색
- Sparse Graph에 효율적

**단점:**
- 두 정점 간 연결 확인이 느림
- 구현이 약간 복잡

**적합한 경우:**
- Sparse Graph (간선이 적은 그래프)
- 대부분의 실제 상황
- 그래프 탐색 알고리즘 (DFS, BFS)

### Adjacency Matrix vs Adjacency List

| 특성 | Adjacency Matrix | Adjacency List |
| --- | --- | --- |
| 공간복잡도 | O(V²) | O(V + E) |
| 간선 확인 | O(1) | O(degree) |
| 전체 간선 탐색 | O(V²) | O(V + E) |
| 인접 정점 탐색 | O(V) | O(degree) |
| 적합한 그래프 | Dense | Sparse |

## Graph 탐색

### 1. DFS (Depth First Search, 깊이 우선 탐색)

**전략:** 한 방향으로 끝까지 탐색 후 backtrack

**구현:**
- Stack 또는 재귀
- 방문 체크 배열 필요

**시간복잡도:**
- 인접 행렬: O(V²)
- 인접 리스트: O(V + E)

**특징:**
- 경로 탐색에 유리
- 모든 노드 방문 보장
- 사이클 탐지

**활용:**
- 미로 찾기
- 위상 정렬
- 사이클 탐지
- 경로 존재 여부 확인
- 연결 요소 찾기

### 2. BFS (Breadth First Search, 너비 우선 탐색)

**전략:** 가까운 정점부터 레벨별로 탐색

**구현:**
- Queue 사용
- 방문 체크 배열 필요

**시간복잡도:**
- 인접 행렬: O(V²)
- 인접 리스트: O(V + E)

**특징:**
- **최단 경로 보장** (무가중치 또는 동일 가중치)
- 레벨별 탐색
- 모든 노드 방문 보장

**활용:**
- 최단 경로 찾기
- 레벨 순회
- 네트워크 전파
- Connected Component 개수

**DFS vs BFS 선택:**
- 최단 경로 필요: BFS
- 모든 경로 탐색: DFS
- 메모리 제약: DFS (재귀 스택 vs 큐)
- 그래프가 깊음: BFS
- 그래프가 넓음: DFS

## Minimum Spanning Tree (MST)

**정의:** 그래프의 모든 정점을 사이클 없이 연결하는 부분 그래프 중 가중치 합이 최소인 트리

**특징:**
- 간선 개수: V - 1 (V는 정점 수)
- 사이클 없음
- 모든 정점 연결

**활용:**
- 네트워크 설계 (최소 비용으로 모든 노드 연결)
- 도로망 설계
- 전기 회로 설계

### 1. Kruskal Algorithm

**전략:** 간선을 가중치 순으로 정렬하여 사이클이 생기지 않으면 선택

**알고리즘:**
1. 모든 간선을 가중치 오름차순 정렬
2. 가중치가 가장 작은 간선부터 선택
3. 사이클을 만들지 않으면 MST에 추가
4. V-1개의 간선 선택 시 종료

**사이클 판단:**
- Union-Find (Disjoint Set) 자료구조 사용
- 두 정점이 같은 집합이면 사이클 발생

**시간복잡도:** O(E log E)
- 정렬: O(E log E)
- Union-Find: O(E + V log V) ≈ O(E)
- 전체: O(E log E) (정렬이 지배적)

**특징:**
- 간선 중심 알고리즘
- Sparse Graph에 효율적
- Union-Find 구현 필요

### 2. Prim Algorithm

**전략:** 시작 정점에서 출발하여 MST를 확장

**알고리즘:**
1. 임의의 시작 정점 선택
2. MST에 속한 정점에서 나가는 간선 중 최소 가중치 선택
3. 선택한 간선으로 연결된 정점을 MST에 추가
4. 모든 정점이 MST에 포함될 때까지 반복

**구현:**
- Priority Queue (Min Heap) 사용
- 방문 체크 배열 필요

**시간복잡도:** O(E log V)
- 우선순위 큐 사용 시
- 인접 리스트 기반

**특징:**
- 정점 중심 알고리즘
- Dense Graph에 효율적
- 연결된 그래프에만 적용 가능

**Kruskal vs Prim:**

| 특성 | Kruskal | Prim |
| --- | --- | --- |
| 전략 | 간선 선택 | 정점 확장 |
| 자료구조 | Union-Find | Priority Queue |
| 시간복잡도 | O(E log E) | O(E log V) |
| 적합한 그래프 | Sparse | Dense |
| 구현 난이도 | 보통 | 쉬움 |

**면접 포인트:**
- MST는 유일하지 않을 수 있음 (같은 가중치의 간선 존재 시)
- 두 알고리즘 모두 Greedy 방식
- Kruskal은 전체 간선 고려, Prim은 현재 트리에서 확장
- 실제로는 그래프 밀도에 따라 선택

# 시간복잡도 총정리

자료구조 선택 시 각 연산의 시간복잡도를 고려해야 합니다. 다음은 주요 자료구조의 평균/최악 시간복잡도입니다.

## Array (배열, Vector)

- 참조(접근): **O(1)**
- 탐색: O(n)
- 맨 끝 삽입/삭제: O(1) (amortized)
- 중간 삽입/삭제: O(n)

**특징:**
- Random Access 가능
- Vector의 push_back()은 평균 O(1) (동적 확장 시 O(n)이지만 amortized)

## Linked List (연결 리스트)

- 참조(접근): O(n)
- 탐색: O(n)
- 삽입/삭제: **O(1)** (위치를 알고 있을 때)
    - 실제로는 위치 찾기에 O(n) 추가

**특징:**
- Sequential Access만 가능
- 포인터 조작만으로 삽입/삭제

## Stack (스택)

- Push (삽입): **O(1)**
- Pop (삭제): **O(1)**
- Top (최상단 조회): **O(1)**
- 탐색: O(n)
- n번째 원소 접근: O(n)

## Queue (큐)

- Enqueue (삽입): **O(1)**
- Dequeue (삭제): **O(1)**
- Front (맨 앞 조회): **O(1)**
- 탐색: O(n)
- n번째 원소 접근: O(n)

## Map (Red-Black Tree 기반)

- 참조: **O(log n)**
- 탐색: **O(log n)**
- 삽입: **O(log n)**
- 삭제: **O(log n)**

**특징:**
- 정렬된 상태 유지
- 균형 이진 탐색 트리 기반

## Set (Red-Black Tree 기반)

- 참조: **O(log n)**
- 탐색: **O(log n)**
- 삽입: **O(log n)**
- 삭제: **O(log n)**

**특징:**
- 중복 불가
- 정렬된 상태 유지

## Hash Table (해시 테이블)

- 평균 시간복잡도:
    - 참조: **O(1)**
    - 탐색: **O(1)**
    - 삽입: **O(1)**
    - 삭제: **O(1)**
- 최악 시간복잡도:
    - 참조: O(n)
    - 탐색: O(n)
    - 삽입: O(n)
    - 삭제: O(n)

**특징:**
- 충돌이 많으면 성능 저하
- Load Factor 관리 중요
- Separate Chaining + Tree 변환 시 최악의 경우 O(log n)

## Heap (힙)

- 최댓값/최솟값 조회: **O(1)**
- 삽입: **O(log n)**
- 삭제 (루트): **O(log n)**
- 임의 원소 탐색: O(n)

**특징:**
- Complete Binary Tree
- 우선순위 큐 구현

## Binary Search Tree (BST)

- 평균:
    - 탐색: **O(log n)**
    - 삽입: **O(log n)**
    - 삭제: **O(log n)**
- 최악 (편향 트리):
    - 탐색: O(n)
    - 삽입: O(n)
    - 삭제: O(n)

## Balanced BST (AVL, Red-Black Tree)

- 탐색: **O(log n)** 보장
- 삽입: **O(log n)** 보장
- 삭제: **O(log n)** 보장

**특징:**
- 자동으로 균형 유지
- 모든 연산 O(log n) 보장

## Graph

- **인접 행렬:**
    - 간선 확인: **O(1)**
    - 전체 탐색: O(V²)
    - 공간: O(V²)
- **인접 리스트:**
    - 간선 확인: O(degree)
    - 전체 탐색 (DFS/BFS): **O(V + E)**
    - 공간: O(V + E)

## 자료구조 선택 가이드

**빠른 접근이 필요:** Array, Hash Table
**빈번한 삽입/삭제:** Linked List, Hash Table
**정렬된 데이터:** BST, Map, Set
**최댓값/최솟값 반복 접근:** Heap
**LIFO 필요:** Stack
**FIFO 필요:** Queue
**우선순위 관리:** Priority Queue (Heap)
**관계 표현:** Graph

## 추가 학습 주제

### 더 공부하면 좋은 자료구조

- **Trie**: 문자열 검색, 자동완성
- **Segment Tree**: 구간 쿼리
- **Fenwick Tree (BIT)**: 누적 합 쿼리
- **Disjoint Set (Union-Find)**: 집합 연산
- **Skip List**: 확률적 균형 트리
- **Bloom Filter**: 확률적 멤버십 테스트

### 심화 개념

- **Amortized Analysis**: 분할 상환 분석
- **Cache-Oblivious Algorithm**: 캐시 독립적 알고리즘
- **Persistent Data Structure**: 영속 자료구조
- **Lock-Free Data Structure**: 잠금 없는 자료구조
